# Codex 执行规范

## 中文源码编码规范

1. 避免使用 PowerShell 读写管线修改包含中文的源码文件，尤其不要用 `Get-Content` 读取后再通过脚本整文件写回。
2. 处理包含中文的源码时，优先使用 `apply_patch` 做局部补丁，尽量避免整文件重写。
3. 如果必须整文件重写，必须明确按 UTF-8 读取和写入，不能依赖 Windows 默认编码。
4. 修改完成后，必须对相关文件运行乱码扫描，检查 `鍘`、`褰`、`绛`、`鎸`、`寰`、`宸`、`鏂`、`棰`、`瀵`、`鐘` 等常见 mojibake 片段。
5. 一旦发现乱码，先确认是否为本次编辑引入，再用明确 UTF-8 的方式修复；不要继续基于乱码文本做业务修改。

## 回答格式

1. 所有英文组件名、英文变量名、英文技术名词、高级语法，必须紧跟中文括号注释。
2. 示例：`SpaceVisualStage.vue`（星球舞台组件）、`GSAP`（动画库）、`defineExpose`（Vue 暴露组件方法 API）、`syncVisualLayers`（视觉层同步函数）。
3. 专业术语每次出现都必须标注中文括号注释，避免只在首次出现时解释。
4. 代码里的关键注释必须清晰说明“为什么这样做”，避免只描述“做了什么”。
5. 回答保持简洁、专业、不啰嗦，优先给结论，再补充必要原因。

## 文件说明

1. 涉及文件时，使用明确文件名并加中文说明。
2. 示例：`mock-interview-space.vue`（模拟面试星际页面主文件）。

## 命令说明

1. 涉及命令时，命令保持原样，后面补充中文说明。
2. 示例：`npm run build`（执行生产构建验证）。

## 结果总结

1. 代码执行完成后，优先按“完成内容 / 验证结果 / 后续建议”三段输出。
2. 如果发现风险，直接标明“风险”，并说明影响和建议处理方式。

## 全局代码质量硬约束

1. 所有开发都必须遵守模块化、组件化、高内聚、低耦合、单一职责。
2. 代码必须简洁、高效、可读，优先解决当前真实需求，不得因为强调“设计”而过度设计。
3. 禁止编写兜底代码、冗余代码、猜测性代码、无明确需求的兼容代码。
4. 禁止为了“保证代码看起来绝对能运行”而堆叠不必要的判断、分支、默认值或适配逻辑。
5. 不清楚接口响应字段时，不得猜测多个字段名，例如同时兼容 `id`（标识字段）、`_id`（数据库标识字段）、`uid`（用户标识字段）、`key`（键名）、`uuid`（唯一标识字段）等；必须先询问用户、查看真实接口或读取已有类型定义。
6. 不清楚业务边界、状态归属、是否需要兼容历史数据时，必须先询问或查证，不得自行猜测实现。
7. 禁止新增暂时用不到的抽象层、管理器、工厂、适配器、注册表、配置系统或未来扩展点。
8. 优先复用项目已有模式；只有当前真实需求需要时，才新增组件、组合函数、类型或工具函数。
9. 注释只解释非显而易见的原因，不写描述代码表面行为的废话注释。
10. 完成前必须自查：是否存在猜字段、猜需求、重复分支、无证据兜底、过度抽象；如存在，应删除或先向用户确认。

## Debug 纪律

1. 遇到问题时，先定位根因，再决定改动点；禁止靠猜测连续叠补丁。
2. 每次修复前必须明确记录：现象、初步根因、验证办法、预期影响面。
3. 同一问题连续两次验证失败后，第三次开始前必须停止局部修补，转为做结构审查、依赖关系复盘和根因分析。
4. 未完成根因确认前，不得继续添加补偿逻辑、兜底分支、临时样式或第二套实现。
5. 如果根因证据不足，必须先补充日志、断点、代码路径检查或最小复现，再继续修改。

## 开发记录机制

1. 非琐碎任务开始前，必须先读取 `CODEX.md`（开发规范）和 `docs/开发记录.md`（开发记录）。
2. `docs/开发记录.md`（开发记录）必须持续保存以下内容：架构决策、技术选择、验证方式、已知失败、失败原因、修正经验、当前边界约束。
3. 任务执行过程中，只要产生新的关键决策、踩坑、失败修复、验证结论，就必须同步更新 `docs/开发记录.md`（开发记录）。
4. 任务完成前必须检查开发记录是否与当前代码一致；如果记录已过时、遗漏关键变更或与实现冲突，必须先更新记录，再结束任务。
5. 后续任务默认以 `docs/开发记录.md`（开发记录）作为连续上下文入口，避免重复犯错和上下文丢失。

## Git 提交规范执行规则

1. 只要涉及 `git commit`（Git 提交），提交前必须先阅读并遵守 `docs/Git提交规范.md`（Git 提交规范文档）。
2. 提交信息必须以 `feat`（新增功能）、`fix`（问题修复）、`docs`（文档更新）、`refactor`（重构）、`style`（样式调整）、`chore`（杂项维护）、`perf`（性能优化）、`test`（测试）之一开头。
3. 冒号后面的提交标题必须使用中文，且必须直接描述“本次主要改了什么”，禁止使用笼统标题，例如“改一下页面”“修复 bug”“更新文档”。
4. 当一次提交涉及多个页面、多个模块或一个完整阶段时，必须优先使用“标题 + 正文”的多行提交格式，正文按“本次主要改动点”列出，不写无结构长句。
5. 提交前必须根据暂存区（staging area，Git 暂存区）实际内容整理提交信息，不得脱离暂存区改动猜测提交标题或正文。
6. 如果当前任务同时包含功能、样式、文档、修复等多类改动，必须先判断是否应拆分提交；不能拆分时，提交标题必须聚焦本次主目标，正文再补充其余关键点。

## mock-interview-space 开发边界

1. 涉及 `mock-interview-space.vue`（模拟面试星际页面主文件）后续开发时，必须先阅读并遵守 `docs/18.mock-interview-space后续开发边界规范.md`（模拟面试星际页后续开发边界规范）。
2. `mock-interview-space.vue`（模拟面试星际页面主文件）只做页面装配和场景编排，不再承接新的业务 UI（用户界面）、业务状态、场景样式或动画细节。
3. 新增需求必须优先落到对应 `Space*Scene.vue`（场景组件）、`SpaceContentPanel.vue`（内容面板组件）、`useMockInterviewSpace*.ts`（模拟面试星际页组合函数）或 `mockInterviewSpaceScenes.ts`（模拟面试星际页场景配置文件）。
4. 用户提到“宇宙页面”“星际页面”“轨道页面”“这个页面”“截图里的页面”时，默认指 `/showcase/mock-interview-space`（宇宙模拟面试页），不是 `src/views/workbench`（旧工作台页面目录）。
5. 开发 `/showcase/mock-interview-space`（宇宙模拟面试页）时，默认禁止修改 `src/views/workbench/*`（旧工作台页面文件）、`src/components/workbench/*`（旧工作台组件文件）和工作台专项页相关文件，除非用户明确要求“改工作台页面”或点名具体旧页面文件。
6. 如果任务名称和旧页面名称相同，例如“专项训练页面”“资料库”“报告页”，必须先判断它是否处在宇宙页面场景内；只要上下文来自 `/showcase/mock-interview-space`（宇宙模拟面试页），就应改 `src/components/showcase/mock-interview-space`（宇宙页面组件目录）下的场景组件。
