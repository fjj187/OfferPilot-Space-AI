# 16.mock-interview-space 模拟面试场景真实 session 接入与报告闭环开发文档

这份文档用于承接宇宙页下方“模拟面试场景”的下一阶段开发。

前面几轮已经把几件关键事情铺出来了：

- 总览场景开始像真实启动台
- 资料库场景开始像真实资料工作区
- 报告场景已经接入第一版真实摘要

但现在最核心的训练主链路里，还有一个最重要的部分没有完全做实：

**宇宙页里的 mock 场景，还没有彻底承接真实训练 session。**

---

## 这次开发要解决什么

这次的重点不是继续美化 mock 场景，而是把：

**真实训练过程**

接入宇宙页里的模拟面试场景。

也就是说，这一轮要解决的是：

- 当前 mock 场景不再只靠一套本地简化状态运行
- 它要和真实 `session` 保持一致
- 从资料库进入 mock 时，真正带着当前资料上下文进入训练
- mock 结束后，能够稳定落到当前宇宙报告场景

---

## 为什么现在该优先做 mock 场景

现在宇宙页已经逐渐形成主链路：

- 总览负责启动
- 资料库负责承接训练资料
- 报告负责训练后的结果承接

但真正最关键的中间环节还是：

**模拟面试本身。**

如果 mock 场景还没有彻底接真实 session，就会出现几个问题：

- 总览和资料库传进来的上下文，不一定真正影响训练过程
- mock 场景和旧 `mock-interview.vue` 可能仍然不是同一场训练
- 训练结束后的报告，和宇宙页里展示的过程不够完全对齐

所以这一轮的价值不是“补一个页面”，而是把整条主链路的中段做实：

**总览 -> 资料库 -> 模拟面试 -> 报告**

---

## 当前问题到底是什么

当前宇宙页里的 mock 场景已经有：

- 当前题干展示
- 引用资料展示
- 追问提示
- 弱项信号
- 消息流式区域
- 回答输入区

但它还没有完全建立在真实 session 上。

也就是说，当前更像是：

**“带真实 UI 能力的训练演示场景”**

而不是：

**“真实训练场景本身”**

这会导致两个层面的割裂：

### 1. 训练状态割裂

例如这些内容还可能没有完全统一：

- 当前题目索引
- 已提交题目
- 当前追问索引
- hint 是否展开
- weaknessTags
- 当前 session id

### 2. 训练闭环割裂

例如这些链路还没有完全打通：

- 从资料库选中的资料，是否真的决定 mock 上下文
- mock 结束后，是否能稳定切到当前宇宙报告场景
- 继续训练时，是否真的能回到上次 session

---

## 这次开发的核心目标

这一轮建议明确做成：

**宇宙 mock 场景真实 session 承接版**

也就是优先解决：

- mock 场景读取真实 session
- mock 场景更新真实 session
- mock 场景和资料上下文真实联动
- mock 场景结束后跳转到宇宙报告场景

先不急着做：

- 更复杂的 AI 追问策略
- 题目级评分可视化
- 完整逐题复盘面板
- 多轮对比训练分析

---

## 这次建议承接哪些真实数据

mock 场景下一步应该优先承接这些数据来源：

- `PersistedWorkbenchContext`
- `PersistedInterviewSession`
- 当前资料库里激活的文档
- 当前 topic / mode / source

mock 场景至少应该和这些字段建立一致关系：

### 1. 训练身份信息

- session id
- topic
- mode
- source
- sourceDocumentId

### 2. 当前训练过程状态

- questionCount
- currentQuestionIndex
- submittedQuestionIds
- answeredCount
- followUpIndex
- hintVisible

### 3. 当前训练结果信号

- weaknessTags
- 当前追问建议
- 当前完成状态

---

## 这次 mock 场景要变成什么

这次不建议重写一个新的 mock 页面结构。

更合适的方向是：

**保留当前宇宙 mock 场景的视觉结构，但把背后的状态与动作替换成真实训练逻辑。**

也就是说：

- UI 层保留宇宙页当前这一套布局
- 业务层改成真实 session 驱动

---

## 这次重点要打通的 4 条关系

### 1. 资料库 -> mock

用户在资料库里当前选中的资料，要真正影响 mock 场景：

- 当前引用资料显示什么
- 当前训练 source 是什么
- 当前 sourceDocumentId 是什么

### 2. 总览 -> mock

用户从总览进入时，要能区分：

- 新开一轮训练
- 继续上一次未完成训练

也就是说，mock 场景不能只看“当前展示”，而要看“当前训练状态”。

### 3. mock -> report

mock 结束后，要稳定进入当前宇宙页里的报告场景，而不是只跳去旧页。

这次建议做到：

- 完成训练后切到 `report` 场景
- 当前报告场景优先读取刚完成的 session
- 如果已有 summary，直接承接 summary

### 4. mock -> mock

用户中断后返回，或者继续训练时，要能回到上次位置：

- 当前题目位置
- 已提交题目
- 当前弱项标签
- 当前 session 身份

---

## 建议的数据读取顺序

宇宙页里的 mock 场景建议按这个顺序组织状态：

1. 先判断是否存在当前 in-progress session
2. 如果有，就优先恢复这轮 session
3. 如果没有，再根据当前 topic / mode / source 创建新 session
4. 当前资料上下文优先取 active document
5. 提交答案后同步更新 session
6. 完成训练后写入 summary，并切换到 report 场景

这样可以保证：

- 继续训练是稳定的
- 新开训练是明确的
- 结束后报告承接是自然的

---

## 这次和旧 `mock-interview.vue` 的关系

旧 `mock-interview.vue` 已经有一套更完整的训练逻辑和持久化闭环。

这次不应该在宇宙页里再手写一套新逻辑，而是建议：

- 复用旧 mock 页已有的 session 读写逻辑
- 复用已有的完成训练 / 生成 summary 逻辑
- 让宇宙页负责场景化承接，不再自己发明一套 session 规则

一句话：

**mock 业务逻辑尽量共用，宇宙页只负责展示方式和场景承接。**

---

## 建议的开发顺序

### 第 1 步：梳理宇宙 mock 场景里的本地简化状态

先明确当前哪些状态还是 mock 场景自己维护的，例如：

- 当前题目
- 提交状态
- 弱项信号
- 追问索引

把这些状态分成两类：

- 应该继续保留在 UI 层的
- 应该改成 session 驱动的

### 第 2 步：对齐旧 mock 页已有的真实 session 逻辑

重点确认这些逻辑哪些可以直接复用：

- session 查找
- session 创建
- session 更新
- 完成训练
- 生成报告摘要

### 第 3 步：先让宇宙 mock 场景读取真实 session

第一轮只要求做到：

- 当前训练身份真实可读
- 当前题目位置真实可读
- 当前资料来源真实可读
- 当前弱项真实可读

### 第 4 步：让宇宙 mock 场景提交后更新真实 session

至少同步这些内容：

- answeredCount
- submittedQuestionIds
- followUpIndex
- weaknessTags
- currentQuestionIndex

### 第 5 步：打通 mock -> report 闭环

完成训练后要做到：

- 写入已完成 session
- 写入 report summary
- 切到宇宙页 `report` 场景

---

## 完成后的目标状态

做完这一轮之后，宇宙页里的 mock 场景应该具备这些能力：

- 它不再只是“看起来像训练”
- 它真的是训练过程本身
- 当前资料会真实影响训练上下文
- 当前 session 会真实记录训练进度
- 完成训练后会真实落到报告场景

---

## 这次不优先做什么

为了避免范围失控，这一轮先不做：

- 更复杂的 AI 评分规则
- 多维度训练图表
- 逐题评分看板
- 高级历史筛选
- 更复杂的专项刷题联动

这轮最关键的只有一句话：

**让宇宙页里的模拟面试场景，真正成为那一轮训练 session 本身。**

---

## 一句话结论

前面几轮已经解决了“从哪里开始训练”和“训练结束后结果落到哪里”，这一轮要解决的是：

**训练过程本身如何在宇宙页里真正接住真实 session，并和报告形成完整闭环。**
