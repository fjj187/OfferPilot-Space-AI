# 10.宇宙面试页接入与共享状态架构开发流程.md

## 一、为什么现在进入这一阶段

在上一阶段，项目已经完成了模拟面试主链路里的 SSE 流式对话与消息渲染架构：

- `mock-interview.vue` 已接入真实流式消息
- `useInterviewStream.ts` 已完成页面层与流式能力桥接
- 消息队列、消息解析、Markdown 渲染组件已经具备
- session summary / report summary 的第一阶段持久化已打通

也就是说：

**现在问题已经不再是“能不能流式对话”，而是“如何把已有面试能力自然接入新的沉浸式宇宙页面”。**

这一阶段之所以成立，是因为产品方向中途发生了一次明显变化：

- 原来的主面试页是偏工作台式、结构化、白卡片布局
- 新增了一个宇宙主题的沉浸式面试页 `mock-interview-space.vue`
- 它不只是一个视觉 demo，而是开始承载真实面试内容

因此，这一阶段真正要解决的是：

1. 新宇宙页面已经做了哪些内容承接
2. 它和原 `mock-interview.vue` 的关系是什么
3. 如何避免两页各写一套独立状态逻辑
4. 如何把“宇宙页”从展示壳升级成“真正共用同一场面试”的页面

---

## 二、这一阶段中途发生了什么变化

### 1. UI 风格发生切换

项目中途新增了新的沉浸式宇宙风格页面：

- `src/views/showcase/mock-interview-space.vue`

这个页面与原工作台式页面不同，它强调：

- 首页级沉浸感
- 场景切换
- 轨道导航
- 更强的视觉主题统一
- 将“模拟面试”包装成一个更有氛围的体验入口

### 2. 视觉层已经开始承接真实内容

这个宇宙页已经不只是空壳，目前已经接入了部分真实面试能力，包括：

- `MessageList`
- `AnswerInputPanel`
- `useInterviewStream`
- 当前题目文案
- 追问提示
- 弱项信号
- 流式消息展示

也就是说：

**宇宙页已经开始承接真实业务内容，但状态来源还没有和原面试页合并。**

---

## 三、目前已经完成了什么

### 1. SSE 流式能力已经完成

当前项目已经具备完整的前端流式对话基础能力：

- `src/composables/useInterviewStream.ts`
- `src/services/sse/interview-stream.ts`
- `src/services/sse/sse-client.ts`
- `src/services/sse/sse-types.ts`
- `src/services/message/interview-message-parser.ts`
- `src/services/message/interview-message-queue.ts`

这些能力说明：

- 流式请求控制已经可用
- token 增量处理已经可用
- requestAnimationFrame 消费队列已经具备
- UI 消息展示链路已经跑通

### 2. 面试主页面已经接入真实消息流

`src/views/workbench/mock-interview.vue` 已完成：

- 真实消息列表展示
- 用户输入触发流式请求
- streaming/error 状态暴露
- follow-up / hint / submit / finish 等面试流程
- session summary / report summary 的基础持久化

### 3. 宇宙页已经接入“部分真实面试能力”

`src/views/showcase/mock-interview-space.vue` 当前已经具备：

- 流式消息面板
- 回答输入面板
- 主问题展示
- 推荐追问
- 弱项提示
- mock 面试主区域嵌入宇宙页内容区

但目前仍然存在一个结构性问题：

**宇宙页使用的是一套“单独复制出来的简化面试状态”，不是原面试页的同一套状态。**

---

## 四、当前最大的架构问题是什么

不是 SSE 没做完，也不是 UI 没接上。

当前最大的问题是：

**同一个“模拟面试业务”，在两个页面里维护了两套状态。**

### 现在的状态分裂表现

#### 原工作台面试页里有完整状态

例如：

- 当前题目
- 当前题目索引
- followUpIndex
- hintVisible
- submittedQuestionIds
- weaknessTags
- currentSessionId
- route/query 同步
- persistence
- finishInterview/report 跳转

#### 宇宙页里有一套简化状态

例如：

- `primaryQuestion`
- `mockAnswerDraft`
- `mockFollowUpIndex`
- `mockSubmittedQuestionIds`
- `mockWeaknessSignals`

这会带来几个问题：

1. 两页不是同一场面试
2. 状态无法真正共享
3. 后续维护成本会越来越高
4. 一旦业务规则变化，需要改两份逻辑

---

## 五、这一阶段的目标

这一阶段不再继续扩展 SSE 本身，而是做：

**“宇宙页接入真实面试状态”的共享状态架构整理。**

具体目标：

1. 把原 `mock-interview.vue` 的面试状态逻辑抽离
2. 形成可复用的 `useMockInterviewSession`
3. 让 `mock-interview.vue` 和 `mock-interview-space.vue` 共用同一套面试 session
4. 宇宙页只负责“展示方式”，不再单独维护业务状态
5. 保持已有 `useInterviewStream` 不变，作为流式能力层继续复用

---

## 六、这一阶段建议的职责拆分

### 1. 流式能力层：继续保留

- `useInterviewStream.ts`

它负责：

- 消息列表
- 流式请求
- streaming 状态
- error 状态
- 停止生成
- 消息队列与 UI 同步

这一层已经是对的，不需要推翻。

### 2. 面试业务状态层：新增

建议新增：

- `src/composables/useMockInterviewSession.ts`

它负责：

- route/query 同步
- 当前 topic / mode / source
- 当前题目列表
- 当前题目索引
- follow-up 状态
- hint 状态
- submitted ids
- weakness tags
- session persistence
- finishInterview
- report 跳转

### 3. 页面层：各自负责展示

#### `mock-interview.vue`

保留工作台布局，用共享状态渲染原页面

#### `mock-interview-space.vue`

保留宇宙沉浸布局，用共享状态渲染宇宙页面

也就是说：

**两页共用同一个“脑子”，只换不同“皮肤”。**

## 九、宇宙页接入时应该怎样改

### 当前宇宙页要删掉的内容

应该逐步移除宇宙页里自己维护的这套状态：

- `primaryQuestion`
- `mockAnswerDraft`
- `mockFollowUpIndex`
- `mockSubmittedQuestionIds`
- `mockWeaknessSignals`
- `initializeMockConversation`
- `submitMockAnswer`
- `clearMockAnswer`
- `rotateMockFollowUp`

### 当前宇宙页要改成什么

改为直接消费共享 composable 暴露的数据，例如：

- `currentQuestion`
- `currentFollowUp`
- `answerDraft`
- `setAnswerDraft`
- `submitAnswer`
- `clearAnswer`
- `weaknessTags`
- `messages`
- `isStreaming`
- `streamError`

### 最终效果

宇宙页中展示的：

- 当前题目
- 追问建议
- 回答输入
- 流式消息
- 弱项信号

都来自同一个 session。

这样用户无论从：

- 工作台面试页进入
- 宇宙页进入
- 总览页进入

只要 session 相同，就进入的是同一场面试。

## 十、建议的开发顺序

### 第 1 步：抽出 `useMockInterviewSession.ts`

目标：

- 先让 `mock-interview.vue` 不再自己持有业务状态
- 页面继续可运行

### 第 2 步：让 `mock-interview-space.vue` 接同一个 composable

目标：

- 用共享状态替换掉宇宙页当前本地 mock 状态
- 保留宇宙页现有视觉壳

### 第 3 步：统一入口路由参数

目标：

- 从 overview / library / showcase 进入时
- topic / mode / source / questionCount 规则一致

### 第 4 步：统一 finish / report 链路

目标：

- 无论从哪个页面完成面试
- 都能进入同一个 report 结果流

### 第 5 步：最后再考虑消息持久化

当前先不急。

---

## 十一、这一阶段先不要急着做什么

### 1. 不要先把宇宙页做成第三套业务逻辑

这是当前最危险的方向。

### 2. 不要先做“宇宙页专属 session”

会让状态进一步分裂。

### 3. 不要先做完整消息历史持久化

这一步仍然可以晚一点再做。

---

## 十二、这一阶段完成后的结果

完成后，项目会进入一个更稳定的状态：

- `mock-interview.vue` 是工作台式面试页
- `mock-interview-space.vue` 是沉浸式宇宙面试页
- 两页共用同一套面试状态
- 两页共用同一套 SSE 流式能力
- 后续扩展 report / history / session replay 会更顺

也就是说：

**这一步不是“又做了一个页面”，而是把已有面试能力从单页应用，升级成“同一业务多视图承载”。**

---

## 十三、结论

上一阶段已经把：

**SSE 流式对话与消息渲染架构**

搭好了。

这一阶段真正要做的是：

**把新增的宇宙面试页，从“接入了流式对话的展示页”，升级成“与主面试页共用同一场面试的真实业务页”。**

这会让项目从“功能能跑”进入“结构开始成熟”的阶段。
