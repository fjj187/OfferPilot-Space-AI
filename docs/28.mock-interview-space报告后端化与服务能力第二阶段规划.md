# 28. mock-interview-space 报告后端化与服务能力第二阶段规划

## 文档目的

`27` 阶段已经完成“真实流式链路 + 最小会话读写闭环”。  
`28` 阶段的目标，不再是证明后端存在，而是让这套最小后端真正开始承接更稳定的产品数据闭环。

这份文档用于明确下一阶段方向，避免继续在 `27` 范围里混做。

---

## 一、`28` 阶段的核心目标

`28` 的一句话目标是：

`把报告能力从“前端临时拼装”推进到“后端可生成、可读取、可稳定消费”的第二阶段服务闭环。`

与 `27` 相比，重点变化不是“再多几个接口”，而是：

1. 后端开始承接报告数据，而不只是会话消息。
2. 前端历史页和报告页开始统一消费后端真实结果。
3. 当前最小文件存储继续可用，但数据结构开始为数据库化和未来 C++ 扩展留更清晰边界。

---

## 二、为什么 `28` 要先做报告后端化

当前系统虽然已经能：

1. 真实流式面试。
2. 保存 session / message。
3. 读取历史会话和会话详情。
4. 让历史页、报告页优先消费后端真实消息。

但报告能力仍然主要停留在：

1. 前端根据本地 summary 和消息临时组织文案。
2. 历史报告列表并未完全由后端主导。
3. 后端还没有“报告”这一类稳定对象。

这会导致一个明显问题：

`消息已经后端化了，但复盘结论还没有真正后端化。`

所以 `28` 最应该优先补的是：报告对象本身。

---

## 三、`28` 阶段的边界

### `28` 应该做的事

1. 建立最小 report summary 存储结构。
2. 建立报告列表和报告详情接口。
3. 让报告页优先消费后端 report summary。
4. 让历史页右侧报告列表优先消费后端 summary。
5. 统一 session 与 report 的关联关系。
6. 保持 mock / real 双链路仍然可用。

### `28` 不应该急着做的事

1. 数据库替换 JSON。
2. 复杂用户系统。
3. 检索 / RAG / 向量库。
4. 自动题目编排引擎。
5. 高并发优化和复杂部署治理。
6. 完整多维打分体系。

也就是说，`28` 仍然是“最小闭环第二阶段”，不是完整平台化。

---

## 四、推荐的阶段成果定义

`28` 完成后，项目应达到以下状态：

### 4.1 对用户可见

1. 一轮模拟结束后，报告不再只是前端临时拼装。
2. 历史页能看到真实的报告摘要列表。
3. 报告页能从后端直接读取对应 summary 和会话详情。
4. 重开页面或重启服务后，历史报告仍然存在。

### 4.2 对工程侧可见

1. `backend/` 中新增最小 report 存储与读取能力。
2. 报告接口契约冻结。
3. 前端报告页、历史报告列表都已优先消费后端结果。
4. session / report 的关系模型更清晰，后续更容易迁移数据库。

---

## 五、推荐新增的后端能力

### 5.1 最小 report 存储

建议在 `backend/src/storage/` 中补一个与 session 并列的 report 存储文件，例如：

```text
backend/data/interview-reports.json
```

建议最小结构：

```ts
interface StoredInterviewReportSummary {
  id: string
  sessionId: string
  threadId: string
  topic: string
  questionTitle: string
  summaryHeadline: string
  summaryBody: string
  weaknessTags: string[]
  answeredCount: number
  totalCount: number
  createdAt: string
  updatedAt: string
}
```

### 5.2 推荐接口

建议新增：

1. `GET /api/interview/reports`
2. `GET /api/interview/reports/:sessionId/:threadId`
3. `POST /api/interview/reports/generate` 或在面试结束动作中内部生成

其中第一阶段不一定非要做复杂异步任务，完全可以先做：

`根据已有 session / message 组织最小 summary 并保存。`

---

## 六、推荐的前端改造方向

### 6.1 历史页

当前历史页已经能读真实 session。  
下一步应让右侧“历史复盘摘要”列表优先读后端 report summary，而不是主要依赖本地持久化。

### 6.2 报告页

当前报告页已经能结合后端 session detail 做回填。  
下一步应调整为：

1. 优先读取 report summary。
2. 再读取 session detail 作为证据补充。
3. 当 report summary 不存在时，再 fallback 到当前临时拼装逻辑。

### 6.3 面试结束动作

当前“结束本轮模拟”更像前端状态收束。  
`28` 阶段建议把它推进成：

1. 前端触发“结束本轮”。
2. 后端根据当前 session 生成最小 report summary。
3. 前端跳转报告页时优先读取这个 summary。

---

## 七、推荐开发顺序

### 第 1 步

定义 report summary 数据结构和接口契约。

### 第 2 步

实现后端本地 JSON report 存储。

### 第 3 步

实现报告列表 / 详情读取接口。

### 第 4 步

前端历史页右侧报告列表接后端 summary。

### 第 5 步

前端报告页改成“summary 优先，session detail 补充”。

### 第 6 步

面试结束动作与 report summary 生成衔接起来。

---

## 八、`28` 阶段最值得关注的风险

1. 不要把报告结构一次设计得太重。
2. 不要提前做复杂分析引擎。
3. 不要为了“像数据库”而过度抽象当前 JSON 存储。
4. 不要破坏 `27` 已经稳定下来的流式协议。

`28` 的目标不是“做聪明的分析系统”，而是“做稳定的报告数据闭环”。

---

## 九、最终结论

`27` 解决的是“真实会话链路和最小后端存在性”。  
`28` 要解决的是“报告能力真正进入后端数据闭环”。

因此，下一阶段最应该做的事情不是继续扩张 mock 展示，而是：

`让 session、report、history、report-page 四者全部建立在统一的后端真实数据基础上。`
