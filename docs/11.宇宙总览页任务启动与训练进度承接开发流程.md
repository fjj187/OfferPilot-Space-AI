# 11.宇宙总览页任务启动与训练进度承接开发流程.md

## 一、为什么现在开始做总览页

在前一阶段，项目已经完成了两件关键事情：

- SSE 流式对话与消息渲染架构已经打通
- `mock-interview.vue` 与 `mock-interview-space.vue` 已开始共用同一套面试 session 状态

这意味着：

**现在最缺的已经不是对话能力，而是一个真正承担“启动、承接、继续训练”职责的总览入口。**

原有总览页的问题不是没有内容，而是内容太多、太散、太像后台工作台首页：

- 资料卡很多
- 引导卡很多
- 队列卡很多
- 同屏并列区域太多

它确实把信息讲全了，但不够适合成为沉浸式面试体验的入口。

因此，这一阶段要做的不是“再做一个更复杂的总览页”，而是：

**把原总览页收敛成一个简单、简洁、可继续训练的任务启动页。**

---

## 二、这一阶段的核心目标

新的总览页要承接原有业务价值，但不延续原来的卡片堆叠结构。

这一阶段的目标是：

1. 承接原有总览页已有的训练状态与入口能力
2. 删除大量并排卡片和冗余说明
3. 用更少的区域表达更明确的任务状态
4. 让用户一进总览页就知道：
   - 当前在练什么
   - 现在进行到哪里
   - 下一步该点哪里继续

---

## 二点五、这一阶段的实际开发落点

这一阶段虽然讨论的是“总览页”，但首轮开发与验收入口不直接放在 `workbench/overview.vue`，而是：

- 先在 `src/views/showcase/mock-interview-space.vue` 完成结构验证
- 通过 `http://localhost:2050/showcase/mock-interview-space` 直接看效果
- 待结构、信息密度和动作关系验证通过后，再回落到正式业务页

这样做的原因是：

- 当前视觉和交互验收主要发生在 showcase 页
- `mock-interview-space.vue` 已经承担“宇宙化总览入口”的展示职责
- 如果直接改正式 `overview.vue`，用户在当前验收入口看不到结果，反馈链路会断开

因此，本阶段文档中的“总览页重构”，在实施层面应理解为：

**先重做 showcase 里的宇宙总览启动页，再决定如何同步到正式 workbench 页面。**

---

## 三、原总览页里哪些东西要保留

原总览页并不是都要推翻。

真正应该保留的是这些业务含义：

- 当前训练主题
- 当前训练模式
- 当前资料来源
- 当前是否存在未完成 session
- 当前训练进度
- 最近一次训练结果摘要
- 最近弱项信号
- 去模拟面试 / 去报告 / 去资料页的主入口

也就是说：

**要承接原总览页的“状态”和“动作”，而不是承接它的“卡片数量”。**

---

## 四、原总览页里哪些东西要收掉

以下内容不适合继续原样保留：

- 多个资料卡同时占据大面积首屏
- 多个引导卡并列抢注意力
- 训练队列和复盘摘要同时展开
- 大量解释性文案
- 首页级页面里堆很多二级信息模块

这些信息不是没用，而是应该被压缩成摘要、次级入口，或者放到后续页面里。

---

## 五、新总览页应该变成什么样

新总览页建议收成 4 个部分。

### 1. 主任务区

这是总览页第一屏最重要的部分，只保留：

- 当前主题
- 当前模式
- 当前资料来源
- 当前进度
- 一个主按钮：
  - `继续面试`
  - 或 `开始本轮模拟`

如果存在未完成 session，主按钮优先变成“继续面试”。

### 2. 轨道概览区

用非常轻量的流程轨道表达整个面试路径：

- 总览
- 模拟面试
- 答题策略
- 反馈
- 结果

这里不再用复杂卡片，而是用简洁节点让用户理解当前位置和后续路径。

### 3. 训练摘要区

只展示最重要的训练结果摘要，例如：

- 最近完成题数
- 最近一次训练时间
- 最近弱项信号

这部分替代原总览页里大量展开的队列卡和复盘卡。

### 4. 次级入口区

只保留少量辅助动作：

- 查看资料
- 查看报告
- 换主题

不再做多块并列重卡片。

---

## 六、这一阶段的数据承接方式

新总览页的数据来源不应该重新造一套，而是直接承接现有持久化结果：

### 1. 来自 `WorkbenchContext`

用于承接：

- 当前主题
- 当前文档来源
- 当前模式

### 2. 来自 `InterviewSession`

用于承接：

- 是否存在未完成 session
- 当前进度
- 当前题目位置
- 最近弱项标签

### 3. 来自 `ReportSummary`

用于承接：

- 最近一次训练摘要
- 已完成题数
- 最近结果时间

---

## 七、页面结构建议

### 顶部主区

- 任务标题
- 一段非常短的状态说明
- 主题切换 chips
- 主按钮与次按钮

### 中段流程区

- 一条简洁的训练轨道
- 节点只显示名字和当前位置

### 下段摘要区

- 当前进度
- 最近弱项
- 最近结果

---

## 八、这一阶段不再追求什么

这一版总览页先不要追求：

- 把所有旧卡片都迁过来
- 首页信息最全
- 首页承担所有说明职责
- 复杂多列布局

这版更重要的是：

**信息足够少，但动作足够明确。**

---

## 九、建议开发顺序

### 第 1 步：抽取原总览页要保留的数据字段

保留：

- active topic
- current mode
- source
- in-progress session
- latest report summary
- weakness tags

### 第 2 步：重做总览页布局

去掉原来的：

- hero 卡
- library 卡矩阵
- guidance 卡列表
- practice queue 卡列表

改成更轻的主任务布局。

这里的“总览页布局”在本阶段优先指：

- `src/views/showcase/mock-interview-space.vue` 的首屏结构
- 不是正式 `src/views/workbench/overview.vue` 的卡片堆叠页

### 第 3 步：保留关键导航动作

- 继续面试
- 开始本轮模拟
- 打开资料页
- 打开报告页

### 第 4 步：最后再考虑是否把这页也宇宙化

第一版先把结构做对，再决定是否完全切到宇宙视觉语言。

---

## 十、这一阶段完成后的效果

完成后，总览页会从：

**信息面板集合页**

变成：

**训练任务启动页**

用户进入之后，不再需要先扫一堆卡片，而是能更快进入主链路。

---

## 十一、结论

原总览页的价值不是它有很多卡片，而是它承担了：

- 训练状态承接
- 入口分发
- 进度提醒

新总览页要做的不是把这些东西变得更多，而是：

**保留状态，压缩展示；保留动作，减少干扰。**

这样它才能真正成为面试链路里的简洁入口，而不是另一个复杂首页。
