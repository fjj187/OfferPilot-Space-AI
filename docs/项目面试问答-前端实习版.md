面试问答

第 1 题

问：介绍一下这个项目，你负责什么？

答：
  这是 Vue 3（框架）+ Vite（构建工具）的 AI 模拟面试 Web 应用，主页面是 frontend/src/views/showcase/mock-interview-space.vue。用户导入 Markdown 资料、按章节生成题目、逐题模拟面试，AI 面试官回复通过 SSE（Server-Sent Events，服务端推送）流式显示，并可查看报告。

  项目使用 AI 辅助开发（vibe coding）。我主要负责：流式消息从接口到页面的链路（useInterviewStream.ts、interview-message-queue.ts）、模拟面试状态与刷新恢复（useMockInterviewSpaceMockState.ts、entryMode 与 sessionConfig 设计）、资料 Markdown 分块驱动出题（material-chunk-splitter.ts）。其余 UI 多在 AI 草稿上由我联调、改结构与修 bug。未参与的部分（若属实）：backend Provider 内部实现、legacy 工作台。


第 2 题

问：和 frontend/src/views/chat.vue 有什么区别？

答：
  chat.vue 是单轮通用问答演示，可通过 Vite 代理访问 /deepseek、/spark 等大模型。主产品是面试工作流：资料分块出题、按题作答、feedbackStyle（反馈风格）、切题规则、报告复盘，不是无限轮聊天。


第 3 题

问：用了 AI 写代码，那你具体做了什么？

答：
  AI 适合生成样板 UI 和重复逻辑。我负责方案和核心实现：SSE 消费分层、rAF（requestAnimationFrame，按帧调度）消息队列防卡顿、多入口状态字段（entryMode、mockSessionConfig）、资料分块规则。联调时会改掉 AI 常见问题，例如每个 chunk 直接触发 DOM 更新、多个 watch（侦听器）同时改同一套布尔状态。能结合具体文件说明实现细节。


第 4 题

问：从提交答案到文字显示，经过了哪些模块？

答：
  1. mock-interview-space.vue 或 useMockInterviewSpaceMockState.ts 组装题目、答案、资料上下文
  2. useInterviewStream.ts 创建 streaming 状态的消息并调用 startInterviewStream
  3. interview-stream.ts 发起 POST /api/interview/stream（开发时经 frontend/vite.config.ts 代理到 backend）
  4. sse-client.ts 解析 SSE 帧，产出 chunk 事件
  5. interview-message-queue.ts 按帧合并 chunk，更新 displayContent
  6. MarkdownMessage.vue：markdown-it 转 HTML 后 DOMPurify.sanitize（DOM 净化），再渲染


第 5 题

问：为什么用 SSE，不用 WebSocket？

答：
  面试官回复是服务端向浏览器单向推送 token，SSE 基于 HTTP 即可，与「POST 一次答案再收流」匹配。WebSocket（全双工长连接）更适合需要客户端高频上行的场景。前端实现分传输层 sse-client.ts、控制层 interview-stream.ts、编排层 useInterviewStream.ts。


第 6 题

问：chunk、done、error 事件是什么？能改吗？

答：
  这是前后端约定的流式契约：chunk 携带内容片段，done 表示本轮流结束，error 表示失败。前端类型在 frontend/src/services/sse/sse-types.ts，后端在 backend/src/types/interview.ts。实习项目理解即可；生产环境单方修改会导致联调失败。


第 7 题

问：mock 和 remote 两种模式有什么区别？

答：
  interview-stream.ts 中 resolveInterviewStreamEndpoint：有流地址则 remote（开发默认 /api/interview/stream），无地址则在前端 buildMockInterviewReply + splitTextToReader 模拟打字机，不请求网络。两种模式对 UI 统一为 InterviewStreamEvent（start / chunk / done / error）；remote 用 consumeSSEStream，mock 用 consumeStreamText。


第 8 题

问：为什么需要 backend？前端不能直接调大模型吗？

答：
  API Key 不宜长期暴露在前端；统一 /api 便于本地 mock 与切换后端实现。我主要负责前端消费流式响应。backend 通过 interview-provider-factory.ts 选择 MockInterviewProvider、RemoteLLMProvider 或 CppEngineProvider，前端不感知具体 Provider 类名。


第 9 题

问：流式输出卡顿怎么处理的？

答：
  LLM（Large Language Model，大语言模型）chunk 频率可能高于屏幕刷新。若每个 chunk 都改 Vue 响应式字段，会导致频繁重排。InterviewMessageQueue 将 delta 先入队，每帧 requestAnimationFrame flush 一次再 patch 消息，文件在 frontend/src/services/message/interview-message-queue.ts。


第 10 题

问：AI 返回的 Markdown 直接 v-html 可以吗？

答：
  不可以。MarkdownMessage.vue 先用 markdown-it 转 HTML，再 DOMPurify.sanitize，最后 v-html，防止 XSS（Cross-Site Scripting，跨站脚本）。报告页 ReportReviewMarkdown.vue 同样处理。


第 11 题

问：状态存在哪里？为什么用 composable 而不是全部 Pinia？

答：
  面试状态与宇宙页场景强相关，放在 composables/showcase 更清晰。Pinia（状态仓库）在项目中用得较轻。需要跨刷新保留的业务上下文写入 localStorage，经 workbench-storage.ts 与 useWorkbenchPersistence.ts 读写。复杂流程放在 composable，避免子组件直接改父级业务状态。


第 12 题

问：entryMode 和 mockSessionConfig 是什么？解决什么问题？

答：
  从资料库、专项练习、直接面试等多个入口进入时，若只记「正在面试」，刷新后可能恢复错题组或错配置。entryMode 记录入口类型（direct / practice / material），mockSessionConfig 记录本场快照（资料 id、题数、难度、feedbackStyle 等），一并写入 PersistedWorkbenchContext。恢复时用 sessionConfigKey 匹配会话，避免串台。类型定义在 frontend/src/types/workbench.ts。


第 13 题

问：你们说的状态机是什么意思？

答：
  指业务上遵守有限状态机思想：同一时刻处于明确阶段之一（如面试进行中、等待 AI 回复、本轮结束）；切换通过 launch、finish、restore 等少数入口函数，而不是多个 watch 各自改 isXxx 布尔值。单次提交答案后，消息还有子状态 streaming → done，在 useInterviewStream.ts 的 onEvent 中处理。entryMode + sessionConfig 是多入口的上下文快照，与主流程正交。


第 14 题

问：资料很长，怎么出题？

答：
  material-chunk-splitter.ts 按 Markdown 标题层级切分，单块上限约 1000 字符，输出 heading、level、字符偏移。material-question-builder.ts 由 chunk 生成题目，组卷项带 sourceChunkId，报告可通过 resolve-report-reference-answer.ts 回溯资料章节。这是按章节出题的轻量方案，不是完整向量检索 RAG（Retrieval-Augmented Generation，检索增强生成）。


第 15 题

问：联调时流式没有字或一直转圈，你怎么排查？

答：
  1. 浏览器 Network 查看 POST /api/interview/stream 是否有持续响应、事件是否为 chunk
  2. 确认 pnpm dev 与 pnpm dev:backend 均已启动，代理目标与 VITE_INTERVIEW_BACKEND_ORIGIN 一致
  3. 看消息 status 是否卡在 streaming，done 是否到达
  4. Console 查看 sse-client 或 fetch 报错
  5. 检查 workbenchContext 恢复是否覆盖当前会话。先复现再定位一层，避免多处加兜底分支。


第 16 题

问：遇到过什么难点？怎么解决的？

答：（按真实经历选一条填写）

  难点 A：流式回复时消息列表卡顿。
  原因：每个 chunk 直接更新 DOM。
  解决：引入 InterviewMessageQueue，用 rAF 每帧合并 delta。
  文件：interview-message-queue.ts、useInterviewStream.ts。

  难点 B：刷新后面试配置与题组不一致。
  原因：多入口共用一套布尔状态，恢复逻辑分散。
  解决：增加 entryMode、mockSessionConfig、sessionConfigKey，launch/finish 走 useMockInterviewFlow.ts 单一入口。


第 17 题

问：ref 和 computed 有什么区别？父子组件怎么通信？

答：
  ref（响应式引用）存可变状态；computed（计算属性）由已有状态推导，不重复存储。父组件通过 props（组件入参）向下传，子组件通过 emit（事件抛出）向上报意图；跨层业务流程放在 composable，例如 useInterviewStream，而不是在子组件内直接修改父组件的业务数据。


第 18 题

问：是不是大部分代码都是 AI 写的？

答：
  AI 提高了 UI 样板和重复逻辑的效率。流式分层、rAF 队列、多入口状态字段、分块规则由我参与设计与实现；关键文件能讲清数据流并独立修 bug。未参与的后端引擎实现会明确说明，只描述前端如何消费 /api 契约。


第 19 题

问：如果不用 AI，你还能实现负责的那几块吗？

答：
  可以，速度会慢一些。核心是理解数据流与状态边界，而不是记住某次 prompt（提示词）。实现依赖 Vue 3、TypeScript、fetch 与 ReadableStream（可读流）等已有知识。


第 20 题

问：这个项目有什么局限？你还缺什么？

答：
  局限：按题注入上下文，不是无限轮聊天；资料分块为规则切分；MVP 以本地联调为主。我尚未深入：生产监控、压测、backend Provider 与 C++ 引擎内部（若未参与）。希望通过实习在规范工程化与业务场景里继续补全。
