# HTTP（超文本传输协议）请求层高可用增强方案

## 一、目标

本方案用于增强 `frontend/src/services/http/request.ts`（统一请求客户端）的可用性与稳定性，在不新增第二套 `axios`（HTTP 请求库）实例、不改现有业务接口契约的前提下，补齐以下三项基础能力：

1. `TokenBucket`（令牌桶）并发控制
2. 全局默认 `retry`（重试）策略
3. `circuit breaker`（熔断器）重试熔断能力

目标不是重写当前请求层，而是在现有 `request.ts`（统一请求工具）上做渐进增强，让后续所有 `REST API`（非流式接口） 默认获得更稳定的失败恢复与流量保护能力。

---

## 二、当前现状

当前项目的非流式请求已经统一收口到 `frontend/src/services/http/request.ts`（统一请求客户端），并具备以下能力：

- `axios.create`（创建请求实例）
- 请求 / 响应 `interceptor`（拦截器）
- `Bearer Token`（携带 Bearer 前缀的令牌）注入
- 按请求配置 `retry`（重试）
- `errorType`（错误类型）归一
- 文件下载响应处理

当前仍存在三个明显缺口：

### 1. 缺少统一并发保护

虽然请求都走同一个入口，但当前没有总量级并发控制。
当后台看板、会话列表、报告列表、专项题池等页面同时发起请求时，短时间内可能把浏览器侧和后端侧都压到尖峰状态。

### 2. 默认重试策略不统一

现在部分服务层手动传入 `retry`（重试）配置，部分接口不传。
这样会导致“同类请求稳定性策略不一致”，后续继续加接口时也容易遗漏。

### 3. 连续失败后没有熔断保护

如果某个接口对应的后端服务已经异常，例如认证聚合接口或后台统计接口持续 `500`（服务器内部错误）/ `503`（服务不可用），前端仍会继续按原策略重试。
这会放大雪崩效应，也会让用户长时间等待没有意义的重试。

---

## 三、设计原则

### 1. 只增强 `request.ts`（统一请求工具）

所有高可用能力优先下沉到统一请求层，不在每个服务文件各自复制一套并发、重试、熔断逻辑。
这样做是为了避免后续接口越来越多时，策略分散、参数漂移、行为不一致。

### 2. 默认可用，按请求可覆盖

全局应提供默认 `retry`（重试）与默认并发控制策略，但允许单个请求通过 `AxiosRequestConfig`（请求配置类型）显式覆盖。
这样既能让大多数接口“开箱可用”，又不会堵死长耗时接口或高风险写操作的特殊需求。

### 3. 不误伤已有业务语义

`clear`（清空）、`delete`（删除）、`generate`（生成）这类可能产生副作用的请求，不应因为“统一默认重试”被隐式重复提交。
默认策略要优先保证安全，再追求自动恢复。

### 4. 失败要尽快暴露，不做无意义等待

`circuit breaker`（熔断器）存在的意义，不是让系统更复杂，而是在“已经明显持续异常”的场景下，尽快停止请求和重试，减少用户等待与服务雪崩。

---

## 四、目标能力设计

## 4.1 `TokenBucket`（令牌桶）并发控制

### 目标

在请求真正发出前做统一排队，限制同时进行中的请求数量，例如默认最多 `10`（十个）并发。

### 推荐行为

- 有可用令牌时立即发请求
- 超过上限时进入等待队列
- 请求结束后归还令牌，并唤醒下一个等待请求
- 若请求在排队阶段就被 `AbortSignal`（中止信号）取消，应立即移出队列

### 推荐落点

推荐在 `request.ts`（统一请求工具）内部新增轻量 `TokenBucket`（令牌桶）类，作为统一请求实例的前置守卫。
请求拦截器负责“申请令牌”，响应拦截器和错误拦截器负责“释放令牌”。

### 推荐默认配置

```ts
maxConcurrentRequests: 10
queueTimeoutMs: 15000
```

### 为什么放在这一层

因为并发保护本质上是“请求发送前的基础设施能力”，不属于某一个具体业务接口。
如果把这层逻辑散落在 `admin-api.ts`（后台接口文件）、`interview-report-api.ts`（面试报告接口文件）等服务文件里，后续维护会非常分裂。

### 注意事项

- 文件下载请求默认也会占用令牌，避免大批下载把普通接口全部挤死
- 长耗时接口可以通过配置单独关闭或放宽并发限制
- 队列等待超时后应返回明确错误，不要无限等待

---

## 4.2 全局默认 `retry`（重试）策略

### 目标

让大多数“安全可重试”的请求，在遇到短暂网络波动、网关抖动、临时超时时自动恢复，而不是每个服务层手动重复写一次配置。

### 推荐默认策略

```ts
maxRetries: 3
retryDelayMs: 800
backoffMultiplier: 1.6
retryableStatusCodes: [408, 429, 500, 502, 503, 504]
```

### 推荐行为

- `GET`（读取请求）默认启用全局重试
- 明确无副作用的 `HEAD`（请求头读取）/ `OPTIONS`（协商请求）可复用默认策略
- `POST`（创建或提交请求）、`PUT`（整体更新请求）、`PATCH`（局部更新请求）、`DELETE`（删除请求）默认不自动重试
- 只有当调用方显式声明安全时，副作用请求才允许开启重试

### 推荐配置方式

推荐把“是否应用默认重试”做成请求层内部规则，而不是要求每个服务文件都写：

```ts
retry: {
  mode: 'default' | 'off' | 'custom'
}
```

如果不想引入新的 `mode`（模式）字段，也可以保持当前结构，只在请求拦截器里补齐未显式传入的默认值。

### 为什么要全局默认

因为项目已经进入“接口数量持续增加”的阶段。
如果每个服务文件都手动维护 `INTERVIEW_API_RETRY_CONFIG`（面试接口重试配置）、`ADMIN_API_RETRY_CONFIG`（后台接口重试配置）这类常量，最终只会形成大量近似但不完全相同的策略副本。

### 不建议的做法

- 不建议对所有请求一刀切默认重试
- 不建议把重试次数放在拦截器闭包里共享
- 不建议在服务层随手写不同的默认延迟，除非业务确实有差异

---

## 4.3 `circuit breaker`（熔断器）重试熔断

### 目标

当某个接口在短时间内连续失败超过阈值时，临时阻止后续请求继续发送，避免前端反复命中一个已知异常的后端能力。

### 典型场景

- `/api/admin/dashboard`（后台看板接口）后端暂时不可用
- `/api/interview/reports`（报告列表接口）持续返回 `503`
- 某个聚合接口依赖下游服务异常，导致短时间内连续 `500`

### 推荐行为

以“接口维度”维护熔断状态，而不是全局共用一个开关。

建议状态流转如下：

1. `closed`（关闭态）：正常放行请求
2. 连续失败达到阈值，例如 `3` 次后，进入 `open`（打开态）
3. `open`（打开态）期间直接拒绝请求，不再真正发出
4. 熔断窗口结束后进入 `half-open`（半开态）
5. 半开态只放一个探测请求
6. 探测成功则恢复为 `closed`（关闭态），失败则重新进入 `open`（打开态）

### 推荐默认配置

```ts
failureThreshold: 3
cooldownMs: 30000
halfOpenMaxRequests: 1
```

### 推荐熔断键

推荐使用“请求方法 + 标准化路径”作为熔断维度，例如：

```ts
GET /api/admin/dashboard
GET /api/interview/reports
```

这样做是为了避免一个异常接口把整个站点的请求都熔断掉。

### 哪些错误应计入失败

建议计入：

- `timeout`（超时）
- `network`（网络错误）
- 可重试 `5xx`（服务端错误）
- `429`（请求过多）

建议不计入：

- 用户主动取消
- `401`（未授权）
- `403`（禁止访问）
- 参数错误 `400`（错误请求）/ `422`（参数校验失败）

原因很简单：这些错误通常不是“服务不可用”，而是业务侧或用户侧输入问题，不应触发系统级熔断。

## 五、推荐实现落点

推荐新增或扩展以下结构：

```text
frontend/src/services/http/request.ts（统一请求客户端）
  -> createTokenBucket（创建令牌桶函数）或 TokenBucket（令牌桶类）
  -> resolveDefaultRetryConfig（默认重试配置解析函数）
  -> createCircuitBreakerRegistry（熔断器注册表函数）
  -> shouldCountAsCircuitFailure（是否计入熔断失败函数）
```

如果后续代码量继续增长，再考虑拆分为：

```text
frontend/src/services/http/
  request.ts（统一请求入口文件）
  request-token-bucket.ts（请求令牌桶文件）
  request-retry.ts（请求重试文件）
  request-circuit-breaker.ts（请求熔断文件）
```

当前阶段不建议一上来就拆目录。
如果 `request.ts`（统一请求工具）仍能保持清晰，可优先在单文件内完成第一版落地。

## 六、推荐的配置扩展

为 `AxiosRequestConfig`（请求配置类型）补充以下可选字段会更稳妥：

```ts
interface RequestConcurrencyConfig {
  enabled?: boolean
  queueTimeoutMs?: number
}

interface RequestCircuitBreakerConfig {
  enabled?: boolean
  failureThreshold?: number
  cooldownMs?: number
  halfOpenMaxRequests?: number
  key?: string
}
```

然后挂到请求配置中：

```ts
concurrency?: RequestConcurrencyConfig
circuitBreaker?: RequestCircuitBreakerConfig
```

这样调用方可以按接口做显式覆盖，但默认行为仍由请求层统一托管。

## 七、实施顺序建议

### Phase 1（第一阶段）：补齐全局默认重试

先把默认 `retry`（重试）策略统一到请求层。
这是最小改动、收益最高的一步，也最容易验证。

完成标准：

- `GET`（读取请求）默认自动应用统一重试策略
- 显式传 `retry`（重试配置）时仍以调用方为准
- 现有服务层不需要继续重复声明同类默认值

### Phase 2（第二阶段）：接入 `TokenBucket`（令牌桶）

把并发保护接到请求发起前。
重点验证“排队、释放、取消、超时”四条链路。

完成标准：

- 超过并发上限时会排队
- 请求完成后令牌一定释放
- 排队阶段取消不会残留僵尸队列项
- 队列超时能返回明确错误

### Phase 3（第三阶段）：接入 `circuit breaker`（熔断器）

最后补熔断，因为它的状态机最复杂，也最需要避免误判。

完成标准：

- 同一路径连续失败达到阈值后进入熔断
- 熔断窗口内快速失败，不再真实发请求
- 冷却结束后允许一次半开探测
- 探测成功可自动恢复

## 八、风险与边界

### 1. 风险：副作用请求被误重试

如果把 `POST`（创建或提交请求）默认也纳入自动重试，可能导致重复创建、重复生成或重复清空。
因此默认重试应优先限制在安全请求方法上。

### 2. 风险：令牌未释放造成假死

如果在异常链路、取消链路、文件下载链路中遗漏释放令牌，后续所有请求都可能卡在等待队列。
所以令牌释放必须走统一 `finally`（最终收尾）路径。

### 3. 风险：熔断粒度过粗

如果熔断键只按域名或模块区分，一个接口异常可能把整个后台模块都短暂封死。
因此建议至少细化到“方法 + 路径”。

### 4. 边界：不处理 `SSE`（服务端发送事件）流式链路

本方案只覆盖 `request.ts`（统一请求工具）所在的非流式 `REST API`（接口）层。
`fetch + ReadableStream`（请求加可读流） 这条流式链路仍由 `frontend/src/services/sse`（流式服务目录）单独维护，不纳入本次请求层熔断与并发队列。

---

## 九、验证清单

建议落地后至少验证以下场景：

| 场景 | 验证目标 |
|---|---|
| 普通 `GET` 请求短暂失败 | 能按默认 `retry`（重试）自动恢复 |
| `POST` 生成类请求失败 | 默认不自动重试，避免重复提交 |
| 多个后台页同时加载 | 总并发被 `TokenBucket`（令牌桶）限制住 |
| 排队中的请求被取消 | 可正确出队，不阻塞后续请求 |
| 同一路径连续 `503` | 达到阈值后进入 `circuit breaker`（熔断器）打开态 |
| 熔断冷却结束 | 允许半开探测请求 |
| 半开探测成功 | 熔断状态恢复正常 |
| 半开探测失败 | 重新进入熔断打开态 |

---

## 十、结论

如果项目要把 `request.ts`（统一请求工具）真正提升为“高可用请求层”，建议优先补齐三件事：

1. `TokenBucket`（令牌桶）并发控制，避免瞬时洪峰
2. 全局默认 `retry`（重试）策略，减少重复配置
3. `circuit breaker`（熔断器）能力，避免持续失败时雪崩

这三项能力都应放在统一请求层，而不是散落到各业务 `service`（服务层）文件中。
这样后续无论新增后台接口、面试会话接口还是报告接口，都能默认继承一致的高可用行为。
