# 后台管理模块：模型管理页面开发文档

## 一、目标

本方案用于在现有后台管理模块中新增模型管理页面，让管理员可以集中维护用户可选的大模型配置，减少用户手动填写模型、接口地址和密钥的成本。

核心目标：

- 管理员可以新增、编辑、停用和测试大模型配置。
- 用户在模拟面试、专项刷题、报告生成等场景中可以直接选择已启用模型。
- 系统可以配置默认模型，避免每次对话都手动填写模型参数。
- 第一阶段先完成后台配置与前端展示闭环，后续再让 `remote provider`（远程模型提供器）真实读取后台模型配置。
- 不修改 `POST /api/interview/stream`（模拟面试流式接口）现有 `SSE`（服务端发送事件）契约。

## 二、当前现状

项目当前已具备后台管理基础能力：

- `/admin/dashboard`（后台数据看板页）
- `/admin/sessions`（后台会话管理页）
- `/admin/reports`（后台报告管理页）
- `user/admin`（普通用户 / 管理员）角色区分
- `Bearer Token`（请求头令牌）鉴权
- `request.ts`（统一请求工具）请求收口

大模型调用现状：

- `backend/src/providers/mock-provider.ts`（本地模拟提供器）用于无真实模型时的演示。
- `backend/src/providers/remote-llm-provider.ts`（远程大模型提供器）读取 `INTERVIEW_REMOTE_BASE_URL`（远程模型基础地址）、`INTERVIEW_REMOTE_MODEL`（远程模型名）、`INTERVIEW_REMOTE_API_KEY`（远程模型密钥）等环境变量。
- `backend/src/providers/cpp-engine-provider.ts`（C++ 引擎提供器）为后续 C++ 服务接入预留。
- `frontend/.env.template`（前端环境变量模板）仍保留部分模型密钥示例，但正式模型密钥不应长期放在浏览器侧。

因此，模型管理页的长期方向应是：管理员在后台维护模型配置，后端负责安全存储与调用，前端只消费可选模型列表和默认模型标识。

## 三、第一阶段范围

### 1. 模型列表管理

模型管理页展示当前可用模型配置。

建议字段：

- `modelId`（模型标识）：模型配置唯一标识。
- `displayName`（展示名称）：给用户看的模型名称。
- `provider`（模型供应商）：例如 `openai-compatible`（OpenAI 兼容接口）、`deepseek`（DeepSeek 模型）、`moonshot`（月之暗面模型）、`qwen`（通义千问模型）、`ollama`（本地模型）。
- `baseUrl`（接口基础地址）：模型服务根地址。
- `modelName`（真实模型名）：真实请求里使用的模型名称。
- `apiKey`（接口密钥）：仅保存加密值或脱敏展示。
- `enabled`（启用状态）：控制用户侧是否可选择。
- `isDefault`（默认模型）：控制未手动选择时使用哪个模型。
- `supportsStream`（是否支持流式）：用于判断能否承接 `SSE`（服务端发送事件）对话。
- `enableThinking`（是否开启思考模式）：适配部分混合思考模型。
- `createdAt`（创建时间）与 `updatedAt`（更新时间）：用于审计和排序。

核心能力：

- 新增模型配置。
- 编辑模型名称、供应商、地址、模型名、启用状态。
- 更新 `apiKey`（接口密钥）时只允许重新输入，不回显原始密钥。
- 设置默认模型。
- 停用模型，停用后不出现在用户可选列表。
- 删除模型建议后置到第二阶段，第一阶段优先使用停用，降低误删风险。

### 2. 模型连通性测试

管理员保存模型前或保存后，可以执行一次轻量测试。

测试目标：

- 校验 `baseUrl`（接口基础地址）是否可访问。
- 校验 `apiKey`（接口密钥）是否有效。
- 校验 `modelName`（真实模型名）是否可用。
- 校验是否支持 `stream`（流式响应）。

建议测试接口：

```text
POST /api/admin/models/:modelId/test
```

测试接口只返回健康状态和错误摘要，不返回完整模型输出。

建议响应字段：

- `ok`（是否成功）
- `latencyMs`（延迟毫秒数）
- `message`（测试结果说明）
- `checkedAt`（检查时间）

### 3. 用户侧模型选择

用户不直接填写模型参数，而是从启用模型列表中选择。

建议用户侧只消费：

- `modelId`（模型标识）
- `displayName`（展示名称）
- `provider`（模型供应商）
- `isDefault`（是否默认）

用户侧不应拿到：

- `apiKey`（接口密钥）
- 完整未脱敏密钥
- 后端内部鉴权信息

第一阶段可以先只在后台完成模型配置页，不立即改模拟面试主链路。第二阶段再把模型选择接到会话启动、专项刷题和报告生成流程。

## 四、不做范围

第一阶段先不做以下内容：

- 不修改 `POST /api/interview/stream`（模拟面试流式接口）路径与 `chunk/done/error`（分片 / 完成 / 错误）事件格式。
- 不让浏览器直连第三方大模型接口。
- 不在前端保存真实 `apiKey`（接口密钥）。
- 不做复杂计费、额度、租户隔离和模型调用审计。
- 不支持普通用户自建公共模型，普通用户最多只能选择管理员启用的模型。
- 不把模型配置写入 `frontend/.env`（前端环境变量文件）作为运行时配置来源。

## 五、路由与导航设计

建议在现有后台路由下新增：

```text
/admin/models
```

路由职责：

- `/admin/models`（模型管理页）：管理模型配置、默认模型、启用状态和连通性测试。

建议路由元信息：

```ts
meta: {
  requiresAuth: true,
  requiredRole: 'admin',
  title: '模型管理',
}
```

后台侧边栏建议新增“模型管理”菜单项，位置可放在“报告管理”之后。

## 六、页面与组件拆分

建议落点：

```text
frontend/src/views/admin/models.vue（后台模型管理页）
frontend/src/services/admin/admin-model-api.ts（后台模型接口服务）
backend/src/types/model-config.ts（模型配置类型）
backend/src/storage/model-config-store.ts（模型配置存储）
backend/src/controllers/admin-model-controller.ts（后台模型控制器）
backend/src/routes/admin-model-routes.ts（后台模型路由）
```

第一阶段也可以先把接口挂入现有 `admin-routes.ts`（后台路由文件），等接口增多后再拆 `admin-model-routes.ts`（后台模型路由文件）。

页面职责：

- `models.vue`（后台模型管理页）：负责列表、筛选、表单弹窗、确认操作。
- `admin-model-api.ts`（后台模型接口服务）：负责请求后端模型接口。
- `model-config-store.ts`（模型配置存储）：负责持久化模型配置，不让页面直接读写文件。
- `RemoteLLMProvider`（远程大模型提供器）：后续只通过模型配置解析后的运行时参数发起调用。

状态边界：

- 表单草稿、弹窗开关、测试中状态留在页面组件内部。
- 模型配置真实数据以服务端存储为准。
- 默认模型只允许有一个，后端保存时必须保证唯一性。
- 用户侧模型选择只保存 `modelId`（模型标识），不复制模型完整配置。

## 七、后端接口草案

后台接口建议：

```text
GET /api/admin/models
POST /api/admin/models
PUT /api/admin/models/:modelId
PATCH /api/admin/models/:modelId/status
PATCH /api/admin/models/:modelId/default
POST /api/admin/models/:modelId/test
```

用户侧只读接口建议：

```text
GET /api/models/enabled
```

接口说明：

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/admin/models` | 管理员查看全部模型配置 |
| `POST` | `/api/admin/models` | 新增模型配置 |
| `PUT` | `/api/admin/models/:modelId` | 编辑模型配置 |
| `PATCH` | `/api/admin/models/:modelId/status` | 启用或停用模型 |
| `PATCH` | `/api/admin/models/:modelId/default` | 设置默认模型 |
| `POST` | `/api/admin/models/:modelId/test` | 测试模型连通性 |
| `GET` | `/api/models/enabled` | 用户侧读取可选模型列表 |

注意：

- 管理员接口必须经过 `requireAuth`（登录校验中间件）和 `requireRole('admin')`（管理员角色校验）。
- 用户侧接口只返回启用模型的安全字段。
- 写接口默认不启用自动重试，避免重复创建或重复更新。

## 八、数据结构建议

服务端保存结构建议：

```ts
export type ModelProvider =
  | 'openai-compatible'
  | 'deepseek'
  | 'moonshot'
  | 'qwen'
  | 'ollama'
  | 'custom'

export interface ModelConfig {
  modelId: string
  displayName: string
  provider: ModelProvider
  baseUrl: string
  modelName: string
  encryptedApiKey?: string
  apiKeyPreview?: string
  enabled: boolean
  isDefault: boolean
  supportsStream: boolean
  enableThinking: boolean
  createdAt: string
  updatedAt: string
  lastTestedAt?: string
  lastTestStatus?: 'success' | 'failed'
  lastTestMessage?: string
}
```

说明：

- `encryptedApiKey`（加密后的接口密钥）只在后端保存。
- `apiKeyPreview`（密钥脱敏预览）只展示前后少量字符，例如 `sk-***abcd`。
- `isDefault`（默认模型）由后端强约束唯一。
- `supportsStream`（是否支持流式）决定能否接入模拟面试流式反馈。

## 九、密钥安全边界

模型管理页必须把 `apiKey`（接口密钥）当作敏感信息处理。

基础要求：

- 前端表格只展示脱敏密钥。
- 编辑模型时不回填原始密钥。
- 更新密钥时使用单独输入框，留空代表不修改。
- 后端日志不打印原始密钥。
- 后端返回值不包含原始密钥。
- 后续正式环境应使用服务端加密密钥，避免明文落盘。

第一阶段如果仍使用 `backend/data/*.json`（后端 JSON 数据文件）保存配置，也必须明确这是本地演示方案；正式部署前应迁移到数据库或安全密钥服务。

## 十、与大模型调用链路的关系

推荐分两步接入：

### Phase 1：配置管理闭环

先完成后台模型配置的增删改查和测试能力。

这一阶段不改 `RemoteLLMProvider`（远程大模型提供器）的运行时取值，仍可继续读取环境变量，保证主链路稳定。

### Phase 2：运行时读取默认模型

让 `RemoteLLMProvider`（远程大模型提供器）从后端模型配置中读取默认模型。

推荐新增一个解析函数：

```ts
resolveActiveModelConfig（解析当前可用模型配置函数）
```

解析优先级建议：

1. 请求显式携带的 `modelId`（模型标识）
2. 后台设置的默认模型
3. 环境变量 `INTERVIEW_REMOTE_*`（远程模型环境变量）兜底

这样可以兼容旧配置，也能逐步迁移到后台模型管理。

### Phase 3：用户按场景选择模型

在模拟面试、专项刷题、报告生成等入口增加模型选择。

注意：如果要把 `modelId`（模型标识）加入 `POST /api/interview/stream`（模拟面试流式接口）请求体，需要走前后端与 C++ 协作变更流程，不能单方修改冻结契约。

## 十一、实施步骤

### Phase 1：后台页面与配置存储

目标：管理员可以维护模型配置。

- 新增 `/admin/models`（模型管理页）路由。
- 后台侧边栏新增模型管理入口。
- 新增模型列表、筛选、启用状态和默认模型标记。
- 新增模型新增 / 编辑弹窗。
- 新增后端模型配置存储。
- 新增后台模型管理接口。
- 新增模型连通性测试接口。

### Phase 2：默认模型接入远程调用

目标：系统不用再依赖手动环境变量填写模型。

- 新增 `resolveActiveModelConfig`（解析当前可用模型配置函数）。
- `RemoteLLMProvider`（远程大模型提供器）优先读取后台默认模型。
- 环境变量保留为兜底。
- 模型配置缺失时返回明确错误提示。
- 后台显示当前默认模型是否可用。

### Phase 3：用户侧模型选择

目标：用户可以按场景选择启用模型。

- 新增 `GET /api/models/enabled`（启用模型列表接口）。
- 用户侧模型选择只保存 `modelId`（模型标识）。
- 模拟面试入口显示默认模型和可选模型。
- 专项刷题、报告生成逐步接入模型选择。
- 如需修改流式接口请求体，先更新协作文档并完成三方确认。

### Phase 4：审计与运维增强

目标：让模型管理适合长期维护。

- 记录模型配置变更日志。
- 记录模型测试结果历史。
- 增加模型调用失败统计。
- 增加模型供应商标签和备注。
- 增加模型排序和推荐标记。

## 十二、验证清单

功能验证：

- 管理员可以进入 `/admin/models`（模型管理页）。
- 普通用户不能进入模型管理页。
- 可以新增模型配置。
- 可以编辑模型展示名称、地址、模型名和启用状态。
- 更新密钥后列表只显示脱敏值。
- 设置一个默认模型后，其他模型自动取消默认。
- 停用默认模型时必须要求重新选择默认模型，或后端自动选择下一个可用模型。
- 模型测试成功时显示延迟与成功状态。
- 模型测试失败时显示可理解错误，不暴露密钥。

安全验证：

- 后端响应不包含原始 `apiKey`（接口密钥）。
- 浏览器开发者工具看不到原始密钥。
- 后端日志不输出原始密钥。
- 非管理员访问后台模型接口返回 `403`（禁止访问）。
- 未登录访问后台模型接口返回 `401`（未授权）。

边界验证：

- 不影响现有后台会话管理和报告管理。
- 不影响 `POST /api/interview/stream`（模拟面试流式接口）现有事件格式。
- 未配置模型时，`mock provider`（本地模拟提供器）仍可用于演示。
- 配置错误模型时，主链路能给出明确错误，而不是页面白屏。

## 十三、风险与处理

### 风险 1：密钥泄露

影响：第三方模型额度被盗用，正式环境风险高。

处理：

- 原始密钥只允许后端接收和存储。
- 前端永远只展示脱敏密钥。
- 正式环境引入加密落盘或密钥服务。

### 风险 2：模型配置影响主链路

影响：管理员配置错误后，模拟面试无法正常反馈。

处理：

- 默认模型切换前必须允许测试。
- `RemoteLLMProvider`（远程大模型提供器）保留环境变量兜底。
- 后台显示最近一次测试状态。

### 风险 3：流式契约被误改

影响：前端、Node.js（后端运行时）和 C++（后端语言）联调失败。

处理：

- 第一阶段不改流式请求体。
- 需要传 `modelId`（模型标识）时，先走协作文档规定的契约变更流程。

## 十四、推荐优先级

建议优先顺序：

1. 新增后台模型管理页与导航入口。
2. 新增后台模型配置接口和后端存储。
3. 做好密钥脱敏与默认模型唯一性。
4. 增加模型连通性测试。
5. 让 `RemoteLLMProvider`（远程大模型提供器）读取后台默认模型。
6. 用户侧再接入模型选择。

这样可以先把配置能力做稳，再逐步接入真实调用链路，避免模型管理页和流式对话主链路同时大改。
