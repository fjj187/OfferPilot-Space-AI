> 适用对象：前端（浏览器端）、Node.js（服务端 JavaScript 运行时）网关、C++（后端语言）服务重构同学  
> 基准代码时间：2026-06-08  
> 基准代码范围：`frontend/`（前端目录） + `backend/`（Node 网关目录）  
> 说明：本文基于当前真实代码抽取，重点覆盖模拟面试主链路、历史会话、报告生成、弱项补练题池，以及最关键的 `SSE`（服务端发送事件）流式推送协议。

## 1. 整体业务场景与数据流说明

### 1.1 业务场景

当前服务端主要承接的是一套“模拟面试 + 流式反馈 + 历史沉淀 + 报告生成 + 弱项补练”的业务闭环，核心场景如下：

1. 前端发起模拟面试回答请求。
2. 服务端基于当前题目、资料上下文、用户回答，实时生成面试官反馈。
3. 服务端通过 `SSE`（服务端发送事件）持续推送回答片段，前端边收边渲染。
4. 本轮面试结束后，前端再请求生成报告。
5. 服务端基于会话记录和报告内容生成弱项补练题池。

### 1.2 当前系统角色

| 角色 | 职责说明 |
|------|----------|
| 前端（Vue 3（前端框架）） | 负责页面交互、发起 HTTP（超文本传输协议）请求、接收 `SSE`（服务端发送事件）流 |
| Node 网关（Express（Node 服务框架）） | 统一对外 API（接口）入口、参数校验、会话/报告存储、转发流式结果 |
| C++ 服务 | 未来重构后的核心推理服务、资料处理服务、流式反馈服务，由 Node 网关内部调用 |

### 1.3 完整数据流

#### 场景 A：模拟面试流式反馈

```text
前端提交回答
  -> POST /api/interview/stream
  -> Node 校验请求参数
  -> Node 调用 provider（当前可为 mock / remote / 未来 cpp）
  -> provider 持续产出 chunk（片段）
  -> Node 以 SSE 事件格式写回 chunk / done / error
  -> 前端边接收边渲染消息
  -> Node 在流结束后把会话消息写入本地存储
```

#### 场景 B：报告生成

```text
前端结束本轮面试
  -> POST /api/interview/reports/generate
  -> Node 从已存储会话中聚合本轮内容
  -> 生成报告摘要 JSON（结构化数据）
  -> 返回 report（报告对象）给前端
  -> 同时持久化到 backend/data/interview-reports.json
```

#### 场景 C：弱项补练题池

```text
前端在报告页点击生成补练题
  -> POST /api/interview/practice-pool/generate
  -> Node 根据报告、弱项标签、资料上下文生成题池
  -> 返回 pool（题池对象）
  -> 前端用该题池继续进入下一轮训练
```

## 2. 前端主动请求接口契约（HTTP 部分）

## 2.1 通用约定

### 基础信息

| 项目 | 当前约定 |
|------|----------|
| 服务基地址（开发环境） | 前端访问 `http://localhost:2048/api/*`，由 `Vite`（前端构建工具）代理到 `http://localhost:3030/api/*` |
| 服务基地址（Node 实际监听） | `http://localhost:3030` |
| 请求体格式 | `application/json; charset=utf-8` |
| 响应格式 | 普通接口返回 JSON（结构化数据）；流式接口返回 `text/event-stream; charset=utf-8` |
| 鉴权方式 | 当前版本无鉴权，无 `Token`（令牌）、无 `Cookie`（会话凭证）依赖 |
| 字符编码 | UTF-8（统一编码格式） |

### 通用错误响应结构

```json
{
  "code": "INVALID_REQUEST",
  "message": "sessionId is required."
}
```

字段说明：

| 字段 | 类型 | 含义 | 示例值 |
|------|------|------|--------|
| `code` | `string`（字符串） | 错误码 | `"INVALID_REQUEST"` |
| `message` | `string`（字符串） | 错误说明 | `"sessionId is required."` |

### 正常响应

| 项目 | 内容 |
|------|------|
| HTTP 状态码 | `200 OK` |
| 响应形式 | `SSE`（服务端发送事件）流 |

具体推送格式见下文“第 3 部分 服务端主动推送消息契约”。

### 错误响应

#### 非流式校验错误

当必填字段缺失时，服务端直接返回普通 JSON（结构化数据）错误，而不会进入流式推送：

| HTTP 状态码 | 错误码 | 含义 |
|------------|--------|------|
| `400 Bad Request` | `INVALID_REQUEST` | 请求字段缺失或非法 |

示例：

```json
{
  "code": "INVALID_REQUEST",
  "message": "sessionId is required."
}
```

#### 流式过程中出错

流已建立后，服务端通过 `error` 事件推送：

```json
{
  "code": "INTERNAL_STREAM_ERROR",
  "message": "Unknown stream error."
}
```

## 3. 服务端主动推送消息契约（推送部分）

## 3.1 推送方式

当前主链路使用 `SSE`（服务端发送事件），不是 `WebSocket`（全双工长连接协议），也不是普通 HTTP 长轮询。

### 选择 `SSE`（服务端发送事件）的原因

1. 当前场景是服务端单向持续输出文本片段，天然适合 `SSE`（服务端发送事件）。
2. 前端已按 `event + data` 的标准 `SSE`（服务端发送事件）帧解析。
3. 现有 Node 网关实现、前端消费实现、后续 C++ 重构目标都应保持该契约不变。

---

## 3.2 推送消息格式

> 这一节是给 C++（后端语言）同学最需要重点对齐的部分。  
> 结论先写在前面：**前端实际识别的是 `SSE`（服务端发送事件）事件名 + JSON（结构化数据）字段内容，必须保持下面格式不变。**

### 3.2.1 统一消息模型

虽然底层传输是 `SSE`（服务端发送事件）帧，但从协议理解上，可以统一看成下面这 3 类消息：

| 消息类型 | `event`（事件名） | JSON 字段 | 用途 |
|----------|-------------------|-----------|------|
| 文本增量消息 | `chunk` | `content` | 推送一段新的回答文本 |
| 结束消息 | `done` | 无 | 表示本次回答已经结束 |
| 异常消息 | `error` | `code`、`message` | 表示本次回答失败 |

### 3.2.2 统一 JSON 结构总表

#### 结构 A：`chunk`（文本增量消息）

```json
{
  "content": "你提到了 Proxy，但还可以继续说明依赖收集。"
}
```

| 字段名 | 数据类型 | 含义 | 示例值 |
|--------|----------|------|--------|
| `content` | `string`（字符串） | 当前这一帧新增的文本片段，前端会直接拼接到现有消息末尾 | `"你提到了 Proxy，但还可以继续说明依赖收集。"` |

#### 结构 B：`done`（结束消息）

```json
{}
```

| 字段名 | 数据类型 | 含义 | 示例值 |
|--------|----------|------|--------|
| 无 | 无 | 当前协议下 `done` 事件只表示“结束”，不携带业务字段 | 无 |

#### 结构 C：`error`（异常消息）

```json
{
  "code": "REMOTE_STREAM_ERROR",
  "message": "Upstream model timeout."
}
```

| 字段名 | 数据类型 | 含义 | 示例值 |
|--------|----------|------|--------|
| `code` | `string`（字符串） | 错误码，供前端区分错误类型 | `"REMOTE_STREAM_ERROR"` |
| `message` | `string`（字符串） | 错误说明，供前端提示用户 | `"Upstream model timeout."` |

### 3.2.3 前端真实消费规则

前端当前是按下面规则解析的，这部分不要改：

1. 如果 `event`（事件名）是 `chunk`，前端读取 JSON（结构化数据）里的 `content` 字段。
2. 如果 `event`（事件名）是 `done`，前端认为本轮流式输出结束。
3. 如果 `event`（事件名）是 `error`，前端读取 JSON（结构化数据）里的 `code` 和 `message` 字段。
4. `chunk` 的字段名必须是 `content`，不能改成 `text`、`delta`、`message`、`token` 等其他名字。
5. `error` 的字段名必须是 `code` 和 `message`，不要改成 `errorCode`、`msg` 等其他名字。

### 传输层格式

服务端返回头：

```http
Content-Type: text/event-stream; charset=utf-8
Cache-Control: no-cache, no-transform
Connection: keep-alive
```

服务端按如下格式输出：

```text
event: chunk
data: {"content":"你提到了 Proxy，"}

event: chunk
data: {"content":"但还可以继续说明依赖收集。"}

event: done
data: {}
```

下面是一轮正常回答的完整推送示例：

```text

event: chunk
data: {"content":"但还可以继续说明依赖收集。"}

data: {"content":"如果继续往下讲，最好补充 trigger 的触发链路。"}
```

```text

```

#### 事件 1：`chunk`

含义：推送一段新的文本内容，前端直接拼接展示。

| 字段 | 类型 | 含义 | 示例值 |

含义：本次流式回复正常结束。

## 3.3 推送触发条件

### `chunk`（片段事件）触发时机

满足以下任一条件即可向前端持续推送：

1. 模型生成出新的文本片段。
2. C++ 服务内部按固定分块粒度产出文本。
3. 下游流式模型返回新的增量内容。

### `done`（结束事件）触发时机

以下情况应推送 `done`：

1. 当前回答完整生成结束。
2. 服务端确认不会再有新文本片段。

### `error`（错误事件）触发时机

以下情况应推送 `error`：

1. 请求参数合法，但流式处理中途失败。
2. 下游模型服务异常。
3. C++ 服务超时、崩溃、格式解析失败。

## 3.4 心跳 / 重连机制

### 当前线上代码现状

当前实现中：

1. 没有应用层心跳包。
2. 没有专门的心跳响应事件。
3. 前端没有内建自动重连逻辑。
4. 前端支持用户主动中止，底层使用 `AbortController`（中止控制器）。

### 当前推荐处理方式

#### 客户端主动取消

前端在用户点击“停止生成”时，会主动中断连接。  
这类中断不应视为错误告警。

#### 超时处理建议

当前协议未固定超时字段，建议 C++ 重构时内部约定：

1. 单次流式请求服务端超时建议先定为 `60s` 到 `120s`。
2. 若超时，返回 `error` 事件，例如：

```json
{
  "code": "STREAM_TIMEOUT",
  "message": "Interview stream timeout after 120 seconds."
}
```

#### 自动重连建议

由于当前业务是“单次问答请求 -> 单次回复流”，不是持续订阅型流，所以**不建议前端对同一次回答自动重连**。  
如果连接断开，建议由用户重新提交当前回答，避免重复生成、重复落库或上下文错乱。

#### 心跳建议

如果 C++ 服务后续确实需要长耗时推理，建议内部保留两种方案二选一：

1. 保持当前协议不变，不额外发送心跳，只要持续有 `chunk` 即可。
2. 新增注释型心跳帧（例如 `: ping`），但**不要影响现有 `chunk` / `done` / `error` 契约**。

| 流式序列化方式 | `SSE`（服务端发送事件）帧 + `data: JSON字符串` |
| 中文内容 | 允许直接传输 UTF-8 中文，不需要转 GBK（国标编码）等本地编码 |

---

## 4.3 时间戳格式

当前后端统一使用：

| 项目 | 约定 |
|------|------|
| 时间格式 | `ISO 8601`（国际标准时间字符串） |
| 时区 | UTC（世界协调时间） |
| 精度 | 毫秒级 |
| 示例 | `"2026-06-08T11:00:00.000Z"` |

### C++ 实现建议

1. 统一由服务端生成 UTC（世界协调时间）时间字符串。
2. 不建议返回本地时区时间字符串，例如 `"2026/06/08 19:00:00"`。
3. 所有存储落盘时间字段建议统一为 `createdAt`（创建时间）和 `updatedAt`（更新时间）。

## 5.2 联调步骤建议

### 建议顺序

1. 先启动 Node 网关，确认 `GET /api/health` 正常。
2. 先用假数据联通普通 HTTP 接口，例如 `reports/generate`、`practice-pool/generate`。
3. 再联通最核心的 `POST /api/interview/stream` 流式接口。
4. 确认 `chunk`、`done`、`error` 三类事件都能按约定输出。
5. 最后再接入真实前端页面验证中止、异常、连续问答等场景。

### C++ 重构接入建议

建议采用下面的分层方式：

```text
前端 -> Node 网关（对外协议不变） -> C++ 内部服务（实现可重构）
```

不建议：

1. 前端直接调用 C++ 服务。
2. C++ 服务直接修改对外 URL（地址）、字段名、事件名。
3. 为了方便 C++ 内部实现而改动前端现有消费协议。

## 5.3 异常问题排查方式

### 问题 1：前端收不到流

排查顺序：

1. 检查请求是否真正打到 `/api/interview/stream`。
2. 检查响应头是否是 `text/event-stream`。
3. 检查返回内容是否包含双换行分隔的 `SSE`（服务端发送事件）帧。
4. 检查事件名是否仍为 `chunk` / `done` / `error`。

### 问题 2：前端能连上，但 UI（用户界面）不更新

重点排查：

1. `chunk` 事件的 JSON 字段是否仍叫 `content`。
2. 是否误返回成一次性完整 JSON，而不是多帧 `SSE`（服务端发送事件）。
3. 是否在服务端输出中混入了额外非 `SSE`（服务端发送事件）文本。

### 问题 3：流中途断开

重点排查：

1. C++ 下游是否超时或崩溃。
2. Node 网关是否中途关闭连接。
3. 是否因用户主动点击“停止生成”触发了正常中断。

### 问题 4：报告或题池字段对不上

重点排查：

1. 字段名是否与本文档完全一致。
2. 枚举值是否被中文化或改名。
3. 时间字段是否仍为 UTC（世界协调时间）字符串。
4. 数组字段是否被错误返回为单对象或空字符串。

### 问题 5：中文乱码

重点排查：

1. 服务端输出是否统一 UTF-8。
2. C++ JSON（结构化数据）序列化是否按 UTF-8 编码。
3. 落盘或日志环节是否误用了本地编码。

## 6. 对 C++ 后端同学的明确结论

1. 对外协议请继续由 Node 网关承接，前端只认 `/api/*`。
2. 最关键的重构接口是 `POST /api/interview/stream`，其对外返回必须继续保持 `SSE`（服务端发送事件）形式。
3. `SSE`（服务端发送事件）事件名固定为 `chunk` / `done` / `error`，其中 `chunk` 的数据字段固定为 `content`。
4. 当前没有鉴权、没有应用层心跳、没有自动重连，这三点如果后续要增强，请先保持兼容，不要直接改掉现有契约。
5. 如果只是重写底层实现，前端原则上不需要改代码；如果要改字段或事件名，必须先同步前端一起改契约。
