# 后台管理模块：模型管理页面后续开发文档

## 一、文档定位

本文档承接 [`docs/37.后台管理模块-模型管理页面开发文档.md`](./37.后台管理模块-模型管理页面开发文档.md)（后台模型管理页面开发文档），用于规划模型管理页的后续开发。

当前仓库已经完成第一阶段闭环：

- `/admin/models`（后台模型管理页）已经接入后台侧边栏。
- `frontend/src/views/admin/models.vue`（后台模型管理页文件）已经具备模型列表、新增、编辑、启停、设为默认和连通性测试能力。
- `frontend/src/services/admin/admin-api.ts`（后台接口服务文件）已经收口模型管理请求。
- `backend/src/types/model-config.ts`（模型配置类型文件）已经定义模型配置数据结构。
- `backend/src/storage/model-config-store.ts`（模型配置存储文件）已经提供本地 JSON（JavaScript Object Notation，轻量数据交换格式）持久化能力。
- `backend/src/services/admin-service.ts`（后台服务文件）已经处理默认模型唯一性、密钥脱敏和模型测试结果回写。

因此，40 号文档不重复规划第一阶段页面，而是定义下一阶段如何把模型配置安全、渐进地接入真实大模型运行链路。

## 二、当前边界

### 1. 已完成能力

模型管理页目前已经能满足管理员维护模型配置的基本需求：

- 管理员可以查看全部模型配置。
- 管理员可以新增和编辑模型配置。
- 管理员可以启用或停用模型。
- 管理员可以设置唯一默认模型。
- 管理员可以发起模型连通性测试。
- 前端只展示 `apiKeyPreview`（接口密钥脱敏预览），不回显原始 `apiKey`（接口密钥）。

### 2. 尚未完成能力

后续仍需要补齐以下能力：

- `RemoteLLMProvider`（远程大模型提供器）尚未读取后台默认模型配置。
- 用户侧尚未提供 `GET /api/models/enabled`（启用模型列表接口）。
- 模拟面试主流式链路已补齐 `modelId`（模型标识）请求透传，`RemoteLLMProvider`（远程大模型提供器）会按当前选择的模型解析运行时配置；当前仍未把该字段下沉到 `cpp`（C++ 引擎）提供器消费。
- 模型调用失败、测试历史、配置变更尚未形成审计记录。
- `encryptedApiKey`（加密后的接口密钥）当前仍是本地演示级处理，正式环境需要迁移到更安全的密钥方案。

### 3. 冻结契约

后续开发必须继续遵守 [`docs/前后端与C++协作对齐.md`](./前后端与C++协作对齐.md)（前后端与 C++ 协作对齐文档）：

- 不单方修改 `POST /api/interview/stream`（模拟面试流式接口）路径。
- 不单方修改 `chunk/done/error`（分片 / 完成 / 错误）等 `SSE`（服务端发送事件）载荷契约。
- 不让前端直连第三方模型接口或 C++（后端语言）服务端口。
- `POST /api/interview/stream`（模拟面试流式接口）请求体现已扩展可选 `modelId`（模型标识），但 `SSE`（服务端发送事件）`chunk/done/error`（分片/完成/错误）事件结构保持不变；后续如需让 `cpp`（C++ 引擎）链路也消费该字段，仍需继续走三方协作变更流程。

## 三、后续目标

后续开发目标分三层推进。

### 1. 配置可用

后台维护的默认模型可以被后端运行时读取，管理员不用再依赖手工修改环境变量切换远程模型。

### 2. 选择可见

普通用户在业务入口可以看到已启用模型和默认模型，但只能拿到安全字段，不能拿到 `baseUrl`（接口基础地址）和 `apiKey`（接口密钥）。

### 3. 调用可控

模拟面试、专项刷题和报告生成逐步支持指定模型，同时保留环境变量兜底和 `mock provider`（本地模拟提供器）演示能力，避免错误配置直接打断主链路。

## 四、总体架构

建议后续保持单网关结构：

```text
frontend/（前端）
  ↓ /api/*
backend/（Node 网关）
  ↓ resolveActiveModelConfig（解析当前可用模型配置函数）
RemoteLLMProvider（远程大模型提供器）
  ↓
第三方大模型服务
```

职责边界：

| 层级 | 职责 |
|---|---|
| `frontend`（前端） | 展示可选模型、提交用户选择的 `modelId`（模型标识） |
| `backend`（Node 网关） | 校验模型权限、解析模型配置、隐藏密钥、调用模型服务 |
| `model-config-store`（模型配置存储） | 保存后台模型配置，不承接调用逻辑 |
| `RemoteLLMProvider`（远程大模型提供器） | 只消费解析后的运行时配置，不直接暴露存储细节 |

## 五、Phase 2：默认模型接入远程调用

### 1. 目标

让 `RemoteLLMProvider`（远程大模型提供器）优先使用后台设置的默认模型。

### 2. 推荐新增文件

```text
backend/src/services/model-config-resolver.ts
```

建议职责：

- 读取全部模型配置。
- 查找请求指定的 `modelId`（模型标识）。
- 查找已启用默认模型。
- 回退到 `INTERVIEW_REMOTE_*`（远程模型环境变量）。
- 统一返回运行时模型配置。

### 3. 解析优先级

推荐使用以下优先级：

1. 请求显式携带的 `modelId`（模型标识）。
2. 后台启用状态下的默认模型。
3. `INTERVIEW_REMOTE_BASE_URL`（远程模型基础地址）、`INTERVIEW_REMOTE_MODEL`（远程模型名）、`INTERVIEW_REMOTE_API_KEY`（远程模型密钥）环境变量。
4. 无可用模型时返回明确错误。

### 4. 运行时配置类型

建议新增内部类型：

```ts
export interface RuntimeModelConfig {
  modelId?: string
  displayName: string
  provider: ModelProvider
  baseUrl: string
  modelName: string
  apiKey?: string
  supportsStream: boolean
  enableThinking: boolean
  source: 'admin' | 'env'
}
```

说明：

- `RuntimeModelConfig`（运行时模型配置）只允许在后端内部流转。
- `apiKey`（接口密钥）不得返回给前端。
- `source`（配置来源）用于日志和排查，不进入用户侧公开接口。

### 5. 不改内容

本阶段仍不修改 `POST /api/interview/stream`（模拟面试流式接口）请求体。默认模型接入只改变后端选择远程模型配置的来源。

## 六、Phase 3：用户侧启用模型列表

### 1. 目标

让普通用户可以读取已启用模型列表，用于后续业务入口展示。

### 2. 建议接口

```text
GET /api/models/enabled
```

建议响应字段：

```ts
export interface EnabledModelConfigDTO {
  modelId: string
  displayName: string
  provider: ModelProvider
  isDefault: boolean
}
```

### 3. 安全约束

用户侧接口禁止返回：

- `baseUrl`（接口基础地址）
- `modelName`（真实模型名）
- `apiKey`（接口密钥）
- `encryptedApiKey`（加密后的接口密钥）
- `apiKeyPreview`（接口密钥脱敏预览）
- `lastTestMessage`（最近测试结果说明）

原因：用户只需要选择模型，不需要理解或持有模型调用参数。

### 4. 前端建议落点

```text
frontend/src/services/model/model-api.ts
frontend/src/composables/model/useEnabledModels.ts
```

说明：

- `model-api.ts`（模型接口服务文件）只负责请求用户侧模型列表。
- `useEnabledModels.ts`（启用模型组合函数）负责加载状态、错误状态和默认模型派生。
- 不把模型选择状态混入后台 `admin-api.ts`（后台接口服务文件）。

## 七、Phase 4：业务入口模型选择

### 1. 接入顺序

建议按风险从低到高接入：

1. 报告生成入口。
2. 专项刷题生成入口。
3. 模拟面试流式反馈入口。

原因：

- 报告生成和专项刷题通常是非流式 `REST API`（表述性状态转移接口）链路，契约变更风险较低。
- 模拟面试流式反馈涉及 `SSE`（服务端发送事件）和 C++（后端语言）协作，必须最后接入。

### 2. 用户侧状态归属

推荐规则：

- 当前页面临时选择的模型留在组件或组合函数内部。
- 需要跨页面恢复的模型选择只保存 `modelId`（模型标识）。
- 不复制完整模型配置到 `localStorage`（浏览器本地存储）。
- 不把 `apiKey`（接口密钥）或 `baseUrl`（接口基础地址）写入任何浏览器持久化。

### 3. 模拟面试流式接口接入现状

当前仓库已完成前端与 Node（网关）侧最小接入，现状如下：

- `backend/src/types/interview.ts`（后端面试流类型文件）与 `frontend/src/services/sse/sse-types.ts`（前端流式类型文件）均已补齐可选 `modelId`（模型标识）。
- `frontend/src/composables/showcase/useMockInterviewSpaceMockState.ts`（宇宙模拟面试状态组合函数）发起主面试流式请求时，已透传当前 `selectedModelId`（当前选中模型标识）。
- `backend/src/providers/remote-llm-provider.ts`（远程大模型提供器）已改为 `resolveActiveModelConfig(request.modelId)`（按请求模型标识解析当前可用模型配置函数），因此 `remote`（远程模型）模式会真正按所选模型运行。
- `POST /api/interview/stream`（模拟面试流式接口）的路径未变，`SSE`（服务端发送事件）`chunk/done/error`（分片/完成/错误）结构未变。

后续若要继续扩展到 `cpp`（C++ 引擎）链路，仍必须完成：

- 更新冻结契约说明。
- 更新 `backend/src/types/interview.ts`（后端面试类型文件）。
- 更新 `frontend/src/services/sse/sse-types.ts`（前端流式类型文件）。
- 确认 C++（后端语言）服务是否需要消费或透传该字段。
- 同时验证 `mock`（本地模拟）、`remote`（远程模型）和 `cpp`（C++ 引擎）三种 `Provider`（提供器）模式。

## 八、页面体验优化

模型管理页后续可以补齐以下管理体验。

### 1. 列表信息增强

建议新增：

- 最近测试时间。
- 最近测试状态。
- 当前默认模型摘要。
- 启用模型数量。
- 支持流式模型数量。
- 供应商筛选。
- 测试失败筛选。

### 2. 表单校验增强

建议增强：

- `baseUrl`（接口基础地址）必须是合法 `http`（超文本传输协议）或 `https`（安全超文本传输协议）地址。
- `modelName`（真实模型名）不能为空。
- `displayName`（展示名称）同一供应商下不建议重复。
- `ollama`（本地模型）可允许无 `apiKey`（接口密钥）。
- 远程供应商默认要求填写 `apiKey`（接口密钥）。

### 3. 操作反馈增强

建议补齐：

- 测试中禁用重复点击。
- 测试失败展示可理解原因。
- 停用默认模型时提示默认模型会自动切换或被清空。
- 新增模型后默认回到列表顶部或高亮新增行。

## 九、密钥安全升级

当前 `encryptedApiKey`（加密后的接口密钥）仍属于本地演示级处理。后续正式化建议分三步。

### 1. 短期

- 后端响应永远不返回原始 `apiKey`（接口密钥）。
- 后端日志禁止打印原始 `apiKey`（接口密钥）。
- 连通性测试错误信息需要清洗密钥片段。

### 2. 中期

- 使用服务端加密密钥对 `apiKey`（接口密钥）加密落盘。
- 加密密钥从环境变量读取。
- 本地开发允许降级为演示存储，但必须有明显注释。

### 3. 长期

- 迁移到数据库或密钥管理服务。
- 支持密钥轮换。
- 支持操作审计。
- 支持按环境区分开发、测试和生产模型配置。

## 十、审计与观测

建议后续新增轻量审计能力：

| 事件 | 建议记录字段 |
|---|---|
| 新增模型 | 操作者、模型标识、供应商、时间 |
| 编辑模型 | 操作者、模型标识、变更字段、时间 |
| 启停模型 | 操作者、模型标识、目标状态、时间 |
| 设置默认模型 | 操作者、旧默认模型、新默认模型、时间 |
| 连通性测试 | 操作者、模型标识、结果、延迟、时间 |
| 调用失败 | 模型标识、供应商、错误类型、时间 |

建议新增存储文件：

```text
backend/src/storage/admin-audit-store.ts
```

第一阶段只记录后台模型管理操作即可，不急于把所有业务调用都纳入审计。

## 十一、异常策略

### 1. 默认模型不可用

处理建议：

- 如果后台默认模型已停用，则查找下一个启用模型。
- 如果没有启用模型，则回退环境变量。
- 如果环境变量也不可用，则返回明确错误。

### 2. 模型测试失败

处理建议：

- 不自动停用模型。
- 更新最近测试失败状态。
- 页面给出错误摘要。
- 不阻止管理员手动保存配置。

### 3. 调用运行失败

处理建议：

- `RemoteLLMProvider`（远程大模型提供器）返回统一错误。
- 前端展示通用可理解提示。
- 后端记录模型标识和供应商，但不记录密钥。
- 不自动切换到其他模型继续生成，避免一次请求产生不可预测结果。

## 十二、验证清单

### Phase 2 验证

- 后台设置默认模型后，`RemoteLLMProvider`（远程大模型提供器）能读取该配置。
- 默认模型停用后，不再被运行时选中。
- 没有后台默认模型时，环境变量仍可兜底。
- 无可用远程模型时，错误提示明确。
- `POST /api/interview/stream`（模拟面试流式接口）请求体和 `SSE`（服务端发送事件）事件格式未变化。

### Phase 3 验证

- `GET /api/models/enabled`（启用模型列表接口）只返回启用模型。
- 用户侧响应不包含 `apiKey`（接口密钥）、`baseUrl`（接口基础地址）和 `modelName`（真实模型名）。
- 默认模型在用户侧列表中唯一。
- 没有启用模型时，前端能展示空状态。

### Phase 4 验证

- 报告生成入口可以选择模型。
- 专项刷题入口可以选择模型。
- 模拟面试入口已可透传 `modelId`（模型标识）且 `remote`（远程模型）模式按所选模型运行。
- `mock`（本地模拟）、`remote`（远程模型）和 `cpp`（C++ 引擎）模式均可按预期运行。

### 安全验证

- 浏览器开发者工具看不到原始 `apiKey`（接口密钥）。
- 后端日志不输出原始 `apiKey`（接口密钥）。
- 普通用户无法访问 `/api/admin/models`（后台模型管理接口）。
- 未登录用户无法访问后台模型管理接口。
- 连通性测试失败信息不会泄露密钥片段。

## 十三、推荐实施顺序

建议后续按以下顺序推进：

1. 新增 `model-config-resolver.ts`（模型配置解析服务文件）。
2. 让 `RemoteLLMProvider`（远程大模型提供器）读取后台默认模型。
3. 新增 `GET /api/models/enabled`（启用模型列表接口）。
4. 新增用户侧模型列表服务和组合函数。
5. 先接入报告生成和专项刷题。
6. 继续评审并决定是否把 `modelId`（模型标识）下沉到 `cpp`（C++ 引擎）链路。
7. 补齐模型管理审计与密钥加密落盘。

这个顺序可以先让后台配置真正产生运行时价值，同时把冻结的流式契约放到最后处理，降低联调风险。
