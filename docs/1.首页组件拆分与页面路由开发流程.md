# 首页组件拆分与页面路由开发流程

> 版本：v1.0  
> 日期：2026-05-09  
> 适用项目：PrepChain / AI 面试知识库训练平台  
> 目标：把当前首页从单文件静态页面，逐步拆成可复用组件，并建立后续业务页面可扩展的路由结构

---

## 一、这份文档的用途

这份文档不是产品说明，而是开发执行顺序说明。

主要用于解决两个问题：

1. 当前首页都堆在 `src/views/chat.vue` 里，后续继续开发会越来越难维护
2. 当前项目只有一个 `/chat` 页面，无法承载资料库、模拟面试、专项刷题、复盘报告等业务页面

因此，当前阶段的开发重点是：

- 先拆首页组件
- 再建立页面路由

这两个步骤完成后，项目会从“单页样式稿”进入“可持续开发的前端工程结构”。

## 四、总开发原则

在执行这两个任务时，必须遵守以下原则：

### 4.1 第一版只拆展示，不急着抽 store

当前 `chat.vue` 里的静态数据先保留在页面层，例如：

- `sidebarItems`
- `quickTopics`
- `libraryCards`
- `practiceQueue`
- `guidanceCards`
- `interviewFlow`

第一阶段不要急着放到 Pinia。

原因：

- 当前目标是清理页面结构
- 不是做全局状态管理重构
- 数据先留在页面里，组件通过 `props` 接收即可

### 4.2 先按区域拆，不要拆得太碎

第一版组件拆分应该先按页面区域拆，不要一开始就拆成几十个小组件。

错误做法：

- 一个标题单独一个组件
- 一个小图标单独一个组件
- 一个数字单独一个组件

推荐做法：

- 先按“左侧栏 / 顶部栏 / Hero / 卡片区 / 右侧辅助区”拆

### 4.3 路由先搭壳，不急着做全部功能

第二阶段做页面路由时，允许先创建空白页面占位。

也就是说：

- 先把页面文件建好
- 先让左侧导航可跳转
- 具体内容后续逐页填充

1. 降低 `chat.vue` 的体积

## 5.3 第一阶段组件拆分顺序

不要一次性全拆，按下面顺序走。

### 第一步：先拆最明显的独立区域

优先拆：

1. `WorkbenchSidebar.vue`
2. `WorkbenchHeader.vue`
3. `LibraryCard.vue`
4. `GuidanceCard.vue`

原因：

- 拆完后页面体积会明显下降
- 对后续路由改造也最有帮助

### 第二步：再拆中间主舞台模块

继续拆：

5. `OverviewHero.vue`
6. `TopicChips.vue`
7. `FlowStepCard.vue`

### 第三步：最后拆右侧剩余模块

再拆：

8. `PracticeQueueCard.vue`
9. `ReviewSnapshotCard.vue`

## 5.4 每个组件第一版应该怎么处理数据

第一版数据不要放 store，先由页面层管理，再通过 `props` 传下去。

### 示例一：LibraryCard

在父页面中当前有：

```ts
const libraryCards = [
  {
    title: '前端八股总纲.md',
    meta: '128 个知识点 · 已切片',
    desc: '覆盖 Vue、TS、浏览器、性能优化和工程化，适合作为主资料库。',
    accent: 'from-[#6aa7ff] to-[#8ed0ff]'
  }
]
```

那么 `LibraryCard.vue` 第一版只需要接收：

- `title`
- `meta`
- `desc`
- `accent`

父组件负责传值，子组件只负责展示。

### 示例二：GuidanceCard

当前父页面中有：

```ts
const guidanceCards = [
  {
    title: '开始一轮模拟面试',
    desc: '基于你当前资料库，生成 10 道前端高频题并逐题追问。',
    tone: 'primary'
  }
]
```

那么 `GuidanceCard.vue` 第一版只需要接收：

- `title`
- `desc`
- `tone`

第一版组件不要自己处理业务逻辑，只做展示。

## 5.5 拆组件的执行步骤

### 步骤 1：备份当前页面结构思路

在开始拆之前，先确认当前页面作为“总览页”的结构已经稳定。

当前这版布局应默认保留：

- 左侧导航栏
- 顶部工具栏
- 中间 Hero + 资料概览
- 右侧 AI 辅助栏

### 步骤 2：复制出目标片段

拆某个组件时，不要先大删代码。

推荐做法：

1. 在 `chat.vue` 中找到该区域的模板片段
2. 新建组件
3. 把这段模板复制进去
4. 把原片段替换为新组件引用

### 步骤 3：提炼 props

将新组件中硬编码的数据替换为 `props`。

例如：

- 标题文本
- 描述文本
- 颜色类名
- 激活状态
- badge 值

### 步骤 4：局部样式迁移

如果样式只属于这个组件，就一起迁过去。

如果是多个区域共用样式，再考虑保留在页面层或抽成公共样式。

当前阶段不要求做完整样式抽象，优先保证拆完页面不乱。

### 步骤 5：在父页面中注册并替换

把原来大块 HTML 替换为：

- `<WorkbenchSidebar />`
- `<WorkbenchHeader />`
- `<LibraryCard />`
- `<GuidanceCard />`

### 步骤 6：每拆完一个组件就立即检查页面

检查项：

1. 页面是否还能正常渲染
2. 样式是否错位
3. Props 是否传全
4. v-for 是否正常
5. 是否有类名丢失

不要一次拆很多再一起排错。

---

## 5.6 拆组件完成后的目标结果

拆完第一版后，`chat.vue` 应该变成一个“页面组装层”。

它应该主要负责：

- 定义当前页面静态数据
- 组织各个组件
- 组合页面布局

而不是继续承担所有具体结构和样式细节。

理想状态是：

- `script` 中保留页面级数据
- `template` 中以组件拼装为主
- 样式体积明显减少

## 6.1 建页面路由前的目标

当前页面已经具备工作台雏形，因此第二步不应继续把所有功能塞进一个 `chat.vue` 里，而是要建立多页面结构。

目标是把平台拆成以下页面：

1. 总览
2. 资料库
3. 模拟面试
4. 专项刷题
5. 复盘报告
6. 训练历史

---

## 6.2 路由设计思路

当前路由只有：

- `/chat`

后续推荐改造成“工作台主路由 + 多子页面”的结构。

### 推荐路由路径

```text
/workspace/overview
/workspace/library
/workspace/mock-interview
/workspace/practice
/workspace/report
/workspace/history
```

如果你想保持原 `/chat` 兼容，也可以先做一层过渡：

```text
/chat -> /workspace/overview
```

推荐方向是最终统一到 `workspace` 语义下，因为这个页面已经不是单纯聊天页了。

## 6.3 页面文件结构建议

建议在 `src/views` 下建立业务目录：

```text
src/views/workbench/
  overview.vue
  library.vue
  mock-interview.vue
  practice.vue
  report.vue
  history.vue
```

说明：

- `overview.vue`：承接当前亮色首页
- 其余页面第一阶段可先做占位

---

## 6.4 是否需要单独的 Layout

需要。

因为这些页面会共用：

- 左侧导航
- 顶部搜索栏
- 统一工作台背景
- 主体内容区容器

所以推荐增加一个工作台布局组件，例如：

```text
src/components/Layout/WorkbenchLayout.vue
```

这个 Layout 负责：

1. 左侧栏
2. 顶部栏
3. 中间内容插槽

也就是说：

- 左侧导航不要每页都写一遍
- 顶部 header 不要每页都写一遍
- 各业务页面只负责中间主体内容

---

## 6.5 路由搭建执行顺序

### 步骤 1：先新增页面文件

先把这些页面文件建出来：

- `overview.vue`
- `library.vue`
- `mock-interview.vue`
- `practice.vue`
- `report.vue`
- `history.vue`

第一版允许先用非常简单的占位内容，例如：

- 页面标题
- 一段说明文字

目的是先把路由通路搭起来。

### 步骤 2：把当前 `chat.vue` 转为总览页

当前页面视觉上已经更像总览工作台，因此建议：

- 把当前页面内容迁移到 `overview.vue`

此时：

- `chat.vue` 可以后续废弃或转为兼容跳转
- `overview.vue` 成为真正的首页总览页

### 步骤 3：新增工作台 Layout

把这些公共区域抽到 Layout：

- 左侧导航栏
- 顶部 header
- 主体容器壳子

这样各个业务页就只填自己的中间内容。

### 步骤 4：修改 `child-routes.ts`

把当前单页结构改成多子路由结构。

也就是说，把原来：

- 一个 `/chat`
- 一个 `ChatIndex`

改成：

- 一个 `workspace` 主路由
- 多个业务子路由

### 步骤 5：处理默认重定向

默认进入工作台时，应跳转到：

- `/workspace/overview`

### 步骤 6：让左侧菜单真正接路由

当前左侧菜单只是静态按钮。

后续要让它具备真实导航功能：

- 点击后跳转对应页面
- 当前页高亮

推荐方式：

- 菜单项中增加 `routeName` 或 `path`
- 点击时 `router.push`
- 或直接使用 `RouterLink`

---

## 6.6 路由阶段的注意事项

### 注意 1：先不要把所有页面做完

当前目标只是“路由结构搭起来”，不是一次性做完整业务页。

### 注意 2：Layout 和页面职责要分清

Layout 负责：

- 外框
- 通用导航
- 通用 header

页面负责：

- 各自业务内容

### 注意 3：不要重复写侧栏和顶部

否则后面维护会变得很痛苦。

## 十、最后建议

这两个任务一定要分开看：

- **拆组件** 是为了让页面结构变清晰
- **建路由** 是为了让项目结构变合理

不要把“拆组件、改路由、接真实数据、做业务功能”混在一个步骤里一起做。

最稳的方式是：

1. 先稳定 UI
2. 再拆组件
3. 再建路由
4. 再接功能

这样你后面回看这个项目时，也能很清楚每一步为什么要做。

## 十四、这次改造后应该怎么理解目录职责

### 14.1 `src/components/workbench/`

这里放：

- 可复用的工作台 UI 模块
- 纯展示组件
- 接收 `props` 的局部模块

这里不应该放：

- 全局状态
- 路由配置
- 大页面级业务流程

### 14.2 `src/views/workbench/`

这里放：

- 工作台页面
- 页面级静态数据
- 页面拼装逻辑

这里适合放：

- `overview.data.ts`
- `workbench.data.ts`

这里不适合放：

- 全局共享业务状态 store

### 14.3 `src/store/`

这里目前还是 Pinia 根目录。

当前先不要把首页静态展示数据塞进来。

后续真正适合进 store 的，是这些业务状态：

- 当前资料库数据
- 当前训练会话
- 当前答题进度
- 当前复盘结果
- 页面间共享筛选条件

## 十五、你后面要从哪里继续开发

如果你下一次继续做这个项目，建议从下面顺序接着做：

### 下一步优先级 1：把总览页从旧页面完全切换过来

虽然现在新结构已经搭起来了，但你后续开发应以：

- `src/views/workbench/overview.vue`

为准，不再以旧的 `src/views/chat.vue` 作为主要工作文件。

### 下一步优先级 2：开始做资料库页

重点文件：

- `src/views/workbench/library.vue`

后续这里会承接：

- md/docx 导入
- 文档列表
- 标签筛选
- 文档预览

### 下一步优先级 3：开始接本地数据或真实业务状态

等页面结构稳定后，再考虑引入：

- `src/store/library/`
- `src/store/interview/`
- `src/store/report/`

这样的业务 store 结构。
