## 二、目录归类规范

### 前后端

- **前端**：`frontend/src/` + `frontend/package.json`
- **后端**：`backend/src/` + `backend/package.json`
- **根目录**：`pnpm-workspace.yaml` + 根 `package.json`（统一 `pnpm dev` / `pnpm dev:backend`）
- 已完成：`frontend/` 与 `backend/` 并列，根目录不再散落 `src/`、Vite 配置等

### 新代码落点（强制）

| 类型 | 目录 | 禁止 |
|------|------|------|
| 页面入口 | `views/showcase/` | 不在 `views/workbench/` 新增 |
| 宇宙页 UI | `components/showcase/mock-interview-space/` | — |
| 领域 UI | `components/<domain>/` | — |
| 状态 / 流程 | `composables/`（宇宙页用 `composables/showcase/`） | 不在已删除的 `hooks/` 新增 |
| API / SSE | `services/` | — |
| 纯函数 | `utils/` | 不把复杂业务编排塞入 utils |

### 组件三级决策

```
是否仅宇宙页使用？
  ├─ 是 → components/showcase/mock-interview-space/
  └─ 否 → 是否含业务语义？
           ├─ 是 → components/<domain>/
           └─ 否 → components/<通用>/
```

## 五、开发边界（Agent / 人工均适用）

1. 默认只改 **showcase / 宇宙页** 相关路径，除非任务明确要求 legacy。
2. 不改 `views/workbench/`、`components/workbench/` 的业务逻辑，除非修复阻塞性 bug 或执行删除计划。
3. 新 composable 命名：`useXxx`，放在与场景匹配的子目录。
4. 前后端重复工具（如 `is-unknown-answer.ts`）保持语义一致；大合并非当前优先级。

## 六、建议的下一阶段（可选）

当宇宙页功能稳定、legacy 确认零引用时，可开一个独立 PR：

1. 删除 workbench 目录与路由块（`child-routes.ts` 中 `/workspace` 整段）
2. 删除 `LEGACY_WORKBENCH_ENABLED` 及相关守卫
3. 清理 `types/workbench.ts` 等仅 legacy 使用的类型（需先 grep 确认）

在此之前，**保持「隐藏但不删」** 是当前最安全策略。
