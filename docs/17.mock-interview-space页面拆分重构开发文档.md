# 17.mock-interview-space页面拆分重构开发文档

## 1. 背景

当前文件 [src/views/showcase/mock-interview-space.vue](C:/Users/Administrator/Desktop/前端项目/chatgpt-vue3-light-mvp-main/src/views/showcase/mock-interview-space.vue) 已经达到约 `4020` 行，实际承担了过多职责，已经不适合继续在单文件内迭代。

这个文件当前同时包含：

- 宇宙场景静态配置 `scenes`
- 模拟面试业务状态与提交逻辑
- 总览场景承接逻辑
- 资料库场景承接逻辑
- 报告场景承接逻辑
- 轨道切换与视觉层动画
- 自动播放与滚动控制
- 整页模板
- 整页样式

继续在这里直接改，会越来越容易出现：

- 改一个场景，误伤另一个场景
- UI 样式覆盖链越来越长
- 很难定位滚动、动画、场景切换的副作用
- 新同学几乎无法接手
- 后续做测试、复用、性能排查成本都很高

这份文档的目标不是“纯理论上的最佳拆分”，而是给出一套适合当前仓库、可逐步落地、风险可控的拆分方案。

---

## 2. 当前问题分布

从结构上看，这个文件大致分成 5 类代码：

### 2.1 场景配置

对应区域：

- `SceneTheme / SceneItem / VisualSlot / VisualLayerState / OrbitSlot`
- `const scenes = [...]`

问题：

- 这些内容本质是配置，不应该和页面渲染逻辑、滚动逻辑耦合在一起
- 每次改文案、配色、轨道信息都要翻进主页面文件

### 2.2 场景业务状态

对应区域：

- mock 面试状态
- overview 入口状态
- library 资料状态
- report 报告状态

问题：

- 一个页面同时维护四个子页面的数据状态
- `computed / ref / watch` 数量非常多，阅读门槛高
- 业务逻辑和视觉切换混在一起

### 2.3 轨道与滚动控制

对应区域：

- orbit 切换
- autoplay
- header fade
- scroll 动画
- content reveal

问题：

- 这些是“页面舞台控制逻辑”，应该单独集中管理
- 当前和 mock / report / library 业务状态混杂
- 修改滚动逻辑时容易误触场景逻辑

### 2.4 模板层

对应区域：

- `<template>` 内包含 overview / library / mock / report 所有场景结构

问题：

- 模板阅读成本极高
- 想单独改 mock 面试区，也必须在巨大的 template 里找
- 组件边界不清晰

### 2.5 样式层

对应区域：

- 当前文件中自带的大量 scoped SCSS

问题：

- 样式量很大，且覆盖层级深
- 多场景样式放在一起，后续维护容易互相污染
- mock 相关样式尤其应该收敛到 mock 场景自己的组件内

## 3. 拆分目标

目标不是一步把 4000 行拆成 50 个文件，而是先把“变化频率高、最容易继续变大的部分”拆出去。

建议拆分目标如下：

1. `mock-interview-space.vue` 只保留“舞台容器”和“场景切换编排”
2. 每个场景的业务 UI 单独成为一个场景组件
3. 轨道动画 / 自动播放 / 滚动控制抽成 composable
4. `scenes` 配置抽到独立配置文件
5. 大块场景样式跟随组件下沉

拆完后，这个 view 文件最好收敛到：

- 场景入口装配
- 顶层 refs
- 轨道切换调用
- 场景组件挂载

理想控制在：

- 主 view：`500 ~ 900` 行
- 单场景组件：`200 ~ 500` 行
- 单个 composable：`100 ~ 300` 行

## 5. 推荐拆分边界

### 5.1 第一优先级：先拆配置

最先拆：

- `SceneTheme`
- `SceneItem`
- `scenes`

建议落到：

- `src/constants/showcase/mockInterviewSpaceScenes.ts`

收益：

- 立刻减轻主文件体积
- 后续调文案、色彩、轨道信息更安全
- 不影响模板和交互，风险最低

### 5.2 第二优先级：先拆 mock 场景

最应该先拆的场景是 `mock`，因为它目前变化最频繁，也最容易继续扩展。

建议抽成：

- `SpaceMockScene.vue`

它内部承接：

- 题目区
- 提示区
- 问答内容区
- 回答草稿区
- mock 专属样式

主页面只保留：

- `mock` 场景数据输入
- `submit / clear / stop` 事件对接

收益：

- 你后续改题目区、草稿区、滚动按钮，都不需要再钻 4000 行文件
- mock 的样式污染面大幅缩小

### 5.3 第三优先级：拆滚动与轨道逻辑

建议把这部分抽成 composable：

- `useMockInterviewSpaceOrbit.ts`
- `useMockInterviewSpaceScroll.ts`

其中：

`useMockInterviewSpaceOrbit.ts` 负责：

- orbitOrder
- activeIndex / displayIndex / copySceneIndex
- autoplay
- goToNext / goToPrev
- requestSceneChange / navigateToScene
- 轨道相关定时器

`useMockInterviewSpaceScroll.ts` 负责：

- header fade
- content reveal target
- auto scroll
- scrollToContentPreview
- mock 页面题目置顶滚动

收益：

- 滚动问题和业务问题解耦
- 后续再改“悬浮箭头滚到哪里”时，不会碰到场景业务状态

### 5.4 第四优先级：拆 report / library / overview 场景

建议一场景一个组件：

- `SpaceOverviewScene.vue`
- `SpaceLibraryScene.vue`
- `SpaceReportScene.vue`

这些组件里可以继续复用已有组件：

- `DocumentListItem`
- `DocumentPreviewPanel`
- `ImportActionCard`
- `LibraryFilterTabs`
- `LibraryStatCard`
- `MessageList`
- `AnswerInputPanel`

### 5.5 第五优先级：拆视觉壳层

最后再拆这些“纯舞台层”组件：

- `SpaceHeader.vue`
- `SpaceOrbitNav.vue`
- `SpaceVisualStage.vue`
- `SpaceScrollCapsule.vue`

原因：

- 这部分虽然大，但变更频率通常低于 mock 场景业务
- 放到后面拆，风险更可控

## 7. 推荐的 props / emits 边界

这里是最关键的部分。拆组件时不要把整个大对象一把塞进去，否则只是“文件拆了，耦合没拆”。

### 7.1 `SpaceMockScene.vue`

建议 props：

```ts
interface SpaceMockSceneProps {
  primaryQuestion: QuestionItem | null
  hintText: string
  messages: InterviewMessage[]
  scrollVersion?: string
  answerDraft: string
  submitted: boolean
  streaming: boolean
  streamError?: string
}
```

建议 emits：

```ts
{
  'update:answerDraft': [value: string]
  submit: []
  clear: []
  stop: []
}
```

不要把：

- `mockSubmittedQuestionIds`
- `mockWeaknessSignals`
- `mockFollowUpIndex`

这类内部状态直接暴露到场景组件里，尽量由 composable 或上层聚合后只传结果。

### 7.2 `SpaceLibraryScene.vue`

只传当前场景真正需要的数据：

- tabs
- filtered documents
- selected document
- import feedback
- actions

不要把整个 `useLibraryWorkspaceState()` 的返回值对象整包传递。

### 7.3 `SpaceReportScene.vue`

建议只传：

- 头部 meta
- 概览 stats
- 快照 items
- 建议列表
- 历史记录列表
- 事件 action

## 8. 样式拆分建议

当前样式不建议继续全部放在 `mock-interview-space.vue`。

建议原则：

### 8.1 页面壳层样式保留在主 view

保留内容：

- 星空背景
- hero-stage
- header
- orbit 壳层
- 通用过渡动画

### 8.2 场景内部样式跟随场景组件

例如：

- `.mock-question-card`
- `.mock-conversation-shell`
- `.mock-draft-shell`

应该跟着 `SpaceMockScene.vue`

### 8.3 局部深度覆盖尽量就地收口

当前大量：

- `:deep(.message-list)`
- `:deep(.n-input ...)`
- `:deep(.n-button ...)`

这类覆盖最好跟着使用它们的组件走，不要留在主 view 里。

否则后续一改 `AnswerInputPanel`，主 view 又要补覆盖，维护体验会持续变差。

## 9. 拆分时最容易踩的坑

### 9.1 不要一上来拆样式

先拆结构和状态，再拆样式。

原因：

- 结构不稳时先搬样式，极容易出现“搬完又改回去”

### 9.2 不要先拆 orbit 视觉层

orbit 动画链路复杂、牵涉 gsap 和多个 ref。

它虽然大，但不一定是当前最大的维护痛点。先拆 mock 场景收益最高。

### 9.3 不要整包透传 composable 返回值

例如不要这样：

```vue
<SpaceLibraryScene v-bind="libraryState" />
```

这样会让组件边界继续失控。

### 9.4 不要同时做“拆分 + 视觉重构”

拆分阶段只做结构迁移，不顺便重设计。

否则你很难判断 bug 是：

- 拆分导致的
- 还是视觉修改导致的

### 9.5 每阶段都要做页面级回归

至少手动回归：

- overview
- library
- mock
- report
- 轨道切换
- 悬浮滚动按钮
- mock 输入与提交流

拆完第一轮后，`mock-interview-space.vue` 应该大致只保留：

- 页面基础容器

## 13. 结论

这个文件现在最适合的拆法，不是“按技术层纯粹拆”，而是：

1. 先拆配置
2. 再拆 mock 场景
3. 再拆 mock 状态
4. 再拆滚动
5. 再拆 orbit
6. 最后补齐其他场景组件

原因很简单：

- mock 场景是当前最常改、最容易继续膨胀的区域
- 配置和滚动是低风险高收益的拆分点
- orbit 复杂但稳定，适合放后面

如果按这个顺序做，能在不打断当前页面开发的情况下，逐步把这个大文件收敛到可维护状态。
