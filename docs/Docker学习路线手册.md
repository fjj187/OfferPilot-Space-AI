# Docker学习路线手册

这份手册只聚焦前端开发和 `Node.js（服务端 JavaScript 运行时）` 相关的 `Docker（容器化平台）` 学习内容。

目标不是一次学全，而是按从易到难的顺序，学到能独立把前端项目放进容器、运行、调试、构建和部署。

## 一 学习目标

学完这条路线后，你应该能做到这些事

- 看懂 `Dockerfile（镜像构建文件）`
- 用 `docker build（构建镜像命令）` 构建前端或 `Node.js（服务端 JavaScript 运行时）` 项目镜像
- 用 `docker run（启动容器命令）` 跑一个前端开发容器
- 用 `docker compose（多容器编排命令）` 同时启动前端和后端
- 理解前端项目里端口 映射 挂载 环境变量 热更新 的常见问题
- 能写一个前端生产环境镜像并交给 `Nginx（静态资源服务器）` 托管

## 二 学习顺序总览

建议按下面顺序学

1. `Docker（容器化平台）` 基础概念
2. 单容器运行 `Node.js（服务端 JavaScript 运行时）` 项目
3. `Dockerfile（镜像构建文件）` 语法
4. 前端项目容器化
5. `docker compose（多容器编排命令）`
6. 前端开发调试
7. 前端生产构建与部署
8. 进阶优化

## 三 第一阶段 基础概念

这一阶段只需要把核心名词弄清楚。

### 1. `image（镜像）`

可以理解成容器运行前的模板。

它里面通常包含

- 操作系统基础环境
- `Node.js（服务端 JavaScript 运行时）`
- 项目依赖
- 启动命令

### 2. `container（容器）`

容器是镜像运行后的实例。

你可以把镜像理解为类，把容器理解为对象。镜像负责定义，容器负责运行。

### 3. `Dockerfile（镜像构建文件）`

这是一个文本文件，用来描述“镜像应该怎么做出来”。

比如

- 用哪个基础镜像
- 把哪些文件复制进去
- 安装什么依赖
- 启动时执行什么命令

### 4. `registry（镜像仓库）`

用来存放镜像。

常见场景

- 从远程拉基础镜像
- 把自己项目镜像推送到仓库

### 5. `volume（数据卷）`

用于持久化数据，或者把宿主机目录挂到容器里。

前端开发里最常见的用途是源码挂载，也就是让容器直接使用你本地的项目代码。

### 6. `network（容器网络）`

容器之间通信依赖网络。

比如前端容器访问后端容器时，通常不是访问 `localhost（本机回环地址）`，而是访问服务名。

### 7. `port mapping（端口映射）`

把容器里的端口映射到宿主机。

例如

```bash
docker run -p 2048:2048 my-app
```

这表示把宿主机 `2048` 端口映射到容器 `2048` 端口。

### 本阶段你要会什么

- 说清楚镜像和容器的区别
- 知道 `Dockerfile（镜像构建文件）` 是干什么的
- 知道为什么前端开发经常要用挂载和端口映射

## 四 第二阶段 先会跑一个 Node.js 项目

这一阶段目标很简单，先不要急着学复杂编排，先学会“把一个 `Node.js（服务端 JavaScript 运行时）` 项目跑进容器”。

### 1. 你要认识的命令

`docker pull（拉取镜像命令）`
用于拉取远程镜像。

`docker build（构建镜像命令）`
用于根据 `Dockerfile（镜像构建文件）` 生成镜像。

`docker run（启动容器命令）`
用于启动一个容器。

`docker ps（查看容器命令）`
用于查看正在运行的容器。

`docker logs（查看日志命令）`
用于查看容器输出日志。

`docker exec（进入容器执行命令）`
用于进入正在运行的容器做调试。

### 2. 你要理解的过程

一个 `Node.js（服务端 JavaScript 运行时）` 项目被容器化，通常会经历这几个步骤

1. 选择基础镜像
2. 复制项目文件
3. 安装依赖
4. 暴露端口
5. 启动服务

### 3. 最小示例

```Dockerfile
FROM node:22-alpine

WORKDIR /app

COPY package.json ./

RUN npm install

COPY . .

EXPOSE 3000

CMD ["npm", "run", "dev"]
```

### 4. 陌生语法解释

`FROM（指定基础镜像）`
表示这份镜像基于哪个已有镜像构建。

`WORKDIR（设置工作目录）`
表示后续命令默认在哪个目录执行。

`COPY（复制文件）`
表示把宿主机文件复制到镜像里。

`RUN（构建时执行命令）`
表示在构建镜像时执行一次命令，常见于安装依赖。

`EXPOSE（声明端口）`
表示镜像内的应用会使用哪个端口。

`CMD（容器启动命令）`
表示容器启动时默认执行什么命令。

### 本阶段你要会什么

- 能看懂一个最小 `Dockerfile（镜像构建文件）`
- 能解释 `FROM（指定基础镜像）` 和 `CMD（容器启动命令）`
- 能自己把一个简单 `Node.js（服务端 JavaScript 运行时）` 项目跑起来

## 五 第三阶段 学 Dockerfile

这部分是重点，前端和 `Node.js（服务端 JavaScript 运行时）` 容器化的核心就在这里。

### 1. 常用指令

`ENV（环境变量）`
给镜像或容器设置环境变量。

`ARG（构建参数）`
给构建过程传参数。它主要在构建时生效，不等于运行时环境变量。

`ENTRYPOINT（固定入口命令）`
定义容器主入口。比 `CMD（容器启动命令）` 更强。

`USER（指定运行用户）`
指定容器内用哪个用户运行进程。

### 2. `CMD（容器启动命令）` 和 `RUN（构建时执行命令）` 的区别

`RUN（构建时执行命令）`
是在构建镜像时执行。

`CMD（容器启动命令）`
是在容器启动时执行。

你可以这样理解

- `RUN（构建时执行命令）` 是做镜像
- `CMD（容器启动命令）` 是跑容器

### 3. `ARG（构建参数）` 和 `ENV（环境变量）` 的区别

`ARG（构建参数）`
主要给构建阶段用，比如切换安装源或打包模式。

`ENV（环境变量）`
主要给运行阶段用，比如 `NODE_ENV（Node 环境变量）`

### 4. 为什么先复制 `package.json（项目依赖清单）`

很多示例会先写

```Dockerfile
COPY package.json ./
RUN npm install
```

这样做的原因是利用 `layer cache（镜像层缓存）`。

`layer cache（镜像层缓存）`
表示 Docker 会缓存每一步构建结果。

如果依赖文件没变，安装依赖这一步就能复用缓存，构建更快。

### 本阶段你要会什么

- 能解释常见 `Dockerfile（镜像构建文件）` 指令
- 知道构建阶段和运行阶段的区别
- 能看懂为什么镜像构建顺序会影响速度

## 六 第四阶段 前端项目容器化

从这里开始，内容和你的学习目标强相关。

### 1. 前端项目的两种典型模式

第一种是开发模式

- 在容器里运行 `Vite（前端开发服务器）`
- 使用源码挂载
- 保留热更新能力

第二种是生产模式

- 容器里执行 `pnpm build（前端生产构建命令）`
- 生成 `dist（构建产物目录）`
- 用 `Nginx（静态资源服务器）` 提供访问

### 2. 开发容器的关注点

`bind mount（目录挂载）`
把本地源码目录挂到容器里，改代码就能马上生效。

`hot reload（热更新）`
代码改动后页面自动刷新或局部更新。

`HMR（热模块替换）`
热更新的一种更细粒度形式，只替换变化模块，不整页刷新。

`watch（文件监听）`
开发服务器监听文件变化的机制。

Windows 环境里，挂载后的监听有时不稳定，这时可能要用轮询。

`polling（轮询）`
不是等系统主动通知，而是定时检查文件是否变化。稳定性更高，但更耗性能。

### 3. 你的项目里最该关注什么

这个仓库是 `Vue 3（前端框架）` + `Vite（前端开发服务器）` + `TypeScript（JavaScript 类型系统）` + `pnpm workspace（pnpm 工作区）`

所以你要重点学这些点

- 如何在镜像里安装 `pnpm（包管理器）`
- 如何处理根目录和 `frontend/（前端目录）` 的工作区关系
- 如何暴露 `2048` 端口
- 如何让 `Vite（前端开发服务器）` 在容器里被外部访问

### 本阶段你要会什么

- 区分前端开发容器和生产容器
- 知道为什么前端开发常常需要目录挂载
- 知道 `HMR（热模块替换）` 在容器里为什么可能失效

## 七 第五阶段 学 docker compose

当你不止一个服务时，就该学 `docker compose（多容器编排命令）` 了。

### 1. 它解决什么问题

如果你要同时启动这些服务

- `frontend（前端服务）`
- `backend（后端服务）`
- 未来可能还有 `ollama（本地模型服务）`

一个个手动 `docker run（启动容器命令）` 会很麻烦。

`docker compose（多容器编排命令）` 可以统一描述这些服务怎么启动。

### 2. 你要认识的字段

`services（服务列表）`
定义有哪些服务。

`build（构建配置）`
定义镜像如何从本地项目构建。

`ports（端口映射列表）`
定义端口映射。

`volumes（挂载配置）`
定义目录挂载或数据卷。

`environment（环境变量列表）`
定义服务运行时的环境变量。

`depends_on（依赖服务）`
定义启动顺序依赖。

### 3. 最小示例

```yaml
services:
  frontend:
    build:
      context: .
      dockerfile: ./frontend/Dockerfile
    ports:
      - "2048:2048"
    volumes:
      - .:/app
    command: pnpm dev
```

### 4. 陌生语法解释

`context（构建上下文目录）`
表示构建镜像时，Docker 能访问哪些文件。

`command（运行命令覆盖项）`
表示覆盖镜像里默认的 `CMD（容器启动命令）`

### 本阶段你要会什么

- 能看懂一个基础 `compose.yaml（容器编排文件）`
- 知道前端和后端为什么能通过服务名互相访问
- 能理解一个服务依赖另一个服务是什么意思

## 八 第六阶段 前端开发调试

这一阶段是实战里最容易卡住的部分。

### 1. 常见问题一 页面打不开

先检查这些点

- 容器是否启动
- 端口是否映射
- `Vite（前端开发服务器）` 是否监听了 `0.0.0.0（监听所有网络接口地址）`

`0.0.0.0（监听所有网络接口地址）`
表示允许容器外访问。

如果只监听 `localhost（本机回环地址）`，宿主机通常访问不到。

### 2. 常见问题二 改代码不刷新

先检查这些点

- 是否做了源码挂载
- 容器里的工作目录是否正确
- 文件监听是否正常
- 是否需要开启 `polling（轮询）`

### 3. 常见问题三 前端访问不到后端

先检查这些点

- 是否错误写成 `localhost（本机回环地址）`
- 前端容器里访问后端时，地址是否改成服务名
- 端口是否写对

### 4. 常见问题四 依赖安装很慢

先检查这些点

- 是否正确利用缓存层
- 是否每次都无意义复制全量文件
- 是否把 `node_modules（依赖目录）` 处理混乱

### 5. 常见调试命令

```bash
docker ps
docker logs 容器名
docker exec -it 容器名 sh
docker compose ps
docker compose logs
```

### 本阶段你要会什么

- 能用日志定位基本问题
- 能分辨是端口问题 挂载问题 还是服务地址问题
- 能进入容器做排查

## 九 第七阶段 前端生产构建与部署

这部分适合你学完开发容器后再看。

### 1. `multi-stage build（多阶段构建）`

指一个 `Dockerfile（镜像构建文件）` 里分多个阶段。

前端项目常见做法是

1. 用 `Node.js（服务端 JavaScript 运行时）` 镜像安装依赖并执行构建
2. 用 `Nginx（静态资源服务器）` 镜像承接 `dist（构建产物目录）`

这样做的好处是生产镜像更小 更干净。

### 2. 为什么前端生产环境常用 Nginx

因为打包后的前端通常就是静态文件

- `html（超文本标记语言文件）`
- `css（层叠样式表文件）`
- `js（JavaScript 脚本文件）`

`Nginx（静态资源服务器）` 很适合托管这类文件。

### 3. 你要理解的部署链路

本地代码

到

`Dockerfile（镜像构建文件）`

到

`docker build（构建镜像命令）`

到

镜像仓库

到

服务器拉取并运行

### 本阶段你要会什么

- 理解前端生产镜像为什么通常不是直接跑 `Vite（前端开发服务器）`
- 理解 `multi-stage build（多阶段构建）` 的价值
- 知道 `dist（构建产物目录）` 在部署中的位置

## 十 第八阶段 进阶学习

这些内容放到最后学，前面不熟时不要急着碰。

### 1. 镜像优化

要学会关注

- 镜像大小
- 构建速度
- 缓存命中率

### 2. 安全基础

要知道这些概念

`non-root user（非 root 用户）`
不要总让容器内进程用最高权限运行。

`least privilege（最小权限原则）`
只给当前任务需要的最小权限。

### 3. 数据与持久化

虽然你主要做前端，但如果后面接 `Node.js（服务端 JavaScript 运行时）` 服务，最好理解

- 配置文件挂载
- 日志目录挂载
- 数据目录持久化

### 4. 持续集成

`CI（持续集成）`
表示把构建 测试 检查放到自动流程中。

以后你可以把这些动作放进容器

- `pnpm lint（代码检查命令）`
- `pnpm test（测试命令）`
- `pnpm build（生产构建命令）`
