# 47.总览页数据可视化大屏真实数据接入方案

## 一、结论

当前总览页数据可视化大屏不是直接接入后端真实聚合数据，而是由 `SpaceOverviewScene.vue`（总览场景组件）调用 `useInterviewAnalyticsDashboardData.ts`（面试分析大屏数据组合函数），从本地持久化的面试会话和复盘报告中派生数据；当本地没有可用记录时，再回退到 `interviewAnalyticsDemoData.ts`（面试分析演示数据文件）。

因此，现阶段更准确的判断是：大屏组件已经具备数据驱动能力，但真实后端统计接入层还没有完成。

## 二、当前依据

### 2.1 展示组件只接收数据

`InterviewAnalyticsDashboard.vue`（面试分析大屏组件）通过 `props`（组件入参）接收 `data`（数据）、`loading`（加载中）、`error`（错误）和 `active`（激活状态），内部只负责图表布局、状态展示和 `ECharts`（图表库）配置生成。

这说明大屏组件本身没有直接请求接口，也不应该在组件内部拉取业务数据。

### 2.2 页面容器负责传入数据

`SpaceOverviewScene.vue`（总览场景组件）负责接入 `useInterviewAnalyticsDashboardData.ts`（面试分析大屏数据组合函数），并把 `dashboardData`（大屏数据）传给 `InterviewAnalyticsDashboard.vue`（面试分析大屏组件）。

当前数据流如下：

```text
浏览器本地持久化数据
  -> useInterviewAnalyticsDashboardData.ts（面试分析大屏数据组合函数）
  -> SpaceOverviewScene.vue（总览场景组件）
  -> InterviewAnalyticsDashboard.vue（面试分析大屏组件）
  -> BaseECharts.vue（基础图表组件）
```

### 2.3 仍存在演示数据兜底

`useInterviewAnalyticsDashboardData.ts`（面试分析大屏数据组合函数）会在当前时间范围没有本地训练记录时返回 `interviewAnalyticsDemoData`（面试分析演示数据）。

这保证页面可预览，但也说明它还不是完整的后端真实统计链路。

### 2.4 服务层尚无总览统计接口

当前 `frontend/src/services/`（前端服务层目录）下没有独立的 `analytics.ts`（分析数据服务文件）或等价总览统计服务封装；项目已有接口主要集中在模拟面试、报告、题池、后台管理等模块。

## 三、目标结构

真实数据接入后，建议保持展示层低耦合，按下面结构推进：

```text
Node API（后端接口）
  -> frontend/src/services/analytics/analytics-api.ts（分析数据接口服务）
  -> frontend/src/utils/analyticsMapper.ts（分析数据映射工具）
  -> SpaceOverviewScene.vue（总览场景组件）
  -> InterviewAnalyticsDashboard.vue（面试分析大屏组件）
  -> BaseECharts.vue（基础图表组件）
```

分层职责如下：

| 层级 | 推荐文件 | 职责 |
| --- | --- | --- |
| 后端接口 | `GET /api/analytics/overview`（分析总览接口） | 聚合面试、报告、题池、训练记录等统计数据 |
| 服务层 | `frontend/src/services/analytics/analytics-api.ts`（分析数据接口服务） | 统一发起请求、处理基础错误 |
| 类型层 | `frontend/src/types/analytics.ts`（分析数据类型文件） | 定义后端 `DTO`（数据传输对象）和前端视图模型 |
| 映射层 | `frontend/src/utils/analyticsMapper.ts`（分析数据映射工具） | 把后端字段转成大屏组件需要的结构 |
| 页面层 | `SpaceOverviewScene.vue`（总览场景组件） | 管理加载、错误、空数据、筛选条件 |
| 展示层 | `InterviewAnalyticsDashboard.vue`（面试分析大屏组件） | 只负责渲染和事件抛出 |

## 四、后端接口草案

> 注意：本节是新增接口建议，不是已冻结契约。真正落地前需要按 `docs/前后端与C++协作对齐.md`（三方协作边界文档）走评审流程。

建议新增聚合接口：

```http
GET /api/analytics/overview?range=30d
```

查询参数建议：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `range`（时间范围） | `7d` / `30d` / `all` | 对应总览页现有时间筛选 |

响应结构建议：

```ts
export interface AnalyticsOverviewResponse {
  summary: AnalyticsSummaryMetricDto[]
  scoreTrend: AnalyticsScorePointDto[]
  abilityRadar: AnalyticsAbilityMetricDto[]
  topicDistribution: AnalyticsDistributionDto[]
  practiceQuestionTypeDistribution: AnalyticsDistributionDto[]
  weaknessRanking: AnalyticsWeaknessRankDto[]
  trainingHeatmap: AnalyticsHeatmapDayDto[]
  activityFeed: AnalyticsActivityDto[]
  aiSummary?: string
  generatedAt: string
}
```

建议字段范围：

| 模块 | 内容 |
| --- | --- |
| 核心指标 | 面试总次数、完成次数、平均得分、弱项标签数量、训练时长 |
| 评分趋势 | 最近 7 到 14 次训练得分 |
| 能力雷达 | 表达能力、逻辑结构、专业知识、岗位匹配、抗压能力、复盘吸收 |
| 分布统计 | 训练主题分布（Vue 3 / TypeScript / 浏览器等）、专项题型分布（概念理解 / 代码分析 / 场景追问） |
| 弱项排行 | 高频弱项、出现次数、补练建议 |
| 训练热力 | 最近 14 天或 30 天训练活跃度 |
| 最近动态 | 最近面试、报告、专项训练记录 |
| AI 摘要 | 可选的阶段性训练总结 |

## 五、前端类型建议

后端 `DTO`（数据传输对象）和前端视图模型需要分开定义，避免大屏组件直接依赖后端字段。

```ts
export interface AnalyticsOverviewViewModel {
  dashboardData: InterviewAnalyticsDashboardData
  source: 'remote' | 'local' | 'demo'
  generatedAt?: string
}
```

其中 `InterviewAnalyticsDashboardData`（面试分析大屏数据结构）继续复用现有 `frontend/src/types/interview-analytics.ts`（面试分析类型文件），不要把 `ECharts option`（图表配置项）暴露给页面或后端。

## 六、前端落地步骤

### 6.1 新增服务层

新增 `frontend/src/services/analytics/analytics-api.ts`（分析数据接口服务文件），只负责请求后端聚合接口。

建议方法：

```ts
export const fetchAnalyticsOverview = (params: {
  range: InterviewAnalyticsTimeRange
}) => {
  return request<AnalyticsOverviewResponse>({
    url: '/api/analytics/overview',
    method: 'GET',
    params
  })
}
```

### 6.2 新增映射层

新增 `frontend/src/utils/analyticsMapper.ts`（分析数据映射工具），把 `AnalyticsOverviewResponse`（分析总览响应）转成 `InterviewAnalyticsDashboardData`（面试分析大屏数据结构）。

映射规则需要明确：

| 后端字段 | 前端字段 |
| --- | --- |
| `summary`（汇总指标） | `metrics`（指标卡片） |
| `scoreTrend`（评分趋势） | `scoreTrend`（评分趋势） |
| `abilityRadar`（能力雷达） | `abilityRadar`（能力雷达） |
| `topicDistribution`（训练主题分布） | `topicDistribution`（训练主题分布） |
| `practiceQuestionTypeDistribution`（专项题型分布） | `practiceQuestionTypeDistribution`（专项题型分布） |
| `weaknessRanking`（弱项排行） | `weaknessRanking`（弱项排行） |
| `trainingHeatmap`（训练热力） | `trainingHeatmap`（训练热力） |
| `activityFeed`（训练动态） | `activityFeed`（训练动态） |

### 6.3 改造组合函数

优先改造 `useInterviewAnalyticsDashboardData.ts`（面试分析大屏数据组合函数），让它支持三类数据源：

```text
remote（后端真实数据）
  -> local（本地持久化派生数据）
  -> demo（演示数据兜底）
```

建议保留当前本地派生逻辑作为降级方案。这样本地后端未启动、接口未完成或 `GitHub Pages`（静态站点托管）演示环境下，页面仍可正常展示。

### 6.4 页面只管理状态

`SpaceOverviewScene.vue`（总览场景组件）只做四件事：

1. 维护时间范围。
2. 触发重新加载。
3. 展示 `loading`（加载中）、`error`（错误）、`empty`（空数据）状态。
4. 把 `dashboardData`（大屏数据）传给 `InterviewAnalyticsDashboard.vue`（面试分析大屏组件）。

不要把字段映射、接口细节、后端兼容逻辑写进页面模板。

## 七、空数据与降级策略

推荐策略如下：

| 场景 | 页面表现 | 数据来源 |
| --- | --- | --- |
| 后端接口成功且有数据 | 展示真实统计 | `remote`（后端真实数据） |
| 后端接口成功但为空 | 展示空状态，可提示先完成训练 | `remote`（后端真实数据） |
| 后端接口失败但本地有记录 | 展示本地派生统计，并提示本地数据 | `local`（本地数据） |
| 后端接口失败且本地无记录 | 展示演示数据或空状态，按产品选择 | `demo`（演示数据） |
| 静态演示环境 | 禁用真实接口，直接使用本地或演示数据 | `local` / `demo` |

## 八、边界约束

1. 不在 `InterviewAnalyticsDashboard.vue`（面试分析大屏组件）中请求接口。
2. 不在 `BaseECharts.vue`（基础图表组件）中理解业务字段。
3. 不直接修改已冻结的 `POST /api/interview/stream`（模拟面试流式接口）或 `SSE`（服务端事件流）契约。
4. 不让前端直连 `C++`（后端语言）服务。
5. 不在前端猜测多个后端字段名做兼容，例如同时兼容 `score`、`avgScore`、`average_score`。
6. 不为了接入真实数据删除演示数据兜底，除非确认所有部署环境都有后端服务。

## 九、验收标准

真实数据接入完成后，至少满足：

1. `InterviewAnalyticsDashboard.vue`（面试分析大屏组件）仍然只通过 `props`（组件入参）接收数据。
2. `frontend/src/services/analytics/analytics-api.ts`（分析数据接口服务文件）存在，并且页面不直接拼接请求地址。
3. `frontend/src/utils/analyticsMapper.ts`（分析数据映射工具）存在，后端响应不直接穿透到图表组件。
4. 时间范围切换可以重新拉取或重新计算数据。
5. `loading`（加载中）、`error`（错误）、`empty`（空数据）都有明确展示。
6. 后端不可用时，页面不崩溃。
7. 本地演示环境和 `GitHub Pages`（静态站点托管）构建不依赖真实后端。
8. 未修改冻结的面试流式接口和 `SSE`（服务端事件流）字段。

## 十、推荐实施顺序

1. 先补后端接口契约文档，确认 `GET /api/analytics/overview`（分析总览接口）字段。
2. 新增前端 `types`（类型定义）、`service`（服务层）、`mapper`（映射工具）。
3. 改造 `useInterviewAnalyticsDashboardData.ts`（面试分析大屏数据组合函数），加入远程数据源和降级策略。
4. 小幅调整 `SpaceOverviewScene.vue`（总览场景组件），把加载状态传给大屏。
5. 保留本地派生数据和演示数据，完成多环境验证。
