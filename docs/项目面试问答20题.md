一、开场与业务（第 1～4 题）
────────────────────────────────────────


第 1 题：用 30 秒介绍项目和你的职责

面试官可能问：
  介绍一下这个项目，你在里面做了什么？

建议答：
  这是基于 Vue 3（框架）+ Vite（构建工具）的 AI 模拟面试 Web 产品，主入口是 frontend/src/views/showcase/mock-interview-space.vue（宇宙模拟面试页）。用户可导入 Markdown 资料、按章节分块生成题池、逐题模拟面试，面试官回复通过 SSE（Server-Sent Events，服务端推送）流式输出，并生成报告与弱项追踪。仓库是 Monorepo（单体仓库）：frontend 负责交互与流式渲染，backend 是 Node（Node.js）Express 网关，用 Provider（提供者）工厂切换 mock、远程大模型或 C++ 引擎。我主要负责：（请按实际填写，例如）宇宙页场景与状态机、useInterviewStream 流式链路、资料分块出题、多入口会话恢复。


第 2 题：和 Chat 对话页有什么区别

面试官可能问：
  不就是套壳 ChatGPT 吗？和 chat 页有什么不同？

建议答：
  frontend/src/views/chat.vue 是早期单轮 Chat 演示，可走 /deepseek、/spark 等 Vite 代理直连大模型，偏通用问答。主产品是面试工作流：资料驱动出题、按题作答、feedbackStyle（反馈风格：追问/纠偏/引导）、切题规则、结束后报告。价值是把「看资料 → 练题 → AI 面试官反馈 → 复盘」串成闭环，不是无限上下文聊天。


第 3 题：业务价值与目标用户

面试官可能问：
  解决什么痛点？为什么需要资料分块？

建议答：
  痛点是面试准备缺少「按自己资料练 + 结构化反馈」。导入 Markdown 后，frontend/src/services/material/material-chunk-splitter.ts 按标题层级切到约 1000 字一块，再经 material-question-builder.ts 生成题池，保证题目可追溯到具体章节（sourceChunkId），而不是泛泛的八股题。


第 4 题：技术选型

面试官可能问：
  为什么 Vue 3？为什么单独做 backend？

建议答：
  Vue 3 Composition API（组合式 API）+ composables 适合把面试状态机从 UI 拆开；Naive UI（组件库）+ UnoCSS（原子化 CSS）跟仓库一致。需要 backend 是因为：API Key 不宜长期裸放前端；SSE 契约要统一；与 C++ 并行时只换 Provider 环境变量，前端仍只请求 POST /api/interview/stream。


────────────────────────────────────────
二、架构与请求链路（第 5～8 题）
────────────────────────────────────────


第 5 题：整体架构一张图

面试官可能问：
  画一下前后端架构。

建议答：
  浏览器访问 Vite 前端（端口 2048）→ 开发时 /api 代理到 backend（端口 3030）→ interview-provider-factory.ts 选 Provider → SSE 流回前端 → sse-client 解析 → InterviewMessageQueue（rAF 队列）→ MarkdownMessage（DOMPurify）展示。前端禁止直连 C++，必须走 /api。


第 6 题：从点击提交到文字显示（10 跳）

面试官可能问：
  用户提交答案后，数据怎么走？

建议答（按顺序）：
  1）mock-interview-space.vue 或 useMockInterviewSpaceMockState.ts 组装题目、资料摘要、feedbackStyle
  2）useInterviewStream.ts 创建 streaming 消息，调用 interview-stream.ts 的 startInterviewStream
  3）interview-stream.ts 判断 mock 或 remote（有无 stream endpoint）
  4）normalizeInterviewStreamRequest 压成一份 prompt
  5）remote 时 fetch POST /api/interview/stream，经 frontend/vite.config.ts 代理到 backend
  6）backend/src/controllers/interview-controller.ts 校验并 setupSSEResponse
  7）InterviewService + createInterviewProvider 产出 chunk/done/error
  8）sse-client.ts 按 \n\n 切帧解析
  9）interview-message-queue.ts 用 requestAnimationFrame 合并 delta 更新 DOM
  10）MarkdownMessage.vue：markdown-it + DOMPurify.sanitize 后渲染


第 7 题：Vite 代理为什么重要

面试官可能问：
  为什么不直接请求 localhost:3030？

建议答：
  开发时浏览器页面在 2048，直接打 3030 会触发 CORS（跨域）且部署环境端口不固定。统一用相对路径 /api，由 Vite proxy 转发到 VITE_INTERVIEW_BACKEND_ORIGIN（默认 http://localhost:3030），生产也可由网关做同样收敛。


第 8 题：前端目录怎么分层

面试官可能问：
  组件、composable、service 怎么划分？

建议答：
  views 只做页面编排；components 只做展示与 emit（事件抛出）；composables 放业务流程和场景状态（如 useMockInterviewSpaceMockState.ts）；services 放有 IO 的逻辑（SSE、分块、storage）；utils 纯函数；types 跨模块类型。宇宙页专用 UI 在 components/showcase/mock-interview-space/，可复用面试组件在 components/mock-interview/。

三、SSE 与流式（第 9～12 题）
────────────────────────────────────────


第 9 题：SSE 和 WebSocket 选型

面试官可能问：
  为什么用 SSE 不用 WebSocket？

建议答：
  面试反馈是服务端单向推 token，SSE 基于 HTTP 足够；复用 /api 与现有网关。WebSocket 适合双向高频协作；我们上行是一次 POST 提交答案，下行才是流。实现分三层：sse-client.ts（传输）、interview-stream.ts（控制）、useInterviewStream.ts（UI 编排）。


第 10 题：SSE 事件契约

面试官可能问：
  chunk 协议是什么？能随便改字段吗？

建议答：
  冻结事件：chunk（data 含 content）、done、error。前端 frontend/src/services/sse/sse-types.ts 与后端 backend/src/types/interview.ts 对齐。单方不能改，否则 Node、C++、前端联调全断；扩展须评审后先改 types 再改实现。


第 11 题：mock 与 remote 双模式

面试官可能问：
  没起 backend 怎么演示？mock 和真接口怎么统一？

建议答：
  resolveInterviewStreamEndpoint：配置了 VITE_INTERVIEW_SSE_URL 用该地址；否则 DEV 下默认 /api/interview/stream；否则 endpoint 为空，走前端 buildMockInterviewReply + splitTextToReader，不经网络。有 endpoint 即 remote，走 consumeSSEStream；无 endpoint 即 mock，走 consumeStreamText。下游统一 InterviewStreamEvent，UI 只认 start/chunk/done/error。


第 12 题：断线、取消、错误与重试

面试官可能问：
  用户刷新或取消怎么办？流断了 UI 卡住怎么办？

建议答：
  fetch 带 AbortSignal，useInterviewStream 可 abort 当前流。后端 response.on close 会 abort Provider。前端若连接结束未收到 done，onDone 补发 done，避免一直 streaming。error 时 flush 队列、展示文案，并写入 retryableState 支持 retryStream。

四、性能与安全（第 13～14 题）
────────────────────────────────────────


第 13 题：rAF 消息渲染队列

面试官可能问：
  流式输出卡顿怎么优化？为什么不用防抖？

建议答：
  LLM chunk 频率可能远高于屏幕刷新，每 chunk 直接改 displayContent 会导致大量重排。frontend/src/services/message/interview-message-queue.ts 按 messageId 缓冲 delta，每帧 requestAnimationFrame flush 一次，合并为一次 patch，把 DOM 更新频率压到约 60fps。防抖适合「停止输入后再触发」，不适合「持续到达的流式 token」。


第 14 题：XSS 与 Markdown 安全

面试官可能问：
  AI 返回的 Markdown 能直接 v-html 吗？

建议答：
  不能。frontend/src/components/message/MarkdownMessage.vue 先 renderMarkdownText（markdown-it），再 DOMPurify.sanitize，最后 v-html。流式内容不可信，必须在展示层消毒。报告侧 ReportReviewMarkdown.vue 同样处理。

五、资料分块与状态（第 15～17 题）
────────────────────────────────────────


第 15 题：资料 Markdown 分块与驱动出题

面试官可能问：
  长文档怎么切成题？和 RAG 有什么关系？

建议答：
  material-chunk-splitter.ts 按 # 标题层级切，单块上限 1000 字，输出 heading、level、charStart/charEnd。material-question-builder.ts 从 chunk 生成 MaterialQuestionItem；组卷带 sourceChunkId。答题和 frontend/src/utils/report/resolve-report-reference-answer.ts 可按 chunk 回溯资料参考答案。这是「按章节出题」的轻量方案，不是完整向量检索 RAG，但溯源清晰、成本低。


第 16 题：多入口工作流状态收敛

面试官可能问：
  从资料库、专项练习、直接面试进入，状态怎么不串？

建议答：
  双层模型在 frontend/src/types/workbench.ts：entryMode（direct / practice / material）+ mockSessionConfig（文档、题型、难度、反馈风格等）。持久化经 workbench-storage.ts、useWorkbenchPersistence.ts；业务真状态在 useMockInterviewSpaceMockState.ts；launch/replay 由 useMockInterviewFlow.ts 统一写 mockEntryMode 和 mockSessionConfig。sessionConfigKey 匹配可恢复会话，避免 A 入口恢复成 B 的配置。


第 17 题：为什么状态不放一个巨大 Store

面试官可能问：
  为什么不用 Pinia 管所有面试状态？

建议答：
  面试态是场景态，和全局主题、布局不同；大量需 localStorage 恢复。放 composable + 明确持久化字段，开始/结束/清空/恢复有唯一入口，避免多个 watch 偷偷改同一份核心状态。Pinia 在 store/ 用得轻，符合 MVP 边界。

六、协作、难点与收尾（第 18～20 题）

第 18 题：Node Provider 工厂与 C++ 并行

面试官可能问：
  怎么接 C++？前端要改吗？

建议答：
  backend/src/services/interview-provider-factory.ts 根据环境变量返回 MockInterviewProvider、RemoteLLMProvider 或 CppEngineProvider。对外仍是 POST /api/interview/stream 与相同 SSE 事件。前端不 import Provider，只消费契约。我若未写 C++ 侧，应说明：我负责前端契约与联调，引擎由后端/C++ 同学实现 Provider。


第 19 题：最难的问题与 STAR 示例

面试官可能问：
  遇到的最大难点？怎么验证的？

建议答（任选一条真实经历，STAR 结构）：

  示例 A（性能）：
    S：模拟面试流式输出时列表明显抖动。
    T：保证打字机体验且不掉帧。
    A：定位到 per-chunk 更新 DOM，引入 InterviewMessageQueue + rAF 合并。
    R：高频 token 下滚动与渲染稳定，联调 mock 与 remote 均通过。

  示例 B（状态）：
    S：从资料入口和练习入口恢复后会话串台。
    T：多入口可恢复且配置不互相覆盖。
    A：补 entryMode + sessionConfig 双层与 sessionConfigKey。
    R：刷新后仍能回到正确题组与反馈风格。

  验证方式：pnpm dev + pnpm dev:backend 联调；无 backend 时走前端 mock 分支演示。


第 20 题：测试、局限与后续规划

面试官可能问：
  怎么测的？上线数据？下一步做什么？

建议答：
  测试：本地双端启动，提交答案观察 SSE 流与 done 收尾；lint（pnpm lint）；关键路径手动回归多入口 launch 与恢复。若是 MVP/demo，诚实说暂无大规模线上指标，以功能闭环和联调通过为主。
  局限：单轮按题注入上下文，非无限聊天；chat 页与面试流是两条链路。
  后续：练习题池与真实 API 对齐、SSE 断线重连策略、报告弱项闭环、监控 error 事件率。

  收尾句：项目体现的是 BFF + 冻结契约下的可替换引擎，以及前端流式 UI 的分层与安全渲染，而不是简单调用一个 Chat API。
