# 45.总览页切轨掉帧优化方案

## 1. 背景

当前 `Mock Interview Space（模拟面试宇宙空间）` 的总览页在切换到其他轨道页时，存在明显卡顿和掉帧问题。

现象有一个比较稳定的触发条件：

- 如果用户只是快速略过 `InterviewAnalyticsDashboard（面试数据可视化大屏组件）` 区域，再回到上方切轨，基本不会掉帧。
- 如果用户向下滚动，让大屏真实进入可视区并完成一轮渲染，再回到上方切轨，就更容易出现切换卡顿、动画不顺、帧率下降。

这说明问题重点不在“轨道切换动画本身”，而在“大屏显示后产生的持续后台负载”。

## 2. 问题判断

结合页面结构和现象，可以初步判断性能瓶颈主要来自以下链路：

1. `ECharts（图表库）` 实例在大屏显示后完成初始化，并持续保留运行态。
2. `ResizeObserver（尺寸监听器）`、`scroll listener（滚动监听）`、`resize listener（窗口尺寸监听）` 等仍在离屏状态继续工作。
3. 图表或大屏内部可能仍保留 `requestAnimationFrame（逐帧调度）`、定时刷新、动态布局计算。
4. 总览页切轨动画与大屏后台任务竞争主线程，导致切换瞬间丢帧。
5. 现有大屏更像“隐藏但未停机”，而不是“离屏即降载”。

一句话概括：当前是“看过一次大屏后，大屏仍在后台消耗渲染预算”，所以用户回到顶部切轨时，主线程已经不干净了。

## 3. 优化目标

本次优化目标不是简单缩短动画时长，而是让切轨前后的主线程负载保持稳定。

目标拆分如下：

- 总览页切换至其他轨道页时，帧率保持高位稳定。
- 大屏仅在真正需要展示时才激活。
- 大屏离开视口后，不再持续占用不必要的计算、监听和重绘资源。
- 不修改既有业务接口，不改动前后端协作边界。
- 优化应优先控制生命周期和渲染策略，而不是先调视觉特效参数。

## 4. 根因假设

### 4.1 高概率根因

`InterviewAnalyticsDashboard（面试数据可视化大屏组件）` 在进入视口后，触发了完整图表初始化；但离开视口后，以下资源没有被有效暂停：

- 图表实例
- 尺寸监听
- 滚动响应
- 重排重绘相关逻辑
- 可能存在的动画或定时任务

于是，用户虽然视觉上已经回到总览顶部，但底部大屏仍在后台“活着”。

### 4.2 为什么“略过不会卡，真实显示后会卡”

因为“略过”通常只触发浅层布局经过，没有触发完整图表渲染链；而“真实显示”则会触发：

- 图表初始化
- 图表尺寸测量
- 数据渲染
- 监听器注册
- 可能的响应式联动更新

一旦这些任务被激活但没有在离屏时回收，后续切轨就会变重。

## 5. 优化总原则

本次方案遵循 4 个原则：

1. 可见才初始化。
2. 离屏就降载。
3. 切轨前先停大屏。
4. 重组件与主切换动画解耦。

## 6. 方案设计

### 6.1 方案一：基于可见性的激活控制

为大屏区域引入 `IntersectionObserver（交叉可见性监听）` 机制，统一管理大屏是否处于激活态。

#### 目标

让大屏从“页面存在即默认参与渲染”改为“进入可视区才参与渲染”。

#### 设计要点

- 首次未进入视口时，大屏图表不初始化。
- 进入视口达到阈值时，才允许图表实例化。
- 离开视口时，向下层图表组件广播“暂停”状态。
- 如果页面切换到其他轨道，即便页面还未滚动，也要强制关闭大屏激活态。

#### 预期收益

- 避免首屏无意义图表初始化。
- 避免离屏图表持续占用渲染资源。
- 降低总览页切轨时的并发主线程压力。

### 6.2 方案二：`BaseECharts（图表基座组件）` 增加离屏停机能力

大屏性能问题通常不是单一页面问题，而是图表基座层没有提供统一的“暂停 / 恢复 / 销毁”机制。

#### 目标

给图表基座建立明确的 3 种状态：

- `inactive（未激活）`
- `active（激活渲染）`
- `suspended（离屏暂停）`

#### 设计要点

当图表不可见或总览页不再处于当前交互焦点时：

- 停止 `resize（尺寸重算）`
- 断开 `ResizeObserver（尺寸监听器）`
- 停止动画帧调度
- 停止定时任务
- 视图重图场景下支持 `dispose（销毁实例）`

当再次可见时：

- 按需恢复监听
- 按需恢复图表实例
- 重新进行一次安全尺寸同步

#### 预期收益

- 解决“组件被隐藏但仍在运行”的根问题。
- 提供后续所有图表页都可复用的统一性能治理能力。

### 6.3 方案三：切轨动作前置“降载闸门”

这是最直接影响体感的一步。

#### 目标

在用户点击轨道切换按钮的瞬间，就先关闭总览页下方大屏的高成本任务，再启动页面切换动画。

#### 设计要点

切轨流程由：

“点击切换 -> 直接切场景”

改为：

“点击切换 -> 总览页大屏降载 -> 下一帧开始切场景动画”

#### 原因

因为当前最贵的不是点击本身，而是切换瞬间和后台任务并发。

#### 预期收益

- 明显减少切轨首帧卡顿。
- 降低动画启动时的主线程争抢。
- 改善“回到顶部后一切轨就顿一下”的主观感受。

### 6.4 方案四：大屏容器级渲染隔离

除生命周期控制外，还应对大屏区域增加容器级渲染隔离。

建议使用：

- `content-visibility（内容可见性优化）`
- `contain（渲染隔离）`

#### 目标

让下半屏大块复杂内容尽量不要影响上半屏轨道导航区的布局、绘制和合成。

#### 预期收益

- 降低离屏区域对全页渲染树的干扰。
- 减少总览页顶部动画被底部复杂 `DOM（文档对象模型）` 拖累的概率。

#### 注意

这个方案是辅助优化，不能替代图表生命周期治理。因为 `CSS（层叠样式表）` 只能减少布局和绘制参与，不能自动停止图表内部逻辑。

### 6.5 方案五：重图表策略分级

不是所有图表都应采用同一策略。

#### 建议分级

##### 轻量图

适合：

- 静态图
- 更新频率低
- 实例恢复成本低

处理方式：

- 离屏暂停，不销毁实例

##### 重量图

适合：

- 大数据量
- 多动画
- 频繁 resize
- 恢复时可接受重建成本

处理方式：

- 离屏直接销毁，回到可视区再重建

#### 预期收益

在性能和恢复速度之间取得平衡，不做“一刀切”。

## 7. 非目标范围

本次优化先不做以下事项：

- 不改后端接口
- 不改 `SSE（服务端推送协议）` 边界
- 不改业务数据结构
- 不先动星球视觉风格和主切页动效设计
- 不先做大规模图表重写

避免为了修性能问题，扩大成架构级重构。

## 8. 验收标准

建议以以下标准作为验收口径。

### 场景一：略过大屏后切轨

预期：

- 保持流畅，无明显顿挫
- 帧率稳定，无肉眼可见掉帧

### 场景二：完整显示大屏后返回顶部切轨

预期：

- 切轨流畅度与“略过大屏”场景接近
- 不应出现明显首帧卡顿
- 不应出现动画被拖慢的主观感受

### 场景三：多次上下滚动后反复切轨

预期：

- 不因多次进入大屏导致越来越卡
- 不出现累积型性能退化
- 大屏重新进入可视区时恢复正常

## 9. 实施顺序建议

建议按以下顺序推进：

1. 接入大屏可见性状态控制。
2. 给 `BaseECharts（图表基座组件）` 增加暂停 / 销毁能力。
3. 将总览页切轨动作与大屏降载联动。
4. 增加容器级渲染隔离。
5. 最后再微调动画参数和恢复节奏。

这样收益最大，且风险可控。

## 10. 风险与注意事项

### 风险一：图表恢复时机不稳定

如果重新进入视口时恢复过早，可能拿到错误容器尺寸，导致图表布局异常。

应对：

- 恢复时增加安全时机控制
- 仅在容器尺寸稳定后执行图表重绘

### 风险二：离屏直接销毁导致回显闪动

如果重图表每次都销毁，回到可视区可能会有轻微重建闪烁。

应对：

- 仅对重图启用销毁策略
- 轻图采用暂停策略

### 风险三：切轨前强制降载影响连续感

如果处理不当，可能出现切轨前状态切换过硬。

应对：

- 降载动作提前一帧完成
- 保证用户感知不到中间态

## 11. 结论

本问题的核心不是“切轨动画太重”，而是“下方大屏在真实显示后没有彻底离屏降载”。

所以本次优化重点应放在：

- 可见性驱动初始化
- 离屏暂停 / 销毁
- 切轨前主动降载
- 大屏区域渲染隔离

只要这四个点做对，切轨帧率会比单纯调动画参数更稳定，也更接近长期可维护方案。
