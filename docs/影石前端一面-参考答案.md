1. MCP 了解吗？在项目里怎么用？

面试回答：

**MCP**（Model Context Protocol，模型上下文协议）是一套让大模型应用能**标准化连接外部工具与数据源**的开放协议，可以理解成「给 AI 用的 USB 接口规范」：主机（Host，例如 Cursor、Claude Desktop）通过 MCP 客户端去连 MCP Server，Server 暴露 **Tools**（可调用函数）、**Resources**（可读文件/数据）、**Prompts**（模板）等能力。

和直接写 REST API 调用的区别：

- **契约统一**：工具描述、参数 schema、错误格式按协议来，换模型或换 Host 不用重写集成层。
- **进程隔离**：MCP Server 常独立进程跑（本地 stdio 或远程 HTTP/SSE），崩溃、权限、密钥不直接进模型上下文。
- **可组合**：一个 Host 可同时挂多个 Server（数据库、Git、浏览器、内部文档），模型在对话里按需选工具。

在前端/全栈项目里的典型用法：

（1）**IDE / Agent 场景**：Cursor 等通过 MCP 让 Agent 读仓库、跑命令、查文档——前端同学更多是**消费方**，负责配好 `.cursor/mcp.json` 或团队 Server 清单。
（2）**产品内嵌 Agent**：Node 网关做 MCP Host，把「查题池、拉面试记录、读资料 Markdown」封成 MCP Tools，模型通过 function calling 调，而不是把业务 SQL 写进 prompt。
（3）**与 RAG 配合**：MCP Resource 暴露向量库检索接口或原始文档片段，模型先 tool 再生成，减少胡编。

我会强调：**MCP 解决的是「模型怎么安全、可扩展地碰外部世界」**；业务鉴权、审计、限流仍要在 Server 侧自己做，协议本身不替代后端安全设计。

解析 / 要点：

```text
用户 / IDE (MCP Host)
        ↓ JSON-RPC / stdio / HTTP+SSE
   MCP Server A (git)   MCP Server B (db)   MCP Server C (docs)
        ↓                      ↓                    ↓
     本地仓库               只读查询              向量检索 / 文件
```

| 概念 | 作用 |
| --- | --- |
| Host | 发起连接、把工具结果塞回对话 |
| Server | 实现具体能力（工具、资源） |
| Tools | 模型可调用的「函数」，带 JSON Schema 参数 |
| Resources | 可订阅/读取的上下文（文件、URI） |

面试加分：对比 **OpenAI Function Calling** —— 思路类似，MCP 是跨厂商、可本地部署的标准层；别和 **MVC**、**MQTT** 混淆。

2. Subagent 了解吗？

面试回答：

**Subagent**（子代理）指在主 Agent（代理）编排下，再派生**职责单一、上下文隔离**的子任务执行体，完成一块工作后把**结构化结论**回传父 Agent，而不是在一个超长对话里硬塞所有步骤。

常见模式：

（1）**任务拆分**：父 Agent 做规划（读需求 → 列步骤），子 Agent A 专门探索代码库，子 Agent B 跑测试，子 Agent C 写文档——并行缩短 wall time。
（2）**上下文隔离**：子 Agent 只带子任务相关文件/指令，避免主对话 token 爆炸，也降低「前面错误结论污染后面」的风险。
（3）**工具权限分级**：子 Agent 可只读（explore），父 Agent 才允许写盘、执行 shell，符合最小权限。
（4）**结果汇总**：子 Agent 返回摘要 + 关键路径/代码引用，父 Agent 做决策与合并，用户只看到一层回复。

和 **MCP** 的关系：MCP 是「连外部能力」的协议；Subagent 是「连内部推理/执行单元」的架构。二者可叠加——子 Agent 挂专用 MCP Server（例如只连 CI API）。

在产品里的价值：复杂需求（跨模块重构、联调排错）比单轮 Chat 稳定；代价是调度成本、重复检索、需要好的**任务描述与返回格式**约束。

解析 / 要点：

```text
        用户问题
            ↓
      Parent Agent（规划 / 汇总）
       ↙    ↓    ↘
  Subagent-探索  Subagent-测试  Subagent-文档
       ↘    ↓    ↙
      合并结论 → 用户可见回复
```

| 对比项 | 单 Agent 长对话 | Subagent |
| --- | --- | --- |
| 上下文 | 易膨胀 | 按任务切片 |
| 并行 | 难 | 探索/测试可并行 |
| 风险 | 一步错全程偏 | 可限定子任务范围 |

Cursor 等产品里的 `Task` 工具、`explore` / `shell` 子类型即此类实现。面试时提一句：**要有明确输入/输出契约**，否则子 Agent 结果难以合并。

3. RAG 了解吗？怎么做的？

面试回答：

**RAG**（Retrieval-Augmented Generation，检索增强生成）是在让大模型生成之前，**先从知识库检索相关片段**，拼进 prompt，再生成答案。用来缓解模型幻觉、知识截止、企业内部文档不在训练集里的问题。

标准流水线：

（1）**文档入库**：原始资料（PDF/Markdown/网页）→ 清洗 → **分块**（chunk，例如 500–1000 字，带重叠 overlap 防断句）→ **向量化**（embedding）→ 写入向量库。
（2）**检索**：用户问题 → query embedding → 相似度检索（TopK）→ 可选 **重排序**（rerank，交叉编码器）→ 得到若干 chunk。
（3）**增强生成**：把 chunk 原文 + 引用来源 + 用户问题 组成 prompt → LLM 生成；要求**仅依据引用作答**，不知道就说不知道。
（4）**工程配套**：chunk 元数据（文档 id、章节、页码）、引用标注、缓存、评测（命中率、答案 grounded 比例）。

和 **OfferPilot** 类产品的结合点：资料 Markdown 按标题分块、模拟面试按章节出题，本质就是「结构化 RAG」——不一定上向量库，**标题层级 + 关键词**也是检索策略；上了 embedding 后对口语化提问更稳。

优化我会主动提：**分块策略、Hybrid 检索（向量 + BM25）、查询改写、多轮对话指代消解**。

解析 / 要点：

```text
离线：文档 → 分块 → Embedding → 向量库
在线：问题 → 检索 TopK → 拼 Prompt → LLM → 答案 + 引用
```

| 环节 | 常见坑 |
| --- | --- |
| 分块 | 太大语义杂、太小丢上下文 |
| 检索 | 只向量漏关键词；只关键词漏语义 |
| Prompt | 不约束「据引用回答」易幻觉 |
| 评测 | 无 golden set 无法迭代 |

```typescript
// 伪代码：检索 + 生成
const chunks = await vectorStore.similaritySearch(question, { k: 5 })
const context = chunks.map(c => c.pageContent).join('\n---\n')
const answer = await llm.chat({
  messages: [
    { role: 'system', content: '仅根据下列资料回答，无依据请说明' },
    { role: 'user', content: `资料：\n${context}\n\n问题：${question}` }
  ]
})
```

4. 大文件上传怎么做？

面试回答：

浏览器单次 `POST`（HTTP 提交）整文件不适合大文件：内存占用高、超时、断网要重来。主流是 **分片上传**（把一个大 `File` 切成多段分别传）+ **断点续传** + **并发控制**。

**分片前后对比**（面试可画给面试官看）：

```text
分片前（用户选中的 1 个完整文件）：
  video.mp4  ████████████████████  共 48 MB，一次 POST 容易超时/失败要重头传

分片后（前端按 5 MB 一段切开，逻辑上还是同一个文件）：
  分片 0  █████  bytes 0      ~ 5MB   → POST /upload/part?part=0
  分片 1  █████  bytes 5MB    ~ 10MB  → POST /upload/part?part=1
  ...
  分片 9  █████  bytes 45MB   ~ 48MB  → POST /upload/part?part=9

服务端收齐 10 片后 merge（合并）→ 磁盘上的 video.mp4 与源文件字节一致
```

方案要点：

（1）**前端切片**：用 `File.slice(start, end)` 按固定大小（如 2–5 MB）切成多个 **`Blob` 分片**（二进制片段，见下方代码注释），每片带 **partNumber（分片序号）** 和整文件的 **hash（指纹）**。
（2）**秒传 / 续传**：上传前带 `fileHash`，服务端查是否已有完整文件或已有哪些分片（`uploadedList`），缺的才传。
（3）**并发上传**：限制同时 N 个分片（如 3–6），避免占满带宽和浏览器连接数；单片失败单独重试，带指数退避。
（4）**合并**：全部分片到齐后，服务端 **merge**（按 partNumber 顺序拼接），返回最终 URL。
（5）**体验**：`fetch` + 进度汇总；算 hash 放 **Web Worker**（见第 5 题），不卡页面。

安全与协议：每片带 **uploadId、签名、过期时间**；合并时校验片数与 hash。

解析 / 要点：

```text
选文件 → 计算 fileHash → 查已传分片列表
    → 并发 POST 每一片 (partNumber + 该片 Blob)
    → 服务端 merge → 返回 fileUrl
```

**分片 + 单片上传示意代码**（陌生英文在注释里说明）：

```javascript
/**
 * file：用户 <input type="file"> 选中的完整文件对象（含 name、size）
 * Blob：浏览器里的「二进制大对象」，分片后每一段都是 Blob，可单独放进 FormData 上传
 * slice：从原 File 上「切一刀」，不复制整文件，省内存
 */
const CHUNK_SIZE = 5 * 1024 * 1024 // 每片 5MB
const file = input.files[0] // 例如 video.mp4，48MB

// ---------- 第一步：分片（大文件 → 多个小 Blob）----------
const chunks = []
let partNumber = 0
for (let start = 0; start < file.size; start += CHUNK_SIZE) {
  const end = Math.min(start + CHUNK_SIZE, file.size)
  const chunkBlob = file.slice(start, end) // Blob 分片，只引用原文件的一段字节
  chunks.push({ partNumber, chunkBlob, start, end })
  partNumber++
}
// 48MB / 5MB ≈ 10 个 chunk；chunks[0].chunkBlob 约 5MB，chunks[9] 约 3MB

// ---------- 第二步：单片上传（每片一次 HTTP）----------
async function uploadPart({ uploadId, partNumber, chunkBlob, fileHash }) {
  const form = new FormData() // 表单体，可混文件字段和普通字段
  form.append('uploadId', uploadId) // 本次上传任务 id，服务端用来归并分片
  form.append('partNumber', String(partNumber)) // 第几片，merge 时必须按序
  form.append('fileHash', fileHash) // 整文件指纹，用于秒传/校验
  form.append('chunk', chunkBlob, `part-${partNumber}`) // 该片二进制，字段名 chunk

  await fetch('/api/upload/part', { method: 'POST', body: form })
}

// ---------- 第三步：并发上传 + 通知合并 ----------
const fileHash = await calcHashInWorker(file) // 在 Worker 里算，见第 5 题
const uploadId = await initUpload({ fileHash, fileName: file.name, totalParts: chunks.length })

await Promise.all(
  chunks.map(c => uploadPart({ uploadId, ...c, fileHash }))
)

await fetch('/api/upload/merge', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ uploadId, fileHash, totalParts: chunks.length })
})
// merge 后服务端得到完整 video.mp4
```

| 名词 | 含义 |
| --- | --- |
| `File` | 带文件名的 `Blob`，来自用户选择 |
| `Blob` | 纯二进制块；分片后每片是一个 `Blob` |
| `slice` | 切分，得到新 `Blob`，不一次性读入 48MB |
| `FormData` | 以 multipart 形式 POST 文件片段 |
| `partNumber` | 分片序号，合并时保证顺序 |
| `merge` | 服务端按序号把各片拼回完整文件 |

| 能力 | 实现 |
| --- | --- |
| 断点续传 | 服务端记录已收分片，只传缺失的 `partNumber` |
| 秒传 | 同 `fileHash` 文件已存在则跳过上传 |
| 进度 | 各片 `onprogress` 按字节加权 |
| 校验 | merge 后全文件 hash 与前端一致 |

5. Web Worker 能传递什么类型的参数？

面试回答：

`postMessage`（消息传递 API）在 **主线程与 Worker 线程之间**传的是数据的**拷贝或转移**，两边各有一份（或所有权转给 Worker），**不是**共享同一个 JS 对象引用。

**结构化克隆**（Structured Clone Algorithm，浏览器内置的「可序列化深拷贝」）可传：

- 基本类型：`string`、`number`、`boolean`、`null`、`undefined`、`bigint`
- 普通对象、数组、`Date`、`RegExp`、`Map`、`Set`、`ArrayBuffer`（拷贝一份）
- `File`、`Blob`（第 4 题分片、算 hash 时常用）
- 部分 typed array（类型化数组视图）

**不能 / 不宜直接传**：

- 函数、DOM 节点、`Symbol`、循环引用对象
- 带复杂 prototype 的类实例（方法会丢）

**Transferable**（可转移列表）：`ArrayBuffer` 等可用 `postMessage(data, [transferList])` **转移所有权**，零拷贝，传完后主线程里原 buffer 不可用（`detached`）。

实践：大文件 hash 在 Worker 里读 `File.slice` 得到的 `Blob`，主线程只收 `{ hash, percent }`；别把 Vue 响应式对象整包丢进 Worker。

解析 / 要点：

**Web Worker 是什么**

- 浏览器提供的**独立 JS 线程**，与页面主线程（UI、事件、Vue/React 渲染）并行。
- 每个 Worker 有自己的全局 `self`，**不能直接操作 DOM**，也拿不到 `window`。
- 通过 `new Worker(url)` 或 **模块 Worker** `new Worker(url, { type: 'module' })` 启动。

**作用（为什么要用）**

- 把 **CPU 密集** 任务挪出主线程：大文件 hash、加解密、图片压缩、解析大 JSON、复杂排序。
- 避免主线程长时间占用导致**页面卡顿、掉帧**；第 4 题若在主线程算 48MB 的 MD5，上传按钮会「假死」。

**为什么能传这些类型（机制）**

- 两线程内存隔离，不能共享普通对象，否则会有竞态。
- `postMessage` 走 **结构化克隆**：浏览器把数据序列化成副本（或走 **Transferable** 把 `ArrayBuffer` 所有权划过去）。
- 所以能传的是「可克隆的数据」，不能传「可执行的函数」或「绑在 DOM 上的节点」。

**常见使用例子**

| 场景 | 主线程 | Worker |
| --- | --- | --- |
| 大文件上传 hash | 选文件、上传、更新进度条 | 对每片 `Blob` 算 MD5/SHA |
| 图片压缩 | 展示预览 | `canvas` + `OffscreenCanvas` 缩放 |
| 后台轮询解析 | 更新列表 UI | 解析超大 CSV/日志 |
| 加密 | 提交表单 | AES 处理敏感字段 |

**与第 4 题联动的完整示意**：

```javascript
// ========== main.js（主线程：UI、上传）==========
const worker = new Worker('/hash-worker.js')

worker.postMessage({
  chunkBlobs: chunks.map(c => c.chunkBlob), // Blob[]，结构化克隆，每片可在 Worker 里读
  fileName: file.name
})

worker.onmessage = (e) => {
  const { hash, percent } = e.data // 主线程收到的是副本，不是共享引用
  console.log('算 hash 进度', percent, '结果', hash)
}

// ========== hash-worker.js（Worker 线程：只算 hash，不碰 DOM）==========
self.onmessage = async (e) => {
  const { chunkBlobs, fileName } = e.data
  let digest = ''
  for (let i = 0; i < chunkBlobs.length; i++) {
    const buf = await chunkBlobs[i].arrayBuffer() // Blob → ArrayBuffer，二进制缓冲
    digest += await md5Chunk(buf) // 伪代码：逐片累加 hash
    self.postMessage({ percent: ((i + 1) / chunkBlobs.length) * 100 })
  }
  self.postMessage({ hash: digest, done: true })
}

// ---------- Transferable 示例：把 buffer「过户」给 Worker，主线程不再能读 ----------
const buf = new ArrayBuffer(1024)
worker.postMessage({ buf }, [buf]) // 第二参数：转移列表；传完后 buf.byteLength === 0（已分离）
```

| 类型 | 行为 |
| --- | --- |
| 普通对象 / Blob | 深拷贝（结构化克隆） |
| `ArrayBuffer` | 默认拷贝；也可 Transferable 转移 |
| 函数 / DOM | 不可传 |

6. 大文件分片上传会遇到什么性能问题？（面经原问：「这样分片会遇到什么性能问题」）

> **「这样分片」指什么**：紧接第 4 题——把用户选的**一个大文件**（如 48MB 的 `video.mp4`）用 `File.slice` 切成多个 **`Blob` 分片**，再**每片单独发一次 HTTP 上传**，最后在服务端 **merge** 拼回完整文件。不是数据库分片、也不是 React 组件拆分。

面试回答：

第 4 题那种**大文件分片上传**能缓解超时和断点重传，但实现不当会带来这些**性能与稳定性**问题：

（1）**请求风暴**：分片过小 → HTTP 次数暴增，握手与 Header 开销占比高；过大 → 单片失败重传成本高、内存峰值高。一般要按网速与接口限流动态调 chunkSize。
（2）**并发过高**：浏览器同域连接数有限（HTTP/1.1 约 6），并发太多反而排队；HTTP/2 多路复用好些，但仍要服务端与对象存储扛得住。
（3）**主线程算 hash**：全量 MD5 会长时间占 CPU，页面卡顿；应放 **Web Worker**，或 **抽样 hash**（首尾+中间）做弱校验换速度。
（4）**内存**：`FileReader.readAsArrayBuffer` 一次读整文件会爆内存；必须 **slice 逐片读**。
（5）**服务端 merge**：磁盘顺序拼接大文件 IO 重；对象存储用 **multipart upload API** 更合适；merge 时要防重复片、乱序片。
（6）**弱网重试**：无限重试打满服务端；要有最大重试、退避、单片超时。
（7）**列表接口放大**：续传前「查已传分片」若分片上万，响应体也大，可改 bitmap 或范围查询。

优化口诀：**合适的片大小、受限并发、Worker 算 hash、片级重试、服务端原生分片合并**。

解析 / 要点：

```text
【分片对象】同一个 video.mp4
  不分片：1 次 POST 48MB → 易超时，失败从头传
  分片后：10 次 POST 各约 5MB → 单片失败只重传 5MB

【性能权衡】
  片太小(如 100KB) → 请求次数爆炸，TCP/HTTP 握手开销 > 传数据
  片太大(如 50MB)  → 单片失败重传贵，浏览器瞬时内存高
  并发过高         → 同域连接数打满，反而排队（HTTP/1.1 常见 6 连接）
```

| 现象 | 可能原因（都在「分片上传」语境下） |
| --- | --- |
| 上传慢 | `CHUNK_SIZE` 过小导致请求过多；或并发设太低 |
| 页面卡 | 在主线程对**整文件或每片**算 hash，未用 Worker |
| 内存涨 | 用 `readAsArrayBuffer(整个 file)` 而非 `slice` 逐片读 |
| 合并慢 | 服务端用 Node 单线程顺序写盘；应用层 merge 未用对象存储 multipart API |
| 续传接口慢 | 已传分片列表上万条，JSON 过大 |

7. 什么是 MVVM？MVVM 是怎么监听对象里的数据修改的？

面试回答：

**MVVM**（Model-View-ViewModel）把界面（View）和业务数据（Model）拆开，中间用 **ViewModel** 做绑定：View 只声明「绑什么」，ViewModel 暴露可观察状态，Model 负责真实数据与接口。典型代表：**Vue**、**Knockout**。

数据流：

- **View → ViewModel**：事件（输入、点击）改 ViewModel 状态
- **ViewModel → View**：状态变了，View 自动更新（不用手写 DOM API）

「怎么监听对象里的数据修改」——面试官常期待 **两套实现**：

**Vue 2：Object.defineProperty**

- 初始化时遍历 data 的 key，对每个属性用 `defineProperty` 定义 `getter/setter`
- **dep** 依赖收集：读属性时把当前 **Watcher** 记下来
- **setter** 触发：派发更新，通知 Watcher → 更新 DOM
- 局限：新增属性要用 `Vue.set`；数组下标和 `length` 需特殊处理

**Vue 3：Proxy**

- 用 `Proxy` 代理整个对象，拦截 `get/set/deleteProperty` 等 **11+ 种操作**
- 天然支持新增/删除属性、数组索引、`Map/Set`（配合 reactive API）
- 性能更好在「懒代理」：只有访问到的嵌套对象才递归代理

**React** 严格说更像 **View + ViewModel 合在组件函数里**，靠 setState 触发重渲染，不是经典 MVVM 双向绑定，但思想相近。

面经里答 Proxy 面试官还想听 **defineProperty**，建议两套都讲，并说明 Vue 3 为何迁移。

解析 / 要点：

```javascript
// defineProperty 简化
function defineReactive(obj, key, val) {
  const dep = []
  Object.defineProperty(obj, key, {
    get() { /* 收集 dep */ return val },
    set(newVal) {
      val = newVal
      dep.forEach(fn => fn()) // 触发更新
    }
  })
}

// Proxy 简化
function reactive(obj) {
  return new Proxy(obj, {
    get(target, key) { /* track */ return target[key] },
    set(target, key, value) {
      target[key] = value
      /* trigger */
      return true
    }
  })
}
```

| 对比 | defineProperty | Proxy |
| --- | --- | --- |
| 新增属性 | 需额外 API | 可拦截 |
| 数组 | 需 hack | 原生拦截 |
| 兼容性 | IE 尚可 | 需现代浏览器 |

8. Vue 3 的 ref、reactive、computed 等了解吗？

面试回答：

这是 **Vue 3 Composition API**（组合式 API）的核心响应式与派生 API。

**ref**

- 包一层 `.value` 的响应式引用，基本类型和「需要整体替换」的对象都用它
- 模板里自动解包，不用写 `.value`
- 内部对 object 也会再转成 reactive

**reactive**

- 对**对象/数组/Map/Set** 做 Proxy 代理，深层响应式
- 不能代理普通类型；**不能解构**（解构丢响应式），要用 `toRefs`
- 始终操作同一引用，适合「一大块状态」如表单 model

**computed**

- 带缓存的**派生状态**：依赖的 ref/reactive 不变，多次访问不重新算
- 默认只读；传 `{ get, set }` 可做可写 computed
- 适合过滤列表、拼接展示文案，别在里面写副作用

相关常考点：

- **watch / watchEffect**：watch 显式指定源；watchEffect 自动收集依赖，立即跑一遍
- **toRef / toRefs**：从 reactive 拆出 ref，保持响应式
- **shallowRef / shallowReactive**：大对象只需浅层更新时用，省性能

和 React 对比：Vue 是「细粒度依赖追踪自动更新 DOM」；React 是「setState 触发整组件函数重执行 + diff」。

解析 / 要点：

```typescript
import { ref, reactive, computed, watch } from 'vue'

const count = ref(0)
const state = reactive({ list: [] as string[] })

const doubled = computed(() => count.value * 2)

watch(count, (n) => console.log(n))

// 模板: {{ count }} {{ doubled }}
```

| API | 适用 |
| --- | --- |
| ref | 标量、会被整体替换的引用 |
| reactive | 复杂对象、表单、嵌套结构 |
| computed | 由状态推导的展示值 |
| watch | 异步副作用、精准监听 |

```text
ref/reactive 变 → 依赖收集 → 调度 effect → 组件 patch DOM
computed 变 → 仅 invalidate 缓存 → 下次读时重算
```

9. 手撕：快排、二分查找

面试回答：

**快排**（Quick Sort）分治：选基准 pivot，分区使左边 ≤ pivot、右边 ≥ pivot，再递归左右。平均 O(n log n)，最坏 O(n²)（已有序且总选最边 pivot）；实现时注意 **原地、双指针分区、避免栈溢出**（大数组可随机 pivot）。

**二分**（Binary Search）在**有序**数组上每次砍掉一半，O(log n)。写法要统一 **闭区间 [left, right]** 或左闭右开，避免死循环；找边界变体：第一个 ≥ target、最后一个 ≤ target。

下面给出可直接白板写的 JavaScript 版本。

解析 / 要点：

**快排**

```javascript
function quickSort(arr, left = 0, right = arr.length - 1) {
  if (left >= right) return arr
  const pivot = partition(arr, left, right)
  quickSort(arr, left, pivot - 1)
  quickSort(arr, pivot + 1, right)
  return arr
}

function partition(arr, left, right) {
  const pivotVal = arr[right]
  let i = left
  for (let j = left; j < right; j++) {
    if (arr[j] <= pivotVal) {
      ;[arr[i], arr[j]] = [arr[j], arr[i]]
      i++
    }
  }
  ;[arr[i], arr[right]] = [arr[right], arr[i]]
  return i
}
```

| 要点 | 说明 |
| --- | --- |
| 稳定性 | 上述交换版不稳定 |
| 优化 | 随机 pivot、小数组换插入排序 |
| 空间 | 原地递归 O(log n) 栈 |

**二分（精确命中）**

```javascript
function binarySearch(arr, target) {
  let left = 0
  let right = arr.length - 1
  while (left <= right) {
    const mid = left + ((right - left) >> 1)
    if (arr[mid] === target) return mid
    if (arr[mid] < target) left = mid + 1
    else right = mid - 1
  }
  return -1
}
```

**lowerBound**（第一个 ≥ target，常用于插入位置）：

```javascript
function lowerBound(arr, target) {
  let left = 0
  let right = arr.length
  while (left < right) {
    const mid = left + ((right - left) >> 1)
    if (arr[mid] < target) left = mid + 1
    else right = mid
  }
  return left
}
```

手写检查清单：

- `mid = left + ((right - left) >> 1)` 防溢出
- 循环条件 `left <= right` vs `left < right` 与区间定义一致
- 快排 `partition` 返回 pivot 最终下标，递归边界 `pivot ± 1`
