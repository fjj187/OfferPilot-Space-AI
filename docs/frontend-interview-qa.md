# 前端面试题回答整理

## 1. 公司级项目里做前后端协作时，和上下游、后端一般怎么配合

我一般会把前后端协作拆成几个阶段来看。

第一步是需求对齐。
这一步不是直接开写代码，而是先把几个核心问题问清楚：

- 这个页面服务谁
- 用户操作链路是什么
- 哪些字段是前端展示必须依赖的
- 哪些数据是后端现算，哪些是直接返回
- 接口是实时的、轮询的，还是一次性请求
- 异常场景怎么处理，比如空数据、超时、权限不足、审核中

第二步是接口对齐。
我会重点看这几件事：

- 请求方式，是 `GET`、`POST`、`PUT` 还是 `DELETE`
- 入参结构，字段名、类型、是否必填、默认值
- 出参结构，业务码、提示文案、数据体结构
- 分页约定，比如 `pageNum`、`pageSize`、`total`
- 排序和筛选字段是否固定
- 时间格式、金额单位、布尔值表达方式是否统一

如果项目比较规范，一般会直接看接口文档，比如 Swagger、OpenAPI、YApi、Apifox。
如果文档不完整，我会主动和后端一起把关键字段确认掉，不会等到联调时再猜。

第三步是并行开发。
公司项目里不可能所有接口都等后端开发完才开始做，所以前端一般会先：

- 根据接口文档定义 TypeScript 类型
- 约定请求层和错误处理层
- 用 mock 数据先把页面结构和交互跑通
- 把 loading、空态、异常态一起补上

这样后面联调时，问题会更容易定位，到底是页面逻辑问题，还是接口返回问题。

第四步是联调。
联调时我通常会重点排查：

- 请求参数是不是按接口要求传了
- 请求头有没有带对，比如 token、租户信息、设备信息
- 返回的数据结构是不是和文档一致
- 字段有没有缺失、类型有没有变化
- 状态码和业务码是否符合预期
- 边界情况是否被后端覆盖，比如空数组、null、超长文本

如果有问题，我不会只说“接口不对”，而是会给后端明确结论，例如：

- 请求 URL
- 请求方法
- 入参
- 实际返回
- 预期返回
- 页面表现

这样沟通效率很高，后端也容易复现。

第五步是上线协作。
上线前后我会和测试、产品、后端一起确认：

- 环境变量是否正确
- 域名和代理是否正确
- 鉴权流程是否一致
- 灰度和正式环境配置是否一致
- 埋点、日志、监控是否可用

总结一下，我理解的前后端协作不是“前端等接口”，而是需求对齐、接口对齐、并行开发、联调排障、上线兜底这一整套流程。前端除了写页面，也要主动推进协作，把信息讲清楚，把问题定清楚。

## 2. 你理解的 RESTful 是什么

RESTful 可以理解成一种接口设计风格，不是某个具体框架，也不是某个库。

它的核心思想是：
用资源来描述业务对象，用统一的 HTTP 动词来描述操作。

比如用户、订单、商品，这些都可以看成资源。

常见设计方式：

- `GET /users` 获取用户列表
- `GET /users/1001` 获取某个用户详情
- `POST /users` 新增用户
- `PUT /users/1001` 整体更新某个用户
- `PATCH /users/1001` 局部更新某个用户
- `DELETE /users/1001` 删除某个用户

RESTful 的好处主要有几点：

- URL 更直观，看到路径大概就知道在操作什么资源
- 动词语义统一，团队协作成本更低
- 前后端接口规范更一致
- 更利于缓存、权限、网关、日志这些基础设施做统一处理

它不是死规定，实际业务里很多公司也会有折中写法，比如：

- `POST /user/queryList`
- `POST /order/cancel`

这类接口严格来说没那么 RESTful，但在复杂业务里很常见，因为有些动作本身更像“命令”，不只是对资源做简单增删改查。

所以我对 RESTful 的理解是：
它是一种推荐的资源化接口设计思想，目的是提高接口语义清晰度和系统一致性，不是必须百分百教条执行。

## 3. 简历里写了 JsBridge，它的实现原理是什么，项目里怎么用，是不是基于某个 APP 基座开发，端上同学有没有参与

JsBridge 本质上是 H5 和原生 App 之间的通信桥梁。

因为 WebView 里的 JavaScript 和原生代码是两个运行环境，它们不能天然直接互相调用，所以需要一层桥接机制。

JsBridge 一般分成两个方向：

### 3.1 JavaScript 调原生

常见方式有这几种：

1. URL Scheme 拦截

例如前端构造一个特殊 URL：

```ts
myapp://openPage?name=detail&id=1001
```

WebView 检测到这个 URL 后不真正跳转，而是被原生拦截，解析参数，再执行对应原生能力。

2. 原生直接注入对象

比如 Android 里常见把对象挂到 `window` 上，前端可以直接调用：

```ts
window.NativeBridge.openCamera()
```

3. 消息通道

比如通过 `postMessage` 一类方式，把消息发给原生，由原生统一分发处理。

### 3.2 原生调 JavaScript

常见方式是原生执行一段 JS，或者调用前端提前挂载好的回调函数，例如：

```ts
window.onNativeCallback?.({
  type: 'chooseImage',
  data: { url: 'xxx' }
})
```

### 3.3 JsBridge 的实现原理

如果让我自己概括，核心就三件事：

- 定义通信协议，比如方法名、参数、回调 ID
- 建立双向通信通道
- 做异步回调管理

很多 Bridge 会设计成这种调用形式：

```ts
bridge.call('getToken', { forceRefresh: false }).then((res) => {
  console.log(res)
})
```

底层通常会做这些事：

1. 生成一个唯一的 `callbackId`
2. 把方法名、参数、回调 ID 组装成消息
3. 通过 WebView 通道发给原生
4. 原生执行完后再带着 `callbackId` 回传结果
5. 前端根据 `callbackId` 找到对应 Promise 的 `resolve` 或 `reject`

### 3.4 项目里一般怎么用

项目里最常见的使用场景有：

- 获取登录态或 token
- 获取设备信息、App 版本、系统版本
- 打开原生页面
- 关闭 H5 页面
- 调起拍照、相册、定位、分享
- 获取剪贴板、文件上传能力
- 支付、埋点、页面返回拦截

例如前端进入页面时，可能先通过 JsBridge 获取 token，再去请求业务接口。
又比如点击“去实名认证”，前端不自己写页面，而是调用原生路由跳转到 App 内的实名认证页。

### 3.5 是否基于某个 App 基座开发

如果面试官追问这一点，比较稳妥的回答方式是：

是的，通常这种 H5 页面不是完全独立运行，而是嵌在公司自己的 App 基座里。基座会提供 WebView 容器、登录态、通用导航、分享能力、设备能力和 Bridge 能力。前端主要基于这套约定来开发页面。

“App 基座”可以理解成一个承载 H5 的宿主环境，它会规定：

- 哪些桥方法可以用
- 页面生命周期怎么通知
- 导航栏怎么控制
- 登录态怎么同步
- 错误日志怎么上报

### 3.6 端上同学有没有参与支持

一般是会参与的，而且必须参与。

因为很多能力前端自己做不了，比如：

- 摄像头
- 文件系统
- 定位
- 推送
- 原生路由
- token 注入
- WebView 配置

端上同学一般会负责：

- 定义或实现原生桥方法
- 给出调用协议和参数格式
- 处理不同系统差异，尤其是 iOS 和 Android
- 协助排查 WebView 兼容问题
- 处理 Bridge 调用失败、版本兼容、权限问题

前端这边则更多负责：

- 封装 JS 调用层
- 做能力探测和降级处理
- 做参数校验和异常提示
- 对接业务使用

所以 JsBridge 不是前端单方面的东西，它本质上是前端和端上共同约定出来的一层通信协议。

## 4. 移动端开发里怎么调试，出现 bug 能做哪些排查动作

移动端调试我一般分成几个层次。

### 4.1 本地浏览器调试

如果页面本身是 H5 页面，第一步我会先在本地浏览器跑起来看问题能不能复现。

这一步能排查：

- 页面逻辑错误
- 样式问题
- 接口参数问题
- 基础事件问题
- 路由问题

这时候我会直接用浏览器开发者工具：

- 看 Console 控制台报错
- 看 Network 请求参数和返回结果
- 看 Elements 检查 DOM 和样式
- 打断点看变量值和调用栈

### 4.2 真机或模拟器调试

如果问题只在移动端出现，我会去真机或模拟器上复现。

常见场景有：

- 某些点击事件失效
- 键盘弹起导致布局错乱
- iOS 和 Android 表现不一致
- WebView 环境下白屏
- 原生能力调用失败

调试方式一般有：

- Android 通过 Chrome 远程调试 WebView
- iOS 通过 Safari 连接设备调试 WebView
- 看真机上的控制台输出
- 抓包看请求是否真实发出

### 4.3 联调环境排查

如果是开发环境、测试环境、预发环境的问题，我会检查：

- 当前连接的是哪个环境
- 接口域名是否正确
- 代理是否生效
- token 是否过期
- CORS 或网关配置是否异常
- 本地环境变量是否和目标环境一致

### 4.4 线上问题排查

线上 bug 不一定能直接断点，但可以做很多排查动作：

- 看前端埋点和错误日志
- 看接口失败率和状态码
- 看用户操作路径
- 看特定机型、系统版本、App 版本分布
- 看资源加载是否 404
- 看是否是缓存问题
- 看是否是灰度版本问题

如果公司接了监控平台，比如 Sentry 一类工具，还能看到：

- 报错堆栈
- 浏览器信息
- 系统信息
- 用户行为面包屑
- 接口请求记录

### 4.5 我能做到哪些具体动作

如果面试官问“你能不能自己排查”，我会明确回答能，而且至少能做这些：

- 自己连开发环境或测试环境
- 看浏览器控制台和网络请求
- 打断点调试前端逻辑
- 检查请求参数、响应结果、业务码
- 验证是否是缓存、环境变量、权限、路由问题
- 在真机上远程调试 WebView
- 配合端上同学一起看 Bridge 调用链路

如果再往上一层，我也会主动做复现条件归纳，比如：

- 哪个环境出现
- 哪个账号出现
- 哪个机型出现
- 哪个版本出现
- 稳定复现还是偶现

这类信息对排查效率帮助非常大。

## 5. 你对 SSE 的理解是什么，传输方向、通信特点，以及前端接收流式数据的方式

SSE 全称是 Server-Sent Events，中文可以理解成服务端推送事件。

它是基于 HTTP 的一种服务器向客户端单向持续推送数据的机制。

### 5.1 传输方向

SSE 是单向的：

- 服务端 -> 客户端

客户端可以先发起一个 HTTP 请求建立连接，但连接建立后，主要是服务端持续往客户端推数据。
它不像 WebSocket 那样天然支持双向实时通信。

### 5.2 通信特点

SSE 的典型特点有这些：

- 基于 HTTP，接入成本相对低
- 单向推送，适合服务端持续产出消息
- 文本协议，格式比较简单
- 默认支持断线重连机制
- 更适合消息流、进度流、增量文本流

常见应用场景：

- AI 聊天回复流式输出
- 日志流
- 任务执行进度
- 消息通知

### 5.3 数据格式

服务端返回的内容一般是 `text/event-stream`。

数据长这样：

```text
data: 第一段内容

data: 第二段内容

```

如果有事件名，也可能这样：

```text
event: message
data: hello

event: finish
data: done

```

每个事件块之间通常用空行分隔。

### 5.4 前端怎么接收

最经典的方式是使用浏览器原生 `EventSource`：

```ts
const es = new EventSource('/api/stream')

es.onmessage = (event) => {
  console.log(event.data)
}

es.onerror = () => {
  es.close()
}
```

如果服务端需要 `POST`、自定义请求头、携带更灵活的鉴权信息，原生 `EventSource` 会受限。
这里要注意，`EventSource` 是浏览器提供的 SSE 接收对象，它默认更适合简单的 `GET` 请求。

这时业务里经常会改用 `fetch` + `ReadableStream` 来自己读流：

```ts
const response = await fetch('/api/stream', {
  method: 'POST',
  headers: {
    Authorization: 'Bearer token'
  },
  body: JSON.stringify({ question: '你好' })
})

const reader = response.body?.getReader()
const decoder = new TextDecoder('utf-8')

while (reader) {
  const { done, value } = await reader.read()
  if (done) break
  const chunk = decoder.decode(value, { stream: true })
  console.log(chunk)
}
```

这里几个陌生语法解释一下：

- `ReadableStream` 是浏览器里的可读流对象，表示响应体可以一段一段读取
- `getReader()` 是拿到读取器
- `reader.read()` 每次读取一块二进制数据
- `TextDecoder` 是把二进制内容解码成字符串
- `done` 表示流是否结束
- `value` 是当前读取到的数据块

### 5.5 SSE 和 WebSocket 的区别

简单说：

- SSE 是服务端单向推送
- WebSocket 是双向通信

如果需求是“服务端不断往前端吐消息”，比如 AI 打字机效果，SSE 很适合。
如果需求是“双方频繁双向通信”，比如聊天室、多人协作、对战类场景，WebSocket 更合适。

## 6. Vue2 和 Vue3 的主要区别是什么，为什么 Vue2 的 defineProperty 有局限但很多场景还能响应，Vue2 做了什么补救，Vue3 为什么改成 Proxy

这个问题可以从几个层面来答。

### 6.1 Vue2 和 Vue3 的主要区别

第一，响应式实现不同。

- Vue2 基于 `Object.defineProperty`
- Vue3 基于 `Proxy`

第二，源码架构和性能优化不同。

Vue3 在编译阶段和运行时都做了很多优化，比如：

- 更好的 Tree Shaking
- 更细粒度的依赖收集
- 更高效的更新

第三，开发方式不同。

- Vue2 主要是 Options API
- Vue3 同时支持 Options API 和 Composition API

Composition API 更适合复杂逻辑复用，相关逻辑可以按功能组织，而不是被分散在 `data`、`methods`、`computed`、`watch` 里。

第四，TypeScript 支持更友好。

Vue3 在类型推导、组合式函数、泛型支持这些方面明显更适合中大型项目。

第五，内置能力更多。

Vue3 引入了：

- `Fragment`
- `Teleport`
- `Suspense`
- 更完善的自定义渲染能力

### 6.2 Vue2 的 defineProperty 有什么局限

`Object.defineProperty` 只能劫持已经存在的属性。

所以它有几个经典问题：

1. 对象新增属性监听不到

```ts
obj.a = 1
obj.b = 2
```

如果 `b` 一开始没被劫持，后加的 `b` 就不是响应式的。

2. 删除属性监听不到

```ts
delete obj.a
```

3. 数组下标修改监听不到

```ts
list[1] = 100
```

4. 直接修改数组长度监听不到

```ts
list.length = 0
```

### 6.3 为什么日常很多场景又还能响应

因为平时大多数业务场景里，数据结构往往在初始化时就确定了。

比如表单对象、详情对象、列表项结构，通常都会先把字段声明好：

```ts
data() {
  return {
    form: {
      name: '',
      age: 0
    }
  }
}
```

这种情况下，`name` 和 `age` 在初始化阶段就已经被 Vue2 劫持了，所以后面修改它们当然还能响应。

也就是说，不是 `defineProperty` 没问题，而是很多业务恰好踩不到它的边界。

### 6.4 Vue2 做了什么补救

Vue2 针对这些问题做过一些补救。

1. 提供 `Vue.set` 或 `this.$set`

当你给对象新增属性时，可以这样写：

```ts
this.$set(this.form, 'email', '')
```

这样 Vue 能手动把新属性变成响应式。

2. 重写数组变更方法

Vue2 并不是直接监听每个数组下标，而是重写了这些会修改数组的方法：

- `push`
- `pop`
- `shift`
- `unshift`
- `splice`
- `sort`
- `reverse`

当你调用这些方法时，Vue 能感知变更并触发更新。

比如：

```ts
this.list.splice(1, 1, 100)
```

这种写法就能触发响应。

### 6.5 Vue3 为什么改成 Proxy

因为 `Proxy` 是对整个对象做代理，不只是给某个属性加 getter 和 setter。

它的优势很明显：

1. 可以监听对象新增和删除
2. 可以更自然地监听数组下标和长度变化
3. 可以代理更多操作，比如 `in`、`deleteProperty`
4. 不需要递归遍历所有属性后再劫持，设计上更现代
5. 对 Map、Set 这类数据结构支持更好

所以 Vue3 改成 `Proxy`，核心原因就是：
响应式能力更完整，边界问题更少，扩展性更好，也更适合现代复杂应用。

## 7. 简历里写了参与多个企业级项目、也写了组件库，为什么这里只重点写这两个项目，你具体做过哪些组件

这个问题本质上是在考察你是不是只会“列项目名”，还是能讲清楚沉淀过什么能力。

比较好的回答思路是：

我简历里不是想把所有项目都平铺展开，而是挑最能代表我能力的项目重点写。因为面试时间有限，项目太多反而会稀释重点，所以我会优先写能体现复杂业务理解、前后端协作、组件抽象和工程化能力的项目。

如果继续展开，可以讲两层。

第一层，为什么选这两个项目。

- 一个更偏业务复杂度，能体现我在真实企业场景里的交付能力
- 一个更偏组件和基础建设，能体现我的抽象能力和复用能力

第二层，我具体做过哪些组件。

可以按通用组件和业务组件来讲。

通用组件我做过的方向一般包括：

- 表单组件
- 表格组件
- 弹窗组件
- 上传组件
- 选择器组件
- 分页组件
- 搜索筛选组件
- 权限按钮组件
- 空状态和异常状态组件
- Markdown 预览或富文本展示组件

业务组件一般会更贴近场景，比如：

- 审批流节点组件
- 订单状态时间轴组件
- 风险提示卡片组件
- 用户画像面板
- 数据概览看板组件

如果面试官继续问“你具体做过哪些”，要尽量往“设计细节”上答，而不是只报名字。

例如表格组件可以讲：

- 支持列配置化
- 支持插槽自定义列
- 支持分页、排序、筛选
- 支持行选择、批量操作
- 支持不同业务场景复用

这样就比“我做过表格组件”更有说服力。

## 8. 做表单组件时，是怎么设计表单系统的，表单项怎么设计，值怎么收集，校验怎么做，受控/非受控、默认值、回填这些怎么处理，一个通用表单至少要具备哪些能力

这是个很典型的系统设计题。

我一般会把表单系统拆成三层：

- Form 容器层
- FormItem 表单项层
- 具体字段组件层

### 8.1 Form 容器层

Form 容器负责整张表单的状态管理和对外能力暴露。

它通常要负责：

- 存储表单值
- 提供初始值
- 提供校验入口
- 提供重置能力
- 提供提交能力
- 向 FormItem 提供上下文

在 Vue 里，一般会通过 `provide/inject` 做上下文通信。

这里解释一下：

- `provide` 是父组件向后代组件提供数据
- `inject` 是后代组件接收祖先组件提供的数据

它适合这种“Form 和很多 FormItem 有层级关系”的场景，避免一层层透传 props。

### 8.2 FormItem 表单项层

FormItem 主要负责某一个字段的展示和管理。

它一般会有这些配置：

- `label` 标签名
- `field` 字段名
- `component` 使用什么输入组件
- `rules` 校验规则
- `required` 是否必填
- `defaultValue` 默认值
- `props` 传给具体输入组件的参数

FormItem 会做两件关键事情：

第一，和 Form 容器建立字段注册关系。
比如挂载时注册自己，卸载时注销自己。

第二，建立值的绑定。
它知道自己对应的是 `formModel[field]`，输入组件变化时，把值写回表单状态。

### 8.3 值怎么收集

值收集最常见的方式有两种。

第一种，统一维护一个 `model` 对象。

```ts
const model = reactive({
  name: '',
  age: undefined
})
```

每个表单项都通过 `field` 去读写：

- `model.name`
- `model.age`

提交时直接拿整个 `model` 即可。

第二种，是字段组件各自维护状态，再由 Form 做聚合。

这种方式也可以，但复杂度更高，除非有特殊需求，否则我更倾向第一种集中式数据源。

### 8.4 校验怎么做

校验一般分为：

- 同步校验
- 异步校验

同步校验比如：

- 必填
- 长度限制
- 正则格式

异步校验比如：

- 用户名是否重复
- 编号是否已存在

常见做法是每个字段配置 `rules`：

```ts
rules: [
  { required: true, message: '请输入姓名' },
  { pattern: /^1\\d{10}$/, message: '手机号格式不正确' }
]
```

表单提交时执行整体验证，失焦或输入时执行单项验证。

一个比较完整的校验系统通常要支持：

- `validate()` 校验整表
- `validateField(field)` 校验单字段
- `clearValidate()` 清除校验状态
- `resetFields()` 重置值和错误信息

### 8.5 受控和非受控怎么理解

受控和非受控这个概念在 React 里提得更多，但前端组件设计里通用。

受控组件可以理解成：
值由外部传入，组件内部不私自决定最终值，变化通过事件通知外部。

比如：

```ts
<MyInput :value="form.name" @update:value="handleChange" />
```

非受控组件可以理解成：
组件内部自己维护状态，外部只给一个初始值。

在企业级表单系统里，我更倾向受控模式，因为：

- 数据流更清晰
- 更方便统一收集和回填
- 更方便联动和校验

### 8.6 默认值和回填怎么处理

默认值一般分两种来源：

1. 配置默认值
2. 接口回填值

初始化时通常会做一次合并：

- 先放组件默认值
- 再放业务传入初始值
- 最后用接口返回值回填覆盖

这里要注意区分“首次初始化”和“编辑态回填”。

比如新增表单时用默认值。
编辑表单时要把接口详情数据正确映射回字段。

如果有嵌套字段，还要支持路径式取值，例如：

- `user.name`
- `address.city`

### 8.7 一个通用表单至少具备哪些能力

我觉得至少要有这些：

- 字段配置化
- 值收集
- 默认值初始化
- 编辑态回填
- 同步和异步校验
- 错误提示展示
- 提交与重置
- 字段联动
- 动态显示和隐藏
- 插槽扩展
- 布局控制
- 禁用和只读态

如果再往企业级一点，还会需要：

- 表单权限控制
- 大表单分区渲染
- 复杂字段组合
- 草稿保存
- 脏数据检测
- 与搜索表单、弹窗表单复用同一套模型

## 9. 你对 HTTP 缓存的理解是什么，强缓存和协商缓存怎么区分，强刷为什么能拿到最新资源，如果想让某个资源请求时不要走缓存，该怎么做

HTTP 缓存的目标是减少重复请求，加快资源加载，降低服务端压力。

浏览器缓存大体可以分成两类：

- 强缓存
- 协商缓存

### 9.1 强缓存

强缓存命中时，浏览器不会发请求到服务器，直接用本地缓存。

常见响应头有：

- `Cache-Control`
- `Expires`

其中更常用的是 `Cache-Control`。

例如：

```text
Cache-Control: max-age=3600
```

表示 3600 秒内资源有效，浏览器可直接使用。

### 9.2 协商缓存

协商缓存的特点是：
浏览器会发请求到服务器，但会带上缓存标识，问服务器“我本地这个还能不能用”。

如果能用，服务器返回 `304 Not Modified`，响应体通常为空，浏览器继续用本地缓存。

常见配套头有：

- `Last-Modified` 和 `If-Modified-Since`
- `ETag` 和 `If-None-Match`

其中：

- `Last-Modified` 基于最后修改时间
- `ETag` 基于资源唯一标识，精度通常更高

### 9.3 强缓存和协商缓存怎么区分

核心区别就是：

- 强缓存命中，不发请求
- 协商缓存命中，会发请求，但服务器返回 304

### 9.4 强刷为什么能拿到最新资源

浏览器强制刷新时，通常会跳过本地强缓存，重新向服务器发请求。

这时浏览器会要求重新验证资源，甚至直接重新拉取资源，所以更容易拿到最新内容。

本质上不是浏览器自动知道资源变了，而是强刷改变了缓存使用策略，让浏览器重新验证或重新拉取资源。

### 9.5 如果想让某个资源不要走缓存，怎么做

可以分前后端两边来看。

前端常见做法：

1. 给请求 URL 加时间戳或版本号

```ts
/api/data?t=1710000000000
```

因为 URL 变了，浏览器会把它当成新资源。

2. 对静态资源做文件指纹

比如：

- `app.a1b2c3.js`
- `style.x9y8z7.css`

打包构建工具一般会自动做这件事。

服务端常见做法：

1. 设置禁止缓存响应头

```text
Cache-Control: no-store
```

表示不要缓存，最彻底。

2. 使用：

```text
Cache-Control: no-cache
```

这里要注意：
`no-cache` 不是“不缓存”，而是“每次使用前都要向服务器协商验证”。

3. 配合协商缓存策略控制资源更新。

### 9.6 实战里怎么理解

一般来说：

- HTML 文件缓存会保守一些，避免用户一直拿旧入口
- JS、CSS、图片这类静态资源通常走强缓存 + 文件 hash
- 实时性很高的接口数据会关闭缓存或改成协商缓存

## 10. 请手写一个防抖函数，并运行验证一下。实际业务里怎么使用 debounce，平时有没有用过 Lodash

### 10.1 防抖是什么

防抖的意思是：
事件被频繁触发时，不要每次都立刻执行函数，而是等“停止触发一段时间后”再执行最后一次。

典型场景：

- 输入框搜索
- 窗口 resize
- 频繁点击按钮

### 10.2 手写一个基础版防抖函数

```ts
function debounce(fn, delay) {
  let timer = null

  return function (...args) {
    const context = this

    if (timer) {
      clearTimeout(timer)
    }

    timer = setTimeout(() => {
      fn.apply(context, args)
    }, delay)
  }
}
```

几个语法解释：

- `...args` 是剩余参数语法，表示把不定数量参数收集成数组
- `this` 指当前函数调用时的上下文
- `fn.apply(context, args)` 表示用指定的 `this` 和参数数组来调用函数
- `clearTimeout(timer)` 是清除上一次还没执行的定时器

### 10.3 运行验证示例

下面这个例子里，短时间连续触发三次，最后只会执行一次：

```ts
function debounce(fn, delay) {
  let timer = null

  return function (...args) {
    const context = this

    if (timer) clearTimeout(timer)

    timer = setTimeout(() => {
      fn.apply(context, args)
    }, delay)
  }
}

const log = debounce((value) => {
  console.log('执行了', value, Date.now())
}, 300)

log('a')
log('b')
log('c')
```

最终只会打印最后一次的 `c`。

### 10.4 实际业务里怎么用

我在业务里比较常用的几个场景：

1. 搜索输入框

用户输入关键字时，不会每敲一个字就请求一次接口，而是停顿 300 毫秒后再发请求。

2. 窗口尺寸变化

页面 resize 时会不断触发事件，可以防抖后再重新计算布局。

3. 按钮防重复提交

比如保存按钮、导出按钮，短时间内连续点击可以做防抖或节流。

### 10.5 有没有用过 Lodash

用过，实际业务里很常见。

Lodash 的 `debounce` 优点是比手写基础版更完善，通常支持：

- `leading` 立即执行
- `trailing` 结束后执行
- `cancel` 取消
- `flush` 立即触发一次

所以面试时可以说：

我能手写基础版理解原理，业务里如果需求比较标准，也会直接用 Lodash，提高稳定性和开发效率。

## 11. 你了解 Jest、测试用例、单元测试吗？单元测试是做什么的

了解。

Jest 是 JavaScript 生态里很常见的测试框架，常用于写单元测试，也支持 mock、断言、覆盖率统计这些能力。

测试用例可以理解成：
你为了验证某段代码行为是否符合预期，写的一组输入、执行和断言。

例如一个加法函数：

```ts
function add(a, b) {
  return a + b
}
```

测试用例可能是：

```ts
test('1 + 2 should be 3', () => {
  expect(add(1, 2)).toBe(3)
})
```

这里几个语法解释一下：

- `test` 表示定义一个测试
- `expect` 表示拿到实际结果
- `toBe` 表示断言结果应该严格等于某个值

### 11.1 单元测试是做什么的

单元测试的目标是验证最小可测试单元的正确性。

这个“单元”通常可以是：

- 一个函数
- 一个工具方法
- 一个组件的某个逻辑
- 一个状态管理里的 action

它的核心价值有这些：

1. 防止回归

你改了一段代码后，测试可以帮你检查原有功能有没有被改坏。

2. 提高重构信心

有测试兜底时，重构更稳，不容易因为改结构而改出隐藏 bug。

3. 固化边界条件

像空值、异常值、极端输入这些边界情况，可以直接写进测试里。

4. 帮助理解代码行为

好的测试本身也像文档，别人看测试就能知道函数预期行为。

### 11.2 我对单元测试的实际理解

不是所有代码都必须写很多测试。
通常更值得写测试的是：

- 工具函数
- 复杂业务计算逻辑
- 公共组件
- 容易出回归问题的模块

而纯样式页面、变化很快的简单展示层，测试投入要看收益。

## 12. 请你写一个爬楼梯，或者直接讲思路。状态转移和边界条件是什么

这是经典动态规划题。

题目通常是：
每次可以爬 1 阶或 2 阶，问爬到第 n 阶有多少种方法。

### 12.1 思路

假设 `dp[n]` 表示爬到第 `n` 阶的方法数。

那么最后一步只有两种可能：

- 从第 `n - 1` 阶爬 1 步上来
- 从第 `n - 2` 阶爬 2 步上来

所以状态转移方程就是：

```text
dp[n] = dp[n - 1] + dp[n - 2]
```

### 12.2 边界条件

边界条件要先定好。

常见定义方式：

- `dp[1] = 1`
- `dp[2] = 2`

解释一下：

- 到第 1 阶只有 1 种走法
- 到第 2 阶有 2 种走法，分别是 `1 + 1` 和 `2`

### 12.3 代码实现

```ts
function climbStairs(n) {
  if (n <= 2) return n

  let prev1 = 1
  let prev2 = 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2
    prev1 = prev2
    prev2 = current
  }

  return prev2
}
```

这里是空间优化写法，没有真的开一个 `dp` 数组。

语法解释：

- `let` 表示变量可以重新赋值
- `for` 循环表示从 3 一直计算到 `n`
- `current` 表示当前台阶的方法数
- `prev1` 表示前一个前状态，也就是旧的 `dp[i - 2]`
- `prev2` 表示前一个状态，也就是旧的 `dp[i - 1]`

### 12.4 为什么这题适合动态规划

因为它同时满足两个特征：

1. 最优子结构

当前问题的答案依赖更小子问题的答案。

2. 重复子问题

比如算 `dp[5]` 时会用到 `dp[4]`、`dp[3]`，而算 `dp[4]` 时又会用到 `dp[3]`、`dp[2]`。

所以用动态规划能避免重复计算。
