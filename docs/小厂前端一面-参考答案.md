# 小厂前端一面 · 参考答案

1. 代码题：用 `fetch` 请求接口后，为什么读不到返回的业务数据？

面试回答：

我**先讲正确写法**，再对照面试官常给的 7 种错误代码找 bug。

---

**正确写法**：

```javascript
async function loadUser() {
  const res = await fetch('/api/user', {
    credentials: 'include',                              // ✅（7）需要时带 Cookie
    headers: { Authorization: `Bearer ${token}` }      // ✅（7）或显式带 Token
  })
  if (!res.ok) throw new Error(`HTTP ${res.status}`)   // ✅（5）4xx/5xx 要先拦
  const data = await res.json()                        // ✅（1）（4）await 且只读一次
  return data.data                                     // ✅（2）从解析结果取字段，不是解构 fetch
}

// 调用时也要 await，不能在外面立刻读变量                        // ✅（3）
const user = await loadUser()
console.log(user.name)
```

**要点**：`fetch` → 判 `res.ok` → `await res.json()` 一次 → 再取业务字段；非 JSON 改用 `.text()` / `.blob()`（对应坑 6）。

---

**对照：面试官常考的 7 个坑**（代码里标 ❌）

**（1）忘了 `await res.json()`，把 Promise 当对象用**

```javascript
fetch('/api/user').then(res => {
  const data = res.json()   // ❌ 漏 await，data 是 Promise，不是 JSON 对象
  console.log(data.data)    // ❌ undefined：Promise 上没有 .data
})
```

**原因**：`res.json()` 也是异步的，必须 `await` 或再 `.then` 一层。

---

**（2）当成 axios，直接解构 `{ data }`**

```javascript
async function getUser() {
  const { data } = await fetch('/api/user')  // ❌ fetch 返回 Response，没有 data 字段
  return data                                 // ❌ undefined
}
```

**原因**：`{ data }` 是 axios 习惯；原生 `fetch` 要先 `await res.json()` 再取字段。

---

**（3）异步还没结束就读结果**

```javascript
let userData
fetch('/api/user')
  .then(res => res.json())
  .then(data => { userData = data })

console.log(userData.name)  // ❌ 此时请求多半还没回来，userData 是 undefined
```

**原因**：`fetch` 非阻塞；`console.log` 不会等上面的 `then` 执行完。

---

**（4）响应体读了两次**

```javascript
const res = await fetch('/api/user')
const a = await res.json()
const b = await res.json()  // ❌ TypeError: body stream already read
```

**原因**：`Response.body` 只能消费一次，第一次 `json()` 后流就关了。

---

**（5）不判断 HTTP 状态，404/500 也当成功**

```javascript
const res = await fetch('/api/user')  // 假设实际返回 404，body 是 HTML 错误页
const data = await res.json()         // ❌ Unexpected token '<'，HTML 不是 JSON
console.log(data.data)                // 走不到这里
```

**原因**：`fetch` 只有网络失败才 `reject`；4xx/5xx 仍会 resolve，要先 `if (!res.ok)`。

---

**（6）解析方式和 Content-Type 不匹配**

```javascript
const res = await fetch('/api/export')  // 服务端返回 text/csv 或空 body
const data = await res.json()           // ❌ 不是 JSON 却硬 parse，抛 SyntaxError
```

**原因**：看 `Content-Type`；纯文本用 `.text()`，文件用 `.blob()`。

---

**（7）跨域 / 鉴权没配好，根本收不到正常响应**

```javascript
// 场景 A：跨域
fetch('https://other-domain.com/api/user')  // ❌ 无 CORS 头 → 浏览器直接拦，then 不进

// 场景 B：要带 Cookie / Token 却没带
fetch('/api/user', {
  credentials: 'omit',                      // ❌ 默认 omit，Cookie 不发，后端当未登录
})
```

**原因**：Network 先看 CORS 红字还是 401；跨域要服务端放 `Access-Control-Allow-Origin`，登录态要 `credentials: 'include'` 或 Bearer Token。

解析 / 要点：

| 坑 | 面试官代码特征 | 现象 |
| --- | --- | --- |
| （1） | `res.json()` 没 await | `undefined` |
| （2） | `const { data } = await fetch(...)` | `undefined` |
| （3） | `fetch` 外面立刻读变量 | `undefined` |
| （4） | 两次 `res.json()` | stream already read |
| （5） | 不判断 `res.ok` | JSON parse 报错 |
| （6） | 非 JSON 也 `.json()` | SyntaxError |
| （7） | 跨域 / 没带 Cookie·Token | CORS 报错或 401 |

**fetch vs axios**：axios 拦截器常直接给你 `res.data` 并在 4xx 时 reject；`fetch` 都要自己写。

2. 页面输入 URL 到内容显示，经历了什么过程？

面试回答：

从地址栏输入 URL 到屏幕出现内容，可分 **导航 → 网络 → 解析 → 渲染** 四段。

（1）**用户输入与导航**

- 浏览器判断是搜索词还是 URL，补全 `https://`。
- 查 **HTTP 缓存**（强缓存 `Cache-Control`；协商缓存 `ETag` / `Last-Modified`），命中可能跳过网络。

（2）**DNS 与连接**

- **DNS 解析** 域名 → IP。
- **TCP 三次握手**；HTTPS 再 **TLS 握手**。
- HTTP/2/3 协商多路复用或 QUIC。

（3）**HTTP 请求与响应**

- 发 **请求报文**（Method、Path、Headers、Body）。
- 收 **响应报文**（见第 3 题）：状态行、响应头、Body。
- **301/302** 跟 `Location` 重定向；**304** 用本地缓存。

（4）**解析与渲染**

- HTML → **DOM**；CSS → **CSSOM** → **渲染树**。
- **Layout（回流）→ Paint（绘制）→ Composite（合成）**。
- JS 执行可能触发重排/重绘；并行加载图片、字体、XHR/fetch 等。

一句话：**URL → DNS → TCP/TLS → HTTP → 解析 HTML/CSS/JS → 布局绘制 → 显示**。

解析 / 要点：

```text
地址栏 URL → 缓存? → DNS → TCP → TLS
  → HTTP Request/Response
  → DOM + CSSOM → Render Tree → Layout → Paint → Composite
```


| 阶段  | 常考点                         |
| --- | --------------------------- |
| 缓存  | 强缓存 vs 304 协商               |
| 阻塞  | CSS 阻塞渲染；script defer/async |
| 性能  | 关键渲染路径、LCP                  |

3. HTTP 响应报文的具体内容是什么？

面试回答：

HTTP 响应报文由 **状态行 + 响应头 + 空行 + 消息体** 组成。

（1）**状态行**

```http
HTTP/1.1 200 OK
```

协议版本、**状态码**、原因短语。

（2）**响应头（常见）**


| Header                        | 含义         |
| ----------------------------- | ---------- |
| `Content-Type`                | body 类型与编码 |
| `Content-Length`              | body 字节长度  |
| `Cache-Control` / `ETag`      | 缓存策略       |
| `Set-Cookie`                  | 写 Cookie   |
| `Location`                    | 重定向地址      |
| `Access-Control-Allow-Origin` | CORS       |


（3）**空行**：`\r\n\r\n` 分隔头与 body。

（4）**消息体**：HTML、JSON、文件二进制，或空（**204 No Content**）。

**完整 JSON 示例**：

```http
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 57

{"code":0,"message":"ok","data":{"id":1,"name":"demo"}}
```

解析 / 要点：

```text
HTTP/1.1 200 OK    ← 状态行
Header: value        ← 响应头
                     ← 空行
{ "data": ... }      ← 消息体
```

- 与第 1 题联动：`fetch` 返回 `Response`，headers 与 body 分开读。
- 与第 5 题联动：状态行里的数字即 HTTP 状态码。

4. 元素水平居中有哪些方式？

面试回答：

先分清 **行内** 还是 **块级**、**单个** 还是 **多个** 元素。

（1）**`text-align: center`**：父元素居中 **inline / inline-block** 子内容。

（2）**`margin: 0 auto`**：子元素 **定宽 block**，左右 margin 自动平分。

（3）**Flex（最常用）**

```css
.parent {
  display: flex;
  justify-content: center;
}
```

（4）**Grid**

```css
.parent {
  display: grid;
  place-items: center;
}
```

（5）**绝对定位 + `transform`**

```css
.child {
  position: absolute;
  left: 50%;
  transform: translateX(-50%);
}
```

（6）**绝对定位 + 定宽高 + `margin: auto`**

**口诀**：常规用 **Flex**；文字用 **text-align**；定宽卡片用 **margin auto**；弹层用 **absolute + transform/grid**。

解析 / 要点：


| 场景     | 推荐                     |
| ------ | ---------------------- |
| 多个按钮一排 | flex + justify-content |
| 固定宽表单  | margin: 0 auto         |
| 全屏蒙层   | grid place-items       |


注意：`text-align` 只对 inline 级生效；`margin auto` 需要宽度；绝对定位脱离文档流。

5. HTTP 状态码：五个种类与常见码

面试回答：

按首位分 **五类**：


| 类别      | 范围      | 含义      |
| ------- | ------- | ------- |
| **1xx** | 100–199 | 信息，协议继续 |
| **2xx** | 200–299 | 成功      |
| **3xx** | 300–399 | 重定向     |
| **4xx** | 400–499 | 客户端错误   |
| **5xx** | 500–599 | 服务端错误   |


**常见码**：

- **200 OK**、**201 Created**、**204 No Content**
- **301** 永久重定向、**302** 临时、**304 Not Modified**（协商缓存）
- **400 Bad Request**、**401 Unauthorized**（未登录）、**403 Forbidden**（无权限）、**404 Not Found**、**429** 限流
- **500 Internal Server Error**、**502 Bad Gateway**、**503 Service Unavailable**、**504 Gateway Timeout**

前端习惯：**401** 跳登录；**403** 提示权限；**5xx** 友好重试；`fetch` 对 404 不 throw，需看 `res.ok`。

解析 / 要点：

```text
1xx 继续 | 2xx 成功 | 3xx 跳转 | 4xx 客户端错 | 5xx 服务端错
```

- **401 vs 403**：没认证 vs 没权限——高频考点。
- **304**：不是错误，是用本地缓存副本。

6. LLM 应用里的状态管理怎么做？

面试回答：

LLM（Large Language Model，大语言模型）产品除了普通页面态，还有 **对话历史、流式中间态、工具结果、会话恢复**，要分层建模。

（1）**会话级（Session）**：`sessionId`、场景、题序、用户配置；跨组件用 **Pinia** 或 **composable（组合函数）**；需刷新恢复的可写 **localStorage**。

（2）**消息级（Messages）**：`messages[]`，含 `role`、`content`、`threadId`、`status`（streaming/done/error）。SSE（Server-Sent Events）流式时 **往当前 assistant 消息 append chunk**，不要每个 token 一条。

（3）**流控级（Stream）**：`isStreaming`、`AbortController`（停止生成）、错误与重试；传输层与 UI 解耦。

（4）**派生态**：当前线程消息、能否提交等用 **computed（计算属性）** 推导，不重复存。

**单一写入入口**：`appendMessage` / `updateMessage` / `clearMessages`，避免多处 `watch` 偷偷改同一状态。

若有项目经验可举例：对话页 messages + 流式 queue + localStorage 恢复 session；结束/切场景时明确哪些 state reset。

解析 / 要点：

```text
Session → Messages → Stream Control → Derived(computed)
```


| 状态               | 持久化    |
| ---------------- | ------ |
| 消息列表             | 可选摘要入库 |
| streaming buffer | 否      |
| 用户手动配置           | 是      |


小厂若做 AI 产品，面试官可能追问：**流式 UI 卡顿怎么办**——可答 rAF 合并增量、节流渲染。

7. 内容超出上下文（Context）限制怎么办？（按优先级压缩）

面试回答：

模型 **上下文窗口**（如 8K/128K tokens）有限，超限要 **按优先级保留、从低到高压缩或丢弃**。

**P0 — 必须保留**

- System 指令（角色、格式、安全边界）
- **当前用户问题 / 当前任务**
- 正在使用的 tool/function schema

**P1 — 高优先**

- **最近 N 轮对话**（滑动窗口）
- RAG 检索 **TopK 相关 chunk**，带引用

**P2 — 可摘要**

- 更早历史 → **LLM 摘要**成几百字「会话摘要」替换原文
- 冗长 tool 输出 → 只留关键字段或摘要

**P3 — 先删**

- 无关旧 tool trace、重复 few-shot、整篇文档原文（改分块检索）

**工程手段**：发送前 **token 预算**；分层上下文（system / 摘要 / 近期 / 检索）；metadata 存用户偏好，别每轮重复啰嗦。

解析 / 要点：

```text
超窗 → 保 P0 → 保 P1 近 K 轮 + RAG
     → P2 摘要旧对话 → P3 删冗余
```


| 策略       | 适用    |
| -------- | ----- |
| 滑动窗口     | 多轮聊天  |
| 摘要压缩     | 长会话   |
| RAG TopK | 知识库问答 |


本场公司还问了 **任务怎么拆解喂给 AI**——可一并答：大需求拆成「目标 / 约束 / 文件范围 / 验收标准」再交给 Agent，比一整段模糊 prompt 稳。

8. Git：一套完整的推送流程（常用命令）

面试回答：

（1）**看状态**

```bash
git status
git diff
git diff --staged
```

（2）**同步远端（协作时先做）**

```bash
git pull --rebase origin main
```

（3）**暂存与提交**

```bash
git add .
git commit -m "feat: 具体说明"
```

（4）**推送**

```bash
git push origin feature/xxx
# 首次：git push -u origin feature/xxx
```

（5）**PR / MR** Review 后合并。

**完整示例**：

```bash
git checkout -b feat/dark-mode
# 改代码...
git status
git add src/
git commit -m "feat: 深色模式切换"
git pull --rebase origin main
git push -u origin feat/dark-mode
```

**常用补充**：`git log --oneline -10`、`git stash`、`git branch -vv`；冲突后改文件 → `git add` → `git rebase --continue`。

解析 / 要点：

```text
改代码 → status/diff → pull --rebase → add → commit → push → PR
```

- **push 被拒**：远端有新提交，先 pull/rebase 再 push。
- **fetch vs pull**：fetch 只下载；pull = fetch + merge/rebase。

9. 什么是 API？

面试回答：

**API**（Application Programming Interface，应用程序编程接口）是 **模块之间约定好的调用方式**，调用方用统一规则使用能力，不必关心内部实现。

（1）**代码 API**：函数、类、浏览器 **DOM API**、Vue 的 **Composition API**。

（2）**Web / HTTP API**：前后端通过 HTTP 通信，常见 **RESTful**：

- URL 表资源
- Method 表动作（GET 查、POST 建、PUT/PATCH 改、DELETE 删）
- JSON 传数据
- 状态码表结果（见第 5 题）

（3）**契约**：字段、类型、错误码、鉴权（Bearer Token）需前后端对齐。

（4）**vs SDK**：API 是规范；SDK 是封装好的客户端库。

前端日常说的「调 API」多指 **请求后端 HTTP 接口**（Axios/fetch），不是直连数据库。

解析 / 要点：

```text
调用方 --契约(路径/方法/JSON)--> 服务端
       <-- 状态码 + 响应体 ------
```

一句话：**API 是合作的「插座标准」**，形状一致才能对接。

10. 如何实现：点击切换按钮切换主题为深色模式？

面试回答：

三件事：**状态存哪、CSS 怎么切、要不要记住**。

（1）**CSS 变量 + `html` class**

```css
:root { --bg: #fff; --text: #1a1a1a; }
html.dark { --bg: #141414; --text: #e8e8e8; }
body { background: var(--bg); color: var(--text); }
```

```javascript
function toggleDark() {
  const dark = !document.documentElement.classList.toggle('dark')
  localStorage.setItem('theme', dark ? 'dark' : 'light')
}
```

（2）**跟随系统**：`matchMedia('(prefers-color-scheme: dark)')`；用户未手动选时跟系统，选了则 localStorage 优先。

（3）**Vue 项目**：`isDark` ref + 根组件 `NConfigProvider :theme`（Naive UI）；或 UnoCSS/Tailwind 的 `dark:` 前缀。

（4）**防闪烁**：`index.html` 内联 script，首屏渲染前读 localStorage 加 `class="dark"`。

解析 / 要点：

```text
点击 → 改 theme state → html.classList / CSS 变量
     → localStorage 持久化
```


| 方案                   | 说明                  |
| -------------------- | ------------------- |
| CSS 变量               | 全局一处改色              |
| class on html        | 配合 Tailwind dark 模式 |
| prefers-color-scheme | 系统主题联动              |

11. AI 工具会取代程序员吗？

面试回答：

**不会整体取代，会改分工**——重复、模式固定的编码被压缩，**判断、架构、质量、协作** 更值钱。

（1）**AI 擅长的**：样板代码、测试骨架、文档、正则、常见 bug 提示、按 spec 生成 CRUD。

（2）**难替代的**：需求取舍、架构与状态设计、安全（XSS/鉴权）、联调排障、线上责任。

（3）**角色变化**：从「纯手写」到 **「拆任务 → 喂给 AI → review → 测试 → 合入」**。本场公司 **十分看重 AI 使用能力**，招的是 **会用 AI 提效的人**，不是会被一键替换的人。

（4）**我的做法**：边界清晰让 AI 出初稿；核心逻辑与安全 **人主导**；合入前必跑 lint/测试。

解析 / 要点：

```text
AI = 加速器 | 程序员 = 方向盘 + 责任方
```


| 被压缩     | 被放大         |
| ------- | ----------- |
| 重复 CRUD | 系统设计、Review |
| 抄文档式编码  | 任务拆解、验证     |


态度：**拥抱工具 + 强调工程判断**，别说「完全不会取代」或「全靠 AI 写」。

12. 未来规划

面试回答：

（可按个人情况改；结构：**短期能交付 → 中期深度 → 长期方向**，并 **贴合 AI 型小厂**）

**短期（0–1 年）**

- 夯实 **Vue/TS + 工程化**（组件化、性能、联调）。
- 熟练 **AI 协作**：会把需求拆成子任务、写清约束与验收，用 Cursor/ChatGPT 提效但 **自己 review**。
- 能独立负责一条前端业务线（页面 + 接口 + 状态）。

**中期（1–3 年）**

- 成为业务线 **核心前端**：复杂交互、AI 对话/流式 UI、状态与持久化设计。
- 拓展 **全链路**：懂一点 Node/部署，能和后端对齐接口，参与方案而不只接 UI。

**长期（3 年+）**

- 往 **资深 / Tech Lead** 发展：规范、带人、平衡体验与交付。
- 持续跟 **AI 产品化**（Agent、RAG、上下文管理）和前端新技术。

**和该公司契合（示例）**：贵司重视 AI 落地，我希望在前端侧把 **「好用的产品交互 + 高效的 AI 协作开发」** 结合起来，既交付快，也保证质量可控。

解析 / 要点：


| 维度    | 面试官想听                |
| ----- | -------------------- |
| 稳定性   | 规划连贯，非空喊口号           |
| AI 匹配 | 任务拆解、review 习惯（本场高频） |
| 现实    | 短期可执行的具体目标           |


**避免**：空泛「学新技术」；只谈薪资；规划与岗位无关。  
**加分**：提到 **如何把大需求拆给 AI**、可量化目标（性能、上线节点）。
