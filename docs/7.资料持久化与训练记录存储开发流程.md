# 7.资料持久化与训练记录存储开发流程

## 一、为什么现在开始做持久化

到目前这个阶段，项目已经完成了这些关键前置条件：

- 工作台页面结构已经稳定
- 总览页、资料页、模拟面试页已经形成主链路
- 页面职责已经通过 `6.1.对话主链路与页面职责收敛文档` 明确下来
- `mock-interview.vue` 已经成为后续主 AI 对话页

这意味着现在已经不是“先把页面画出来”的阶段了，而是开始进入：

**让数据真正留得住、流得动、能承接后续页面开发** 的阶段。

如果现在还不做持久化，后面会立刻遇到这些问题：

1. 导入的资料刷新后丢失
2. 当前训练主题刷新后丢失
3. 模拟面试做了一半，页面刷新后记录消失
4. 报告页和历史页没有真实数据来源
5. 后面接 SSE 时，消息流上下文没有本地承接层

所以现在开始做持久化，不是“补充功能”，而是：

**为后面报告页、历史页、真实 AI 问答打地基。**

---

## 二、这一阶段持久化的目标

这一阶段的目标不是一步做到“完整数据库产品”，而是先做一版：

**前端本地可用、结构清晰、后续可迁移的本地持久化方案。**

也就是说，本阶段先解决：

- 用户导入的资料能留住
- 当前工作上下文能留住
- 模拟面试的训练记录能留住
- 后续报告页和历史页有数据可读

而先不解决：

- 多设备同步
- 真正服务端存储
- 文档全文搜索引擎
- 向量检索
- 真正的 docx / 文件夹解析落地

---

## 三、这一阶段推荐用什么存

### 结论

这一阶段建议分两层：

1. **简单结构先用 `localStorage`**
2. **文档列表和训练记录优先按“可迁移到 IndexedDB”的模型来设计**

### 为什么不一上来就全用 IndexedDB

因为你现在最重要的是把数据结构定义清楚，而不是先把存储方案做重。

如果一开始就上 IndexedDB，会带来这些额外复杂度：

- API 封装更多
- 调试成本更高
- 你后面回看开发过程时不够直观

### 为什么又不能完全随便写 `localStorage`

因为你这个项目不是简单主题切换页，而是后面要承接：

- 资料导入记录
- 面试轮次记录
- 复盘报告承接

如果存储结构太随意，后面迁移到 IndexedDB 或后端时会很痛苦。

### 所以本阶段的策略是

**存储介质可以先轻，但数据模型必须先规整。**

---

## 四、这一步到底先持久化什么

这一阶段建议先做四类数据。

### 1. 资料导入记录

这是最优先的一类。

至少要存：

- 文档 id
- 文档标题
- 文档类型
- 文档来源
- 导入时间
- 关联主题
- 是否为当前激活文档

#### 为什么先存这个

因为 `library.vue` 现在已经有：

- 导入反馈
- 自动滚动到新文档
- 当前文档上下文切换

但如果刷新后全没了，这一层体验其实还是假的。

---

### 2. 当前训练上下文

至少要存：

- 当前主题
- 当前选中文档 id
- 当前页面来源
- 当前模式（比如 guided / standard）

#### 为什么要单独存这一类

因为它不完全等于文档列表，也不完全等于面试记录。

它更像：

**当前工作台正在围绕什么资料、什么主题工作。**

这是后面从资料页进入模拟面试页时最关键的桥梁。

---

### 3. 模拟面试轮次记录

至少要存：

- 轮次 id
- 主题
- 进入模式
- 关联资料来源
- 当前题目数
- 已回答题数
- 薄弱点标签
- 开始时间
- 结束时间
- 状态（进行中 / 已完成 / 中断）

#### 为什么这一类现在就该做

因为 `mock-interview.vue` 已经具备：

- 当前题切换
- 提示
- 追问
- 回答草稿
- 结束跳转

这时候训练记录如果不落地，后面报告页就只能继续看假数据。

---

### 4. 报告页承接摘要

这一阶段可以先只存最轻量的版本：

- 关联轮次 id
- 主题
- 已完成题数
- 薄弱点标签列表
- 来源页面

#### 为什么现在只存摘要

因为报告页还没正式开发完。

所以现阶段不用急着存完整报告内容，先把“报告页能承接什么基础数据”保留下来就够了。

---

## 五、这一阶段先不要持久化什么

为了避免做重，这些东西先不要碰。

### 1. 文档全文内容

先不要把完整 md / docx 正文全部持久化。

原因：

- 当前导入还是前端演示态
- 真正的 docx / 文件夹解析还没定下来
- 全文存储会很快把本地结构变复杂

### 2. SSE 消息流全文

先不要做完整会话消息持久化。

原因：

- 真实 SSE 还没接
- 你现在还没有稳定的消息模型
- 先存轮次摘要更稳

### 3. 完整报告 HTML / Markdown

先不要存整份报告富文本。

原因：

- 报告页还没开发
- 现在存整份内容太早
- 后面格式很可能还会改

---

## 六、建议的数据结构模型

这一部分不是最终类型文件，但现在先把模型概念定清楚。

### 1. 文档资料模型

建议后面定义在：

- `src/types/workbench.ts`

建议结构：

```ts
interface PersistedLibraryDocument {
  id: string
  title: string
  docType: 'md' | 'docx'
  topicKeys: string[]
  source: string
  importedAt: string
  isActive?: boolean
}
```

### 2. 当前工作上下文模型

```ts
interface PersistedWorkbenchContext {
  activeTopic: string
  activeDocumentId: string
  currentMode?: 'standard' | 'guided'
  sourcePage?: string
  updatedAt: string
}
```

### 3. 模拟面试轮次模型

```ts
interface PersistedInterviewSession {
  id: string
  topic: string
  mode: 'standard' | 'guided'
  source: string
  questionCount: number
  answeredCount: number
  weaknessTags: string[]
  startedAt: string
  finishedAt?: string
  status: 'in_progress' | 'completed' | 'aborted'
}
```

### 4. 报告摘要模型

```ts
interface PersistedReportSummary {
  id: string
  sessionId: string
  topic: string
  source: string
  weaknessTags: string[]
  answeredCount: number
  createdAt: string
}
```

## 七、文件目录应该怎么放

这一阶段建议新加三类目录。

### 1. 类型目录

- `src/types/workbench.ts`

这里统一放：

- 文档模型
- 上下文模型
- 训练记录模型
- 报告摘要模型

### 2. 存储工具目录

建议新增：

- `src/utils/storage/`

里面建议拆：

- `src/utils/storage/workbench-storage.ts`

这个文件负责：

- 读取本地存储
- 写入本地存储
- 更新局部数据
- 提供固定 key

### 3. 页面层数据桥接目录

建议新增：

- `src/composables/useWorkbenchPersistence.ts`

这个 composable 负责：

- 页面进入时读取持久化数据
- 页面变更时同步写回
- 为 `library.vue` / `mock-interview.vue` 提供统一接口

---

## 八、为什么这一阶段先推荐 composable，不急着上 store

这一点很重要。

### 当前不要直接把持久化逻辑塞进页面

如果直接写在：

- `library.vue`
- `mock-interview.vue`

里面，后面会变成：

- 页面逻辑
- 路由承接
- UI 状态
- 本地存储读写

全部搅在一起。

### 当前也不要一上来就大抽 Pinia

因为现在你还在定数据边界。

如果现在就把全部持久化逻辑抽到 Pinia，后面很容易出现：

- store 太胖
- 页面层调试变困难
- 不知道哪些状态是本地 UI 态，哪些是真正业务态

### 所以这一步推荐

**页面层 + composable + storage utils**

也就是：

1. 页面负责使用
2. composable 负责桥接
3. storage 工具负责真正读写

这是一种很适合当前项目阶段的中间态结构。

## 九、推荐的开发顺序

### 第 1 步：先定义类型和 storage key

先做：

- `src/types/workbench.ts`
- `src/utils/storage/workbench-storage.ts`

目标：

- 把要存什么先定义清楚
- 固定 key 命名

例如可以约定：

- `offerpilot.library.documents`
- `offerpilot.workbench.context`
- `offerpilot.interview.sessions`
- `offerpilot.report.summaries`

### 第 2 步：先接资料页持久化

文件重点：

- `src/views/workbench/library.vue`
- `src/views/workbench/library.data.ts`
- `src/composables/useWorkbenchPersistence.ts`

本步目标：

- 导入新资料后写入本地存储
- 页面初始化时优先读本地资料列表
- 当前激活文档也能恢复

### 第 3 步：再接当前上下文持久化

重点存：

- 当前主题
- 当前文档
- 当前模式

这样从资料页切到模拟面试页时，上下文就更稳了。

### 第 4 步：接模拟面试页轮次记录

文件重点：

- `src/views/workbench/mock-interview.vue`
- `src/views/workbench/mock-interview.data.ts`
- `src/composables/useWorkbenchPersistence.ts`

本步目标：

- 开始面试时生成 session
- 提交回答时更新 answeredCount
- 结束本轮时写 completed 状态
- 薄弱点标签一并写入

### 第 5 步：给报告页准备基础读取能力

重点：

- `src/views/workbench/report.vue`

先不做完整报告，只做：

- 能根据 query / sessionId 读到对应摘要

### 开发前

- 导入资料刷新就丢
- 当前主题刷新就丢
- 模拟面试记录刷新就丢
- 报告页和历史页没有真实承接数据

- 历史页有后续开发基础

## 十二、这一阶段特别注意的事项

### 1. 不要把“假数据文件”和“用户本地数据”混成一份

比如：

- `library.data.ts`
- `mock-interview.data.ts`

这些文件现在是演示数据来源，不应该直接承担“用户本地数据仓库”的职责。

它们更适合做：

- 默认种子数据
- 页面初始展示数据

而用户实际导入的数据应该来自 storage。

### 2. 不要急着做自动同步所有字段

先存核心字段就够了。

例如：

- 当前草稿全文先不急着存
- 每条追问消息先不急着存

否则你会很快把存储层做复杂。

### 3. 要考虑“可恢复”和“可清理”

持久化不只是“写进去”，还要考虑：

- 页面怎么恢复
- 用户怎么切换上下文
- 后面怎么清空历史数据

所以这一步在工具层最好预留：

- `get`
- `set`
- `remove`
- `clear`

### 4. key 命名要从现在就规范

因为后面一旦页面变多、存储项变多，如果 key 名字乱了，会非常难维护。

---

## 十三、这一阶段结束后，下一步做什么

完成这一步之后，最自然的下一步有两个方向：

### 方向一：继续实现持久化对应代码

也就是正式开发：

- 类型文件
- storage 工具
- composable
- 资料页接入
- 模拟面试页接入

### 方向二：开始开发报告页真实承接

前提是第一阶段持久化已经把 session 摘要写下来了。

所以更推荐的顺序仍然是：

1. 先做持久化代码
2. 再做报告页真实数据承接

---

## 十四、结论

这一阶段最重要的不是“把所有数据都存起来”，而是：

**先把资料、上下文、训练轮次这三种核心数据稳定地落到本地。**

只要这三层稳住，后面的：

- SSE 对话
- 报告页
- 历史页
- 更完整的 AI 主链路

都会顺很多。
