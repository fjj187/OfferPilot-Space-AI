# 前端开发环境本地后端跨域代理与转发配置说明

这份文档专门说明这次本地联调时，为什么要改前端请求地址、Vite 代理、环境变量模板和后端 CORS。

它不是一份泛泛而谈的概念文档，而是按照这次真实开发里遇到的问题来写。以后你回头看，可以直接把它当成一份本地前后端联调排障笔记。

当前本地开发环境的真实情况是这样的：

前端运行在：

```txt
http://localhost:2048
```

后端运行在：

```txt
http://localhost:3030
```

模拟面试真实流接口是：

```txt
POST /api/interview/stream
```

这里最核心的问题不是接口有没有写出来，而是浏览器能不能真的从前端页面里把请求发到后端，并稳定收到 SSE 流。

发现的第一个问题

前端虽然已经支持真实接口地址，但默认情况下不一定会真的请求本地后端。

改动前的前端逻辑可以理解成这样：

```ts
const resolveInterviewStreamEndpoint = () => import.meta.env.VITE_INTERVIEW_SSE_URL?.trim() || ''
```

这段代码的意思是：

如果配置了 `VITE_INTERVIEW_SSE_URL`，就走真实接口。
如果没有配置，就返回空字符串。

后面的逻辑又是：

```ts
const resolveInterviewStreamMode = (): InterviewStreamMode => (
  resolveInterviewStreamEndpoint() ? 'remote' : 'mock'
)
```

这意味着只要 `VITE_INTERVIEW_SSE_URL` 没填，前端就会自动回退到本地 mock 流。

这段逻辑本身没有错，但它有一个很实际的问题：

本地开发时，哪怕后端已经起好了，只要环境变量没填，页面仍然会继续走 mock。这样开发者很容易误以为自己已经在联调真实接口，其实页面根本没请求后端。

也就是说，问题不在“代码不能跑”，而在“默认行为不利于联调”。

后来改成了这样：

```ts
const resolveInterviewStreamEndpoint = () => {
  const configuredEndpoint = import.meta.env.VITE_INTERVIEW_SSE_URL?.trim() || ''
  if (configuredEndpoint) return configuredEndpoint

  return import.meta.env.DEV ? '/api/interview/stream' : ''
}
```

改完之后的行为是：

如果手动配置了 `VITE_INTERVIEW_SSE_URL`，优先用手动配置。
如果没有手动配置，但当前是开发环境，就默认请求 `/api/interview/stream`。
如果既没配置，又不是开发环境，才继续回退到 mock。

这一步解决的问题是：

前端在本地开发环境下，会默认优先走真实链路，而不是悄悄回退到 mock。

这不是在增加复杂度，而是在把“真实后端已经存在”这件事真正接到页面里。

对应文件：

```txt
src/services/sse/interview-stream.ts
```

发现的第二个问题

就算前端愿意去请求真实接口，浏览器也不一定会让它直接请求成功。

原因是前端和后端端口不同：

```txt
前端  http://localhost:2048
后端  http://localhost:3030
```

这在浏览器看来是两个不同的源。

只要是不同源请求，浏览器就会开始检查跨域规则。这里又因为请求是：

```txt
POST
Content-Type: application/json
```

浏览器通常会先发一个预检请求 `OPTIONS`，问后端：

我能不能从 `http://localhost:2048` 发这个请求过去？

如果后端没有正确返回跨域头，浏览器会直接拦掉正式请求。这个时候你在页面上看到的现象通常不是“业务报错”，而是“请求压根没发过去”。

这说明一个很重要的事实：

命令行里能 `curl` 通，不代表浏览器里就一定能通。

发现的第三个问题

如果前端直接写完整后端地址，例如：

```ts
fetch('http://localhost:3030/api/interview/stream', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify(payload)
})
```

虽然理论上可以请求到后端，但开发环境里会把跨域问题暴露得非常直接，而且以后切换环境也会更麻烦。

本地联调时，一个更顺手的做法是：

让前端仍然请求自己的相对路径：

```txt
/api/interview/stream
```

然后让 Vite 开发服务器把这个请求代理转发到本地后端。

也就是说，浏览器看到的是：

```txt
http://localhost:2048/api/interview/stream
```

但 Vite 实际上会把它转发给：

```txt
http://localhost:3030/api/interview/stream
```

这就是代理的意义。

不是改业务协议，而是在开发环境里帮你做一次转发。

所以前端代理配置改成了这样

```ts
const interviewBackendOrigin = env.VITE_INTERVIEW_BACKEND_ORIGIN || 'http://localhost:3030'

server: {
  port: 2048,
  proxy: {
    '/api': {
      target: interviewBackendOrigin,
      changeOrigin: true
    }
  }
}
```

这段配置的作用是：

前端在开发环境访问 `/api/...` 时，不再由前端自己处理，而是交给 Vite 转发到 `3030` 端口的后端服务。

改完之后会怎样：

你在页面代码里不需要写死 `http://localhost:3030`。
本地开发时，页面可以像访问同源接口一样访问 `/api/interview/stream`。
如果以后本地后端端口改了，只要调整环境变量 `VITE_INTERVIEW_BACKEND_ORIGIN` 即可。

这一步解决的问题是：

把开发环境下的真实后端接入方式标准化了，不需要每次都改业务代码里的请求地址。

对应文件：

```txt
vite.config.ts
```

环境模板为什么也要改

之前模板里的流式接口地址还是旧端口：

```txt
VITE_INTERVIEW_SSE_URL=http://localhost:3001/api/interview/stream
```

这和当前实际后端 `3030` 已经不一致了。

如果模板不改，就会出现一种很常见的问题：

代码已经按新方案接好了，结果新人或后续自己照着旧模板配置，仍然把请求打到错误端口。

后来改成了这样：

```txt
VITE_INTERVIEW_SSE_URL=
VITE_INTERVIEW_BACKEND_ORIGIN=http://localhost:3030
```

这里的意思是：

开发环境默认不强制手填真实 SSE 地址，让前端优先走 `/api/interview/stream`。
Vite 代理的目标后端地址单独放在 `VITE_INTERVIEW_BACKEND_ORIGIN`。

这样做的好处是：

前端请求入口和本地后端实际地址分离了。
页面代码只关心 `/api/interview/stream`。
开发代理才关心后端真正在哪个端口。

这一步让配置关系更清楚。

对应文件：

```txt
.env.template
```

代理还不够，后端为什么还要加 CORS

很多人第一次接本地前后端时，会以为只要加了 Vite 代理，就完全不需要管后端跨域。

实际上代理是开发期手段，不是后端能力本身。

后端如果完全不支持 CORS，会带来几个问题：

当你不经过 Vite，而是直接从别的前端地址请求它时，浏览器会拦。
当以后部署方式变化，不再依赖当前这层代理时，还会再次遇到同样的问题。
你也没办法确认后端本身是否具备被浏览器访问的最基础能力。

所以后端还是应该具备基本的跨域支持。

原来的后端入口大致是这样：

```ts
const app = express()

app.use(express.json({
  limit: '1mb'
}))
```

这里只有 JSON 解析，没有跨域处理。

后来改成了这样：

```ts
import cors from 'cors'

const app = express()

app.use(cors({
  origin: true
}))

app.use(express.json({
  limit: '1mb'
}))
```

这里 `origin: true` 的作用是：

允许后端根据当前请求的来源动态返回 `Access-Control-Allow-Origin`。

对本地联调来说，这意味着当浏览器从 `http://localhost:2048` 发请求过来时，后端可以正确回应跨域头和预检请求。

同时，`backend/package.json` 里也补上了依赖：

```json
{
  "dependencies": {
    "cors": "^2.8.5"
  },
  "devDependencies": {
    "@types/cors": "^2.8.19"
  }
}
```

这一步解决的问题是：

不是只有命令行能打通后端，而是浏览器环境下也具备合法访问它的基础能力。

对应文件：

```txt
backend/src/app.ts
backend/package.json
```

为什么这些改动是一起发生的

因为这些问题不是独立的，而是一条链。

只改前端默认地址，不改代理，页面仍然可能跨域失败。

只改代理，不改后端 CORS，虽然当前开发环境多数情况下能借代理绕过去，但后端本身仍然没有浏览器访问能力。

只改后端 CORS，不改前端默认入口，页面又可能继续悄悄走 mock，根本不发真实请求。

所以这几步必须连起来看：

前端先默认愿意走真实接口。
开发服务器负责把 `/api` 转发到本地后端。
后端自己也具备最基础的跨域响应能力。
环境模板再把这套关系表达清楚。

这样整条链路才真正成立。

这次改完之后，本地开发链路变成了这样

页面里的请求地址：

```txt
/api/interview/stream
```

Vite 开发服务器收到后：

```txt
http://localhost:2048/api/interview/stream
```

Vite 代理转发到：

```txt
http://localhost:3030/api/interview/stream
```

后端返回 SSE：

```txt
event: chunk
data: {"content":"..."}

event: done
data: {}
```

前端再把这些 SSE 内容按已有逻辑渲染到消息流里。

也就是说，现在这条链路已经不再是“前端自己演”，而是：

前端页面
Vite 代理
本地 Express 后端
SSE 返回
前端渲染

为什么这样能解决问题

因为真正的问题从来不是“有没有写接口”，而是“页面能不能稳定、可控地接到这个接口”。

现在这套方案解决的是四层问题。

第一层，前端默认行为问题。
开发环境下默认走真实接口，不再轻易误回退到 mock。

第二层，开发期地址管理问题。
页面使用相对路径 `/api/interview/stream`，不需要把真实后端地址写死在业务代码里。

第三层，浏览器跨域问题。
Vite 代理和后端 CORS 一起把浏览器访问链路补完整了。

第四层，后续可维护性问题。
环境模板、代理目标、前端请求入口三者关系清晰，后续换端口或换部署方式时，修改点也更明确。

最后总结

这次不是单纯“为了跨域去加一个配置”，而是把本地前后端联调从不稳定的半手工状态，整理成一套可重复的开发方式。

现在这套方式的核心是：

前端开发环境默认优先走真实接口。
前端页面统一请求 `/api/interview/stream`。
Vite 在开发期把 `/api` 代理到本地 Express 后端。
后端自己也支持浏览器跨域访问。

如果以后你再碰到“本地前后端都有，但页面联调就是不顺”的问题，优先检查的也应该是这四层：

前端默认到底走的是 mock 还是 real。
请求地址是不是写死了错误端口。
开发服务器有没有做代理转发。
后端有没有正确响应 CORS 和预检请求。
