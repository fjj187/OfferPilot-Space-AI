# 20.mock-interview-space 视觉一致性收尾与后续开发文档

## 1. 背景

`/showcase/mock-interview-space`（宇宙模拟面试页）已经完成总览、资料库、模拟面试、专项刷题、复盘报告几个主要轨道场景的接入。

最近一轮开发重点从“是否有内容”转向“切换是否稳定、标题是否统一、内容区是否不挤”。目前已经处理过以下问题：

1. 胶囊下滑逻辑统一，模拟面试场景不再走单独滚动分支。
2. 首屏标题上方的 `eyebrow`（眉标小字）已经移除。
3. 内容区标题上方的场景小字已经移除。
4. 专项刷题场景右侧 `Focus`（焦点）圆球已经移除。
5. 模拟面试场景右侧 meta chip（元信息胶囊）已经改为自适应多列。

这说明当前页面已经进入视觉一致性收尾阶段。下一步不应急着扩功能，而应先把标题区、内容区、切换动画和滚动行为稳定下来，避免后续每新增一个场景都要重新修 UI（用户界面）。

---

## 2. 当前状态

### 2.1 已完成

1. `mock-interview-space.vue`（模拟面试星际页面主文件）保持页面装配职责。
2. `SpaceContentPanel.vue`（宇宙内容面板组件）负责按场景挂载具体内容。
3. `SpaceMockScene.vue`（模拟面试场景组件）已承接问题、回答、消息流和右侧元信息。
4. `SpacePracticeScene.vue`（专项刷题场景组件）已承接弱点、专区、题型、题数和难度选择。
5. `SpaceLibraryScene.vue`（资料库场景组件）已承接资料导入与资料筛选。
6. `SpaceReportScene.vue`（复盘报告场景组件）已承接报告摘要与历史入口。
7. `useMockInterviewSpaceScroll.ts`（宇宙页面滚动组合函数）已恢复统一下滑逻辑。

### 2.2 仍需要收尾

1. 各场景内容区标题高度仍依赖各组件自身样式，缺少统一结构。
2. `SpaceMockScene.vue`（模拟面试场景组件）的标题区右侧元信息仍可能随资料标签数量变化而影响视觉重心。
3. `SpaceContentPanel.vue`（宇宙内容面板组件）里仍有较多场景判断和 props（组件属性）透传。
4. 首屏圆圈切换时，标题、摘要、轨道视觉的动效节奏还可以继续统一。
5. 页面暂未形成明确的“场景头部规范”，后续新增场景容易再出现高度不一致问题。

## 3. 下一步开发目标

下一轮建议优先做“视觉一致性收尾”，目标不是增加新功能，而是让现有五个场景切换时稳定、整齐、可维护。

具体目标：

1. 抽出统一的 `SpaceSceneHeader`（宇宙场景头部组件）或统一头部样式片段。
2. 统一内容区标题、说明、右侧辅助信息的布局规则。
3. 统一首屏标题区和内容区标题区的字号、行高、间距。
4. 统一五个轨道节点切换后的内容区首屏高度。
5. 保持胶囊下滑目标一致，不再为单个场景写特殊滚动逻辑。

---

## 4. 建议开发顺序

### 4.1 第一阶段：场景头部统一

优先处理内容区头部，不要先动业务卡片。

建议新增或整理：

1. `SpaceSceneHeader.vue`（宇宙场景头部组件）。
2. 支持 `title`（标题）、`body`（说明）、`aside`（右侧区域插槽）。
3. 模拟面试场景的 meta chip（元信息胶囊）放入 `aside`（右侧区域插槽）。
4. 专项刷题场景不传 `aside`（右侧区域插槽），避免强行撑高。
5. 资料库、复盘报告、总览场景逐步迁移到同一头部结构。

验收标准：

1. 五个场景内容区大标题顶部位置一致。
2. 大标题下方说明文字间距一致。
3. 没有标题上方小字。
4. 没有为某一个场景写单独的标题高度补丁。

### 4.2 第二阶段：模拟面试元信息区收口

模拟面试右侧 meta chip（元信息胶囊）数量会随资料、知识点变化，需要控制展示密度。

建议处理：

1. 最多展示 8 个 chip（元信息胶囊）。
2. 优先级为阶段、难度、模式、题源、资料、类型、主题、知识点。
3. 超出数量时使用 `+N`（剩余数量）胶囊收口。
4. 长文件名只在 chip（元信息胶囊）内省略，不撑开标题区。

验收标准：

1. 元信息区最多 2 到 3 行。
2. 不因为资料名过长导致标题区挤压。
3. 不因为知识点过多导致标题区高度失控。

### 4.3 第三阶段：切换稳定性检查

每次点击圆圈切换场景时，需要检查视觉稳定性。

检查项：

1. 首屏大标题是否跳动。
2. 内容区大标题是否跳动。
3. 胶囊点击后是否总是向下滚动到内容入口。
4. 内容区首屏是否不会突然出现大空白。
5. 移动端布局是否不横向溢出。

验收标准：

1. 连续点击五个轨道节点，页面高度节奏稳定。
2. 模拟面试、专项刷题、资料库三个主要内容场景没有明显错位。
3. `npm run build`（执行生产构建验证）通过。

## 5. 职责边界

### 5.1 `mock-interview-space.vue`（模拟面试星际页面主文件）

只负责：

1. 页面整体装配。
2. 当前场景状态。
3. 滚动、自动轮播、轨道切换编排。

不负责：

1. 单个场景 UI（用户界面）。
2. 单个场景标题样式。
3. 单个场景业务卡片布局。

### 5.2 `SpaceContentPanel.vue`（宇宙内容面板组件）

负责：

1. 根据 `displayScene.id`（当前展示场景标识）挂载场景组件。
2. 向场景组件传入必要 props（组件属性）。
3. 承接场景事件并向上抛出。

后续可优化：

1. 减少过长 props（组件属性）列表。
2. 将场景数据组装逻辑前移到组合函数或局部 computed（计算属性）。
3. 保持本组件只做“场景路由层”。

### 5.3 `Space*Scene.vue`（宇宙场景组件）

负责：

1. 自己场景内的 UI（用户界面）。
2. 自己场景内的交互。
3. 自己场景内的样式。

不应负责：

1. 全页面滚动策略。
2. 轨道节点状态。
3. 其他场景的布局兼容。

---

## 6. 不建议现在开发的内容

当前不建议立刻开发：

1. 新增更多轨道节点。
2. 重写星球动画。
3. 接入复杂题库生成逻辑。
4. 大规模重构旧工作台页面。
5. 为专项刷题单独做新页面。

原因：

1. 当前主要问题是视觉和布局一致性，不是功能数量不足。
2. 宇宙页面已经具备主链路雏形，继续加功能会放大现有布局不稳定问题。
3. 先稳定头部、滚动和切换，后面接题库和报告会更稳。
