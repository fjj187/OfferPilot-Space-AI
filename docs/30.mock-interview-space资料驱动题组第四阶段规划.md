# 30. mock-interview-space 资料驱动题组第四阶段规划

## 文档目的

`28` 完成报告后端化；`29` 规划 **弱项 → 专项题组 → mock 按序练**（全局题库 + 规则编译）。  

用户已确认走 **路线 B：做深训练资料**，并明确：

1. **不希望系统随机出题**（难度跳变、不可解释）。
2. 希望 **导入资料后，能围绕资料内容练**，在模拟面试里问答。
3. 「训练资料」指 **解析 / 切分 / 建题 / 编组**，不是训练 AI 模型。

本文定义 **第四阶段：资料驱动题组**，与 `29` 共用同一套 **题组执行层**，仅 **题组来源与编译规则** 不同。

> **读法**：第九节 **A～F 已冻结（2026-05-24，全按推荐默认）**；实施顺序 **E1：先 29 题组执行层，再 30 资料池**。

---

## 一、`30` 阶段的一句话目标

**上传资料 → 准备练习题（建池）→ 预览题组 → 模拟面试按序问答；题目来自资料，不靠全局随机抽题。**

对用户可见的变化：

1. 资料不再只是「选中一个文件名」，而是 **「已生成 N 道练习题，可预览」**。
2. 开练前能看到 **本轮将练哪些题、来自哪一节、难度如何**。
3. 模拟面试 **按题组顺序** 推进；AI 反馈仍带资料摘要 / 片段（现有 `sourceDocument*` 能力保留并加强）。
4. 难度通过 **编译模式** 控制，而不是从全局池 `shuffle` 随机抽。

---

## 二、与「训练资料」说法的对齐

| 用户说法 | 工程含义（本文采用） | 不做 |
|----------|----------------------|------|
| 训练资料 | 对单份资料做 **准备**：切 chunk、生成题目、写入资料题池 | 微调模型、向量训练 |
| 训练好所有导入资料 | 每份资料状态为 **已准备**，题池可复用 | 一次性批处理全库后才允许开练（MVP 按 **单资料** 准备即可） |
| 从资料里抽题 | **规则编译题组**，按章节或难度递进取 N 题 | `Math.random()` 从全库抽 |
| 模拟面试问答 | mock 只消费 **当前资料题组** | 资料页只预览不参与开练 |

**不推荐** 单独做一个含义模糊的「训练资料」大按钮；推荐在资料场景用明确动作：

- **「生成练习题」**（准备）
- **「预览题组并开始模拟面试」**（开练）

总览轨道若需要入口，文案应为 **「去准备资料」** 而非「训练资料」。

## 三、为什么需要 `30`（当前资料能力缺口）

| 环节 | 当前 | 缺口 |
|------|------|------|
| 导入 | ✅ rawText、summary、tags | — |
| 选题 | ⚠️ 标签 / 类型弱匹配 `questionBank` | 题与资料正文无稳定对应 |
| 出题 | ❌ | 题来自静态题库，非资料衍生 |
| 开练 | ⚠️ 带资料上下文调 AI | 用户感知不到「按资料练」 |
| 难度 | ⚠️ 题库自带 difficulty，与资料无关 | 易跳档、不可控 |
| 29 专项题组 | 规划弱项编译 | 与资料正交，不替代资料出题 |

## 四、与 `29` 的分工（共用执行层）

### 4.1 统一抽象：`TrainingQuestionGroup`（题组执行层）

`29` 与 `30` 均产出 **同一结构** 的题组，供 `useMockInterviewSpaceMockState` 消费：

```ts
type TrainingQuestionGroupSource = 'practice_weakness' | 'material_document'

interface TrainingQuestionGroupItem {
  questionId: string        // 全局唯一，资料题可用 material-q-{docId}-{chunkId}
  order: number
  title: string
  prompt: string
  difficulty: 'easy' | 'medium' | 'hard'
  questionType?: 'concept' | 'code' | 'scenario'
  focusArea?: PersistedPracticeFocusArea
  matchReason: string       // 展示用，如「来自资料第 2 节 · 响应式」
  sourceChunkId?: string    // 资料题必填
  sourceDocumentId?: string
}

interface TrainingQuestionGroup {
  id: string
  source: TrainingQuestionGroupSource
  items: TrainingQuestionGroupItem[]
  currentIndex: number
  status: 'pending' | 'in_progress' | 'completed'
  createdAt: string
  updatedAt: string
  // 来源快照，便于恢复
  planSnapshot?: PersistedPracticePlan           // source === practice_weakness
  documentSnapshot?: { documentId: string; name: string }  // source === material_document
  compileOptions?: MaterialGroupCompileOptions
}
```

**持久化挂载**（`PersistedWorkbenchContext`）：

```ts
interface PersistedWorkbenchContext {
  // ...现有
  practicePlan?: PersistedPracticePlan | null
  practiceQuestionGroup?: TrainingQuestionGroup | null  // 29 命名可统一到此
  materialQuestionPool?: MaterialQuestionPool | null  // 资料题池（准备结果，可复用）
}
```

### 4.2 两套编译器、一套执行器

| 阶段 | 编译器 | 输入 |
|------|--------|------|
| 29 | `practice-question-group-builder` | `practicePlan` + 报告弱项 + `questionBank` |
| 30 | `material-question-group-builder` | 当前资料 + `materialQuestionPool` + 用户编译选项 |

**mock 优先级（推荐）：**

1. 若存在 `trainingQuestionGroup.source === 'material_document'` 且 `status !== completed` → **只走资料题组**。
2. 否则若 `mockEntryMode === 'practice'` 且有弱项题组 → 走 29。
3. 否则走现有 direct + `questionBank`（资料仅作 AI 上下文，**不作为出题源**）。

### 4.3 实施顺序建议

| 顺序 | 内容 |
|------|------|
| **先** | 29 的题组执行层（mock 按序、进度、预览） |
| **后** | 30 的资料切分 + 建池 + `material-question-group-builder` |

避免 mock 里并行两套 thread 逻辑。

## 五、资料题池与切分（准备阶段）

### 5.1 `MaterialChunk`（资料切片）

```ts
interface MaterialChunk {
  id: string
  documentId: string
  order: number
  heading: string
  level: number          // 1=# 2=## ...
  text: string
  charStart?: number
  charEnd?: number
}
```

**MVP 切分规则：**

1. 按 Markdown 标题 `#` / `##` / `###` 切段。
2. 无标题时长文按 ~800～1200 字滑动窗口切（带 overlap 可选，29 不做 overlap 也行）。
3. `docx` 已转 `rawText` 后同样规则。

### 5.2 `MaterialQuestionItem`（资料题池条目）

```ts
interface MaterialQuestionItem {
  id: string
  documentId: string
  chunkId: string
  order: number
  title: string
  prompt: string
  difficulty: 'easy' | 'medium' | 'hard'
  questionType: 'concept' | 'code' | 'scenario'
  generatedBy: 'rule' | 'llm'
  focusAreas?: PersistedPracticeFocusArea[]
}
```

```ts
interface MaterialQuestionPool {
  documentId: string
  documentVersion: string   // 如 importedAt 或 content hash，资料更新后失效旧池
  chunks: MaterialChunk[]
  questions: MaterialQuestionItem[]
  preparedAt: string
  status: 'idle' | 'preparing' | 'ready' | 'error'
  errorMessage?: string
}
```

### 5.3 MVP 规则出题（推荐默认，不调 LLM）

服务：`src/services/material/material-question-builder.ts`

| chunk 特征 | 生成题目 | 默认难度 |
|------------|----------|----------|
| `level <= 2` 且 heading 非空 | 「请讲解：{heading}」 | medium |
| 含 fenced code | 「阅读并解释以下代码片段…」+ 摘录 | medium/hard |
| 短列表 / 加粗考点 | 概念理解题 | easy |
| 含「为什么 / 原理 / 源码」 | 原理追问型 | hard |

每 chunk **最多 1～2 题**，避免题池爆炸。  
`questionType` 由规则映射，写入 `MaterialQuestionItem`。

### 5.4 阶段 2：LLM 离线出题（可选，非 MVP）

- 触发：用户点「生成练习题」或导入后勾选「深度出题」。
- 每 chunk 一次请求，产出结构化 JSON 落池。
- **开练时不再现生成**，保证同资料同题组可复现。
- 需配置 API；失败回退规则题。

## 六、题组编译（开练阶段，非随机）

服务：`src/services/material/material-question-group-builder.ts`

### 6.1 输入

- `MaterialQuestionPool`（`status === ready`）
- `MaterialGroupCompileOptions`：

```ts
interface MaterialGroupCompileOptions {
  count: number
  difficultyFilter?: Array<'easy' | 'medium' | 'hard'>  // 空 = 不筛
  orderMode: 'chapter' | 'difficulty_ladder' | 'single_difficulty'
  singleDifficulty?: 'easy' | 'medium' | 'hard'       // orderMode === single_difficulty
}
```

### 6.2 输出

- `TrainingQuestionGroup`（`source: 'material_document'`）

### 6.3 选题算法（确定性，禁止 shuffle 凑数）

1. 从 `pool.questions` 过滤 `difficultyFilter`。
2. 排序：
   - **`chapter`（推荐默认）**：`chunk.order` → `question.order`。
   - **`difficulty_ladder`**：`easy < medium < hard`，同档内 `chunk.order`。
   - **`single_difficulty`**：只保留一档，再按章节。
3. 取前 `count` 条；**同一 `questionId` 不重复**。
4. 不足 `count`：**不取模循环**；返回实际条数 + UI 提示「仅找到 N 题」。

### 6.4 与 AI 上下文

每题 `prompt` 可自动附加：

```text
【本题资料依据】{chunk.heading}：{chunk.text 前 400 字}
```

流式请求继续传 `sourceDocumentSummary` / `sourceDocumentExcerpt`（现有字段），与 **题目绑定的 chunk** 对齐（excerpt 优先本题 chunk）。

## 七、前端改造方向

### 7.1 资料场景 `SpaceLibraryScene.vue`

在现有列表 / 预览上增加：

1. **状态徽章**：未准备 / 准备中 / 已生成 N 题 / 失败。
2. **主操作「生成练习题」**：调用切分 + `material-question-builder`，写 `materialQuestionPool`。
3. **次操作「预览并开始模拟面试」**：`disabled` 直到 `pool.status === ready`。
4. **题组预览抽屉 / 区块**：列表展示 title、难度、章节、顺序。
5. **编译选项（轻量）**：题数、顺序模式、难度筛选（默认 `chapter` + 全部难度）。

**保留** 资料列表、导入、预览；**不删除** library 轨道，只改职责为「准备 + 开练入口」。

### 7.2 `mock-interview-space.vue`

- 资料开练：`buildMaterialQuestionGroup(pool, options)` → 写入 `trainingQuestionGroup` + `activeDocumentId` → `startMockRound({ mockEntryMode: 'material' })`（或复用 `direct` + group 来源标记）。
- 与 `handlePracticeStart`（29）并列，**入口不同、执行层相同**。

### 7.3 `useMockInterviewSpaceMockState.ts`

- 新增 `mockEntryMode: 'direct' | 'practice' | 'material'`（或 `trainingQuestionGroup.source` 驱动）。
- `material` 模式：`buildConfiguredQuestionThreads` **只读** `trainingQuestionGroup.items`。
- 顶栏 meta：**资料名 · 第 i / n 题 · {matchReason}**。
- **D1（与 29 一致）**：资料模式默认 **一题一线程**，不展开多 followUp 链（除非单题内 AI 追问，thread 仍一条）。

### 7.4 静态 `questionBank` 角色

- **资料模式**：不出 `questionBank` 的题。
- **无资料题组**：用户若未准备资料就进 mock，提示 **「请先在资料页生成练习题」** 或走 29/普通模式（见拍板 C）。

## 十、阶段成果与验收标准

### 10.1 对用户可见

1. 导入 md → 点「生成练习题」→ 看到 **≥1 道题** 且带章节来源。
2. 预览题组 → 开始模拟面试 → **第一题与预览一致**。
3. 全程 **无「随机换题」感**；刷新后进度仍在。
4. AI 回答能引用 **本题对应章节**（prompt / excerpt 一致）。

### 10.2 对工程侧可见

1. 资料题与 `questionBank` 出题路径分离清晰。
2. 与 29 共用 `TrainingQuestionGroup`，mock 无第三套 thread 逻辑。
3. `pnpm run build` 通过。

### 10.3 最小手测清单

1. 准备一份含 3 个 `##` 标题的 md → 生成练习题 → 题数与章节对应。
2. 选 5 题、章节顺序 → mock 依次出现 5 题，不重复、不跳章。
3. 筛选仅 medium → 不出现 easy/hard（除非池子不足有提示）。
4. 改资料内容重新导入 → 旧 pool 失效，需重新生成。
5. 29 专项补练链路不受破坏（回归）。

---

## 十一、边界与风险

1. **题库体量**：规则题依赖 chunk 数量；短文资料题少，需 UI 明示。
2. **docx 质量**：解析差则 chunk 差；MVP 可提示「建议 md」。
3. **不做 RAG**：MVP 用全文切片 + 固定 excerpt，不上向量检索。
4. **不改 workbench 资料页**（宇宙页边界不变）。
5. **不替代 28 报告后端**；报告仍可推荐「回资料准备某主题」，但 30 不强制报告改接口。

---

## 十二、最终结论

`29` 解决 **弱项驱动的可序题组**；`30` 解决 **资料驱动的可序题组**。  

二者共享 **题组执行层**，避免随机出题；资料侧核心是 **准备（建池）→ 编译（选题组）→ mock（按序练）**。

**A～F 已全部采纳推荐项**；按 **E1** 先完成 29，再启动 30。
