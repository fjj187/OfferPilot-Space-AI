## 一、这份文档的用途

这份文档用于指导你开发资料库页面，不是最终产品说明，而是开发执行顺序文档。

它主要解决三个问题：

1. 资料库页现在还是占位页面，不具备实际功能
2. 项目的后续能力都依赖资料导入，没有资料库就无法进入模拟面试和复盘
3. 资料页如果不先设计好结构，后面接 `md/docx` 导入和解析时会反复返工

这份文档的目标是：

- 先把资料库页页面结构定下来
- 再把导入、列表、筛选、预览这四块能力按顺序做出来
- 最后为后续文档解析和本地存储铺路

---

## 二、资料库页在整个项目里的位置

资料库页是这个项目的“输入层”。

整个平台的业务链路是：

1. 用户导入个人面试资料
2. 系统建立本地资料库
3. AI 基于资料库内容生成题目
4. 用户进入模拟面试或专项刷题
5. 最后输出复盘报告

所以资料库页不是一个普通文件管理页，它承担的是：

- 资料导入入口
- 知识来源管理
- 后续 AI 提问的数据基础

如果这一页没设计好，后面所有训练能力都会受影响。

## 四、这一阶段的开发目标

资料库页面第一版建议只做这些核心能力：

### 必做

1. 导入区域
2. 文档列表区域
3. 标签/主题筛选区域
4. 文档详情预览区域

### 暂缓

1. 真正的 `docx` 深度解析
2. 文档切片后的知识图谱
3. OCR / PDF 支持
4. 云端同步
5. 向量检索

也就是说，第一版资料页应该先做到：

**“能导入 -> 能看到 -> 能筛选 -> 能预览”**

不要第一阶段就试图做成完整知识库系统。

## 五、资料库页第一版推荐结构

建议资料库页仍然延续当前工作台风格，使用“中间主区 + 右侧辅助区”的布局。

### 5.3 页面核心模块

可以拆成下面这些视觉区域：

#### 模块 A：页面头部

展示：

展示：

展示：

- TypeScript
- 工程化
- 项目场景

展示每个文档的卡片或表格项：

- 文档名
- 类型
- 导入时间
- 标签
- 解析状态

#### 模块 F：文档详情预览区

展示：

- 当前选中的文档名
- 文档简介
- 前几段文本预览
- 标签列表
- 预计可用于哪些训练场景

## 六、推荐的开发顺序

资料库页不要一上来就做真实导入逻辑，建议按下面顺序推进。

---

## 6.1 第一步：先做静态页面骨架

这一阶段只做页面布局和视觉模块，不接真实文件。

目标：

- 把资料库页从占位页变成完整 UI 页面
- 保持与总览页一致的风格
- 确认布局、卡片、列表区域没有问题

### 要做的事情

1. 替换当前 `library.vue` 占位页面
2. 加入页面头部
3. 加入导入卡片区
4. 加入统计卡片区
5. 加入筛选标签区
6. 加入文档列表区
7. 加入右侧预览辅助区

### 这一阶段的数据怎么放

和总览页保持一致：

- 先放在页面层
- 如果数据多，就建 `library.data.ts`

建议建立：

```text
src/views/workbench/library.data.ts
```

里面先放静态假数据，例如：

- `libraryStats`
- `documentList`
- `filterTabs`
- `selectedDocumentPreview`

注意：

这一阶段仍然不要急着进 Pinia。

## 6.3 第三步：接入前端文件导入能力

静态页面完成后，才开始接真实交互。

第一版推荐支持：

1. 选择单个文件
2. 选择多个文件
3. 选择文件夹

### 第一版支持格式

- `.md`
- `.docx`

### 第一版目标

做到：

- 用户选择文件后，前端能拿到 `File` 对象
- 列表中能新增一条文档记录
- 右侧能显示基础信息

这一阶段不要求马上完成高质量解析。

---

## 6.4 第四步：实现基础解析

这里的“基础解析”指的是第一版可用，不是最终解析引擎。

### 对 md

第一版建议做：

- 读取文本内容
- 提取纯文本
- 尝试按标题切分
- 生成简单标签

### 对 docx

第一版建议做：

- 能读取文本内容
- 能显示文档摘要

### 暂时不做

- 复杂格式清洗
- 图表提取
- 图像 OCR
- 深层结构理解

---

## 6.5 第五步：接本地存储

当你已经可以导入和读取文件后，下一步就不是继续优化 UI，而是接数据持久化。

推荐顺序：

1. 先记录导入后的文档元信息
2. 再记录文档原文或纯文本
3. 最后再记录切片结果

### 第一版建议保存的内容

每个文档至少有：

- `id`
- `name`
- `type`
- `size`
- `importedAt`
- `rawText`
- `tags`
- `status`

### 存储建议

第一版优先考虑：

- IndexedDB

不建议只用 `localStorage` 存全文，因为后面文档一多会顶不住。

## 七、资料库页的数据结构建议

第一版不用上 store，但应该先把结构想清楚。

建议文档对象结构：

```ts
interface LibraryDocument {
  id: string
  name: string
  type: 'md' | 'docx'
  size: number
  importedAt: string
  rawText: string
  summary: string
  tags: string[]
  status: 'pending' | 'parsed' | 'error'
}
```

建议切片结构：

```ts
interface DocumentChunk {
  id: string
  documentId: string
  title: string
  content: string
  order: number
  topic?: string
}
```

第一阶段可以只先定义，不一定立刻全用上。

### 17.1 `library.data.ts`

这里适合放：

- 页面假数据
- 第一版页面展示数据

- 持久化逻辑
- 跨页面共享状态

这里负责：

- 组装页面
- 处理当前页面的基础交互
- 管理当前页的本地状态

例如：

- 当前筛选项
- 当前选中文档
- 文件导入后的追加行为

### 17.3 `src/components/library/`

这里负责：

- 可复用的资料库 UI 组件
- 只接收 props 的展示模块

不负责：

- 全局状态管理
- 路由跳转逻辑
