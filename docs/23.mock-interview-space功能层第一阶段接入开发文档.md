# 23.mock-interview-space 功能层第一阶段接入开发文档

## 1. 当前阶段判断

基于 `docs/22.mock-interview-space后续开发方向与功能层接入文档.md`（后续开发方向与功能层接入文档）以及最近几轮实际开发，`/showcase/mock-interview-space`（模拟面试星球页）已经完成了第一阶段功能层接入前半段的铺垫：

1. `mock`（模拟面试场景）已经能读取并展示 `practicePlan`（专项训练计划）。
2. `report`（复盘报告场景）已经能触发“按本轮弱项继续补练”。
3. `PersistedWorkbenchContext`（工作台持久化上下文）与 `PersistedReportSummary`（持久化报告摘要）已经共用 `PersistedPracticePlan`（专项训练计划类型）。
4. `report -> feedback -> mock`（报告 -> 专项刷题 -> 模拟面试）的状态链路已经打通到“能工作”的程度。
5. 当前是否真正接入 AI 流式问答，仍取决于 `VITE_INTERVIEW_SSE_URL`（面试流式接口地址）是否存在；没有该地址时，前端仍会退回本地模拟流。

这意味着接下来的工作不应该再继续碎片化改局部卡片，而要正式转入：

**真实流式接入 -> 定向补练增强 -> 全链路联调验收**

为了降低回归风险，建议按三轮推进，而不是一次性把所有能力揉在一起。

---

## 2. 本文目标

本文不是新的视觉文档，而是功能层第一阶段的执行文档。目标是把星球页当前已经有的“看起来成立的训练闭环”推进成“真实、稳定、可验收的训练闭环”。

三轮目标如下：

1. **第一轮：接通真实 AI 流式模拟面试。**
2. **第二轮：把报告续练升级成更细粒度的定向补练。**
3. **第三轮：完成专项刷题、模拟面试、复盘报告的全链路联调与验收。**

---

## 3. 第一轮：真实 AI 流式模拟面试接入

### 3.1 目标

把当前“有接口时走真实流式、没接口时走本地模拟流”的半接入状态，推进为明确、可配置、可观察的真实 AI 流式模拟面试能力。

### 3.2 当前现状

现有 `src/services/sse/interview-stream.ts`（面试流式服务）里已经存在这条判断：

1. 有 `VITE_INTERVIEW_SSE_URL`（面试流式接口地址）就发真实 `fetch`（网络请求）。
2. 没有该地址就走 `buildMockInterviewReply`（本地模拟回复）+ `splitTextToReader`（本地分片阅读器）。

这说明前端壳子已经在，但仍缺少：

1. 环境变量示例文档。
2. 前后端请求 / 响应契约说明。
3. 流式异常、停止、恢复的完整验收。

### 3.3 本轮要做什么

1. 在 `.env.template`（环境变量模板）中补充 `VITE_INTERVIEW_SSE_URL`（面试流式接口地址）示例。
2. 明确 `InterviewStreamRequest`（面试流式请求体）和流式返回结构，优先写入文档或代码注释稳定契约。
3. 让 `mock`（模拟面试场景）能区分：
   - 当前为真实 AI 流式
   - 当前为本地模拟流
4. 验证 `submit`（提交回答）、`stop`（停止生成）、`streamError`（流式错误）、`finish`（结束训练）在真实接口下都能正确工作。
5. 保持 `mock-interview-space.vue`（模拟面试星际页面主文件）继续只做装配，不回灌大段接口流程逻辑。

### 3.4 推荐改动落点

1. `src/services/sse/interview-stream.ts`（面试流式服务）：
   - 明确真实流与本地模拟流的分支边界。
2. `src/composables/useInterviewStream.ts`（面试流式对话组合函数）：
   - 承接真实流式状态。
3. `src/composables/showcase/useMockInterviewSpaceMockState.ts`（星球页模拟面试状态组合函数）：
   - 只消费稳定流式状态，不直接组装接口协议。
4. `.env.template`（环境变量模板）：
   - 补充 `VITE_INTERVIEW_SSE_URL` 示例说明。

### 3.5 验收标准

1. 配置 `VITE_INTERVIEW_SSE_URL` 后，提交回答能进入真实流式追问。
2. 不配置 `VITE_INTERVIEW_SSE_URL` 时，系统仍能明确回退到本地模拟流，不是静默失败。
3. 流式生成中可以停止。
4. 接口错误时页面能显示明确报错，不会把 session（训练轮次）打成半残状态。
5. `npm run build`（生产构建验证）通过，且新增文件通过乱码扫描。

## 4. 第二轮：报告续练升级为定向补练

### 4.1 目标

把当前“按本轮弱项继续补练”的单层续练，升级成更细粒度、更可解释的定向补练，而不是只靠一个弱项标签反复开下一轮。

### 4.2 当前现状

现在已经具备：

1. `report`（复盘报告场景）可以触发继续补练。
2. `practicePlan`（专项训练计划）可以回写进当前上下文。
3. `mock`（模拟面试场景）可以显式展示当前专项训练卡片。

但当前计划仍偏粗：

1. `weaknessTag`（弱项标签）粒度较大。
2. `reason`（生成原因）还停留在解释性文本层面。
3. 训练计划还没有拆成“结构表达 / 案例细节 / 指标结果 / 原理追问”等更细维度。

### 4.3 本轮要做什么

1. 扩展 `PersistedPracticePlan`（专项训练计划类型），让它能表达更细的补练方向。
2. 在 `PersistedReportSummary`（持久化报告摘要）里沉淀更稳定的弱项维度，而不是只存最终文案。
3. 报告场景生成续练计划时，优先复用结构化弱项信号，而不是重新猜。
4. `mock`（模拟面试场景）里的专项训练卡片继续只做展示，不新增业务推导。

### 4.4 建议的细粒度维度

可优先从下面四类开始，不建议第一轮就扩成大而全体系：

1. `structure`（结构表达弱）：
   回答没有清晰分段，没有先结论后展开。
2. `case_detail`（案例细节弱）：
   讲了方案，但缺业务场景、约束、取舍和过程。
3. `result_metric`（结果指标弱）：
   讲了动作，但没有结果、收益或量化指标。
4. `principle_depth`（原理追问弱）：
   能说流程，但经不起“为什么这样做”的追问。

### 4.5 推荐改动落点

1. `src/types/workbench.ts`（工作台类型定义）：
   - 扩展 `PersistedPracticePlan`（专项训练计划类型）与报告弱项结构。
2. `src/composables/showcase/useMockInterviewSpaceReportScene.ts`（星球页报告场景组合函数）：
   - 负责把摘要里的真实弱项结构映射为续练入口。
3. `src/views/showcase/mock-interview-space.vue`（模拟面试星际页面主文件）：
   - 复用既有 `handlePracticeStart`（专项训练启动处理）链路，不新增平行写入路径。

### 4.6 验收标准

1. 同样是“继续补练”，不同类型弱项会导向不同训练重点。
2. 报告页到模拟面试页的专项训练卡片内容，与报告中的弱项判断一致。
3. 续练计划不再只是一个泛标签，而是能解释“为什么练这个”。
4. 刷新页面后，当前续练计划仍可恢复。

## 5. 第三轮：全链路联调与验收

### 5.1 目标

在前两轮能力接入后，对星球页当前主链路做一次完整联调，把“能跑”提升为“稳定可验收”。

### 5.2 这轮为什么不能省

现在的页面已经横跨：

1. `overview`（总览场景）
2. `library`（资料库场景）
3. `mock`（模拟面试场景）
4. `feedback`（专项刷题场景）
5. `report`（复盘报告场景）

任何一个状态链路没收好，都可能出现：

1. 切场景后 session 丢失。
2. 报告读到旧摘要。
3. 续练计划被其他上下文覆盖。
4. 刷新后回到错误场景或错误资料。

所以第三轮必须作为独立验收轮，而不是顺手测一下。

### 5.3 本轮要做什么

1. 跑通完整主链路：
   - `library`（资料库）选资料
   - `mock`（模拟面试）答题
   - `report`（复盘报告）生成摘要
   - `continuePractice`（继续补练）
   - 回到 `mock`（模拟面试）继续训练
2. 验证刷新恢复：
   - 当前资料
   - 当前 topic（主题）
   - 当前 mode（模式）
   - 当前 practicePlan（专项训练计划）
   - 最新 report summary（报告摘要）
3. 验证异常路径：
   - 流式失败
   - 中途停止
   - 报告为空
   - 资料缺失
4. 输出一份最小手测 checklist（联调清单），作为后续继续迭代的基线。

### 5.4 推荐联调清单

1. 从资料库选择不同文档进入模拟面试，题目上下文是否发生变化。
2. 回答提交后，是否进入真实流式或明确回退到本地模拟流。
3. 结束本轮后，报告是否引用本轮真实回答内容。
4. 点击“按本轮弱项继续补练”后，专项训练卡片是否和报告弱项一致。
5. 页面刷新后，当前训练资料、训练计划、报告摘要是否还能恢复。
6. `npm run build`（生产构建验证）是否持续通过。

### 5.5 验收标准

1. 从资料库到模拟面试到报告再到补练的最短闭环稳定可复现。
2. 关键场景切换后不会出现状态错位。
3. 真实 AI 流式与本地模拟流的边界对开发者和测试者都清晰可见。
4. 当前主文件与组合函数职责边界未被破坏。

---

## 6. 三轮之间的依赖关系

建议严格按顺序推进，不建议跳轮。

1. **先做第一轮，再做第二轮**
   因为如果真实流式链路没站稳，第二轮的弱项信号质量就不可靠。
2. **第二轮完成后再做第三轮**
   因为联调验收应该验证最终续练形态，而不是验证一套很快会变掉的临时链路。

简化理解就是：

**先把问答做真，再把补练做细，最后把整条链路做稳。**

---

## 7. 当前不要做的事

在这三轮期间，仍然不建议做：

1. 再开新的场景轨道。
2. 重写星球页视觉骨架。
3. 把大量业务逻辑塞回 `mock-interview-space.vue`（模拟面试星际页面主文件）。
4. 提前扩展账号、云同步、后台题库系统。
5. 在没有文档化契约前同时接多个风格不同的流式接口。

这些事情都会稀释当前“最短真实训练闭环”的推进速度。

---

## 8. 一句话结论

`mock-interview-space`（模拟面试星球页）接下来的功能层第一阶段，应该按三轮推进：

**第一轮接通真实 AI 流式模拟面试，第二轮把报告续练升级为定向补练，第三轮完成整条训练闭环的联调与验收。**
