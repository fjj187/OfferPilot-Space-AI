# 42.Three.js 星球真实化改造方案

## 一、目标

本方案用于规划 `mock-interview-space`（宇宙模拟面试页）星球视觉从 `CSS gradient`（CSS 渐变）叠层升级到 `Three.js`（三维渲染库）的落地路径。

核心目标是：

1. 保留现有页面结构、场景编排和 `GSAP`（动画库）轨道切换逻辑。
2. 只把 `SpaceVisualStage.vue`（星球舞台组件）里的星球本体升级为 `Three.js`（三维渲染库）渲染层。
3. 外围发光、标签、轨道 `UI`（用户界面）继续沿用 `DOM`（文档对象模型）和 `CSS`（层叠样式表）。
4. 第一阶段先做“单星球真实化”，不把整页重做成纯 `WebGL`（网页图形渲染技术）应用。

本方案不直接修改 `mock-interview-space.vue`（宇宙模拟面试页主文件）的状态链路，不修改后端接口，不修改 `SSE`（服务端发送事件）契约。

## 二、现状判断

当前星球视觉主要来自两个位置：

| 位置 | 当前职责 | 当前限制 |
| --- | --- | --- |
| `SpaceVisualStage.vue`（星球舞台组件） | 渲染多个 `.planet-shell`（星球外壳层），通过 `GSAP`（动画库）驱动轨道位移 | 星球材质依赖 `CSS gradient`（CSS 渐变）和阴影，缺少真实光照与地表细节 |
| `mockInterviewSpaceScenes.ts`（宇宙页场景配置文件） | 提供每个场景的 `planet`（星球渐变）、`planetGlow`（星球辉光）、`planetShadow`（星球阴影）等主题参数 | 更适合做色彩主题，不适合承载复杂三维材质 |

这种方案优点是轻、稳定、好接 `GSAP`（动画库），但放大后容易出现平面感和塑料感。要做真实地表起伏、云层自转、熔岩流、风暴带、极光和动态呼吸细节，继续堆 `CSS`（层叠样式表）会越来越吃力。

因此，推荐引入 `Three.js`（三维渲染库），但只替换星球本体渲染，不推翻整页。

## 三、架构落点

### 3.1 保持不变的部分

以下部分继续沿用现有实现：

1. `mock-interview-space.vue`（宇宙模拟面试页主文件）继续只负责页面装配和场景编排。
2. `useMockInterviewSpaceOrbit.ts`（宇宙页轨道控制组合函数）继续负责轨道顺序、自动播放和切换方向。
3. `SpaceOrbitNav.vue`（轨道导航组件）继续负责轨道 `DOM`（文档对象模型）展示。
4. `SpaceContentPanel.vue`（内容面板组件）继续负责内容场景挂载。
5. `GSAP`（动画库）继续负责 `visual-layer`（视觉层）的位移、缩放、透明度和层级动画。

### 3.2 新增或改造的部分

推荐第一阶段只新增一个专用星球组件：

```text
frontend/src/components/showcase/mock-interview-space/SpaceThreePlanet.vue
```

`SpaceThreePlanet.vue`（三维星球组件）只负责一件事：在固定尺寸容器内渲染一个 `Three.js`（三维渲染库）星球。

`SpaceVisualStage.vue`（星球舞台组件）仍然负责：

1. 多个场景星球的轨道层布局。
2. `visual-layer`（视觉层）动画。
3. 登录大爆炸 `playBigBangReveal`（播放大爆炸入场函数）。
4. `syncVisualLayers`（视觉层同步函数）。

也就是说，`SpaceVisualStage.vue`（星球舞台组件）只把原来的 `.planet-shell`（星球外壳层）替换为 `SpaceThreePlanet`（三维星球组件），不接管三维渲染细节。

## 四、为什么不直接手写 WebGL

不建议直接手写 `WebGL`（网页图形渲染技术），原因是：

1. 当前目标是产品级视觉升级，不是图形学实验。
2. `Three.js`（三维渲染库）已经封装好相机、灯光、几何体、材质、纹理、动画循环和资源释放。
3. 后续要接 `PBR material`（基于物理的材质）、`normal map`（法线贴图）、`roughness map`（粗糙度贴图）、`emissive map`（自发光贴图）、`shader`（着色器）和 `post-processing`（后处理）时，`Three.js`（三维渲染库）成本更低。
4. 项目主栈是 `Vue 3`（前端框架）和 `Vite`（前端构建工具），`Three.js`（三维渲染库）能以组件方式接入，边界更清晰。

## 五、阶段规划

> 当前进度：`Phase 1`（第一阶段）已落地到 `SpaceThreePlanet.vue`（三维星球组件），完成 `Three.js`（三维渲染库）基础球体、程序地表纹理、云层自转、场景化纹理差异和 `Fresnel`（菲涅耳）边缘大气光；`Phase 2`（第二阶段）已开始加入场景生态效果层，通过 `effectMesh`（动态效果网格）和 `effectTexture`（动态效果纹理）区分海洋光斑、岩质雾、熔岩裂隙、风暴带和能量经纬线；`SpaceVisualStage.vue`（星球舞台组件）仍保留原轨道层动画职责。

### Phase 1：单星球真实化

目标：只替换星球本体，保持现有轨道和场景切换。

建议能力：

1. 使用 `SphereGeometry`（球体几何体）生成星球主体。
2. 使用 `MeshStandardMaterial`（标准网格材质）建立基础受光效果。
3. 使用程序噪声生成表面纹理，先不依赖外部贴图。
4. 新增一层半透明云层球体，云层与地表分不同速度自转。
5. 新增 `Fresnel`（菲涅耳）边缘光，形成大气层轮廓。
6. 按 `scene.theme`（场景主题）映射每个场景的主色、暗部色和自发光色。

此阶段不引入 `post-processing`（后处理），避免第一版复杂度过高。

### Phase 2：动态生态层

目标：让不同场景星球产生明确差异。

建议能力：

1. `overview`（总览场景）：青蓝海洋感地表、轻云层、柔和大气边缘。
2. `library`（资料库场景）：紫灰岩质地表、慢速雾层、低饱和反光。
3. `mock`（模拟面试场景）：暖色荒漠或熔岩裂隙，弱自发光脉冲。
4. `feedback`（专项刷题场景）：蓝白风暴带、快速云层、局部高光流动。
5. `report`（报告场景）：紫色能量纹理、缓慢发光经纬线。

此阶段可以引入轻量 `shader`（着色器），但仍保持组件边界在 `SpaceThreePlanet.vue`（三维星球组件）内部。

### Phase 3：质量增强

目标：在性能可控的前提下提升作品级冲击力。

建议能力：

1. 引入 `normal map`（法线贴图）或程序法线，让地表有起伏。
2. 引入 `roughness map`（粗糙度贴图），区分海面、岩层和能量区反光。
3. 可选接入 `EffectComposer`（后处理合成器）和 `bloom`（辉光后处理），但只对三维星球局部生效。
4. 为低性能设备提供 `reduced motion`（减少动态效果）降级。

## 六、第一版组件设计

### 6.1 `SpaceThreePlanet.vue`（三维星球组件）职责

`SpaceThreePlanet.vue`（三维星球组件）建议只接收展示参数：

```ts
interface SpaceThreePlanetProps {
  sceneId: SceneItem['id']
  primaryColor: string
  secondaryColor: string
  shadowColor: string
  atmosphereColor: string
  active: boolean
}
```

职责边界：

1. 创建 `Scene`（三维场景）、`PerspectiveCamera`（透视相机）、`WebGLRenderer`（网页图形渲染器）。
2. 创建星球主体、云层和大气层。
3. 维护 `requestAnimationFrame`（浏览器逐帧回调）的自转动画。
4. 在组件卸载时释放几何体、材质、纹理和渲染器。
5. 根据 `active`（是否当前激活）调整自转速度和细节透明度。

不应该做：

1. 不处理轨道切换。
2. 不读取路由。
3. 不写业务状态。
4. 不修改 `SceneItem`（场景配置项）结构。
5. 不直接控制 `GSAP`（动画库）轨道动画。

### 6.2 `SpaceVisualStage.vue`（星球舞台组件）改造方式

建议替换模板中的 `.planet-shell`（星球外壳层）：

```vue
<SpaceThreePlanet
  :scene-id="scene.id"
  :primary-color="scene.theme.primary"
  :secondary-color="scene.theme.secondary"
  :shadow-color="scene.theme.planetShadow"
  :atmosphere-color="scene.theme.atmosphere"
  :active="index === activeSceneIndexBySlot"
/>
```

`planetStyleFor`（星球样式生成函数）仍可保留，用于控制尺寸、位置和透明度。后续可以将原 `background`（背景样式）参数转为 `fallback`（降级展示）样式。

## 七、材质方案

第一版优先使用程序噪声，而不是直接找贴图。

原因：

1. 现有五个场景主题色差异明显，程序噪声更容易统一视觉语言。
2. 避免外部贴图版权、体积和加载失败问题。
3. 可以通过同一套参数生成不同星球，不增加配置维护成本。

建议材质层：

| 层 | 技术 | 作用 |
| --- | --- | --- |
| 地表层 | `shader`（着色器）或 `CanvasTexture`（画布纹理） | 生成大陆、岩层、熔岩或能量纹理 |
| 云层 | 半透明 `SphereGeometry`（球体几何体） | 做缓慢自转和呼吸变化 |
| 大气层 | `Fresnel`（菲涅耳）边缘光 | 强化球体轮廓和空间感 |
| 暗面 | `DirectionalLight`（方向光）加材质粗糙度 | 形成真实明暗关系 |
| 自发光 | `emissive`（自发光材质属性） | 表现熔岩、能量线或城市光 |

## 八、性能边界

第一阶段需要控制以下边界：

1. 同屏最多渲染五个 `Canvas`（画布）或一个共享 `Canvas`（画布）。第一版为了组件简单，可以先使用每个星球一个 `Canvas`（画布），验证性能后再决定是否收敛。
2. 每个星球几何体分段建议从 `64 x 64` 起步，不要一开始上高分段。
3. 非激活星球降低动画速度、云层透明度和像素比。
4. `renderer.setPixelRatio`（设置渲染像素比函数）建议限制到 `Math.min(window.devicePixelRatio, 1.5)`（限制设备像素比）。
5. 页面不可见或组件卸载时必须停止 `requestAnimationFrame`（浏览器逐帧回调）。
6. 支持 `prefers-reduced-motion`（系统减少动态效果偏好）时降低自转和云层动画。

如果后续发现五个独立 `Canvas`（画布）开销过大，再进入第二版共享 `renderer`（渲染器）方案，不在第一阶段提前复杂化。

## 九、落地顺序

推荐按以下顺序执行：

1. 安装 `three`（三维渲染库包）并确认 `Vite`（前端构建工具）构建通过。
2. 新增 `SpaceThreePlanet.vue`（三维星球组件），先做一个基础旋转球体。
3. 在 `SpaceVisualStage.vue`（星球舞台组件）中替换 `.planet-shell`（星球外壳层）。
4. 保留 `.planet-shell`（星球外壳层）相关 `CSS`（层叠样式表）作为临时降级样式，确认稳定后再清理。
5. 接入场景主题色映射，让五个星球保持现有色彩识别。
6. 加云层自转和 `Fresnel`（菲涅耳）光晕。
7. 做移动端和桌面端截图验证。
8. 再考虑是否引入 `shader`（着色器）增强地表细节。

## 十、验证清单

### 10.1 构建验证

1. 执行 `pnpm build`（执行生产构建验证）。
2. 如果本机 `pnpm`（包管理器）不可用，使用项目已有可用命令执行等价构建，并记录原因。

### 10.2 页面验证

1. 打开 `/showcase/mock-interview-space`（宇宙模拟面试页）。
2. 检查 `overview`（总览场景）、`library`（资料库场景）、`mock`（模拟面试场景）、`feedback`（专项刷题场景）、`report`（报告场景）轨道切换是否正常。
3. 检查登录大爆炸 `playBigBangReveal`（播放大爆炸入场函数）是否仍能把星球从中心展开到轨道位。
4. 检查自动播放和手动点击轨道时，星球层级、透明度和位移是否正常。
5. 检查移动端是否出现 `Canvas`（画布）溢出、遮挡内容或黑屏。

### 10.3 性能验证

1. 桌面端观察切换时是否掉帧明显。
2. 移动端观察页面首屏是否过慢。
3. 检查 `WebGLRenderer`（网页图形渲染器）是否在组件卸载时释放。
4. 检查浏览器控制台是否出现 `WebGL context lost`（网页图形上下文丢失）或纹理警告。

## 十一、风险与处理

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 多个 `Canvas`（画布）同时运行导致掉帧 | 轨道切换卡顿，移动端发热 | 降低非激活星球像素比和动画速度；必要时改共享 `renderer`（渲染器） |
| `Three.js`（三维渲染库）体积增加 | 首屏资源变大 | 只引入核心模块，不提前引入重型 `post-processing`（后处理） |
| 星球色彩偏离现有主题 | 页面识别度下降 | 继续从 `scene.theme`（场景主题）派生材质颜色 |
| 大爆炸动画与 `Canvas`（画布）尺寸不同步 | 入场阶段抖动或闪烁 | `SpaceVisualStage.vue`（星球舞台组件）仍统一控制外层尺寸和变换 |
| 资源未释放 | 长时间切换后内存上涨 | `onBeforeUnmount`（Vue 卸载前钩子）里统一释放 `geometry`（几何体）、`material`（材质）、`texture`（纹理）和 `renderer`（渲染器） |

## 十二、边界与不做事项

本阶段不做：

1. 不把整页改成纯 `WebGL`（网页图形渲染技术）应用。
2. 不修改 `mock-interview-space.vue`（宇宙模拟面试页主文件）状态架构。
3. 不修改 `POST /api/interview/stream`（模拟面试流式接口）。
4. 不修改 `SSE`（服务端发送事件）事件结构。
5. 不把星球材质配置扩散到业务状态。
6. 不新增后台配置项。
7. 不提前做复杂 `post-processing`（后处理）链路。

本阶段必须保证：

1. 现有轨道切换体验不回退。
2. 星球真实感明显强于当前 `CSS gradient`（CSS 渐变）方案。
3. 组件边界清晰，三维渲染细节留在 `SpaceThreePlanet.vue`（三维星球组件）内部。
4. 出问题时可以快速回退到原 `.planet-shell`（星球外壳层）方案。

## 十三、面试口述版

如果面试官问“为什么要给星球视觉接 `Three.js`（三维渲染库）”，可以这样说：

> 这个页面原来的星球是 `CSS gradient`（CSS 渐变）和阴影叠出来的，优点是轻，但缺少真实光照、地表起伏和动态云层。我的改造思路不是把整页推翻成 `WebGL`（网页图形渲染技术）应用，而是保留原来的 `Vue`（前端框架）组件结构、`GSAP`（动画库）轨道动画和 `DOM`（文档对象模型）信息层，只把星球本体抽成一个 `Three.js`（三维渲染库）组件。这样既能通过球体几何、光照、云层和 `Fresnel`（菲涅耳）大气层提升真实感，又不会破坏现有业务状态链路。

如果继续被追问“为什么不直接手写 `WebGL`（网页图形渲染技术）”，可以这样说：

> 这个需求的重点是产品视觉质量，不是图形学底层实验。`Three.js`（三维渲染库）已经把相机、灯光、材质、纹理和动画循环封装得很成熟，我可以把精力放在材质表现、性能控制和组件边界上。后续如果需要更强效果，也可以在 `Three.js`（三维渲染库）里逐步接 `shader`（着色器）和 `post-processing`（后处理），比直接手写底层 `WebGL`（网页图形渲染技术）更稳。
