# 46.资料库大模型出题改造方案（前端+Node.js版）

## 0. 当前落地进度

本阶段已完成第一批代码接入：

1. Node.js（服务端运行时）新增 `/api/resource/*`（资料出题接口组）。
2. 后端新增 ResourceQuestionService（资料出题服务）、ResourceQuestionStore（资料出题存储）和 ResourceQuestion 类型定义。
3. 前端新增 resource-question-api（资料出题接口客户端），资料库“生成练习题”已切到 seed（题目种子）式出题链路。
4. 随机组卷调用 `/api/resource/question/random`（随机资料出题接口）。
5. 顺序组卷调用 `/api/resource/question/next`（顺序资料出题接口）。
6. 前端批量生成资料练习题时传入 `fastMode`（快速模式），复用 seed（题目种子）直接成题，避免每道题都调用 LLM（大语言模型）导致长时间等待。
7. chunk（文本切块）仍保留为模型分析和资料溯源依据，不再作为前端直接组题单位。

## 1. 背景

当前资料库出题仍然大量依赖 chunk（文本切块）直接出题，存在以下问题：

- 题目质量不稳定
- 上下文不完整
- 容易出现乱码、断句异常、语义缺失
- 随机出题与顺序出题依赖切片顺序，业务含义偏弱

因此需要将资料库出题链路从“切分式出题”改造为“基于大模型（LLM，大语言模型）理解资料后出题”。

本方案第二版明确约束如下：

- 本次方案不依赖 C++ 后端
- 可由前端 + Node.js 后端独立完成
- 不修改既有 C++ 流式能力边界
- 后续如需接入 C++，只作为可选接入层，不作为本期前置条件

---

## 2. 方案目标

本次改造目标：

1. 资料导入后，不再直接基于 chunk 出题
2. 改为先抽取知识点与题目种子，再由大模型生成正式题目
3. 资料库页面中的“随机出题”和“顺序出题”都改为基于题目种子池运行
4. 前端可直接对接 Node.js 后端完成调用
5. 保留 chunk，但仅作为检索依据和溯源材料，不再作为直接出题单位

---

## 3. 总体架构

新的出题架构分三层：

### 3.1 资料层

负责保存原始资料与切片结果。

包含：

- Resource（资料实体）
- ResourceChunk（资料切片）

职责：

- 保存资料原文
- 供知识提炼时引用
- 供题目溯源时引用

注意：

chunk 继续保留，但不再直接用于随机出题或顺序出题。

### 3.2 语义层

负责把资料转换成可出题的语义结构。

包含：

- ResourceOutline（资料大纲）
- KnowledgePoint（知识点）
- QuestionSeed（题目种子）

职责：

- 把资料转成章节结构
- 提取知识点
- 为后续出题生成稳定的语义单元

### 3.3 出题层

负责真正生成题目并提供给前端展示。

包含：

- GeneratedQuestion（正式题目）
- QuestionSession（出题会话）
- SequenceCursor（顺序游标）

职责：

- 按随机模式取题
- 按顺序模式取题
- 调用大模型生成正式题目
- 记录已出题内容与游标状态

---

## 4. 为什么本期不需要 C++ 后端

本次改造的核心是“业务出题逻辑改造”，不是“底层流式通道改造”。

也就是说：

- C++ 后端如果已经实现了 SSE（服务端推送事件），那是通道能力
- 本次要做的是资料分析、题目种子管理、随机/顺序策略改造
- 这些业务逻辑完全可以先由 Node.js 后端独立承担

因此本期可以采用：

- 前端调用 Node.js 接口
- Node.js 负责调大模型
- Node.js 负责生成题目种子
- Node.js 负责随机/顺序策略
- 前端只负责展示、切换模式、维护页面状态

结论：

本方案与 C++ 后端不冲突，但本期不依赖它。

---

## 5. 推荐落地方式

推荐采用“前端 + Node.js BFF（后端聚合层）”模式。

这里的 BFF（前端聚合后端）职责如下：

- 接收资料分析请求
- 调大模型做知识点提取
- 生成 QuestionSeed（题目种子）
- 维护随机 / 顺序出题逻辑
- 生成正式题目
- 返回结构化结果给前端

### 5.1 前端职责

前端负责：

- 资料库页面交互
- 出题模式切换
- 已出题状态展示
- 顺序游标管理
- 流式渲染或轮询展示
- 题目缓存展示

### 5.2 Node.js 后端职责

Node.js 后端负责：

- 文件解析
- 资料切片
- 资料摘要与知识点提取
- 题目种子生成
- 随机/顺序调度
- 大模型调用
- 结构化题目输出
- 可选的数据库持久化

## 7. 新的出题流程

## 7.1 资料导入流程

1. 用户导入资料
2. Node.js 后端解析文件
3. 后端做 chunk（文本切块）处理
4. 后端调用大模型生成：
   - 资料摘要
   - 章节结构
   - 知识点列表
   - 题目种子列表
5. 持久化保存 QuestionSeed（题目种子）
6. 前端显示“资料已可出题”

这里的关键变化是：

资料导入完成后，系统不再只得到 chunk，而是得到“可出题的种子池”。

## 7.2 随机出题流程

1. 前端点击“随机出题”
2. 前端请求 Node.js 后端随机取题
3. 后端从 QuestionSeed 池中筛选候选种子
4. 根据去重规则选出一个 seed（种子）
5. 后端调用大模型生成正式题目
6. 返回结构化题目给前端
7. 前端展示题目并记录已使用 seedId（种子ID）

## 7.3 顺序出题流程

1. 前端点击“顺序出题”
2. 前端带上 currentCursor（当前游标）
3. 后端按 sequenceMode（顺序模式）查找下一条 QuestionSeed
4. 后端调用大模型生成正式题目
5. 返回题目与 nextCursor（下一个游标）
6. 前端更新游标并展示题目

---

## 8. 随机出题如何修改

当前如果是“随机抽 chunk”，必须改为“随机抽 QuestionSeed”。

### 8.1 新随机维度

随机时建议支持以下维度：

- knowledgePoint（知识点）
- difficulty（难度）
- questionType（题型）
- chapterTitle（章节）

### 8.2 去重策略

建议至少实现以下规则：

- 同一个 seedId 短时间不重复
- 同一个 knowledgePoint 连续命中次数受限
- 同一 chapterTitle 连续命中次数受限
- 同一 questionType 连续命中次数受限

### 8.3 前端状态调整

前端不要再维护：

- currentChunkIndex

前端应改为维护：

- usedSeedIds
- lastQuestionId
- randomFilters
- currentSessionId

---

## 9. 顺序出题如何修改

当前如果是“按 chunk 顺序出题”，必须改为“按语义顺序出题”。

### 9.1 推荐顺序模式

建议顺序模式支持三种：

- by_chapter（按章节）
- by_knowledge（按知识点递进）
- by_difficulty（按难度递进）

### 9.2 默认策略

如果页面暂时不想加太多控件，建议先只保留一个“顺序出题”按钮，后端默认用：

- by_chapter（按章节）

后续再扩展成下拉项或分段切换。

### 9.3 前端状态调整

前端不要再维护：

- currentChunkIndex
- totalChunkCount

前端应改为维护：

- sequenceMode
- currentCursor
- currentSeedId
- hasNextQuestion

---

## 10. 前端页面改造建议

## 10.1 页面交互层

资料库页面建议保留现有两个主要动作：

- 随机出题
- 顺序出题

但内部逻辑切换为“面向种子池”。

### 10.2 新增页面状态

建议页面状态至少包含：

- selectedResourceId
- questionMode
- sequenceMode
- currentSessionId
- currentCursor
- usedSeedIds
- currentQuestion
- generating
- analyzeStatus

### 10.3 页面文案建议

为了减少用户理解成本，页面上不需要强调 chunk（文本切块）。

页面只体现：

- 资料解析中
- 题目生成中
- 随机出题
- 顺序出题
- 下一题
- 重新出题

## 11. Pinia（状态管理库）改造建议

建议拆为两个 store（状态仓库）：

### 11.1 resourceQuestionStore（资料出题状态仓库）

职责：

- 管理当前资料的出题状态
- 管理当前题目
- 管理随机/顺序模式
- 管理游标与已使用种子

建议状态：

- selectedResourceId
- questionMode
- sequenceMode
- currentCursor
- usedSeedIds
- currentQuestion
- loading
- error

### 11.2 resourceAnalyzeStore（资料分析状态仓库）

职责：

- 管理资料分析状态
- 管理资料是否已生成种子
- 管理分析进度与失败重试

建议状态：

- analyzingResourceIds
- analyzeResultMap
- failedResourceIds

## 12. Node.js 接口设计建议

## 12.1 资料分析接口

`POST /api/resource/analyze`

请求体：

```json
{
  "resourceId": "r_001"
}
```

返回：

```json
{
  "success": true,
  "data": {
    "resourceId": "r_001",
    "summary": "......",
    "knowledgePoints": [],
    "seedCount": 28
  }
}
```

用途：

- 导入资料后触发
- 生成知识点与题目种子

---

## 12.2 获取资料出题状态接口

`GET /api/resource/:resourceId/question-meta`

返回：

```json
{
  "success": true,
  "data": {
    "seedCount": 28,
    "analyzeStatus": "ready",
    "sequenceModes": ["by_chapter", "by_knowledge", "by_difficulty"]
  }
}
```

用途：

- 页面初始化时判断资料是否可直接出题

---

## 12.3 随机出题接口

`POST /api/resource/question/random`

请求体：

```json
{
  "resourceId": "r_001",
  "excludeSeedIds": ["s_001", "s_003"],
  "difficulty": "medium",
  "questionType": "qa",
  "fastMode": true
}
```

返回：

```json
{
  "success": true,
  "data": {
    "questionId": "q_001",
    "seedId": "s_011",
    "title": "......",
    "question": "......",
    "answerOutline": ["......"],
    "knowledgePoint": "......"
  }
}
```

如果要做流式，也可以改成 SSE（服务端推送事件）输出。

---

## 12.4 顺序出题接口

`POST /api/resource/question/next`

请求体：

```json
{
  "resourceId": "r_001",
  "sequenceMode": "by_chapter",
  "cursor": "12",
  "fastMode": true
}
```

返回：

```json
{
  "success": true,
  "data": {
    "questionId": "q_002",
    "seedId": "s_012",
    "nextCursor": "13",
    "hasNext": true,
    "title": "......",
    "question": "......",
    "answerOutline": ["......"]
  }
}
```

## 13. 大模型调用策略

## 13.1 不再直接拿 chunk 出题

错误方式：

- 随机抽一段 chunk
- 直接让模型按 chunk 出题

正确方式：

- 先从 QuestionSeed 选语义单元
- 再把 seed 信息 + 相关 chunk 依据一起发给模型
- 让模型生成正式题目

### 13.2 推荐提示词输入结构

建议给模型输入：

- 资料摘要
- 当前章节
- 当前知识点
- 题目类型
- 难度要求
- 相关依据切片
- 输出格式要求

### 13.3 强制结构化输出

必须要求模型输出固定结构，例如：

```json
{
  "title": "题目标题",
  "question": "题目内容",
  "answerOutline": ["答案要点1", "答案要点2"],
  "knowledgePoint": "对应知识点",
  "difficulty": "medium",
  "sourceRefs": ["chunk_12", "chunk_13"]
}
```

这样能降低：

- 乱码
- 字段缺失
- 样式不统一
- 前端渲染异常

## 14. 是否需要流式输出

本期不依赖 C++ 流式能力，但 Node.js 也可以自己实现流式。

### 14.1 简化方案

第一期建议直接返回完整 JSON（结构化对象），先把链路跑通。

优点：

- 实现快
- 调试简单
- 成本低
- 对前端改动小

### 14.2 增强方案

如果你希望保留“题目生成中”的体验，可以在 Node.js 中补一层：

- SSE（服务端推送事件）
- 或 WebSocket（长连接通信）

但这不是本期必须项。

本期优先级应该是：

先把“切分式出题”改成“种子式大模型出题”。

## 15. 数据库存储建议

如果你们已有数据库，可建议增加三张表：

- resource_chunks
- question_seeds
- generated_questions

最少也要把 question_seeds 存下来。

原因很简单：

- 题目种子生成成本较高
- 每次都重新分析资料很浪费
- 随机/顺序都依赖稳定的种子池

如果暂时没有正式库，也可以先用：

- JSON（结构化文本）文件
- SQLite（轻量数据库）
- 本地开发表

本期先跑通业务比追求最终存储方案更重要。

## 16. 兼容现有前端的最小改法

为了减少前端改动，建议走“最小替换”策略。

### 16.1 保留原有按钮

页面上仍然保留：

- 随机出题
- 顺序出题

只替换按钮背后的请求逻辑。

### 16.2 替换原来的数据来源

原来可能是：

- 从 chunkList 中抽内容
- 本地组装题目

现在改成：

- 从后端拿正式题目对象
- 页面只负责展示

### 16.3 保留现有题目展示组件

如果已有题目展示组件，只要它支持接收：

- title
- question
- answerOutline

那大概率不需要大改 UI（界面）。

## 18. 风险与边界

### 18.1 本期不碰 C++ 协作边界

本方案默认：

- 不改 C++ 接口
- 不改 C++ 流式能力
- 不等待 C++ 后端排期

### 18.2 不建议前端直接写完整大模型编排

前端可以发起请求，但不要承担复杂 prompt（提示词）编排、题目种子调度、容错逻辑。

这些逻辑更适合放 Node.js 后端。

### 18.3 不建议继续把 chunk 当顺序依据

如果还保留“按 chunk 顺序出题”，后面很容易再次遇到：

- 题意断裂
- 知识点跳跃
- 训练顺序失真

---

## 19. 本期最终建议

本期推荐结论如下：

1. 前端继续保留现有资料库出题入口
2. 新增 Node.js 出题服务，不依赖 C++ 后端
3. 资料导入后先做“资料分析 -> 题目种子生成”
4. 随机出题改为从 QuestionSeed 池随机抽取
5. 顺序出题改为按章节 / 知识点 / 难度推进
6. chunk 只保留为检索依据与溯源依据
7. 正式题目由 Node.js 调大模型生成并返回前端

---

## 20. 一句话总结

第二版方案的本质是：

不再等待 C++ 后端，直接由 Node.js 承接资料理解、题目种子生成、随机/顺序调度和大模型出题；前端只接结构化题目结果，从而把“资料切分式出题”升级为“语义种子式大模型出题”。
