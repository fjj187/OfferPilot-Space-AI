1. 你觉得哪些代码用 AI 生成比较合适？哪些算「简单代码」？

面试回答：

我觉得 AI 更适合生成「边界清晰、模式固定、可快速验证」的代码，而不是核心业务和架构决策。

简单代码主要包括这几类：

- 样板代码：接口类型定义、DTO 映射、表单校验规则、CRUD 页面骨架
- 重复性工具：日期格式化、防抖节流、深拷贝、正则校验
- 单元测试与 Mock 数据：给定输入输出明确的测试用例
- 样式与布局微调：Flex 居中、响应式断点、常见 UI 组件组合
- 文档与注释：README、API 说明、函数 JSDoc

不算简单、需要人主导的包括：鉴权与安全策略、状态机设计、性能瓶颈定位、跨模块协议（比如 SSE 事件契约）、以及涉及用户体验细节的交互逻辑。我的做法是：AI 出初稿，人做 review、跑测试、对齐项目规范后再合入。

解析 / 要点：

- 面试官想考察的是：你有没有把 AI 当「加速器」而不是「替代思考」的意识。
- 「简单」的判断标准可以记四个词：输入明确、输出可验、模式成熟、风险可控。
- 反例要说清楚：密码加密方案、Token 刷新、XSS 防护这类一旦出错代价高的逻辑，不适合盲信 AI。

```text
简单代码  →  模式固定 + 可自动化测试 + 非安全敏感
复杂代码  →  架构决策 + 安全 + 性能 + 跨团队契约
```

2. 在 OfferPilot Space AI 项目中做了哪些模块？遇到什么难点？

面试回答：

**OfferPilot Space AI** 是基于 **Vue 3 + Vite + TypeScript + Naive UI + Pinia + UnoCSS** 的模拟面试训练产品，主入口为 `/showcase/mock-interview-space` 宇宙展示页（页面标题与品牌名与 `systemTitle` 一致），覆盖总览、资料库、模拟面试、报告四个场景。

我主要负责的模块大致有这些：

- **模拟面试流式反馈**：`useInterviewStream` 组合函数 + `POST /api/interview/stream`，把 SSE 的 chunk / done / error 转成问答消息列表。
- **面试报告与历史**：报告摘要拼装、历史会话恢复与报告预览弹窗联动。
- **资料驱动出题**：资料 Markdown 按标题分块，再组装成题组，支撑「按章节练」。
- **专项刷题题池**：按薄弱点、题型、难度从题池组卷，和模拟面试共用 session 持久化。
- **宇宙页场景编排**：`mock-interview-space.vue` 只做页面装配，业务态下沉到 `useMockInterviewSpace`* 系列组合函数和 `Space*Scene` 场景组件。
- **浏览器端持久化**：`useWorkbenchPersistence` 把 entryMode、题序、报告摘要等写入 localStorage，支持刷新恢复。

如果面试官追问「核心实现」，我会概括成五点：

**第一，SSE 流式三层架构。** 传输层 `sse-client.ts` 用 fetch + ReadableStream 解析 event/data 帧；控制层 `interview-stream.ts` 负责 AbortSignal、mock/remote 切换与重试；处理层 `useInterviewStream` 把 chunk 写入消息队列并更新 UI。

**第二，rAF 消息渲染队列。** 模型 token 推送频率很高，每个 chunk 直接改 DOM 会闪烁。`InterviewMessageQueue` 用 requestAnimationFrame 按帧合并 delta，再更新 `displayContent`，比 setTimeout 节流更贴合渲染节奏。

**第三，资料 Markdown 分块与驱动出题。** `material-chunk-splitter.ts` 按标题层级切分，单块不超过 1000 字，并记录 charStart / charEnd，供资料题组和检索使用。

**第四，多入口 session 状态收敛。** 入口有 direct（直接开练）、practice（专项刷题）、material（资料出题）三种 entryMode；用 `mockSessionConfig` 把「从哪进来」和「这一轮怎么练」拆开，配合 `sessionConfigKey` 避免刷新后练错题。

**第五，Node 网关 + Provider 工厂。** 后端 `createInterviewProvider` 按环境切换 Mock / Remote LLM / C++三种 Provider，对外统一 SSE 契约；Vite 把 `/api` 代理到 Node（3030），前端、Node、C++ 可并行开发。

难点主要有三个：

（1）**流式 Markdown 的安全与性能**：模型输出必须边流边渲染，但又要过 DOMPurify 防 XSS，不能裸 `v-html`，也不能等全文结束再渲染。
（2）**高频 chunk 下的 UI 稳定**：最终用 rAF 队列合并增量，而不是每个 token 触发一次响应式更新。
（3）**多入口 session 恢复**：localStorage 里 entryMode、sessionConfigKey、当前题序、feedbackStyle 任一不一致，刷新后就会出现「入口对了、题目错了」的串台问题。

解析 / 要点（结合本项目代码）：

SSE 三层在本项目中的对应关系：

```typescript

// 控制层：frontend/src/services/sse/interview-stream.ts
```

3. 除了通过 Blob URL 封装编译后代码并注入 iframe 实现所见即所得，还有其他方式吗？

面试回答：

有的，常见替代方案有这些：

（1）srcdoc 直接写入 iframe：不用 Blob URL，把编译后的 HTML 字符串赋给 iframe.srcdoc，生命周期更简单，但要注意字符串转义和内容安全策略（CSP）。
（2）Sandboxed iframe + postMessage：父页面只负责编辑器和通信，子 iframe 跑用户代码，通过 postMessage 传编译结果，隔离性更好，也是 CodeSandbox、StackBlitz 一类产品的思路。
（3）Web Worker / Worker + 虚拟 DOM：在 Worker 里执行或转译代码，主线程只负责渲染结果，适合纯 JS 逻辑预览，DOM 操作类场景支持有限。
（4）运行时编译组件：React 可用 @babel/standalone 或 sucrase 在浏览器里 transform 后 eval/new Function 执行；Vue 可用 @vue/compiler-sfc 在客户端编译。不依赖 iframe，但安全隔离弱，只适合可信环境。
（5）Module Federation / ESM CDN：动态 import 远程模块，配合 import maps，适合多文件项目预览。
（6）服务端编译：把源码 POST 到后端，返回 bundle URL 或 HTML 片段，前端只负责展示 — 安全性和一致性更好，但依赖后端。

**OfferPilot Space AI** 走的是另一条路：模拟面试的模型反馈用 **Markdown 流式渲染 + DOMPurify** 展示，不跑用户提交的任意代码；如果要做在线代码 Playground，我会优先选 sandbox iframe + CSP，其次才是 Blob URL。

解析 / 要点：


| 方案                | 优点        | 缺点                       |
| ----------------- | --------- | ------------------------ |
| Blob URL + iframe | 隔离好、实现成熟  | 需 revokeObjectURL，CSP 要配 |
| srcdoc            | 无 Blob 管理 | 大文档性能差，转义麻烦              |
| Web Worker        | 不阻塞主线程    | 无法直接操作 DOM               |
| 浏览器内 Babel        | 体验流畅      | XSS 风险高                  |
| 服务端编译             | 最安全       | 延迟高、成本高                  |


同源策略下，iframe 内的脚本与父页面隔离，是在线代码 Playground 的主流选择；本项目当前业务不需要这层隔离。

4. 项目写了前后端分离，你是怎么对接的？

面试回答：

前后端分离的核心是：前端只负责页面和交互，后端提供 RESTful API，两者通过 HTTP + JSON 通信，部署也分开。

对接流程我一般是这样的：

（1）接口文档对齐：和后端约定 URL、Method、请求/响应字段、错误码；用 Swagger / Apifox 做联调基线。
（2）前端封装请求层：用 Axios 实例统一 baseURL、超时、拦截器。请求拦截器自动带 Token（Authorization: Bearer xxx）；响应拦截器统一处理 401 跳转登录、业务错误码提示。
（3）环境配置：开发环境 Vite/Webpack devServer 配 proxy 解决跨域；生产环境 Nginx 反向代理 `/api` 到后端服务。
（4）状态管理：登录态、用户信息放 Pinia/Vuex + localStorage；页面组件只调 service 层，不直接散落 fetch。
（5）文件上传、分页、鉴权等特殊接口单独封装，保证类型（TypeScript interface）和 backend DTO 一致。
（6）联调与错误处理：先用 Mock 跑通页面，再切真实接口；对网络错误、空数据、权限不足做统一 UX。

解析 / 要点：

典型 Axios 封装：

```typescript
const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000
})

service.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

service.interceptors.response.use(
  res => res.data,
  err => {
    if (err.response?.status === 401) router.push('/login')
    return Promise.reject(err)
  }
)
```

Vite 开发代理（**OfferPilot Space AI** 同类配置）：

```typescript
proxy: {
  '/api': {
    target: 'http://localhost:3030',
    changeOrigin: true
  }
}
```

前后端分离的好处：并行开发、独立部署、前端可静态 CDN；代价是要自己处理跨域、鉴权、接口版本管理。

5. 什么情况会发生跨域？什么情况下是不同源？

面试回答：

跨域本质是浏览器的同源策略限制：当前页面的「源」去请求另一个「源」的资源时，如果不同源，浏览器会拦截响应（除非服务端正确返回 CORS 头）。

源（origin）= 协议 + 域名 + 端口，三者有一个不同就算不同源。

不同源的例子：

- [https://a.com](https://a.com) 请求 [http://a.com](http://a.com) → 协议不同
- [https://a.com](https://a.com) 请求 [https://b.com](https://b.com) → 域名不同
- [https://a.com:443](https://a.com:443) 请求 [https://a.com:8080](https://a.com:8080) → 端口不同

会触发跨域的常见场景：

- 前端 localhost:2048 调后端 localhost:3030 的 API（本项目 dev 典型场景）
- 页面在 a.com，Ajax 请求 b.com 接口
- 前端读取 iframe 里不同源的 DOM（会被拦）
- 用 canvas 绘制跨域图片后 toDataURL（「污染 canvas」）
- 某些字体、WebSocket 跨域连接

不会算跨域的情况：

- 请求静态资源（img、script、link）一般可以加载，但 JS 读内容仍受限
- 同源请求
- 服务端到服务端（Node 调 API）不受浏览器同源策略限制
- 开发环境用 proxy 把请求「变成同源」

解决方式：后端配 Access-Control-Allow-Origin；或 devServer/Nginx 反向代理；复杂场景用 JSONP（老方案）或 postMessage。

解析 / 要点：

```text
源 = protocol + host + port

https://example.com:443/path
 └─协议─┘ └─域名────┘ └端口┘
```

简单记忆：

- 同源：完全一样
- 跨域：协议/域名/端口任一不同
- CORS 是「服务器允许浏览器放行」的机制，不是绕过同源策略，而是协商后的例外

6. 密码加密、Token、数据库是怎么实现的？

面试回答：

这是典型后端负责、前端配合的链路，我按三层说：

一、密码加密（数据库里绝不存明文）

- 注册时：后端对明文密码做哈希，常用 bcrypt 或 argon2（带随机 salt，防彩虹表）。
- 登录时：取出用户记录的 hash，用 bcrypt.compare(明文, hash) 验证，而不是解密。
- 前端只做长度、复杂度校验，不在前端做「可逆加密」—— 前端 HTTPS 传输即可。

二、Token（无状态鉴权）

- 登录成功后，后端签发 JWT（或 sessionId）。
- JWT payload 一般含 userId、过期时间 exp，用服务端 secret 签名。
- 前端把 Token 存 localStorage 或 httpOnly Cookie（更安全，防 XSS 窃取）。
- 之后每次请求在 Header 带 Authorization: Bearer 。
- 后端中间件验签 + 查 exp；过期走 refresh token 或重新登录。

三、数据库

- 关系型 MySQL / PostgreSQL 存用户表、业务表；用户表字段通常有 id、username、password_hash、created_at。
- ORM 如 Prisma / TypeORM / Sequelize 做模型映射。
- 敏感操作（改密、支付）可加二次验证和操作日志。

前端职责：登录表单 → 调 /login → 存 token → 拦截器自动携带 → 401 清 token 跳登录。

解析 / 要点：

Node.js bcrypt 示例：

```javascript
const bcrypt = require('bcrypt')
const saltRounds = 10

// 注册
const hash = await bcrypt.hash(plainPassword, saltRounds)
await db.user.create({ username, password: hash })

// 登录
const user = await db.user.findByUsername(username)
const ok = await bcrypt.compare(plainPassword, user.password)
```

JWT 签发示例：

```javascript
const jwt = require('jsonwebtoken')
const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET, { expiresIn: '7d' })
```

安全清单：

- 密码：单向哈希 + salt，禁止 MD5/SHA1 裸用
- Token：短有效期 + refresh；敏感接口 httpOnly Cookie 更佳
- 数据库：参数化查询防 SQL 注入；最小权限账号连接 DB

7. Coze、DeepSeek 这些是怎么调用的？

面试回答：

这类大模型本质都是 HTTP API，区别在鉴权方式、请求体和是否流式。

DeepSeek（OpenAI 兼容接口）

- 端点：POST `https://api.deepseek.com/chat/completions`
- Header：`Authorization: Bearer <API_KEY>`
- Body：`model`、`messages`、`stream: true` 开启 SSE
- 流式响应按 `data: {...}\n\n` 解析，直到 `[DONE]`

在 **OfferPilot Space AI** 里，模型调用统一走模拟面试网关，不在浏览器直连：

- 前端只调同源 `POST /api/interview/stream`，不暴露 Key，也不在浏览器侧直连各模型 OpenAPI。
- Node 网关由 `RemoteLLMProvider` 持有 `INTERVIEW_REMOTE_API_KEY`，请求 DeepSeek 后把响应转成统一 SSE 事件 chunk / done / error。

Coze（字节扣子 Bot API）

- 在 Coze 控制台创建 Bot，拿到 bot_id 和 Personal Access Token（PAT）
- 常见两种：一次性对话 POST /v3/chat；流式用 stream 参数或 SSE 端点
- 请求体需传 bot_id、user_id（标识终端用户）、query 或 messages
- 若 Bot 挂了插件/工作流，Coze 侧执行，前端只消费最终文本或中间消息
- 同样建议后端代理，Key 不下发浏览器

统一封装思路（本项目模拟面试侧）：

- Provider 工厂按环境变量切换 mock / remote / cpp
- 对外冻结同一份 SSE 契约：chunk、done、error
- 前端只认统一接口，换模型不改 UI 层

解析 / 要点：

经 Node 网关（本项目实际链路）：

```typescript
// 前端只调同源 /api/interview/stream
// backend/src/providers/remote-llm-provider.ts 持有 INTERVIEW_REMOTE_API_KEY
const response = await fetch(createOpenAICompatibleUrl(), {
  headers: { Authorization: `Bearer ${backendEnv.remoteApiKey}` },
  body: JSON.stringify({ model, stream: true, messages })
})
```

Coze 调用示意：

```typescript
await fetch('https://api.coze.com/v3/chat', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${COZE_PAT}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    bot_id: '7xxxxxx',
    user_id: 'user-001',
    stream: true,
    additional_messages: [{ role: 'user', content: '你好' }]
  })
})
```

注意：API Key 放后端；流式要处理断连重试；限速和 token 计费要在产品层做兜底提示。

8. HTML 语义化知道吗？为什么要语义化？

面试回答：

知道。语义化就是 HTML 标签要「名副其实」：用合适的标签表达内容的结构和含义，而不是全 div + span 堆出来。

常见语义标签：

- 结构：header、nav、main、section、article、aside、footer
- 文本：h1–h6、p、strong、em、blockquote
- 列表：ul、ol、li
- 表单：form、label、input、button（type 正确）
- 媒体：figure、figcaption、video、audio

为什么要语义化：

（1）SEO：搜索引擎更好理解页面层级和重点内容
（2）无障碍（a11y）：读屏软件依赖标签角色，div soup 对视障用户不友好
（3）可维护性：团队协作一眼看懂结构，CSS/JS 钩子更清晰
（4）默认样式与行为：button 自带键盘可聚焦、form 自带提交语义
（5）爬虫与结构化数据：配合 schema.org 更容易做富摘要

反例：用 div 做按钮、用 span 做标题，短期能跑，长期 SEO、a11y、维护成本都会变差。

解析 / 要点：

```html
<!-- 不推荐 -->
<div class="header">
  <div class="title">文章标题</div>
  <div class="btn" onclick="submit()">提交</div>
</div>

<!-- 推荐 -->
<header>
  <h1>文章标题</h1>
  <button type="submit">提交</button>
</header>
```

HTML5 大纲算法（outline）虽浏览器支持有限，但 h1–h6 层级仍影响 SEO 和阅读顺序。

语义化 ≠ 不能用 div：当没有合适语义时，div/span 配合 role 和 aria-* 也是正确做法。

9. 你是怎么做拖拽功能的？

面试回答：
我做拖拽时，核心是把“鼠标位置变化”映射成“元素位置变化”。基本思路是：按下时记录起点，移动时计算偏移量，松开时结束拖拽。如果是列表排序，我会在移动过程中判断当前元素应该落在哪个位置，再做交换或插入。

10. script（脚本标签）放在 head（头部标签）和放在 body（主体标签）底部有什么区别？有没有办法既放在 head 里又不堵塞？

面试回答：
放在 head（头部标签）里，浏览器解析到 script（脚本标签）时可能会暂停 HTML（超文本标记语言）解析，等脚本执行完再继续；放在 body（主体标签）底部，通常能让页面先渲染出来，再执行脚本，首屏体验更稳。

`defer`（延迟执行）特点是：下载不阻塞解析，等 HTML（超文本标记语言）解析完成后按顺序执行。
`async`（异步执行）特点是：下载不阻塞解析，但执行顺序不固定，谁先下载完谁先执行。
在项目里，入口脚本一般更适合 `defer`（延迟执行）思路，而一些独立统计脚本可能用 `async`（异步执行）。

17. 说下 JavaScript（脚本语言）的事件循环（Event Loop，事件循环）。假设微任务执行完之后又有新的微任务会什么样？

21. 写一个防抖函数

面试回答：
防抖（debounce，防抖）就是让一个函数在连续触发时，只执行最后一次。比如输入框搜索、窗口 resize（窗口大小调整）、按钮重复点击这些场景都很常见。
在这个项目里，如果某个输入会频繁触发请求，我会优先用防抖来减少无效调用，避免接口和 UI（用户界面）一起抖动。

解析 / 语法点：
防抖核心是定时器：每次触发先清掉上一次的定时器，再重新计时，只有最后一次停下来后才执行函数。

```javascript
function debounce(fn, delay = 300) {
  let timer = null
  return function (...args) {
    if (timer) clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
    }, delay)
  }
}
```

这里的 `clearTimeout`（清除定时器）用来取消上一次执行，`fn.apply`（函数调用）用来保留当前上下文和参数。防抖和节流（throttle，节流）经常一起问，防抖是“停下来再执行”，节流是“固定时间内最多执行一次”。
