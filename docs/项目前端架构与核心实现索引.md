一、整体项目架构（Monorepo 前后端分离）

仓库根目录 chatgpt-vue3-light-mvp-main（pnpm workspace 单体仓库）

  frontend/     Vue 3 + Vite 前端，端口 2048，开发时 pnpm dev
  backend/      Express + TypeScript 网关，端口 3030，开发时 pnpm dev:backend
  docs/         项目文档（协作边界、目录说明、开发记录等）
  package.json  根脚本入口

请求链路（面试常问）

  浏览器
    -> Vite 开发服务器（frontend，统一 /api 入口）
    -> 代理到 Node 网关（backend，默认 localhost:3030）
    -> Provider 工厂（provider factory）按环境切换
         mock 本地模拟
         remote 远程大模型
         cpp C++ 引擎（并行开发位，契约冻结）
    -> SSE（Server-Sent Events，服务端推送）流式响应回前端

请求链路说明（上图展开）

  典型场景：宇宙页 mock-interview-space 里用户提交答案，触发一轮面试官流式回复。以下按时间顺序写清每一跳做什么、落在哪些文件。

  第 1 跳：页面与组合函数发起

    用户点击提交后，容器页 frontend/src/views/showcase/mock-interview-space.vue 或 frontend/src/composables/showcase/useMockInterviewSpaceMockState.ts 组装题目、资料摘要、feedbackStyle（反馈风格）等上下文，最终调用 frontend/src/composables/interview/useInterviewStream.ts 的 startStream（或等价入口）。

    useInterviewStream 先创建一条 status 为 streaming 的 assistant 消息，再调用 frontend/src/services/sse/interview-stream.ts 的 startInterviewStream。

  第 2 跳：前端判断 mock 还是 remote

    frontend/src/services/sse/interview-stream.ts 里 resolveInterviewStreamEndpoint（解析流式接口地址）规则：

      若 .env 配置了 VITE_INTERVIEW_SSE_URL，则用该完整地址（可指向独立网关）。

      否则开发环境 DEV 为 true 时，默认 POST 相对路径 /api/interview/stream。

      否则（例如未配地址的生产构建）endpoint 为空，整段流在前端本地完成，不发起网络请求。

    resolveInterviewStreamMode：只要 endpoint 非空即为 remote，否则为 mock。

    两种模式对下游解析器不同：remote 走 consumeSSEStream（按 SSE 帧解析）；mock 走 consumeStreamText（按纯文本 ReadableStream 读 chunk）。

  第 3 跳：请求体归一化

    normalizeInterviewStreamRequest 把 questionTitle、题序、切题规则、资料摘要/标签/片段、feedbackStyle 指令、候选人 answer 压成一份 prompt，避免 questionPrompt 与 prompt 重复发送。类型见 frontend/src/services/sse/sse-types.ts 的 InterviewStreamRequest，与 backend/src/types/interview.ts 对齐。

  第 4 跳：浏览器到 Vite（仅 remote 且走 /api 时）

    fetch POST http://localhost:2048/api/interview/stream，Body 为 JSON，带 AbortSignal（用户取消时可 abort）。

    Vite server.proxy 把以 /api 开头的路径转发到 VITE_INTERVIEW_BACKEND_ORIGIN，默认 http://localhost:3030，配置在 frontend/vite.config.ts。

    同源策略下前端只认 /api，不直接写 3030 端口，避免 CORS（跨域）和部署环境差异。

    补充：frontend/src/views/chat.vue 等旧对话页的大模型 Key 可能走 /deepseek、/spark 等另一条代理，与面试流 /api/interview/stream 是不同链路；面试联调只关心 /api。

  第 5 跳：Node 网关接收与校验

    backend 路由注册 POST /api/interview/stream，进入 backend/src/controllers/interview-controller.ts 的 streamInterviewController。

    validateInterviewRequest 校验 sessionId、messageId、threadId、topic、questionTitle、questionPrompt、answer、prompt 等必填；失败返回 400 JSON，不走 SSE。

    通过则 setupSSEResponse：Content-Type 为 text/event-stream，Connection keep-alive，随后进入业务流。

  第 6 跳：Provider 工厂与 InterviewService

    createInterviewProvider() 读环境变量（如 backend .env 的 PROVIDER）：remote 用 RemoteLLMProvider，cpp 用 CppEngineProvider，默认 MockInterviewProvider。实现在 backend/src/services/interview-provider-factory.ts。

    InterviewService.streamInterview 先 recordInterviewUserMessage 落库，再 async for await 消费 provider 产出的事件；累积 assistant 全文后在 done 时 recordInterviewAssistantMessage。前端永远不感知具体 Provider 类名，只认 SSE 事件名与 data 形状。

  第 7 跳：Provider 产出与 SSE 写出

    各 Provider 统一产出 InterviewProviderEvent：type 为 chunk 时带 content 字符串片段；结束时 type 为 done；异常为 error。

    controller 用 writeSSEEvent 写标准 SSE 行：event: chunk 与 data: {"content":"..."}，帧之间空行分隔；与 backend/src/utils/sse.ts 一致。客户端断开时 response.on close 会 abort 上游生成，避免孤儿流。

  第 8 跳：响应回到前端传输层

    frontend/src/services/sse/interview-stream.ts 的 createFetchReader 把 response.body 经 TextDecoderStream 得到 ReadableStreamDefaultReader。

    frontend/src/services/sse/sse-client.ts 的 consumeSSEStream 循环 read，缓冲区按 \n\n 切帧，parseSSEFrame 解析 event 与 data 行，回调 onEvent。

    同上 interview-stream.ts 把 frame.event 为 chunk 的 data JSON 解出 content，封装为统一 InterviewStreamEvent { type: 'chunk', messageId, chunk }；done、error 同理。若连接结束但未收到 done 事件，onDone 会补发 done，防止 UI 一直 streaming。

  第 9 跳：前端控制层与 rAF 队列

    frontend/src/composables/interview/useInterviewStream.ts 的 onEvent 分支：

      start：登记 messageId，messageQueue.register，绑定 patchMessage 更新 displayContent。

      chunk：frontend/src/services/message/interview-message-parser.ts 的 parseInterviewStreamChunk 归一化后，frontend/src/services/message/interview-message-queue.ts 的 InterviewMessageQueue.enqueue；内部 requestAnimationFrame 每帧合并 delta 再写 DOM，降低 LLM 高频 token 卡顿。

      done：flushNow、unregister、消息 status 改 done，isStreaming 置 false。

      error：同样 flush，写入错误文案，可记入 retryableState 供重试。

  第 10 跳：展示与安全渲染

    frontend/src/components/message/MessageList.vue 渲染 frontend/src/components/message/MessageBubble.vue；Markdown 格式走 frontend/src/components/message/MarkdownMessage.vue：markdown-it 转 HTML 后 DOMPurify.sanitize，再 v-html，防止流式内容中的 XSS。

  两条常见分支对照（面试可一句话带过）

    分支 A（本地全前端 mock，无 endpoint）

      buildMockInterviewReply 在 frontend/src/services/sse/interview-stream.ts 内根据资料、feedbackStyle、unknownAnswerStreak 拼整段 Markdown 回复，splitTextToReader 模拟打字机切块；不经 Vite 代理、不经 Node。适合 GitHub Pages 或未起 backend 时演示。

    分支 B（开发联调 / 真实后端，endpoint 为 /api/interview/stream）

      完整走上图：Vite 代理 -> Node -> Provider -> SSE 回前端 -> rAF 队列 -> DOMPurify 展示。切换 mock、remote、cpp 只改 backend 环境，前端路径与事件契约不变。

  环境与命令速查

    pnpm dev          前端 2048，DEV 下默认 remote 打 /api/interview/stream

    pnpm dev:backend  后端 3030，需与代理目标一致

    frontend/.env 中 VITE_INTERVIEW_SSE_URL  显式指定流地址时强制走该 URL 的 remote 模式

    frontend/.env 中 VITE_INTERVIEW_BACKEND_ORIGIN  改代理目标，默认 localhost:3030

原则（协作边界）

  前端禁止直连 C++，必须走 /api
  POST /api/interview/stream 与事件 chunk / done / error 为冻结契约，单方不可改
  类型权威：backend/src/types/interview.ts、practice-pool.ts


二、前端技术栈与分层思想

技术栈

  Vue 3（框架）+ script setup + TypeScript
  Vite 8（构建与 dev server）
  Pinia（状态仓库，全局轻量使用）
  Naive UI（组件库）
  UnoCSS（原子化样式）+ Sass（全局与 Markdown 样式）
  pnpm workspace（包管理）

分层（自上而下）

  views/           页面入口，路由挂载点
  components/      展示 UI，按「宇宙专用 / 领域共享 / 通用」三级放置
  composables/     组合函数：业务流程、场景状态、可复用逻辑（不放 DOM）
  services/        有 IO 的领域服务：SSE、API 封装、资料分块、本地存储
  types/           跨模块 TypeScript 类型
  utils/           纯函数，无副作用
  store/           Pinia 全局态（相对轻，大量业务在 composable + localStorage）
  router/          路由与守卫（legacy 工作台拦截、默认重定向宇宙页）
  config/          环境变量与产品开关（如 LEGACY_WORKBENCH_ENABLED）
  styles/          全局 SCSS、Markdown 主题、Naive 变量

默认产品入口

  / 重定向到 /showcase/mock-interview-space（宇宙模拟面试页）
  旧工作台 /workspace/* 代码保留，默认 frontend/src/config/product.ts 关闭访问

八、图中五项核心实现 — 代码落点

1. SSE 流式架构 + 安全 Markdown

   三层解耦

     传输层   frontend/src/services/sse/sse-client.ts
              按 \n\n 切帧、解析 event/data 行、ReadableStream 逐块 onChunk

     控制层   frontend/src/services/sse/interview-stream.ts
              resolveInterviewStreamMode：无 VITE_INTERVIEW_SSE_URL 则 mock，否则 remote POST /api/interview/stream
              mock 用 splitTextToReader 模拟打字机；remote 走 consumeSSEStream

     契约层   frontend/src/services/sse/sse-types.ts
              InterviewStreamEvent：start | chunk | done | error

     编排层   frontend/src/composables/interview/useInterviewStream.ts
              startInterviewStream -> parseInterviewStreamChunk -> messageQueue.enqueue

     解析层   frontend/src/services/message/interview-message-parser.ts

   安全 Markdown（DOMPurify 过滤 XSS）

     frontend/src/components/message/MarkdownMessage.vue
       renderMarkdownText + DOMPurify.sanitize 后 v-html
     frontend/src/components/report/ReportReviewMarkdown.vue
       报告场景同样消毒
     Markdown 渲染能力来源
       frontend/src/components/MarkdownPreview/plugins/markdown（markdown-it 等）

   展示组件链
     frontend/src/components/message/MessageBubble.vue -> MarkdownMessage.vue / 纯文本
     frontend/src/components/message/MessageList.vue 列表与滚动（含 requestAnimationFrame 滚动到底）


2. rAF 消息渲染队列（requestAnimationFrame）

   核心类（面试重点讲这个文件）
     frontend/src/services/message/interview-message-queue.ts
       InterviewMessageQueue：按 messageId 缓冲 delta，每帧 flush 合并为一次 DOM 更新，避免高频 token 卡顿

   调用链
     frontend/src/composables/interview/useInterviewStream.ts 在 chunk 事件里 enqueue
     注册 handler 更新 InterviewMessage.displayContent

   其他 rAF 用途（同属性能优化，非消息队列）
     frontend/src/components/MarkdownPreview/index.vue 打字机逐字展示
     frontend/src/composables/showcase/useMockInterviewSpaceScroll.ts 平滑滚动
     frontend/src/composables/showcase/useMockInterviewSpaceOrbit.ts、frontend/src/views/showcase/space-odyssey.vue 视觉动画

   持久化单一入口
     frontend/src/services/storage/workbench-storage.ts

       读写 workbenchContext；sessionConfigKey 用于匹配可恢复会话

   宇宙页编排（业务真状态）

     frontend/src/composables/showcase/useMockInterviewFlow.ts

     PersistedInterviewSession.sessionConfigKey 与 mock 状态快照联动，避免不同入口恢复错会话


5. Node 网关 + Provider 工厂（前后端协作，前端只消费契约）

   前端统一 API 入口
     frontend/vite.config.ts
       dev proxy：/api -> VITE_INTERVIEW_BACKEND_ORIGIN（默认 http://localhost:3030）

   前端流式入口（不直连 C++）
     frontend/src/services/sse/interview-stream.ts -> POST /api/interview/stream

   后端工厂（面试可对照说明，代码在 backend/）
     backend/src/services/interview-provider-factory.ts
       createInterviewProvider：remote | cpp | mock
     backend/src/providers/
       mock-provider.ts、remote-llm-provider.ts、cpp-engine-provider.ts
     backend/src/controllers/interview-controller.ts
     backend/src/utils/sse.ts
     backend/src/types/interview.ts  契约权威

   并行开发含义
     前端、Node、C++ 共用同一 SSE 事件形态；切换 PROVIDER 环境变量即可换引擎，无需改前端 path


九、面试口述建议（简短）

  架构：Monorepo + Vite 代理 + Node BFF + 可插拔 Provider，前端三层 SSE + rAF 队列 + DOMPurify。

  亮点 1：传输/控制/解析/展示分层，mock 与 remote 同一套事件类型。

  亮点 2：InterviewMessageQueue 把 LLM 高频 chunk 合并到每帧一次 UI 更新。

  亮点 3：资料按标题分块 <=1000 字，chunkId 贯穿出题、答题引用、报告溯源。

  亮点 4：entryMode + sessionConfig 双层，workbenchContext 持久化，多入口可恢复 mock 链路。

  亮点 5：契约冻结在 backend/src/types + frontend/src/services/sse/sse-types.ts，C++ 只实现 Provider 不破坏 API。

  默认演示路径：pnpm dev + pnpm dev:backend，打开 mock-interview-space，走资料导入 -> 分块出题 -> 模拟面试 SSE。
