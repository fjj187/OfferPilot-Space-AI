# Git 提交规范

## 1. 目的

为了让项目提交记录更清晰、后续更容易回看，这个项目统一使用规范化的 Git 提交信息。

这份规范希望解决三个问题：

- 看提交记录时，能快速判断这次改动属于什么类型
- 看提交记录时，能快速知道这次主要改了什么
- 后面写开发过程、复盘文档、项目经历时，可以直接从提交记录里提炼内容

---

## 2. 基本原则

后续提交统一遵循下面几条原则：

1. 提交信息必须带类型前缀
2. 提交标题必须使用中文说明
3. 标题要直接描述“这次主要改了什么”
4. 一次提交尽量只做一类事
5. 如果改动较多，可以使用“标题 + 正文”的多行提交格式

---

## 3. 基本格式

最基础的格式如下：

```bash
<type>: <中文标题>
```

例如：

```bash
feat: 新增总览页主题切换与卡片跳转交互
fix: 修复资料页底部内容被遮挡的问题
docs: 补充资料页参数承接与交互深化开发文档
```

如果这次改动内容比较多，不建议只写一句话，可以用下面这种更完整的格式：

```bash
<type>: <中文标题>

1. 本次主要改动点一
2. 本次主要改动点二
3. 本次主要改动点三
```

例如：

```bash
feat: 新增总览页真实交互与页面跳转能力

1. 新增主题标签切换联动
2. 新增资料卡与右侧引导卡点击跳转
3. 新增总览页到资料页和刷题页的参数传递
```

---

## 4. 提交类型说明

### `feat`

用于新增功能。

适用场景：

- 新增页面功能
- 新增组件交互
- 新增页面联动
- 新增业务流程入口

示例：

```bash
feat: 新增资料页文件导入与预览能力
feat: 新增总览页右侧AI引导卡点击跳转逻辑
```

### `fix`

用于修复问题。

适用场景：

- 修复布局问题
- 修复滚动问题
- 修复交互异常
- 修复参数传递错误

示例：

```bash
fix: 修复总览页右侧卡片掉到底部的问题
fix: 修复资料页滚动区域高度异常问题
```

### `docs`

用于文档新增或文档更新。

适用场景：

- 新增开发文档
- 补充使用说明
- 更新流程文档
- 补充设计文档

示例：

```bash
docs: 新增Git提交规范文档
docs: 完善总览页中间区与右侧区真实交互开发流程
```

### `refactor`

用于代码重构，但不新增功能，也不专门修复 bug。

适用场景：

- 组件拆分
- 页面结构重组
- 数据结构整理
- 提取公共容器或公共逻辑

示例：

```bash
refactor: 拆分工作台总览页组件结构
refactor: 抽离统一内容容器组件WorkbenchContentShell
```

### `style`

用于样式调整，不涉及功能逻辑变化。

适用场景：

- 调整布局
- 调整颜色
- 调整卡片间距
- 调整容器宽高

示例：

```bash
style: 调整总览页右侧引导卡宽度与留白
style: 统一工作台页面外层内容容器样式
```

### `chore`

用于杂项修改，一般是不直接影响核心业务功能的改动。

适用场景：

- 更新配置
- 整理目录
- 修改脚本
- 更新忽略文件

示例：

```bash
chore: 更新gitignore并忽略本地文档目录
chore: 整理项目目录与页面文件命名
```

### `perf`

用于性能优化。

示例：

```bash
perf: 优化资料页筛选计算逻辑
```

### `test`

用于测试相关改动。

示例：

```bash
test: 补充总览页交互逻辑测试用例
```

## 5. 中文标题怎么写

标题不要写得太空，也不要写成流水账。

推荐写法：

- 写页面名
- 写功能点
- 写改动结果

例如：

```bash
feat: 新增总览页主题筛选与资料卡联动逻辑
fix: 修复资料页切换筛选后默认选中文档失效问题
docs: 补充资料页参数承接与交互深化开发文档
refactor: 重构总览页数据结构并补充卡片动作字段
```

不推荐这样写：

```bash
feat: 改一下页面
fix: 修复bug
docs: 更新文档
refactor: 优化代码
```

因为这种写法后面几乎看不出你到底做了什么。

---

## 6. 什么时候不只写一句话

如果满足下面任意一种情况，建议不要只写单行标题，而是补正文：

1. 一个提交同时改了多个页面
2. 一个提交既有组件改动，又有页面逻辑改动
3. 一个提交改了数据结构、页面逻辑和跳转关系
4. 一个提交本身就代表一个比较完整的小阶段

推荐格式：

```bash
feat: 新增工作台页面结构并完善总览页与资料页交互

1. 新增 workbench 相关页面与组件结构
2. 完成总览页主题联动与卡片跳转逻辑
3. 完成资料页基础参数承接能力
```

这种写法的好处是：

- 标题保持简洁
- 正文可以补充更多上下文
- 后面回看时比单行标题更有信息量

---

## 7. 正文怎么写

如果使用多行提交格式，正文建议写“本次主要改动点”，不要写成太长的散文。

推荐写法一：编号列表

```bash
feat: 新增资料页参数承接与文档预览联动

1. 支持根据 topic 和 docType 初始化筛选状态
2. 支持根据 source 自动定位默认文档
3. 优化列表区与预览区的联动体验
```

推荐写法二：短句列表

```bash
fix: 修复工作台页面滚动与双栏布局异常

- 修复总览页右侧卡片掉到底部的问题
- 修复统一容器导致页面无法滚动的问题
- 调整资料页底部留白与滚动区域高度
```

不推荐正文写法：

- 一大段没有结构的长句
- 写得太细，像把 diff 又复述了一遍
- 把无关内容全堆在一个提交正文里

---

## 8. 一次提交尽量只做一类事

推荐：

- 一个提交只做一个主要目标
- 功能开发和样式微调尽量分开
- 文档更新和业务代码尽量分开

例如：

推荐拆开：

```bash
feat: 新增总览页训练队列卡点击跳转逻辑
docs: 补充总览页真实交互开发文档
```

不推荐混在一起：

```bash
feat: 新增总览页训练队列跳转并更新文档顺便调整样式
```

## 9. 适合本项目的常用提交模板

你这个项目后面大概率会反复使用下面这些类型。

### 页面功能开发

```bash
feat: 新增总览页主题切换与卡片跳转交互
feat: 新增资料页导入文档后的自动预览能力
feat: 新增专项刷题页题量与难度参数承接逻辑
feat: 新增模拟面试页问题区与回答区基础结构
```

### 页面问题修复

```bash
fix: 修复总览页右侧卡片布局错位问题
fix: 修复资料页筛选后默认选中文档失效问题
fix: 修复统一内容容器导致页面无法滚动的问题
```

### 文档沉淀

```bash
docs: 新增Git提交规范文档
docs: 完善总览页中间区与右侧区真实交互开发流程
docs: 新增资料页参数承接与交互深化开发文档
```

### 样式微调

```bash
style: 调整工作台页面外层容器底部留白
style: 优化资料页列表区与预览区卡片间距
style: 统一总览页与资料页卡片背景色
```

### 结构调整

```bash
refactor: 拆分资料页列表与预览区域组件职责
refactor: 统一总览页与资料页页面容器结构
```

### 杂项维护

```bash
chore: 更新gitignore并忽略本地开发文档
chore: 补充路由结构并整理页面目录
```

## 10. 一个较大的功能应该怎么拆提交

如果你做的是一个比较大的功能，不建议最后只压成一个非常大的提交。

推荐拆法：

### 第一类：结构先行

```bash
refactor: 拆分工作台总览页组件结构
```

### 第二类：功能接入

```bash
feat: 新增总览页主题联动与卡片跳转逻辑
```

### 第三类：问题修复

```bash
fix: 修复工作台页面双栏布局与滚动异常
```

### 第四类：文档补充

```bash
docs: 补充总览页真实交互开发文档
```

这样拆的好处是：

- 提交记录更清楚
- 回滚更方便
- 后面写阶段总结也更好整理

## 11. 当前项目默认执行规则

后面这个项目的提交信息默认按下面规则写：

1. 必须以 `feat`、`fix`、`docs`、`refactor`、`style`、`chore`、`perf`、`test` 之一开头
2. 冒号后面必须使用中文说明
3. 中文说明必须直接描述本次主要改动
4. 改动较大时，允许使用多行提交正文
5. 正文优先写“本次主要改动点”，不要写过长废话
6. 尽量不要把功能、样式、文档、修复混进同一条提交

---

## 12. 以后让我帮你写提交信息时的默认策略

如果你后面让我直接生成提交信息，我会默认这样处理：

### 小改动

直接给你一条单行提交：

```bash
fix: 修复资料页默认选中文档异常问题
```

### 中等改动

给你一条更完整的单行提交：

```bash
feat: 新增资料页参数承接与文档预览联动
```

### 较大改动

给你一条“标题 + 正文”的多行提交：

```bash
feat: 新增工作台页面结构并完善总览页与资料页交互

1. 新增 workbench 相关页面与组件
2. 完成总览页主题联动与卡片跳转
3. 完成资料页基础参数承接能力
```

## 13. 当前建议

以后这个项目里的提交，不要把“提交信息”只当成一句备注，而要把它当成“开发记录的标题”。

也就是说：

- 小改动可以一句话
- 中等改动一句话也可以，但要写具体
- 阶段性改动建议使用“标题 + 正文”

如果你后面继续让我帮你提交，我会默认优先给你更完整的版本，而不是只写一行很短的标题。
