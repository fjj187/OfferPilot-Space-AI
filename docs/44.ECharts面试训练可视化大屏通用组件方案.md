# 44.ECharts 面试训练可视化大屏通用组件方案

## 一、目标

本文档用于规划一个基于 `ECharts`（可视化图表库）的面试训练数据大屏模块，核心方向是“AI 面试训练数据驾驶舱”。

该模块不是后台管理报表，也不是单一页面的业务组件，而是一个可被多个页面引入的通用展示组件。后续可以放在 `mock-interview-space`（宇宙模拟面试页）、后台数据看板、报告复盘页或单独的大屏路由中使用。

目标有四个：

1. 展示用户面试训练过程中的关键数据，包括训练次数、评分趋势、能力维度、题型分布和岗位分布。
2. 组件保持低耦合，不直接依赖 `Pinia`（状态管理库）、路由、后端接口或具体业务页面。
3. 图表内部封装 `ECharts`（可视化图表库）初始化、销毁、尺寸自适应和主题配置，对外只暴露稳定的 `props`（组件入参）。
4. 第一阶段优先使用静态示例数据或上层传入数据，不修改 `POST /api/interview/stream`（模拟面试流式接口）、`SSE`（服务端发送事件）和 `backend`（后端目录）。

## 二、推荐数据方向

推荐方向：**AI 面试训练数据驾驶舱**。

原因如下：

1. 与项目主题一致：项目核心是模拟面试、资料训练、专项刷题和复盘报告，大屏数据天然来自这些用户行为。
2. 展示价值明确：用户能直观看到自己练了多少、哪些能力变强、哪些题型薄弱。
3. 可复用性强：同一套数据结构既能用于个人复盘，也能用于后台运营看板。
4. 适合 `ECharts`（可视化图表库）：雷达图、折线图、环形图、柱状图和排行图都能覆盖。

不建议第一阶段做泛化的“系统运营大屏”，例如用户访问量、服务器状态、接口耗时等。它们虽然像大屏，但和当前产品主线关系弱，容易变成装饰性页面。

## 三、第一阶段展示模块

第一阶段建议控制在 5 个模块内，避免为了“大屏感”堆过多图表。

| 模块 | 展示内容 | 推荐图表 |
| --- | --- | --- |
| 核心指标区 | 累计面试、训练时长、平均评分、连续训练天数 | 指标卡 |
| 能力雷达区 | 表达能力、逻辑结构、专业知识、岗位匹配、抗压能力 | `Radar`（雷达图） |
| 评分趋势区 | 最近 7 到 14 次训练评分变化 | `Line`（折线图） |
| 题型分布区 | 技术题、项目题、行为题、开放题、压力题占比 | `Pie`（饼图）或 `Doughnut`（环形图） |
| 岗位训练区 | 前端、后端、算法、产品、运营等训练占比 | `Bar`（柱状图） |

可选氛围模块：

1. 实时训练动态：展示“完成一场模拟面试”“生成一组专项题”等动态。
2. 薄弱项排行：展示当前最需要补强的能力标签。
3. 训练热力日历：展示最近一段时间的训练活跃度。

可选模块不建议第一阶段全部实现，应等核心图表稳定后再补。

## 四、组件定位

推荐组件名：

`InterviewAnalyticsDashboard.vue`（面试训练数据大屏组件）

推荐目录：

`frontend/src/components/analytics/InterviewAnalyticsDashboard.vue`（前端通用分析组件目录）

如果后续需要拆分，可以继续放在同级目录：

| 文件 | 职责 |
| --- | --- |
| `InterviewAnalyticsDashboard.vue`（面试训练数据大屏组件） | 负责整体布局、接收数据、组装子图表 |
| `AnalyticsChartPanel.vue`（分析图表面板组件） | 负责统一图表容器、标题、空状态和加载状态 |
| `BaseECharts.vue`（基础图表组件） | 负责 `ECharts`（可视化图表库）实例生命周期和 `option`（图表配置项）渲染 |
| `interviewAnalyticsDemoData.ts`（面试分析示例数据文件） | 提供演示数据，方便页面未接接口时预览 |
| `interview-analytics.ts`（面试分析类型文件） | 定义组件入参和图表数据结构 |

第一阶段可以只新增一个完整组件和一个类型文件；当图表数量继续增加时，再抽出 `BaseECharts.vue`（基础图表组件）。

## 五、低耦合设计原则

### 5.1 只接收数据，不获取数据

`InterviewAnalyticsDashboard.vue`（面试训练数据大屏组件）只通过 `props`（组件入参）接收数据，不在组件内部调用接口。

错误方向：

1. 组件内部直接请求后端接口。
2. 组件内部直接读取 `Pinia`（状态管理库）。
3. 组件内部直接依赖当前路由参数。
4. 组件内部判断自己处于宇宙页、后台页还是报告页。

正确方向：

1. 页面或容器组件负责取数。
2. 页面或容器组件把数据整理为稳定结构。
3. 大屏组件只负责展示。

这样后续任何页面都可以直接引入该组件。

### 5.2 数据结构保持语义化

组件入参不应直接暴露 `ECharts option`（图表配置项），否则调用方需要理解图表实现细节，耦合会变高。

推荐让调用方传业务语义数据：

```ts
interface InterviewAnalyticsDashboardData {
  metrics: InterviewMetricCard[]
  abilityRadar: InterviewAbilityMetric[]
  scoreTrend: InterviewScorePoint[]
  questionTypeDistribution: InterviewDistributionItem[]
  jobPracticeDistribution: InterviewDistributionItem[]
  activityFeed?: InterviewActivityItem[]
}
```

由组件内部把这些数据转换成 `ECharts option`（图表配置项）。

这样做的好处：

1. 调用方只关心“面试训练数据”，不关心图表库细节。
2. 后续从 `ECharts`（可视化图表库）换成其他图表方案时，外部调用不需要大改。
3. 同一份数据可以切换不同主题或不同图表样式。

### 5.3 事件只表达意图

第一阶段建议只保留少量事件：

| 事件 | 含义 |
| --- | --- |
| `metric-click`（指标点击事件） | 用户点击某个核心指标 |
| `chart-click`（图表点击事件） | 用户点击某个图表数据项 |
| `refresh`（刷新事件） | 用户触发刷新 |

组件不直接决定点击后跳转到哪里，也不直接修改上层业务状态。页面接到事件后再决定是否跳转、筛选或打开详情。

## 六、推荐数据结构

### 6.1 核心指标

```ts
interface InterviewMetricCard {
  key: string
  label: string
  value: string
  trendText?: string
  tone?: 'blue' | 'green' | 'orange' | 'red'
}
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `key`（唯一键） | 用于列表渲染和点击识别 |
| `label`（指标名称） | 例如“累计面试” |
| `value`（指标值） | 例如“28 场” |
| `trendText`（趋势文案） | 例如“较上周 +18%” |
| `tone`（色调） | 控制指标卡视觉强调 |

### 6.2 能力雷达

```ts
interface InterviewAbilityMetric {
  name: string
  value: number
  max: number
}
```

建议维度：

1. 表达能力
2. 逻辑结构
3. 专业知识
4. 岗位匹配
5. 抗压能力
6. 复盘吸收

### 6.3 评分趋势

```ts
interface InterviewScorePoint {
  label: string
  score: number
  completedAt?: string
}
```

`label`（标签）可以是“第 1 次”“周一”“06-24”等，具体格式由上层整理。

### 6.4 分布数据

```ts
interface InterviewDistributionItem {
  name: string
  value: number
}
```

题型分布和岗位训练分布都可以复用这一结构。

### 6.5 实时动态

```ts
interface InterviewActivityItem {
  id: string
  title: string
  description?: string
  timeText: string
  type?: 'mock' | 'practice' | 'report' | 'library'
}
```

实时动态只做展示，不负责生成或持久化。

## 七、页面布局建议

大屏布局建议采用“中间主视觉 + 两侧分析面板”的结构。

桌面端结构：

| 区域 | 内容 |
| --- | --- |
| 顶部 | 标题、时间范围、刷新按钮 |
| 左侧 | 核心指标、题型分布 |
| 中间 | 能力雷达、评分趋势 |
| 右侧 | 岗位训练分布、实时动态 |
| 底部 | 薄弱项排行或训练热力概览 |

移动端结构：

1. 单列堆叠。
2. 指标卡优先展示。
3. 图表高度固定，避免内容加载后布局跳动。
4. 实时动态放在最后，避免挤压核心数据。

## 八、视觉风格建议

该模块面向“面试训练分析”，不建议做过度科幻的纯装饰大屏。视觉应该保持专业、清晰、可复盘。

推荐风格：

1. 深色背景，适合大屏氛围。
2. 蓝绿作为主数据色，橙色用于风险或弱项提醒。
3. 图表线条和网格降低透明度，避免压过数据。
4. 指标卡保持清晰数字层级，不堆过多说明文案。
5. 所有图表必须有空状态和加载状态。

不建议：

1. 大面积单一紫色渐变。
2. 过多发光边框。
3. 使用卡片套卡片。
4. 图表标题和数字互相遮挡。
5. 为了氛围牺牲可读性。

## 九、实现步骤建议

### 9.1 第一步：补依赖

在真正开发前确认是否安装 `echarts`（可视化图表库）。

建议依赖：

```bash
pnpm --filter frontend add echarts
```

如果不使用 `vue-echarts`（Vue 图表封装库），可以直接使用 `echarts/core`（图表核心包）做按需引入，减少打包体积。

### 9.2 第二步：定义类型

新增 `frontend/src/types/interview-analytics.ts`（面试分析类型文件），先稳定外部入参结构。

类型先服务当前 5 个图表，不提前设计复杂筛选、权限、后端 DTO（数据传输对象）映射。

### 9.3 第三步：实现基础图表生命周期

如果抽出 `BaseECharts.vue`（基础图表组件），它只负责：

1. 初始化 `ECharts`（可视化图表库）实例。
2. 监听容器尺寸变化。
3. 接收 `option`（图表配置项）并更新图表。
4. 在组件卸载时销毁实例。
5. 透传点击事件。

它不关心面试业务。

### 9.4 第四步：实现大屏组件

`InterviewAnalyticsDashboard.vue`（面试训练数据大屏组件）负责：

1. 接收 `data`（数据入参）。
2. 根据业务语义数据生成各图表 `option`（图表配置项）。
3. 渲染整体布局。
4. 展示加载态、空状态和错误态。
5. 对外抛出点击和刷新事件。

### 9.5 第五步：接入页面

页面接入时只做三件事：

1. 获取或组装数据。
2. 把数据传给 `InterviewAnalyticsDashboard.vue`（面试训练数据大屏组件）。
3. 监听组件事件并决定后续行为。

示例接入方式：

```vue
<InterviewAnalyticsDashboard
  :data="dashboardData"
  :loading="loading"
  @refresh="reloadDashboard"
/>
```

## 十、边界约束

本方案第一阶段不做以下事情：

1. 不修改 `backend`（后端目录）。
2. 不新增或修改任何真实接口。
3. 不修改 `POST /api/interview/stream`（模拟面试流式接口）。
4. 不修改 `SSE`（服务端发送事件）结构。
5. 不把大屏组件绑定到某一个页面。
6. 不把图表数据写入持久化状态。
7. 不为了示例效果伪造多个字段兼容真实接口。

如果后续要接真实数据，应由页面或专门的 `service`（服务层）完成接口适配，再传给通用组件。

## 十一、验收标准

第一阶段完成后，至少满足以下标准：

1. 任意页面都能通过传入 `InterviewAnalyticsDashboardData`（面试分析大屏数据）渲染大屏。
2. 组件不直接 import（导入）任何业务 `store`（状态仓库）。
3. 组件不直接 import（导入）任何接口请求函数。
4. 图表在容器尺寸变化时能自适应。
5. 空数据、加载中和错误状态都有明确展示。
6. 构建通过，且不会因为缺少真实后端数据而报错。
7. 新增代码不影响宇宙页现有状态链路、路由参数和报告生成流程。

## 十二、后续扩展方向

后续可以按优先级扩展：

1. 接入报告数据，展示真实能力评分和弱项标签。
2. 接入专项刷题数据，展示题型正确率和薄弱题型。
3. 接入训练资料数据，展示资料覆盖度和资料驱动训练次数。
4. 增加时间范围筛选，例如近 7 天、近 30 天、全部。
5. 增加导出能力，把当前大屏导出为图片。
6. 增加后台视角，把个人数据聚合为运营看板。

扩展时仍应保持同一原则：数据获取留在上层，图表展示留在通用组件，接口契约变更必须单独评审。
