# 41.SSE 流式中断后重新生成本轮反馈方案

## 一、目标

本方案用于收口 `SSE`（服务端发送事件）流式反馈在网络断开、页面刷新、浏览器关闭或生成过程中异常中断后的用户恢复体验。

本次目标不是“从断点继续写”，而是：

> 检测到上一轮 `assistant`（助手消息）流式反馈未完成时，前端展示“重新生成本轮反馈”按钮；用户点击后，保留原用户回答，移除或替换未完成的 `assistant`（助手消息），重新发起一次完整生成。

也就是说，本方案做的是“断点后重新生成”，不是“断点续写”。

## 二、为什么不做断点续写

当前项目已经具备 `heartbeat`（心跳事件）、`checkpoint`（断点快照事件）、`idempotentKey`（幂等键）和 `checkpoint API`（断点快照接口）的预埋与部分落地，但如果要真正做到“从断点继续生成”，还需要三方同时保证：

| 能力 | 风险 |
| --- | --- |
| `LLM`（大语言模型）上下文可恢复 | 模型本身通常不支持从半句话位置继续同一次生成 |
| `idempotentKey`（幂等键）严格去重 | 复用旧键可能返回旧快照，不复用旧键又会变成新生成 |
| `C++`（后端引擎服务）与 `Node`（后端网关）一致实现 | 任何一端语义不同都会导致恢复行为不稳定 |
| 前端消息队列可无缝拼接 | 半截内容继续拼接容易出现重复、断句、格式损坏 |

因此，本阶段选择更稳定、更符合用户预期的方案：中断后重新生成整条反馈。

## 四、用户场景

### 4.1 网络断开

用户提交回答后，`SSE`（服务端发送事件）正在生成反馈；如果网络断开，当前流最终进入 `error`（错误状态）或连接异常。

期望结果：

1. 未完成 `assistant`（助手消息）标记为失败或中断。
2. 页面展示“重新生成本轮反馈”按钮。
3. 点击后重新发起本轮反馈生成。

### 4.2 页面刷新

用户提交回答后，反馈尚未完成，用户刷新页面。

期望结果：

1. 页面重新进入同一 `sessionId`（会话标识）后检测到 `pendingRecovery`（待恢复流上下文）。
2. 不自动请求 `checkpoint API`（断点快照接口）补渲染旧内容。
3. 展示提示：“检测到上一轮反馈未完成，可重新生成本轮反馈。”
4. 点击后重新发起完整生成。

### 4.3 用户主动停止

用户点击停止生成。

期望结果：

1. 当前 `assistant`（助手消息）标记为 `aborted`（已中止）。
2. 展示“重新生成本轮反馈”按钮。
3. 点击后重新生成整条反馈。

### 4.4 服务端返回错误

后端返回 `error`（错误事件）时，前端展示错误文案和重试入口。

期望结果：

1. 错误文案仍由 `interview-stream-error.ts`（流错误文案文件）统一生成。
2. 按钮文案可以从“重试本轮反馈”统一为“重新生成本轮反馈”。
3. 点击后重新发起完整生成。

## 五、核心设计

### 5.1 统一按钮语义

建议将三种可重试原因统一到“重新生成本轮反馈”：

| 原因 | 当前文案 | 建议文案 | 处理方式 |
| --- | --- | --- | --- |
| `aborted`（主动中止） | 重新生成本轮反馈 | 重新生成本轮反馈 | 移除旧 `assistant`（助手消息）后重发 |
| `error`（错误失败） | 重试本轮反馈 | 重新生成本轮反馈 | 移除旧 `assistant`（助手消息）后重发 |
| `interrupted`（本地中断） | 恢复本轮反馈 | 重新生成本轮反馈 | 不查 `checkpoint`（断点快照），直接重发 |

这样用户不会误解为“继续半截内容”，也能减少前端分支。

### 5.2 重新生成的消息处理

推荐默认策略：

1. 保留原 `user`（用户消息）。
2. 删除旧的未完成 `assistant`（助手消息）。
3. 使用原始 `params`（请求参数）重新调用 `startStream`（启动流函数）。
4. 生成新的 `messageId`（消息标识）和新的 `idempotentKey`（幂等键）。
5. 新反馈作为新的 `assistant`（助手消息）流式写入。

不建议保留旧半截反馈，原因是：

- 半截反馈可能误导用户。
- 新旧两条反馈同时存在会造成“本轮到底以哪条为准”的歧义。
- 报告生成和弱项分析更容易误读重复反馈。

如果后续确实想保留旧内容，可作为二期能力增加“查看上次中断内容”折叠区，但第一阶段不建议做。

### 5.3 `idempotentKey`（幂等键）策略

重新生成时不要复用旧 `idempotentKey`（幂等键）。

原因：

- 旧 `idempotentKey`（幂等键）对应旧的断点快照。
- 复用旧键可能让后端继续返回旧 `checkpoint`（断点快照）或覆盖旧快照。
- 重新生成语义应该是一次新的生成任务。

建议处理：

```ts
const regenerateParams = cloneStreamParams(nextRetry.params)
delete regenerateParams.idempotentKey
return startStream(regenerateParams)
```

`startInterviewStream`（启动面试流函数）会生成新的 `messageId`（消息标识），后端可继续按 `messageId`（消息标识）作为默认 `idempotentKey`（幂等键）。

### 5.4 `checkpoint API`（断点快照接口）处理

本阶段不删除 `checkpoint API`（断点快照接口）相关代码，但 `retryStream`（重试流函数）在 `interrupted`（中断原因）分支不再主动调用：

```ts
restoreInterviewStreamFromCheckpoint（从断点快照恢复流内容函数）
```

原因：

- 当前用户目标是重新生成，不是补渲染旧内容。
- 保留 `checkpoint API`（断点快照接口）可以继续用于调试、未来恢复预览、后端状态排查。
- 减少后端契约变化，不影响 `Node`（后端网关）和 `C++`（后端引擎服务）并行开发。

## 六、建议改造点

### 6.1 `useInterviewStream.ts`（面试流组合函数）

位置：

```text
frontend/src/composables/interview/useInterviewStream.ts
```

建议改动：

1. `retryActionLabel`（重试按钮文案）统一返回“重新生成本轮反馈”。
2. `retryStream`（重试流函数）删除 `interrupted`（中断原因）下的 `checkpoint API`（断点快照接口）恢复分支。
3. `retryStream`（重试流函数）对 `aborted/error/interrupted`（主动中止 / 错误失败 / 本地中断）统一走重新生成。
4. 重新生成前清理旧 `assistant`（助手消息）。
5. 重新生成前清理 `pendingRecovery`（待恢复流上下文）。
6. 重新生成时删除旧 `idempotentKey`（幂等键），让本次生成创建新任务。

建议伪代码：

```ts
const retryStream = async () => {
  const nextRetry = retryableState.value
  if (!nextRetry || isStreaming.value) return ''

  removeMessage(nextRetry.messageId)
  streamError.value = ''
  clearPendingInterviewStreamRecovery()
  pendingRecovery.value = null
  retryableState.value = null

  const regenerateParams = cloneStreamParams(nextRetry.params)
  delete regenerateParams.idempotentKey

  return startStream(regenerateParams) || ''
}
```

注意：如果 `removeMessage`（移除消息函数）没有命中旧消息，也不应该阻断重新生成，因为刷新后旧消息可能已经不在当前本地消息列表里。

### 6.2 `interview-stream-recovery.ts`（流恢复预埋文件）

位置：

```text
frontend/src/services/sse/interview-stream-recovery.ts
```

建议改动：

1. `buildInterviewStreamRecoveryHint`（构建流恢复提示函数）不再提示“补渲染上次已生成内容”。
2. 文案统一为“检测到上一轮反馈未完成，可重新生成本轮反馈。”
3. `canAutoRecoverInterviewStream`（判断是否可自动恢复流函数）可以暂时保留，但当前 UI（用户界面）不再使用“自动恢复”语义。

建议文案：

```text
检测到上一轮流式反馈未完成。你可以点击“重新生成本轮反馈”，系统会保留你的回答并重新生成一条完整反馈。
```

### 6.3 `SpaceMockScene.vue`（模拟面试场景组件）

位置：


当前已有 `retry`（重试事件）入口，优先不新增按钮结构。

建议只调整：

2. 按钮语义统一为重新生成，不再出现“恢复本轮反馈”。
3. 如果按钮旁边有提示文案，要避免“恢复已生成内容”“从断点继续”等说法。

## 七、状态流转

### 7.1 正常生成

```text
startStream（启动流函数）
  -> start（开始事件）
  -> streaming（生成中）
  -> done（完成事件）
  -> 清理 pendingRecovery（待恢复流上下文）
```

### 7.2 网络错误

```text
startStream（启动流函数）
  -> start（开始事件）
  -> streaming（生成中）
  -> error（错误事件）
  -> retryableState.reason = error（可重试状态原因为错误）
  -> 展示“重新生成本轮反馈”
  -> retryStream（重试流函数）
  -> 删除旧 assistant（助手消息）
  -> 新 startStream（启动流函数）
```

### 7.3 页面刷新中断

```text
startStream（启动流函数）
  -> start（开始事件）
  -> 写入 pendingRecovery（待恢复流上下文）
  -> 页面刷新
  -> setActiveSessionId（设置当前会话标识函数）
  -> 从 pendingRecovery（待恢复流上下文）恢复 retryableState（可重试状态）
  -> 展示“重新生成本轮反馈”
  -> retryStream（重试流函数）
  -> 不请求 checkpoint API（断点快照接口）
  -> 新 startStream（启动流函数）
```

### 7.4 用户主动停止

```text
stopStream（停止流函数）
  -> finalizeAbort（中止收尾函数）
  -> retryableState.reason = aborted（可重试状态原因为主动中止）
  -> 展示“重新生成本轮反馈”
  -> retryStream（重试流函数）
  -> 删除旧 assistant（助手消息）
  -> 新 startStream（启动流函数）
```

## 八、边界与不做事项

本阶段不做：

1. 不做自动续流。
2. 不做从 `checkpoint`（断点快照）位置继续生成。
3. 不新增后端接口。
4. 不修改 `POST /api/interview/stream`（模拟面试流式接口）路径。
5. 不修改 `SSE`（服务端发送事件）的 `chunk/done/error/heartbeat/checkpoint`（分片 / 完成 / 错误 / 心跳 / 断点快照）事件结构。
6. 不要求 `C++`（后端引擎服务）新增能力。
7. 不把旧半截 `assistant`（助手消息）纳入报告生成。

本阶段需要保证：

1. 用户回答不丢。
2. 未完成反馈不误当成完成反馈。
3. 重新生成使用新 `messageId`（消息标识）和新 `idempotentKey`（幂等键）。
4. 刷新后能看到重新生成入口。
5. 同一个中断任务不会重复显示多个重试入口。

## 十、面试口述版

如果面试官问“你的项目怎么处理流式生成中断”，可以这样说：

> 项目里已经有 `heartbeat`（心跳事件）、`checkpoint`（断点快照）和 `pendingRecovery`（待恢复流上下文），但我没有把用户体验设计成从半截内容继续生成。因为大模型流式输出本身很难保证从断点位置继续时不重复、不断句，也需要 `Node`（后端网关）和 `C++`（后端引擎服务）严格保证 `idempotentKey`（幂等键）语义。所以我会把中断后的主体验设计成“重新生成本轮反馈”：保留用户原回答，清理未完成的助手消息，重新发起一次完整的 `SSE`（服务端发送事件）请求，并生成新的 `messageId`（消息标识）和 `idempotentKey`（幂等键）。这样用户感知更稳定，前端状态也更容易保证一致。

如果继续被追问“那 `checkpoint`（断点快照）还要不要”，可以这样说：

> `checkpoint`（断点快照）仍然有价值，但本阶段不把它作为用户主流程。它可以用于调试、未来展示上次中断内容，或者后续三方对齐后做更高级的恢复能力。当前主流程选择重新生成，是为了避免半截拼接造成重复内容、格式损坏和报告分析误读。

## 十一、推荐落地顺序

1. 先改 `retryActionLabel`（重试按钮文案），统一为“重新生成本轮反馈”。
2. 再改 `buildInterviewStreamRecoveryHint`（构建流恢复提示函数），去掉“补渲染已生成内容”的描述。
3. 再改 `retryStream`（重试流函数），让 `interrupted`（中断原因）不再查询 `checkpoint API`（断点快照接口）。
4. 确认重新生成时删除旧 `idempotentKey`（幂等键）。
5. 最后补手动测试，覆盖网络断开、页面刷新、主动停止和服务端错误。
