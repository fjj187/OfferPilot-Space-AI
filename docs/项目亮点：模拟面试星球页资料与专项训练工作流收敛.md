# 项目亮点：模拟面试星球页资料与专项训练工作流收敛

## 背景

模拟面试星球页在接入资料库、专项刷题、模拟面试和报告续练之后，页面表面上已经能跑通主链路，但内部语义其实有些打架。

最典型的问题有几类：

- 用户在资料库选了资料后，既可以直接进模拟面试，也可能再去专项刷题页选专区、题型、题数和难度，但系统没有把“资料”和“专项配置”当成两层独立信息处理
- `practicePlan` 一边被当成专项训练配置，一边又被当成当前模式判断依据，导致普通 mock 和专项 mock 容易串状态
- 从专项刷题进入 mock 后，顶部胶囊信息不完整，缺少题型、专项专区、题数这类真正影响本轮训练的信息
- 回答区题号经常固定显示成 `1 / 1`，没有反映用户在专项页里选的 `5 题 / 10 题 / 15 题`
- 资料库页原先只有“去模拟面试”这一种出口，容易让“直接模拟”和“先去专项配置再模拟”两条链路混在一起

这些问题单独看像是几个小 bug，但本质上是工作流和状态模型没有彻底收拢，导致页面在入口语义、状态承接、会话恢复和 UI 展示上都不够稳定。

## 改了什么

核心改动集中在这几处：

- `src/types/workbench.ts`
- `src/composables/useWorkbenchPersistence.ts`
- `src/composables/showcase/useMockInterviewSpaceMockState.ts`
- `src/views/showcase/mock-interview-space.vue`
- `src/components/showcase/mock-interview-space/scenes/SpaceLibraryScene.vue`
- `src/components/showcase/mock-interview-space/SpaceContentPanel.vue`

### 拆出 mock 入口模式和当前会话配置

原先工作台上下文里主要只有：

- `activeDocument`
- `practicePlan`
- `sourcePage`

这不够表达“这轮 mock 到底是普通模拟，还是专项刷题面试；到底应该按什么题型、题数、难度出题”。

所以这次在 `workbench.ts` 里新增了：

- `mockEntryMode`
- `mockSessionConfig`

其中：

- `mockEntryMode` 明确区分当前进入 mock 的入口是 `direct` 还是 `practice`
- `mockSessionConfig` 专门记录本轮真正生效的训练配置，例如 `questionType`、`questionCount`、`difficulty`、`zone`、`activeDocumentId`

这样 mock 页后续展示和出题就不需要再临时从多个旧字段里猜。

### 把资料上下文和专项训练配置分开承接

这次明确把两层语义拆开了：

- 资料决定“围绕什么内容练”
- 专项配置决定“以什么方式练”

具体落地上：

- 从资料库直接进 mock 时，会写入 `mockEntryMode: 'direct'`
- 同时写入只包含资料上下文的 `mockSessionConfig`
- 进入前清空旧的 `practicePlan`

从专项刷题页进入 mock 时：

- 会写入 `mockEntryMode: 'practice'`
- 会把专区、题型、题数、难度完整写进 `mockSessionConfig`
- 会保留当前资料上下文，不再因为进入专项链路把资料丢掉

### 让 mock 页面按真实配置驱动

原先 mock 页很多行为还是隐式依赖 `practicePlan`，这会导致模式判断、出题逻辑和胶囊展示并不稳定。

这次改完后：

- mock 顶部胶囊会优先读取 `mockSessionConfig`
- 专项模式下会增加：
  - `专项专区`
  - `题型`
  - `题数`
- `难度` 也会优先显示当前专项配置的难度，而不是只看题库第一题的默认难度

这样用户从专项刷题页选了什么，进入 mock 后就能在胶囊区一眼看到。

### 让专项模式的题号和题目线程按题数生成

原先 mock 的问题线程更接近“主问题 + 追问”的结构，因此很容易退化成固定的 `1 / 1` 或只围绕一条 question thread 展开。

这次在 `useMockInterviewSpaceMockState.ts` 里补了一层基于配置的线程生成逻辑：

- 如果当前是 `practice` 模式，会根据 `mockSessionConfig.questionCount` 生成对应数量的题目线程
- 例如用户在专项页选了 `5 题`，那 mock 会直接生成 `5` 条线程
- 回答区题号会按真实线程数量显示成 `1 / 5`、`2 / 5`、`3 / 5`

这一步的核心不是单纯改展示文案，而是让题目进度和真实训练计划一致。

### 给会话匹配加上配置指纹

原先 session 查找主要按：

- `topic`
- `mode`
- `source`

这会有个问题：如果同一个主题、同一种来源下，用户分别做过不同题型、不同题数、不同难度的专项训练，旧 session 容易串进新链路。

所以这次给 session 增加了 `sessionConfigKey`，把这类关键信息组合成配置指纹，参与 session 查找与恢复。

这样：

- `Vue + 专项刷题 + 5题 + 场景题`
- `Vue + 专项刷题 + 10题 + 代码题`

就不会被错误地恢复成同一轮会话。

### 把资料库入口拆成两条明确动作

资料库原先只有“去模拟面试”这一种按钮文案，语义太笼统。

这次在资料库场景里直接拆成了：

- `直接模拟面试`
- `去专项刷题`

同时把事件透传链路补到 `SpaceContentPanel.vue` 和主页面装配层。

这一步的价值不只是改字，而是把产品层的工作流显式化，让用户知道：

- 我可以直接围绕资料开始普通 mock
- 也可以先带着资料去专项刷题页配置训练方式，再进入 mock

## 为什么这么改

这次没有继续在现有状态字段上打补丁，而是直接把“工作流语义”和“运行态配置”拆开。

原因很简单：之前的问题不是某一个条件分支写错了，而是系统长期把不同层次的信息混在一起了。

比如：

- 资料既像上下文，又像入口模式
- `practicePlan` 既像推荐配置，又像最终生效配置
- `sourcePage` 既像页面来源，又被拿来间接推断当前训练语义

这样的结构短期能跑，但一旦场景变多，就很容易出现：

- 页面显示的模式和真实模式不一致
- 用户手动选择被默认计划覆盖
- 题号和真实训练计划不一致
- 刷新恢复后会话串场

所以这次的核心思路不是“修一个 bug”，而是把工作流从“隐式推断”改成“显式建模”。

## 改完之后有什么变化

产品体验上，链路会更稳定：

- 用户从资料库直接进 mock，会明确进入普通模拟面试
- 用户从专项刷题页进 mock，会明确进入专项刷题模式
- 专项模式下，胶囊能展示真正生效的题型、专项专区和题数
- 回答区题号能跟上用户在专项页里选的真实题数

代码层面，这次相当于把 mock 的驱动中心从“零散状态推断”收回到了“单一配置对象”：

- 工作台上下文新增 `mockEntryMode` 和 `mockSessionConfig`
- session 恢复新增 `sessionConfigKey`
- mock 展示、模式判断、题目线程生成都开始围绕 `mockSessionConfig` 工作
- 资料库页动作从一个模糊入口拆成两条明确链路

构建层面，这轮修改后页面仍然可以正常通过：

- `vite build` 成功

## 面试中可以怎么讲

可以用这段话概括：

我在模拟面试星球页里做过一次工作流收敛。最初资料库、专项刷题和 mock 虽然能跳转，但资料上下文、专项训练配置和当前会话模式混在一起，导致专项参数带不全、题号固定为 `1/1`、刷新后容易串 session。我后来把系统拆成“资料上下文”和“专项训练配置”两层，并新增 `mockEntryMode` 与 `mockSessionConfig` 作为 mock 的真实驱动配置；同时给 session 增加配置指纹，避免不同专项计划串会话。这样之后，用户从资料库直接进入的是普通模拟，从专项页进入的是带题型、题数、难度和专区的专项 mock，题号也能按 `1/5`、`2/5` 这类真实计划推进。

## 面试官可能追问

### 1. 为什么不直接继续复用 `practicePlan`，还要新增 `mockSessionConfig`？

可以回答：

`practicePlan` 更像“专项训练计划”或“推荐配置”，但 mock 真正运行时需要的是一份稳定、可恢复、可驱动 UI 和 session 的最终配置。把这两者混在一起时，页面很容易在“默认建议”和“用户最后手动选择”之间来回跳。所以我单独加了 `mockSessionConfig`，让运行态只认这一份最终配置。

### 2. 为什么还要加 `mockEntryMode`，直接通过有没有 `practicePlan` 判断不行吗？

可以回答：

不够稳。因为 `practicePlan` 既可能来自报告推荐，也可能来自用户当前专项配置，甚至普通 mock 刷新后也可能残留旧值。用它直接判断模式会让普通 mock 被误判成专项 mock。`mockEntryMode` 的作用就是把“当前这轮是怎么进入 mock 的”这件事显式化。

### 3. 为什么 session 查找还要加配置指纹？

可以回答：

如果只按 `topic + mode + source` 查找，会把同主题下不同专项配置的会话混在一起。比如都是 Vue，但一个是 `5题概念题`，一个是 `10题场景题`，显然不应该恢复成同一轮。所以我把关键配置拼成 `sessionConfigKey`，让恢复逻辑能区分不同训练计划。

### 4. 题号为什么不能只改文案显示？

可以回答：

因为问题不只是显示。用户选了 `5题`，如果底层线程还是只生成一条，那你把文案硬改成 `1/5` 其实是假的，后续切题、恢复、报告统计都会出错。所以我先改线程生成逻辑，让系统真的有 `5` 条可推进的问题线程，再让 UI 显示 `1/5`。

### 5. 为什么资料库页要拆成两个按钮？

可以回答：

因为原先“去模拟面试”把两条不同用户意图混在一起了。有人是想直接围绕资料做普通 mock，有人是想先带着资料去配置专项训练。只有一个按钮时，产品语义含糊，代码里也更容易写成隐式分支。拆成两个动作之后，用户路径和状态模型都更清晰。

### 6. 这次重构有没有代价？

可以回答：

有，主要代价是状态字段和入口写入逻辑变得更明确，短期内改动会比“修一个展示 bug”更大一些。但换来的好处是后续继续接报告续练、题型扩展、刷新恢复和会话追踪时，系统不容易反复打架，长期维护成本更低。

## 为什么不用另一种方案

1. 不继续在旧字段上加判断

继续堆 `if (practicePlan)`、`if (sourcePage === 'practice')` 这类逻辑，短期也许能修掉某一次显示错误，但后面会越来越难维护。因为状态本身没有分层，补丁会越打越多。

2. 不只改文案和胶囊

如果只把顶部胶囊补上“题型”“题数”，但底层线程和 session 仍然按旧逻辑运行，那页面只是看起来像专项 mock，实际上不是。这样会留下更隐蔽的问题。

3. 不把资料和专项关系做成互斥

用户真实心智是“资料决定练什么，专项决定怎么练”。如果把两者做成二选一，用户一旦先选了资料再去专项页，就会遇到资料丢失或模式错乱的问题，不符合使用习惯。

4. 不把报告推荐当成强制配置

报告页给出的 `practicePlan` 适合作为默认建议，但不能覆盖用户后来在专项刷题页的手动选择。否则用户会感觉自己改了题型和题数，系统却还是按旧建议跑。

## 总结

这次改造的核心不是多加几个字段，而是把模拟面试星球页的工作流语义收紧了：资料是训练上下文，专项是训练配置，mock 只认当前这轮真正生效的会话配置。这样之后，资料库、专项刷题、模拟面试和报告续练之间的链路才真正稳定下来，也为后续继续扩展题型、恢复逻辑和报告闭环打下了更可靠的状态基础。
