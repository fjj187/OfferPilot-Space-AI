## 一、SSE（服务端发送事件）流式对话三层解耦

### 和简历表述的对应关系

你的简历写法：

> `SSE`（服务端发送事件）流式对话解耦：分离传输层、控制层与处理层，管理心跳、连接生命周期及多事件类型路由，实现高内聚低耦合。

这句话和项目实现是对应的。项目里没有直接把目录命名成“三层”，但代码职责已经按三层拆开了：底层负责拿流，中间层负责连接生命周期和事件路由，上层负责把事件变成页面消息、错误状态和恢复入口。

### 三层职责

| 分层 | 相关文件 | 负责什么 | 不负责什么 |
|---|---|---|---|
| 传输层 | `sse-client.ts`（SSE 客户端文件）、`fetch-stream-transport.ts`（fetch 流传输文件） | 发起 `POST + SSE`（提交请求加服务端事件流）、读取 `ReadableStream`（可读流）、把原始流拆成帧 | 不关心页面消息怎么展示 |
| 控制层 | `interview-stream-controller.ts`（面试流控制器文件）、`stream-connection-manager.ts`（流连接管理器文件）、`stream-event-router.ts`（流事件路由文件） | 管理 `AbortController`（中止控制器）、连接状态、活动时间、`chunk/done/error/heartbeat/checkpoint`（分片/完成/错误/心跳/断点）事件路由 | 不直接改页面响应式数据 |
| 处理层 | `useInterviewStream.ts`（面试流组合函数）、`interview-message-queue.ts`（面试消息队列文件）、`interview-stream-message-finalizer.ts`（流消息收尾处理器文件） | 把统一事件变成页面消息、错误提示、重试入口、断点恢复和本地持久化 | 不直接解析底层流协议 |

### 实现定位

- `frontend/src/services/sse/sse-client.ts`（SSE 客户端文件）：消费 `ReadableStream`（可读流），把服务端返回的文本流拆成 `SSE frame`（服务端事件帧）。
- `frontend/src/services/sse/fetch-stream-transport.ts`（fetch 流传输文件）：发起远程流请求，并把 `HTTP`（超文本传输协议）错误、网络错误、主动取消分类。
- `frontend/src/services/sse/interview-stream-controller.ts`（面试流控制器文件）：统一启动流、停止流、连接状态标记和错误收口，是三层里的控制中枢。
- `frontend/src/services/sse/stream-connection-manager.ts`（流连接管理器文件）：管理 `AbortController`（中止控制器）、连接状态、最后活动时间。
- `frontend/src/services/sse/stream-event-router.ts`（流事件路由文件）：把 `chunk`（分片）、`done`（完成）、`error`（错误）、`heartbeat`（心跳）、`checkpoint`（断点快照）路由为统一前端事件。
- `frontend/src/services/sse/stream-heartbeat.ts`（流心跳工具文件）：基于连接快照做 15 秒无活动检测。
- `frontend/src/composables/interview/useInterviewStream.ts`（面试流组合函数）：消费统一事件，处理消息渲染、错误提示、重试和断点恢复，不再直接持有底层传输细节。

### 核心实现代码

```ts
// interview-stream-controller.ts（面试流控制器文件）
export const startInterviewStream = (
  payload: InterviewStreamLaunchPayload,
  handlers: InterviewStreamHandlers
): InterviewStreamTask => {
  // 每次发起流都创建独立连接管理器，避免多轮面试流共享连接状态。
  const connectionManager = new StreamConnectionManager()

  // messageId（消息标识）贯穿一整轮流式回复，后续 chunk（分片）都靠它找到同一条消息。
  const messageId = createId()

  // open（打开连接）会创建 AbortController（中止控制器），用于“停止生成”。
  const signal = connectionManager.open()

  // mode（流模式）用于区分远程真实流和本地模拟流，面试时重点讲 remote（远程）链路即可。
  const mode = resolveInterviewStreamMode()
  let hasCompleted = false

  const emitActivity = () => {
    // 每次收到服务端活动都更新时间戳，后面 stream-heartbeat（流心跳工具）会基于它判断是否超时。
    connectionManager.markActive()
    handlers.onEvent({ type: 'activity', messageId, mode })
  }

  if (mode === 'remote') {
    consumeSSEStream({
      signal,
      createReader,

      onStart: () => {
        // 收到开始事件后标记连接活跃，并通知处理层创建一条 streaming（生成中）消息。
        emitActivity()
        handlers.onEvent({ type: 'start', messageId, mode })
      },

      onEvent: (frame) => {
        // 只要服务端有事件返回，就刷新最后活动时间，供心跳检测判断连接是否卡住。
        emitActivity()

        routeInterviewSSEFrame(frame, messageId).forEach((event) => {
          if (event.type === 'done') {
            // done（完成）是正常收尾，控制层负责把连接状态关掉。
            hasCompleted = true
            connectionManager.markClosed()
          }

          if (event.type === 'error') {
            // error（错误）统一在控制层补齐错误对象，处理层只负责展示和重试入口。
            connectionManager.markError()
            event.error = event.error || createFallbackStreamError('Stream request failed')
          }

          if (event.type === 'heartbeat' || event.type === 'checkpoint') {
            // heartbeat/checkpoint（心跳/断点）不直接改消息内容，但说明连接仍然有活动。
            connectionManager.markActive()
          }

          // 控制层只转发统一事件，具体怎么渲染交给 useInterviewStream（面试流组合函数）。
          handlers.onEvent(event)
        })
      }
    })
  }

  return {
    messageId,
    abort: () => {
      // 页面点击“停止生成”时只调用 abort（中止），不用碰底层 reader（读取器）。
      connectionManager.abort()
    },
    getSnapshot: () => connectionManager.getSnapshot()
  }
}
```

```ts
// stream-event-router.ts（流事件路由文件）
export const routeInterviewSSEFrame = (
  frame: SSEFrame,
  messageId: string
): InterviewStreamEvent[] => {
  if (frame.event === 'chunk' || frame.event === 'message') {
    const chunkPayload = parseFrameJson<{ content?: string }>(frame.data)

    return [{
      type: 'chunk',
      messageId,
      // chunk（分片）是实际文本增量，后续会进入消息队列做按帧渲染。
      // 这里兼容 JSON（结构化数据）和纯文本，避免单个异常帧拖垮整条流。
      chunk: chunkPayload?.content || frame.data
    }]
  }

  if (frame.event === 'checkpoint') {
    const checkpointPayload = parseFrameJson<{ idempotentKey?: string; sequence?: number }>(frame.data)

    return [{
      type: 'checkpoint',
      messageId,
      // checkpoint（断点快照）只传递恢复所需元信息，不在这里做恢复，避免路由层承担业务职责。
      idempotentKey: checkpointPayload?.idempotentKey,
      checkpointSequence: checkpointPayload?.sequence
    }]
  }

  return []
}
```

### 30 秒回答

答：我简历里写的 `SSE`（服务端发送事件）流式对话解耦，落到项目里就是把一轮 AI（人工智能）回复拆成三层：传输层只负责 `POST + SSE`（提交请求加服务端事件流）和 `ReadableStream`（可读流）读取；控制层负责 `AbortController`（中止控制器）、连接生命周期、心跳活跃度和多事件类型路由；处理层负责把 `chunk/done/error/heartbeat/checkpoint`（分片/完成/错误/心跳/断点）这些统一事件转成页面消息、错误提示和恢复入口。这样高内聚低耦合，后续加心跳、断点恢复或错误分类时，不需要改页面渲染和底层传输的混合代码。

### 场景题

#### 1. 为什么这里不用一个文件把流式逻辑全写完？

答：因为流式对话同时包含网络读取、连接状态管理、事件路由、消息渲染和异常处理。如果全塞进一个文件，后面加心跳、恢复、重试时很容易牵一发动全身。拆层以后，传输问题看传输层，状态问题看控制层，业务问题看处理层，排查和扩展都会更稳。

#### 2. 如果服务端后面新增 `heartbeat`（心跳事件）怎么办？

答：我这套结构下只要在 `stream-event-router.ts`（流事件路由文件）补一个事件映射，再让 `stream-connection-manager.ts`（流连接管理器文件）更新活动时间即可，`useInterviewStream.ts`（面试流组合函数）只消费统一后的事件，不需要重写底层流读取。

#### 3. 用户点击“停止生成”时，你是怎么做的？

答：控制层持有 `AbortController`（中止控制器），前端点击停止后调用 `abort`（中止）即可。这样不会直接碰渲染逻辑，传输层会被安全中断，控制层再统一把消息收尾成 `aborted`（已中止）状态。

#### 4. 15 秒没返回内容，前端怎么感知？

答：我没有直接判死连接，而是做了被动心跳检测。`stream-heartbeat.ts`（流心跳工具文件）每秒检查一次最近活动时间，超过 15 秒就把状态标成 `inactive`（不活跃），前端给出“连接可能暂时无响应”的提示，但不盲目自动重连。

#### 5. 为什么你这里不是 `WebSocket`（双向长连接协议）？

答：因为这个场景本质上是“前端提交一次问题，服务端持续单向吐结果”，更像 `POST + SSE`（提交请求加服务端事件流）而不是高频双向通信。`SSE`（服务端发送事件）接入更轻，和大模型单轮生成场景也更贴合，同时前端还能结合 `AbortController`（中止控制器）主动停止。

## 二、rAF（按帧调度）消息渲染队列

### 实现定位

- `frontend/src/services/message/interview-message-queue.ts`（面试消息队列文件）：按 `messageId`（消息标识）缓存增量文本。
- `frontend/src/composables/interview/useInterviewStream.ts`（面试流组合函数）：收到 `chunk`（分片）后不直接改页面，而是先 `enqueue`（入队）。

### 核心实现代码

```ts
// interview-message-queue.ts（面试消息队列文件）
export class InterviewMessageQueue {
  private buffers = new Map<string, string[]>()
  private handlers = new Map<string, (delta: string) => void>()
  private frameId: number | null = null

  enqueue(messageId: string, delta: string) {
    if (!delta) return

    const nextBuffer = this.buffers.get(messageId) || []
    nextBuffer.push(delta)
    this.buffers.set(messageId, nextBuffer)
    this.scheduleFlush()
  }

  private scheduleFlush() {
    if (this.frameId !== null) return

    this.frameId = requestAnimationFrame(() => {
      this.frameId = null
      // 同一帧内收到的文本统一合并，降低 Vue（前端框架）响应式更新频率。
      this.flushNow()
    })
  }

  private flushMessage(messageId: string) {
    const buffer = this.buffers.get(messageId)
    const handler = this.handlers.get(messageId)
    if (!buffer?.length || !handler) return

    this.buffers.delete(messageId)
    handler(buffer.join(''))
  }
}
```

### 30 秒回答

答：大模型流式返回时，`token`（文本分片）频率可能高于屏幕刷新频率。如果每来一个分片就直接改响应式数据，会频繁触发重排和重绘，页面会抖。我这里用 `requestAnimationFrame`（按帧调度 API）做了一层消息渲染队列，把同一帧收到的内容先缓冲，再一次性合并渲染，把更新频率压到接近 60 帧节奏，所以长文本输出会更平稳。

### 场景题

#### 1. 为什么不用 `debounce`（防抖）？

答：`debounce`（防抖）适合“停下来再触发”的输入场景，不适合流式输出。模型在持续吐字时，防抖会一直推迟更新，用户会感觉界面没反应。`requestAnimationFrame`（按帧调度 API）是跟浏览器渲染节奏对齐，更适合持续到达的流式分片。

#### 2. 为什么不用 `setTimeout(0)`（零延时定时器）？

答：`setTimeout`（定时器）只是在事件循环里延后，不保证和浏览器绘制同步。`requestAnimationFrame`（按帧调度 API）是在下一帧绘制前执行，天然更适合做 UI 合并更新。

#### 3. 如果一条回复特别长，会不会丢字？

答：不会。队列是按 `messageId`（消息标识）存数组缓冲，`flushNow`（立即刷新）时会把这一帧累计的分片 `join`（合并）后再一次性写入，所以是减少渲染次数，不是减少文本内容。

#### 4. 什么时候会强制立即刷新？

答：通常在 `done`（完成）、`error`（错误）或 `abort`（中止）收尾时，会先把队列里的剩余内容刷掉，再更新最终状态。这样可以避免最后几个字还在缓冲区里，导致用户看到消息缺尾巴。

#### 5. 这个优化本质解决了什么问题？

答：本质上解决的是“高频数据流”和“低频屏幕刷新”之间的节奏不匹配问题。模型可以一秒推很多次，但屏幕不需要一秒重绘那么多次，所以要先缓冲再按帧更新。

## 三、Markdown（标记语言）安全渲染与 XSS（跨站脚本攻击）防护

### 实现定位

- `frontend/src/components/message/MarkdownMessage.vue`（Markdown 消息组件）：先 `renderMarkdownText`（Markdown 渲染函数），再 `DOMPurify.sanitize`（HTML 净化函数），最后才 `v-html`（原始 HTML 渲染指令）。

### 核心实现代码

```vue
<!-- MarkdownMessage.vue（Markdown 消息组件） -->
<script lang="ts" setup>
import DOMPurify from 'dompurify'
import { renderMarkdownText } from '@/components/MarkdownPreview/plugins/markdown'

const props = defineProps<{ content: string }>()

const safeHtml = computed(() => {
  const html = renderMarkdownText(props.content || '')
  // 模型输出属于不可信内容，必须净化后再交给 v-html（原始 HTML 渲染指令）。
  return DOMPurify.sanitize(html)
})
</script>

<template>
  <div class="markdown-message" v-html="safeHtml" />
</template>
```

### 30 秒回答

答：流式消息里虽然要支持代码块、标题、列表这些常见 `Markdown`（标记语言）能力，但不能把模型返回的 HTML 直接塞进页面，否则会有 `XSS`（跨站脚本攻击）风险。所以我这里是先把 `Markdown`（标记语言）转成 HTML，再用 `DOMPurify`（HTML 净化库）做白名单净化，最后才渲染，这样在保留常见展示能力的同时，把脚本注入风险降下来。

### 场景题

#### 1. 为什么不能直接 `v-html`（原始 HTML 渲染指令）？

答：因为模型返回的内容本质上是不可信输入。直接 `v-html`（原始 HTML 渲染指令）会把恶意标签也当成页面结构执行，存在 `XSS`（跨站脚本攻击）风险。

#### 2. 你这套方案会不会把代码高亮也过滤掉？

答：不会一刀切。思路不是“全部转纯文本”，而是“先生成正常 HTML，再对白名单内容净化”。这样像代码块、段落、标题这些常见展示结构还能保留。

#### 3. 如果面试官问：为什么要在展示层做，而不是后端做？

答：后端做净化当然更好，但前端展示层仍然要兜底。因为最终执行 `v-html`（原始 HTML 渲染指令）的是浏览器，前端必须把“不可信内容直接进 DOM”这条链路堵住。

#### 4. 这和普通聊天产品有什么不同？

答：AI 面试场景里模型返回的内容更长，代码块和结构化文本更多，所以不能为了安全把所有格式都牺牲掉。我这里是在安全和可读性之间做平衡，而不是简单纯文本降级。

#### 5. 如果后面支持用户上传资料并参与生成，这个点重要吗？

答：更重要。因为风险来源不再只是模型输出，还包括用户资料内容。只要最终要经过 `v-html`（原始 HTML 渲染指令）进页面，就必须把净化放在稳定链路里。

## 四、对话消息持久化

### 实现定位

- `frontend/src/services/storage/interview-message-store.ts`（面试消息存储文件）：按 `sessionId`（会话标识）写入 `localStorage`（浏览器本地存储）。
- `frontend/src/composables/interview/useInterviewStream.ts`（面试流组合函数）：在消息追加、补丁更新、恢复时统一持久化。

### 核心实现代码

```ts
// interview-message-store.ts（面试消息存储文件）
const INTERVIEW_MESSAGE_STORAGE_PREFIX = 'offerpilot.interview.messages.'

const resolveStorageKey = (sessionId: string) => (
  `${ INTERVIEW_MESSAGE_STORAGE_PREFIX }${ sessionId }`
)

const normalizeMessage = (message: InterviewMessage): InterviewMessage => ({
  ...message,
  // 刷新后原流连接已经断开，恢复时不能继续显示“生成中”。
  status: message.status === 'streaming' ? 'aborted' : message.status,
  displayContent: message.displayContent || message.content,
  content: message.content || message.displayContent
})

export const getPersistedInterviewMessages = (sessionId: string): InterviewMessage[] => {
  const raw = window.localStorage.getItem(resolveStorageKey(sessionId))
  const parsed = raw ? JSON.parse(raw) as InterviewMessage[] : []
  return Array.isArray(parsed) ? parsed.map(normalizeMessage) : []
}

export const setPersistedInterviewMessages = (
  sessionId: string,
  messages: InterviewMessage[]
) => {
  window.localStorage.setItem(
    resolveStorageKey(sessionId),
    JSON.stringify(messages.map(normalizeMessage))
  )
}
```

### 30 秒回答

答：我把对话消息按 `sessionId`（会话标识）持久化到了 `localStorage`（浏览器本地存储）里。这样页面刷新后，用户还能恢复这轮会话的历史消息，不会因为一次刷新把前面生成的内容全丢掉。对于刷新前还处于 `streaming`（生成中）状态的消息，我恢复时会把它转成 `aborted`（已中止），避免误导用户以为它还在继续生成。

### 场景题

#### 1. 为什么按 `sessionId`（会话标识）存，而不是全量混在一起？

答：因为这个项目不是普通闲聊，而是面试流程驱动的多会话场景。按 `sessionId`（会话标识）分桶存储，可以避免不同面试会话互相串内容。

#### 2. 刷新时为什么把 `streaming`（生成中）改成 `aborted`（已中止）？

答：因为浏览器刷新后原来的长连接已经断了，这条消息事实上不可能还在继续生成。如果继续显示成 `streaming`（生成中），会误导用户。

#### 3. 为什么先用 `localStorage`（浏览器本地存储），不用 `IndexedDB`（浏览器结构化存储）？

答：当前阶段消息量还可控，`localStorage`（浏览器本地存储）实现简单、恢复快，足够覆盖“刷新不丢消息”这个核心诉求。后面如果对话很长、数据量变大，再升级到 `IndexedDB`（浏览器结构化存储）更合适。

#### 4. 持久化会不会太频繁？

答：确实要注意频率，所以项目里不是每个底层网络字节都单独落盘，而是跟消息更新链路一起统一保存。配合前面的渲染队列，能避免无意义的高频写入。

#### 5. 如果面试官问：这算不算“对话记忆”？

答：算前端侧的会话记忆，但不是完整的长期记忆系统。它解决的是刷新恢复和本地历史保留，不等于向量检索或 `RAG`（检索增强生成）那种知识记忆。

## 五、HTTP（超文本传输协议）请求层高可用增强

### 实现定位

- `frontend/src/services/http/request.ts`（统一请求客户端）：基于 `Axios`（HTTP 请求库）封装全局默认重试、`TokenBucket`（令牌桶）并发控制、`circuit breaker`（熔断器）保护。

### 核心实现代码

```ts
// request.ts（统一请求工具文件）
class TokenBucket {
  private activeCount = 0
  private queue: PendingAcquireRequest[] = []

  acquire(signal?: AbortSignal) {
    if (this.activeCount < this.maxConcurrentRequests) {
      return Promise.resolve(this.createTicket())
    }

    return new Promise<TokenBucketTicket>((resolve, reject) => {
      const pendingRequest = { id: this.nextId++, resolve, reject, signal }

      if (signal) {
        pendingRequest.abortHandler = () => {
          this.removePendingRequest(pendingRequest.id)
          reject(createRequestError('请求在队列中已取消。', 'ERR_CANCELED'))
        }
        signal.addEventListener('abort', pendingRequest.abortHandler, { once: true })
      }

      this.queue.push(pendingRequest)
    })
  }

  private createTicket(): TokenBucketTicket {
    this.activeCount += 1

    let released = false
    return {
      release: () => {
        if (released) return
        released = true
        this.activeCount = Math.max(0, this.activeCount - 1)
        this.flushQueue()
      }
    }
  }
}
```

```ts
// request.ts（统一请求工具文件）
service.interceptors.request.use(async request => {
  const concurrencyConfig = resolveConcurrencyConfig(request)
  if (concurrencyConfig.enabled) {
    const tokenBucket = resolveTokenBucket(concurrencyConfig)
    const ticket = await tokenBucket.acquire(request.signal)
    // 挂到当前请求配置上，保证成功、失败、取消时都能释放并发令牌。
    request.__releaseConcurrency = ticket.release
  }

  const circuitConfig = resolveCircuitBreakerConfig(request)
  if (circuitConfig.enabled) {
    const circuitKey = resolveCircuitBreakerKey(request)
    request.__circuitBreakerKey = circuitKey
    circuitBreakerRegistry.beforeRequest(circuitKey, circuitConfig)
  }

  return request
})
```

### 30 秒回答

答：我在统一请求层上做了三件事。第一，对 `GET`（读取请求）这类幂等请求默认开启透明重试，减少偶发网络抖动影响；第二，用 `TokenBucket`（令牌桶）限制总并发，避免多个页面同时打接口把前端和服务端一起冲垮；第三，用 `circuit breaker`（熔断器）在某一路径连续失败时临时打开熔断，阻止无意义重试，避免持续雪崩。

### 场景题

#### 1. 为什么只给幂等请求默认自动重试？

答：因为 `GET`（读取请求）失败后重试通常不会带来副作用，但 `POST`（提交请求）可能会重复创建、重复提交，所以默认不能透明重试，除非业务明确允许。

#### 2. 为什么需要 `TokenBucket`（令牌桶）？

答：因为后台页、列表页、报告页可能会并发发起很多请求。如果没有统一并发控制，瞬时洪峰会把浏览器、网关和服务端一起压高。令牌桶本质上是在入口做削峰。

#### 3. 熔断器什么时候会打开？

答：比如同一路径连续出现超时、网络错误，或者持续 `429/5xx`（限流或服务端错误）时，达到阈值就会从 `closed`（关闭）切到 `open`（打开）状态，后续请求短时间内直接失败，等冷却期过后再进入 `half-open`（半开）探测恢复。

#### 4. 如果服务刚恢复，怎么避免瞬间又被打爆？

答：熔断器恢复时不是一次性全部放开，而是先进入 `half-open`（半开）状态，只放少量探测请求。成功了再完全关闭熔断，失败就重新打开。

#### 5. 这个请求层优化解决的核心问题是什么？

答：解决的是“偶发失败可自动恢复，持续失败要及时止损，瞬时高并发要统一削峰”这三个稳定性问题。它不是单点技巧，而是一层可靠性保护网。

### 27. 这个项目现在不只是模拟面试页了吗？你怎么介绍整体范围？

答：现在项目可以概括成“AI（人工智能）模拟面试 + 后台管理”的前后端一体化系统。前台核心是宇宙风格模拟面试页，负责题目展示、流式追问、消息持久化和报告入口；后台新增了 `admin`（管理员）登录、数据看板、会话管理、报告管理，用来查看全量会话和报告数据。

### 28. 后台管理模块是怎么做权限区分的？

### 29. 普通用户和管理员看到的数据范围有什么区别？

答：管理员可以查看全量会话和报告，普通用户只能查看归属于自己的数据。后端给会话和报告补了可选 `owner`（归属用户）字段，新数据会在已登录场景下绑定用户；查询时根据 `authUser.role`（登录用户角色）决定返回范围。

### 30. 为什么后台列表有些状态、分数不是后端真实字段？

答：当前后端真实契约里还没有完整提供 `status`（状态字段）、`score`（分数字段）、`generateStatus`（生成状态字段）这些后台筛选字段，所以前端没有硬猜接口字段，而是基于现有列表数据做派生展示。  
面试里可以说：我区分了“真实接口字段”和“前端派生字段”。真实接口字段用于数据契约，派生字段只用于当前页面筛选和展示，不把它伪装成后端已经支持的能力。

### 31. 刷新页面后“恢复本轮反馈”是怎么做的？

答：前端会把未完成流的 `sessionId`（会话标识）、`threadId`（线程标识）、`idempotentKey`（幂等键）保存成 `pendingRecovery`（待恢复上下文）。刷新后重新绑定同一个会话时，会恢复 `interrupted`（中断）状态，并显示“恢复本轮反馈”入口。  
用户点击恢复时，前端优先调用 `checkpoint API`（断点快照接口），如果命中就补渲染上次已经生成的内容，并把消息收尾成 `aborted`（已中止）。当前版本不自动续写，因为真正自动续流需要后端和 C++（后端语言）引擎都支持同名断点能力。

```ts
// useInterviewStream.ts（面试流组合函数）
if (nextRetry.reason === 'interrupted') {
  const restoreResult = await restoreInterviewStreamFromCheckpoint({
    sessionId: nextRetry.params.sessionId,
    threadId: nextRetry.params.threadId,
    idempotentKey: nextRetry.params.idempotentKey || activeIdempotentKey.value
  })

  if (restoreResult.status === 'restored') {
    // 只补回已经生成的内容，不假装继续生成，避免用户误解流还在运行。
    finalizeRecoveredMessage(nextRetry.messageId, nextRetry.params.threadId, restoreResult.checkpoint.content)
  }
}
```

### 32. 请求层为什么要做重试、并发控制和熔断？

答：因为后台页面和报告页面会同时发多个 `REST API`（表述性状态转移接口）请求，如果没有统一请求层，偶发失败会直接暴露给用户，持续失败又可能被重复请求放大。  
项目现在在 `request.ts`（统一请求工具文件）里做了三层保护：`GET`（读取请求）默认重试处理偶发抖动，`TokenBucket`（令牌桶）限制总并发，`circuit breaker`（熔断器）在同一路径连续失败时短暂拦截，避免雪崩。

### 33. 为什么不直接让前端连 C++（后端语言）服务？

答：项目约束是浏览器统一请求 `/api/*`（接口路径），由 `Vite proxy`（开发代理）转给 `Node.js`（后端运行时）网关，再由后端决定走本地模拟、远程模型或 C++（后端语言）引擎。前端不直接连 C++（后端语言）端口。  
这样做的好处是接口契约稳定，鉴权、日志、错误处理、数据隔离都能在 `Node.js`（后端运行时）这一层统一收口，也方便以后替换模型提供方。

### 34. 你怎么保证新增后台功能没有破坏原来的流式面试链路？

答：我把后台管理和流式面试拆在不同边界里：后台走普通 `REST API`（表述性状态转移接口），流式面试仍然走 `POST /api/interview/stream`（面试流式接口）和 `SSE`（服务端发送事件）事件。新增后台页面时不改 `chunk/done/error`（分片/完成/错误）核心契约；请求层高可用增强也限定在非流式接口，避免影响 `ReadableStream`（可读流）消费链路。  
验证时主要看两件事：`vite build`（生产构建）能过；流式入口仍能正常启动、停止、错误收尾和恢复。

## 七、项目里使用闭包（Closure，闭包）的场景

### 题目：你项目里哪里用到了闭包（Closure，闭包）？

答：我这个项目里比较典型的闭包场景在并发控制和消息队列两块。  
一个是 `frontend/src/services/http/request.ts`（统一请求客户端）里的 `TokenBucket`（令牌桶）并发控制，像 `release`（释放令牌函数）会闭包持有 `released`（是否已释放标记）和当前桶实例状态，保证同一个令牌不会被重复释放；排队请求里的 `abortHandler`（中止处理函数）也会闭包记住当前 `pendingRequest.id`（排队请求标识）和 `reject`（拒绝回调），这样请求在排队阶段被取消时，能精确把自己从队列里移除。
另一个是 `useInterviewStream.ts`（面试流组合函数）里给 `InterviewMessageQueue`（面试消息队列）注册的 `handler`（处理函数），它闭包持有当前 `messageId`（消息标识），所以每次刷新队列时都能把增量文本准确打到对应那条消息上，而不是写错消息。  
所以我会把闭包理解成“函数记住外部上下文的一种能力”，在这个项目里主要用来保存请求上下文和消息上下文。

代码例子：

```ts
// request.ts（统一请求工具文件）
private createTicket(): TokenBucketTicket {
  this.activeCount += 1

  let released = false
  return {
    release: () => {
      if (released) return
      released = true
      // release（释放令牌函数）记住 released（是否已释放标记），防止同一请求重复归还令牌。
      this.activeCount = Math.max(0, this.activeCount - 1)
      this.flushQueue()
    }
  }
}
```

```ts
// request.ts（统一请求工具文件）

  // abortHandler（中止处理函数）记住当前 pendingRequest.id（排队请求标识），能精准移除自己。
  this.removePendingRequest(pendingRequest.id)
  reject(createRequestError('请求在队列中已取消。', 'ERR_CANCELED'))
}
```

```ts

  patchMessage(event.messageId, item => ({
    ...item,
```

### 30 秒版本

答：有，最典型的是并发控制。`TokenBucket`（令牌桶）里返回的 `release`（释放令牌函数）就是闭包，它记住了当前请求对应的释放状态和桶内部计数，保证令牌只释放一次；排队时的 `abortHandler`（中止处理函数）也会记住自己的请求 `id`（标识）和 `reject`（拒绝回调），这样取消时能精准把自己从队列里移除。消息队列注册刷新函数时也是同样思路，它要记住当前 `messageId`（消息标识）。

### 如果面试官继续问：闭包用不好会有什么问题？

答：主要是两个风险。  
第一，闭包持有的数据太大、生命周期太长，容易造成不必要的内存占用；  
第二，如果把本该属于“单次请求”的状态错误地放进共享闭包里，就会互相污染。比如重试次数如果做成全局共享闭包变量，不同请求之间就会串状态。这个项目里我特意把 `__retryCount`（当前请求重试次数）挂到请求配置对象上，而不是放到拦截器共享闭包里，就是为了避免这个问题。

代码例子：

```ts
// request.ts（统一请求工具文件）
const retryRequest = async (error: AxiosError) => {
  const config = error.config
  if (!config || !isRetryableRequestError(error)) {
    throw error
  }

  // 重试次数属于当前请求，不放到全局闭包变量里，避免多个请求串状态。
  config.__retryCount = (config.__retryCount || 0) + 1
  await waitRetryDelay(resolveRetryDelay(config))

  return service(config)
}
```
