# 路由守卫三层治理与参数保真方案

## 一、目标

本方案用于根治当前项目中两类高风险路由问题：

- `router guard`（路由守卫）重复进入导致的死循环、401（未授权）风暴、异常重定向。
- 路由跳转过程中 `query`（查询参数）丢失、`params`（路径参数）刷新失效、动态路由未生效导致的页面白屏或 404（未找到）。

本次方案不只修单个 bug，而是把现有路由体系收敛为“三层治理”：

1. 业务分层：先把路由类型拆清楚。
2. 守卫收口：把登录校验、权限判断、动态路由加载和参数保真统一进一个稳定流程。
3. 架构规范：沉淀通用工具、传参规范和监控口径，避免后续继续返工。

## 二、当前问题背景

结合当前仓库现状，已暴露出以下典型风险：

### 1. 路由守卫死循环风险

当前 `frontend/src/router/permission.ts`（路由守卫文件）会在进入 `requiresAuth`（需要登录）路由时请求 `/api/auth/me`（当前登录态接口）做校验。

若出现以下组合场景：

- 本地 `localStorage`（浏览器本地存储）里还残留登录态；
- `cookie`（浏览器 Cookie）里的 `token`（令牌）已经失效或丢失；
- 守卫失败后没有及时清理本地状态；
- 重定向目标仍会再次进入守卫；

就会触发“守卫反复进入 -> 反复请求 `/me` -> 继续 401 -> 再次重定向”的循环链路。

### 2. 参数丢失风险

当前项目里同时存在以下传参方式：

- `query`（查询参数）
- `params`（路径参数）
- `redirect`（登录后回跳地址）

若没有统一规范，容易出现：

- 刷新页面后 `params` 丢失；
- 动态路由尚未完成注册时就开始匹配，导致参数无效；
- 登录跳转时只保留路径，不保留原始 `query`；
- 守卫里多次 `replace`（替换跳转）导致参数被覆盖。

### 3. 动态路由与权限路由职责混杂

当前项目虽然还不是重度后台系统，但已经存在：

- 公共路由：`/login`（登录页）、`/admin/login`（后台登录页）、404 页。
- 静态权限路由：`/admin/dashboard`（后台看板页）、`/admin/sessions`（后台会话页）、`/admin/reports`（后台报告页）、`/admin/models`（后台模型页）。
- 后续可扩展动态路由：按角色、菜单、开关动态生成的后台业务路由。

如果仍把三类路由混在同一条守卫链路里，会让后续新增后台模块时继续放大路由复杂度。

## 三、方案总览

本方案分为三层：

### 第一层：业务分级，区分公共路由、权限路由与动态路由

目标：先明确“谁应该进守卫、谁应该跳过、谁需要先加载再匹配”。

### 第二层：核心方案落地与细节兜底处理

目标：收敛守卫执行顺序、限制重入次数、保证参数不丢、保证动态路由刷新后仍可恢复。

### 第三层：架构层优化，统一路由管理规范

目标：把这次修复沉淀成长期可复用的路由规范、工具函数和监控指标。

## 四、第一层：业务分级，区分公共路由、权限路由与动态路由

### 1. 公共路由白名单

以下页面不应进入复杂权限校验，直接放行：

- `/login`（登录页）
- `/admin/login`（后台登录页）
- `/404`（未找到页，当前由兜底路由承接）
- `/500`（服务器错误页，如后续新增）

要求：

- 守卫最前面先判断白名单。
- 白名单命中后直接返回 `true`，跳过登录态请求、角色校验和动态路由加载。
- 禁止在公共路由上再次触发“强制跳首页”类兜底逻辑。

### 2. 静态权限路由

当前后台已落地的以下路由属于静态权限路由：

- `/admin/dashboard`
- `/admin/sessions`
- `/admin/reports`
- `/admin/models`

特点：

- 路由表预先存在；
- 只需要校验 `requiresAuth`（需要登录）和 `requiredRole`（所需角色）；
- 不需要依赖后端动态返回菜单再注册。

要求：

- 统一走“登录校验 -> 角色校验 -> 放行”流程。
- 失败后只允许发生一次确定性的重定向。

### 3. 动态菜单路由

后续若引入按角色动态生成的后台菜单路由，必须单独设计：

- 动态路由加载状态；
- 动态路由注册函数；
- 动态路由首次加载后的二次匹配；
- 刷新页面后的恢复逻辑。

要求：

- 动态路由注册逻辑从通用守卫中抽离成独立函数。
- 守卫只判断“是否已完成动态路由准备”，不直接内联大段注册逻辑。

## 五、第二层：核心方案落地与细节兜底处理

这一层是本方案的核心，直接对应“死循环”和“参数丢失”的根因治理。

### 1. 守卫严格限制重入与跳转次数

必须建立以下规则：

- 同一次导航只允许产出一次明确结果：`true`（放行）或一个重定向对象。
- 禁止在同一轮守卫里多次调用 `next()`（旧版写法）或多次返回不同跳转结果。
- 禁止无条件 `replace({ path: '/' })`（替换跳首页）兜底。

建议策略：

- 优先使用 `Vue Router 4`（Vue 路由 4）推荐的 `return` 风格守卫，不混用 `next()`（路由放行回调）。
- 为守卫增加“重入标记”或“动态路由已处理标记”，避免同一目标路由在一次跳转链中重复执行动态注册。
- 当登录校验失败时，立即清理本地登录态，阻断后续再次请求 `/api/auth/me`。

### 2. 登录态校验短路机制

当前 `/api/auth/me`（当前登录态接口）不属于“服务故障”，401 不应走 `circuit breaker`（熔断器）语义，而应走“鉴权短路”语义。

建议新增一层轻量短路机制：

- 当本地有 `auth session`（登录会话）但没有 `token` 时，直接判定为未登录，不发 `/me` 请求。
- 当 `/me` 返回空会话或 401 时，立即清理：
  - `localStorage`（本地存储）中的登录态；
  - `cookie`（浏览器 Cookie）中的 `token`；
  - 内存中的用户态缓存。
- 在短时间窗口内对 `/me` 增加“失败冷却”：
  - 例如 3 到 5 秒内不再重复请求；
  - 直接视为未登录，等待下一次真实用户动作再触发校验。

这样可以避免“鉴权失败风暴”被误认为“接口可重试问题”。

### 3. 参数保真优先级：`query` 优先，`params` 受控使用

本项目后续统一规定：

#### 场景 A：跨登录页、跨权限页回跳

统一使用 `query.redirect`（查询参数回跳地址）传递完整目标地址。

原因：

- `query` 刷新后不丢；
- 可以完整保留路径和查询串；
- 最适合登录后回跳、来源页回退、运营入口跳转。

#### 场景 B：核心业务页面上下文传递

优先使用 `query`，例如：

- `sessionId`
- `threadId`
- `scene`
- `tab`
- `from`

原因：

- 浏览器刷新后参数仍在；
- 便于排查线上问题；
- 更适合当前前端调试和后台管理场景。

#### 场景 C：必须语义化路径的场景

仅当 URL 语义必须体现资源层级时，才允许使用 `params`。

例如：

- `/reports/:sessionId`
- `/sessions/:sessionId/:threadId`

但必须同时满足：

- 页面刷新后可以通过 URL 自身恢复数据；
- 如果需要额外上下文，再配合 `sessionStorage`（会话级存储）补充；
- 禁止把只能存在内存里的业务状态只放在 `params`。

### 4. `params` 刷新丢失兜底

对于必须使用 `params` 的页面，补一层恢复机制：

- 进入页面时，把关键参数快照写入 `sessionStorage`。
- 刷新后若发现动态路由尚未就绪或 `params` 不完整：
  - 先尝试从 `sessionStorage` 恢复；
  - 再执行一次 `router.replace`（替换跳转）进行重匹配。

注意：

- `sessionStorage` 只做恢复兜底，不做主数据源。
- 恢复失败时要给出明确降级，不允许无限重试。

### 5. 动态路由加载后的单次重匹配

若后续引入动态菜单路由，标准流程必须是：

1. 进入守卫。
2. 判断当前目标是否依赖动态路由。
3. 若未加载，则先执行 `ensureDynamicRoutesReady`（确保动态路由已就绪函数）。
4. 动态注册完成后，仅执行一次 `replace(to.fullPath)`（按完整目标地址替换跳转）。
5. 第二次进入守卫时识别“已加载完成”，直接走正常匹配。

关键约束：

- 只允许一次补偿性 `replace`。
- 通过标记位避免无限重进。

## 六、第三层：架构层优化，统一路由管理规范

### 1. 抽离统一路由守卫工具

建议新增统一工具层，例如：

- `createPublicRouteMatcher`（公共路由匹配器）
- `resolveAuthRedirect`（鉴权跳转解析器）
- `ensureDynamicRoutesReady`（确保动态路由已就绪函数）
- `normalizeRouteContext`（路由上下文标准化函数）
- `restoreRouteParamsFromSession`（从会话存储恢复路由参数函数）

目的：

- 让 `permission.ts`（路由守卫文件）只保留主流程；
- 复杂细节拆到独立函数中；
- 便于单测和后续复用。

### 2. 统一路由元信息规范

建议统一每个路由的 `meta`（路由元信息）字段语义：

```ts
meta: {
  public: true,
  requiresAuth: false,
  requiredRole: 'admin',
  dynamic: false,
  preserveQuery: true,
}
```

建议含义：

- `public`（公共路由）：白名单路由，直接放行。
- `requiresAuth`（需要登录）：进入前必须校验登录态。
- `requiredRole`（所需角色）：如 `admin`（管理员）。
- `dynamic`（动态路由）：是否依赖运行时注册。
- `preserveQuery`（保留查询参数）：登录或补偿跳转时是否必须带回原查询参数。

### 3. 统一传参规范

需要形成仓库级约束：

- 登录回跳、来源回退、筛选状态：统一用 `query`。
- 资源层级路径：谨慎使用 `params`。
- 核心业务页面禁止临时拼接匿名 `params`。
- 若必须使用 `params` 且刷新后要恢复，必须同时声明恢复策略。

建议在文档和代码评审中明确：

- 没有刷新恢复方案的 `params` 传参不允许进入核心业务主链路。

### 4. 动态路由生成逻辑独立化

若未来后台菜单扩展，建议把动态路由生成拆成：

- `buildAdminDynamicRoutes`（构建后台动态路由函数）
- `registerDynamicRoutes`（注册动态路由函数）
- `hasRouteRegistered`（路由是否已注册判断函数）

这样可以把“路由生成”和“守卫执行”解耦，降低后期复杂度。

### 5. 前端监控与告警指标

建议补充以下路由层监控：

- 单页面 10 秒内 `/api/auth/me` 请求次数；
- 单页面 10 秒内路由跳转次数；
- 404（未找到）触发率；
- 登录页与后台页之间的来回跳转次数；
- 守卫补偿性 `replace` 次数；
- 动态路由加载失败次数。

告警规则建议：

- 同一路由 10 秒内跳转超过阈值时上报；
- `/me` 在短时间内连续 401 超过阈值时上报；
- 404 或白屏恢复逻辑异常增长时上报。

## 七、建议落地步骤

### Phase 1：先止血，阻断死循环

目标：先停止 401 风暴和异常重定向。

- 守卫接入公共路由白名单。
- 当本地无 `token` 时不再请求 `/me`。
- `/me` 失败后立即清理本地登录态。
- 去掉无条件首页重定向兜底。
- 守卫保证单次导航只返回一个结果。

当前已落地：

- 新增 `frontend/src/router/route-guard-utils.ts`（路由守卫工具文件），收口公共路由判断、登录态清理、鉴权短路和重定向构造。
- `frontend/src/router/permission.ts`（路由守卫文件）已切到“单返回值”守卫风格，不再混杂散落的鉴权分支。
- 当本地有 `auth session`（登录会话）但没有 `token`（令牌）时，守卫会直接清理本地状态并停止请求 `/api/auth/me`（当前登录态接口）。
- 当 `/api/auth/me` 返回空会话或鉴权失败时，守卫会立即清理客户端登录态，并进入失败冷却窗口，避免短时间内持续打 `/me`。
- 对公共路由增加了提前放行逻辑，但不会绕过 `guestOnly`（仅游客可进）页面对“已登录用户回跳”的控制。

Phase 1 剩余可选增强：

- 给 `/api/auth/me` 增加单独的轻量观测埋点，统计 401 频次。
- 为守卫失败冷却增加可配置参数，便于不同环境调节。

### Phase 2：参数保真与动态恢复

目标：解决跳转参数丢失和刷新恢复问题。

- 登录回跳统一走 `query.redirect`。
- 核心业务页统一收口 `query` 传参。
- 必须使用 `params` 的页面接入 `sessionStorage` 恢复兜底。
- 动态路由注册完成后执行单次 `replace(to.fullPath)`。

建议拆解为四个子任务：

1. 登录回跳统一化  
   将普通登录页、后台登录页和权限不足跳转统一接到 `query.redirect`（查询参数回跳地址）语义，避免路径和查询串分离。

2. 核心业务页传参规范化  
   为后台详情页、宇宙页场景切换、报告查看入口建立“优先 `query`、谨慎 `params`”的明确清单。

3. `params` 恢复兜底  
   仅对当前主链路页面新增受控的 `sessionStorage`（会话级存储）恢复能力；旧工作台 `frontend/src/views/workbench/**`（旧工作台页面目录）已冻结，不作为本阶段落点。

4. 动态路由补偿重匹配  
   若未来引入按角色动态菜单，补 `ensureDynamicRoutesReady`（确保动态路由已就绪函数）和单次 `replace(to.fullPath)`（按完整地址替换跳转）补偿逻辑。

当前状态说明：

- `Phase 1`（第一阶段）全局守卫止血已落地。
- `Phase 2`（第二阶段）原先一度尝试在 `frontend/src/views/workbench/**`（旧工作台页面目录）补参数恢复兜底，但这与“旧工作台冻结”边界冲突，现已全部回撤。
- 因此，`Phase 2` 后续只允许落到两类位置：
  - 全局路由层，如 `frontend/src/router/**`（前端路由目录）；
  - 当前主链路页面，如 `frontend/src/views/showcase/mock-interview-space.vue`（宇宙模拟面试页）及其配套场景组件 / 组合函数。

Phase 2 下一步建议：

- 仅梳理 `mock-interview-space`（宇宙模拟面试页）主链路的关键参数恢复，不再扩展 `Workbench*`（旧工作台页面）系列页面。
- 将“哪些页面允许缓存恢复、哪些页面必须严格依赖 URL 自身”的边界补入统一规范，并明确写入“旧工作台冻结，不接受新功能补丁”。
- 视后续后台菜单演进情况，决定是否进入动态路由单次重匹配实现。

### Phase 3：抽象工具与治理规范

目标：把方案变成长期架构能力。

- 抽离守卫工具函数。
- 抽离动态路由注册层。
- 统一 `meta` 元信息语义。
- 补充监控指标与排障口径。

建议拆解为三个子任务：

1. 工具层标准化  
   抽离 `dynamic-route-manager`（动态路由管理文件）和统一跳转构造器；如后续确有主链路需要，再新增受控的路由上下文缓存工具。

2. 路由元信息标准化  
   为 `public`（公共路由）、`dynamic`（动态路由）、`preserveQuery`（保留查询参数）等元信息补类型与注释。

3. 监控口径标准化  
   统一 `/me` 请求频次、路由跳转次数、404 触发率等指标的采集与排障说明。

## 九、不做范围

本方案当前不包含以下内容：

- 不改后端真实鉴权协议。
- 不改 `POST /api/interview/stream`（模拟面试流式接口）相关契约。
- 不在本阶段引入完整权限中心或菜单配置平台。
- 不把 401（未授权）强行纳入 `circuit breaker`（熔断器）语义。

原因：

- 401 的本质是鉴权状态失效，不是服务降级；
- 本次重点是前端路由层治理，不是后端权限体系重构。

## 十、验收标准

### 死循环治理验收

- 本地残留登录态但无 `token` 时，不再重复请求 `/api/auth/me`。
- `/api/auth/me` 返回 401 时，页面最多只发生一次确定性跳转。
- 登录页、后台登录页、404 页不会再进入鉴权死循环。

### 参数保真验收

- 登录后回跳能完整恢复原始目标地址和 `query`。
- 核心业务页面刷新后，关键参数不丢失。
- 动态路由首次加载完成后能正确重匹配，不出现空白页。

### 架构治理验收

- 守卫主流程收敛为清晰的单入口逻辑。
- 动态路由逻辑可独立测试。
- 新增后台模块时不需要复制粘贴复杂守卫分支。

---

## 十一、结论

路由守卫死循环和参数丢失，本质上不是两个孤立 bug，而是同一类“路由职责未分层、守卫流程未收口、传参规范未统一”的架构问题。

因此，本方案不建议继续采用“哪里报错修哪里”的方式，而应按三层策略推进：

1. 第一层先分清路由类型。
2. 第二层收口守卫执行和参数恢复。
3. 第三层沉淀统一工具、规范和监控。

只有这样，后续后台模块扩张、动态菜单接入、登录态演进时，路由系统才不会再次回到“跳转一改就连锁出问题”的状态。
