# 9.SSE流式对话与消息渲染架构开发流程.md

## 一、为什么现在开始做 SSE

到当前这个阶段，项目已经具备了完整的前端产品骨架：

- `library.vue` 负责资料导入与上下文准备
- `mock-interview.vue` 已经成为主 AI 对话舞台
- `report.vue` 已能承接真实复盘结果
- `history.vue` 已能承接 session 与 report summary 列表
- 本地持久化第一阶段已经打通

也就是说，当前项目已经不缺页面，也不缺流程。

现在真正缺的是：

**让 AI 对话从“假数据驱动页面”升级为“真实流式交互体验”。**

所以这一阶段要解决的问题是：

1. 大模型回复如何流式进入页面
2. 高速 token 到达时如何稳定渲染
3. AI 输出的 Markdown 内容如何安全展示
4. 流式消息如何与当前工作台结构融合

这就是为什么，下一阶段最合理的方向是：

**开始搭建 SSE 流式对话与消息渲染架构。**

---

## 二、这一阶段的目标

这一阶段不追求一上来就做成最复杂的多模型系统，而是先做一版：

**可接真实流式响应、结构清晰、后续易扩展的前端 SSE 架构。**

具体目标包括：

- 在 `mock-interview.vue` 接入真实 SSE 主对话
- 把消息结构设计清楚
- 把 SSE 请求控制和消息处理解耦
- 引入消息消费队列，避免高频刷屏抖动
- 引入 Markdown 渲染与安全过滤

---

## 三、这一阶段完成后的用户体验

完成后，用户在模拟面试页应该是这样使用的：

1. 从资料页或总览页进入模拟面试页
2. 发起一轮真实 AI 面试对话
3. 面试官问题不是一次性整块出现，而是流式生成
4. 页面中的消息逐步稳定展示，不闪烁、不抖动
5. 如果 AI 返回列表、代码块、强调内容，可以按 Markdown 正常渲染
6. 用户继续输入回答，进入下一轮追问
7. 最后这些消息和训练结果一起沉淀进报告与历史页

这一步完成后，你的项目才会从“AI 面试工作台”真正升级成“AI 面试应用”。

---

## 四、这一阶段建议优先做哪三项

### 1. SSE 流式对话三层解耦架构

这是最优先的。

建议拆成三层：

#### 传输层

负责：

- 发起 SSE 请求
- 接收原始 chunk
- 连接建立与结束
- 异常中断处理
- 心跳 / 超时控制（第一版可先预留）

#### 控制层

负责：

- 开始一轮对话
- 中止当前流式响应
- 重试
- 维护当前 loading / streaming / error 状态
- 与页面交互层联动

#### 处理层

负责：

- chunk 拼装
- 消息角色分发
- 文本累积
- 事件类型识别
- 输出可直接给 UI 渲染的消息对象

### 为什么一定要三层拆开

因为如果你把所有逻辑都写进 `mock-interview.vue`，后面会非常难维护：

- 页面层会过重
- SSE 与 UI 强耦合
- 很难复用到 `practice.vue`
- 很难接入停止、重试、并发控制

所以这一阶段最重要的不是“先连上流”，而是：

**先把流式响应的职责边界搭对。**

---

### 2. requestAnimationFrame 消息渲染队列

这一项非常值得加，而且很适合写进简历。

原因是：

流式响应返回时，token 往往会高频到达。  
如果每收到一点内容就立刻更新 UI，会带来这些问题：

- 刷屏感太强
- 滚动位置抖动
- 组件频繁重渲染
- 消息一边长一边闪

所以建议做一个：

**基于 requestAnimationFrame 的消息消费队列**

思路是：

- 原始 token 先进入缓冲区
- 每一帧统一消费一小批
- 以自然节奏推进消息内容
- 最终让用户感知更平滑

这既提升体验，也体现你对前端渲染性能有意识。

---

### 3. Markdown 渲染与 XSS 防护

这一项同样推荐加。

因为后面 AI 回复内容很可能包括：

- 分点总结
- 标题层级
- 加粗强调
- 代码块
- 引导列表
- 报告式输出

所以建议支持：

- Markdown 解析
- 代码高亮
- DOMPurify 安全过滤

这样后面无论是：

- AI 面试官追问
- 薄弱点分析
- 报告页总结

都能复用同一套富文本展示能力。

---

## 五、建议先不要急着做哪些能力

### 1. 多模型切换

先别急着做。

原因：

- 当前还在做单条主链路
- 多模型切换会让配置层复杂很多
- 现在最重要的是把“一个模型的对话流”跑稳

### 2. 复杂多角色混流

先别急着做。

比如同时流出：

- 主问题
- 提示卡
- 旁路解释
- 追问建议

这一类“多事件并发到 UI”后面再做。

### 3. 全量消息历史持久化

这一阶段先不做完整消息级持久化。

先把：

- session 摘要
- report summary

跑稳更重要。

## 七、建议的数据结构

### 1. 基础消息模型

建议新增到：

- `src/types/message.ts`

建议结构：

```ts
interface InterviewMessage {
  id: string
  role: 'assistant' | 'user' | 'system'
  content: string
  displayContent: string
  format: 'plain' | 'markdown'
  status: 'streaming' | 'done' | 'error'
  createdAt: string
}
```

### 2. SSE 事件模型

```ts
interface InterviewStreamEvent {
  type: 'start' | 'delta' | 'done' | 'error'
  messageId: string
  delta?: string
  errorMessage?: string
}
```

### 3. UI 队列状态模型

```ts
interface MessageQueueState {
  buffer: string[]
  isFlushing: boolean
  lastFlushAt: number
}
```

## 八、这一阶段建议先接到哪个页面

### 结论

**先接 `src/views/workbench/mock-interview.vue`。**

原因：

- 它已经是主 AI 对话页
- 它已经有输入区、提示区、追问区
- 它是最适合承接真实流式问答的页面

后续如果稳定，再复用到：

- `practice.vue`

但第一阶段不要双页同时接，避免范围过大。

## 九、页面结构应该怎样变化

当前 `mock-interview.vue` 的主区还是：

- 当前问题主卡片
- 回答输入面板
- 操作条

接入 SSE 后，建议逐步演进为：

### 中间主舞台

#### 区域 A：当前轮次上下文条

保留。

#### 区域 B：消息流列表

新增：

- 用户消息
- AI 面试官消息
- 流式生成中的消息

#### 区域 C：回答输入区

保留，但和消息流联动。

#### 区域 D：操作条

保留，后面增加：

- 停止生成
- 重新生成

### 右侧辅助区

继续保留：

- AI 引导
- 文档命中
- 薄弱点跟踪

也就是说：

**中间舞台从“题目卡片页”逐步过渡到“真实消息流主舞台”。**

## 十、建议的开发顺序

### 第 1 步：先定义消息类型和 SSE 类型

先做：

- `src/types/message.ts`
- `src/services/sse/sse-types.ts`

目标：

- 把消息对象与事件对象先定清楚

### 第 2 步：先做最小可用 SSE client

文件：

- `src/services/sse/sse-client.ts`

本步目标：

- 建立连接
- 接收 chunk
- 能通知外层开始、增量、结束、错误

### 第 3 步：做消息解析层

文件：

- `src/services/message/interview-message-parser.ts`

本步目标：

- 把 chunk 变成前端消息事件
- 输出统一结构给 UI

### 第 4 步：做 requestAnimationFrame 队列

文件：

- `src/services/message/interview-message-queue.ts`

本步目标：

- 缓存高频 token
- 逐帧推进到 displayContent
- 降低刷屏抖动

### 第 5 步：做 `useInterviewStream.ts`

文件：

- `src/composables/useInterviewStream.ts`

本步目标：

- 给页面暴露统一流式接口
- 页面层不直接操作 SSE 细节

### 第 6 步：把 `mock-interview.vue` 切到真实消息流 UI

本步目标：

- 中间消息列表显示真实流
- 用户输入能触发一次完整流式请求
- 支持中断
- 支持错误态提示

### 第 7 步：补 Markdown 渲染与安全过滤

文件：

- `src/components/message/MarkdownMessage.vue`

本步目标：

- 支持列表、代码块、强调文本
- DOMPurify 过滤危险内容

## 十一、Markdown 渲染建议

### 建议技术组合

可以考虑：

- `markdown-it` 或 `marked`
- `DOMPurify`
- 代码高亮可后续接 `highlight.js` 或 `shiki`

### 第一版优先支持

- 段落
- 列表
- 加粗
- 代码块
- 行内代码

### 第一版先不急着支持

- Mermaid
- KaTeX
- 自定义复杂嵌入卡片

---

## 十二、这一阶段和持久化的关系

当前 session / report summary 已经持久化了。  
这一阶段 SSE 接入后，要注意：

### 当前先不要立刻把所有消息全文持久化

原因：

- 消息结构还没稳定
- SSE 第一版还在调整阶段
- 全量消息存储会快速增加复杂度

### 当前建议只做：

- 页面内消息流
- session 摘要延续
- report summary 延续

### 后面再考虑：

- 消息级历史存储
- 会话完整回放

---

## 十三、开发前后对比

### 开发前

- 模拟面试页还是假问题驱动
- AI 回复不是实时流式
- 没有消息队列控制
- 没有 Markdown 富文本输出

### 开发后

- 模拟面试页成为真实流式 AI 对话页
- SSE 可以实时返回问题与追问
- 高频消息渲染更平滑
- Markdown 输出更可读、更安全

---

## 十四、这一阶段特别注意的事项

### 1. 不要把 SSE 逻辑全塞进页面

这会直接让 `mock-interview.vue` 失控。

### 2. 队列和 UI 渲染要分开

不要把“消息缓存”和“组件展示”写成一坨。

### 3. Markdown 渲染必须考虑安全过滤

这一点不要后补，应该一开始就纳入方案。

### 4. 第一版流式先跑通单一主链路

也就是：

- 单模型
- 单页面
- 单轮完整体验

先别贪多。

---

## 十五、这一阶段完成后，下一步做什么

做完这一步之后，最自然的下一步就是：

1. 把流式消息与 session 历史进一步结合
2. 把 `practice.vue` 也接上轻量消息流
3. 再考虑消息级持久化与完整会话回看

## 十六、结论

这一阶段真正要做的不是“给项目加个接口”，而是：

**把 `mock-interview.vue` 从假数据训练页，升级成真实 SSE 流式 AI 面试舞台。**

这一步一旦完成，你这个项目的技术深度会明显往上走，后面无论是写简历、讲项目，还是继续扩展历史回放与完整报告，都会更有说服力。
