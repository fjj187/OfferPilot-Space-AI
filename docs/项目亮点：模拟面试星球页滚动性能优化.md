# 项目亮点：模拟面试星球页滚动性能优化

## 背景

模拟面试星球页原本首屏视觉效果比较重：页面同时存在星云背景、发光层、轨道光晕、多颗星球，以及每颗星球内部的大气层、光环、爆闪、波纹、阴影等多层 DOM 特效。切换场景时这些元素会配合 GSAP 做位移、透明度、模糊和层级变化。

实际体验中，页面在首屏上下滚动时会出现明显掉帧。问题不在下面内容区本身，而在首屏视觉层过重：`blur`、`backdrop-filter`、`mix-blend-mode`、多层 `box-shadow`、伪元素高光和大面积渐变叠在滚动区域里，导致浏览器频繁重绘和合成。

## 改了什么

核心改动集中在 `src/views/showcase/mock-interview-space.vue`。

1. 删除高成本背景视觉层

移除了首屏里的这些全局装饰层：

- `space-noise`
- `space-glow glow-a`
- `space-glow glow-b`
- `space-nebula`
- `orbit-glow`

这些层大多依赖大面积渐变、模糊、透明度和混合模式，滚动时合成成本较高。

2. 删除星球内部多层特效

原先每颗星球包含：

- `planet-flare`
- `planet-atmosphere`
- `planet-rings`
- `planet-wave`
- `planet-shadow`

现在每颗星球只保留一个 `.planet-shell` 元素，通过单层 CSS 背景表达星球质感：

```scss
.planet-shell {
  background:
    radial-gradient(circle at 34% 28%, rgb(255 255 255 / 0.42), transparent 0 18%),
    linear-gradient(128deg, transparent 0 42%, rgb(0 0 0 / 0.46) 74%, rgb(0 0 0 / 0.72) 100%),
    var(--planet-texture);
  box-shadow: inset -18px -18px 34px var(--planet-shadow);
}
```

3. 保留轻量切换动效

星球切换交互没有完全删除，仍然保留外层 `visual-layer` 的 GSAP 运动逻辑。但动画只作用在外层容器的：

- `transform`
- `opacity`
- `z-index`

移除了星球本体上的 `blur` 动画和多层子元素的透明度/滤镜变化。

4. 清理无用状态和样式

删除了与旧特效相关的变量、ref、动画和样式，包括：

- `planetShellRefs`
- 星球 blur 状态字段
- `planetVarsFor`
- `glowAStyle`
- `glowBStyle`
- `nebulaStyle`
- `planetTakeover` / `planetRingsTakeover` / `planetFlareTakeover` / `planetWaveTakeover`
- `nebulaSwirlA` / `nebulaSwirlB` / `nebulaCoreFlash`

## 为什么这么改

这次没有继续在旧方案上微调 `opacity` 或防抖时间，而是直接把星球从“实时 DOM 特效合成”改成“单层贴图式渲染”。

原因是瓶颈已经很明确：页面卡顿主要来自浏览器滚动时需要反复处理大面积 `blur`、混合模式、阴影和多个半透明图层。继续调滚动状态只能缓解，不能从根上减少渲染成本。

改成单层星球后，浏览器每帧需要处理的内容少很多：

- DOM 节点更少
- 样式规则更少
- 合成层更简单
- 滚动时不再触发星云、发光层、星球大气层、波纹等重绘
- 场景切换主要回到 GPU 友好的 `transform/opacity`

## 改完之后有什么变化

主观体验上，首屏上下滚动明显更顺，星球切换仍然存在，但视觉从“复杂特效星球”变成了“轻量贴图星球”。也就是说，这次优先保留了页面识别度和交互节奏，同时牺牲了一部分细碎的光效层次。

代码层面变化更直接：

- `mock-interview-space.vue` 变更规模：`12 insertions / 489 deletions`
- 删除了约 30 个首屏高成本视觉 DOM 节点：5 个全局装饰层 + 5 个星球各自 5 个内部特效层
- 每颗星球从 6 层 DOM 降为 1 层 DOM
- 星球切换不再驱动 `filter: blur(...)`
- 滚动状态下不再需要额外降级星云、发光层、轨道 glow 和星球内部特效

构建产物也有下降：

- `mock-interview-space.css`：约 `42.50 KB` 降到 `34.35 KB`
- CSS gzip：约 `7.82 KB` 降到 `6.54 KB`
- CSS 原始体积减少约 `8.15 KB`，下降约 `19.2%`
- CSS gzip 体积减少约 `1.28 KB`，下降约 `16.4%`
- 页面构建通过：`vite build` 成功

## 面试中可以怎么讲

可以用这段话概括：

在模拟面试星球页滚动掉帧问题中，我定位到多层 DOM 星球、星云发光层、`blur`、`mix-blend-mode` 与阴影叠加导致滚动重绘过重；随后将星球改为单层贴图式渲染，移除星云、发光、轨道 glow 及星球大气/光环/波纹等高成本特效，并保留 `transform/opacity` 驱动的切换动效。最终首屏滚动流畅度明显提升，页面 CSS 体积从约 `42.5KB` 降至 `34.4KB`，每颗星球从 6 层 DOM 降为 1 层 DOM，高成本视觉层减少约 80% 以上。

## 面试官可能追问

### 1. 你怎么判断是视觉层导致卡顿，而不是数据或业务逻辑？

可以回答：

我先排查滚动路径，发现滚动时并没有频繁请求接口，也没有大量列表渲染。真正集中在首屏的是多层视觉元素：大面积 `blur`、`backdrop-filter`、`mix-blend-mode`、多层阴影和伪元素。它们都处在滚动中的可视区域，浏览器需要频繁重绘和合成，所以我优先从渲染层面优化，而不是从业务数据入手。

### 2. 为什么不只是加防抖或节流？

可以回答：

防抖和节流只能减少 JS 状态更新频率，但不能减少浏览器每一帧要绘制的视觉内容。如果瓶颈在 paint/composite，单纯节流 JS 收益有限。所以我保留了 rAF 管理滚动状态的思路，但最终把核心优化放在减少 DOM 层级、减少滤镜和减少混合模式上。

### 3. 为什么不用 `will-change` 解决？

可以回答：

`will-change` 适合提示浏览器提前准备合成层，但它不是免费优化。这个页面的问题是层本身太多、太重，如果给大量元素加 `will-change`，可能会增加显存压力，甚至更卡。所以我没有把它当成主要方案，而是先减少需要合成的层。

### 4. 为什么不用 Canvas 或 WebGL？

可以回答：

Canvas/WebGL 能把复杂视觉集中到一个渲染上下文里，长期看可能更适合星球类视觉。但这次目标是快速降低滚动掉帧，同时不大改页面架构。单层 CSS 贴图改动范围更小，风险更低，也能保留现有 Vue/GSAP 的场景切换逻辑。如果后续继续追求更强视觉，可以再把星球渲染迁到 Canvas/WebGL。

### 5. 为什么不直接删除所有星球？

可以回答：

星球是这个页面的主视觉记忆点，直接删除会损失页面的产品感和展示效果。所以我选择保留星球的位置、颜色和切换节奏，但删掉内部高成本特效。这样是在视觉表达和性能之间取一个更平衡的点。

### 6. 为什么不用图片资源替代 CSS 星球？

可以回答：

图片资源也是可行方案，尤其适合更写实的星球。但当前每个场景的星球颜色来自配置主题，CSS 单层贴图可以继续复用现有主题变量，不需要额外维护多套图片资源，也避免新增加载体积。后续如果设计稿确定，可以把 CSS 贴图替换为压缩后的 WebP/AVIF。

### 7. 这次优化有没有代价？

可以回答：

有。代价是星球细节减少了，比如大气层、光环、波纹和爆闪效果被移除，视觉层次没有原来那么丰富。但换来的是滚动稳定性和可维护性提升。对于这个页面来说，用户首先需要顺滑地浏览和进入内容区，所以这是合理取舍。

## 为什么不用另一种方案

1. 不继续堆滚动降级规则

上一版尝试过滚动时临时降低 `blur`、`opacity`、`mix-blend-mode`。它能缓解性能，但会带来滚动时背景闪烁、视觉状态反复切换的问题。继续堆规则会让样式越来越复杂，也不够稳定。

2. 不只优化 JS

JS 滚动状态已经用 `requestAnimationFrame` 和短防抖控制过，内容区阈值也做了缓存。继续压 JS 的收益不如减少渲染层，因为瓶颈主要在浏览器绘制阶段。

3. 不立即引入 WebGL

WebGL 能做得更流畅，但改造成本更高，还需要处理贴图、灯光、尺寸适配、降级方案和交互同步。当前项目更适合先做低风险的结构瘦身。

4. 不用纯静态截图替代

纯图片最省渲染，但场景切换时会失去主题颜色联动和星球运动层次。单层 CSS 星球仍然能跟随每个场景的主题色变化，是更灵活的中间方案。

## 总结

这次优化的核心不是“把动画关掉”，而是把高成本视觉从多层实时合成改成轻量贴图式表达。最终保留了星球切换这个主视觉，同时显著减少了首屏滚动时的 DOM 层级、滤镜、阴影和混合模式压力，属于一次兼顾体验、性能和改造风险的前端渲染优化。
