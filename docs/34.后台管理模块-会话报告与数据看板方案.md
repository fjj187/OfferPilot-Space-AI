# 后台管理模块：会话报告与数据看板方案

## 一、目标

本方案用于在现有 `OfferPilot-Space-AI`（AI 模拟面试项目）中增加一个轻量后台管理模块，优先覆盖会话管理、报告管理、数据看板、登录态与角色区分。第一阶段以低侵入、可演示、可逐步接入真实后端为原则，不改动现有 `SSE`（服务端发送事件）流式对话契约，不影响普通用户模拟面试主链路。

核心目标：

- 管理员可以查看全部面试会话与面试报告，快速定位异常数据。
- 普通用户只能查看自己的会话与报告，不进入后台管理页。
- 后台首页提供轻量数据看板，用于展示面试次数、报告生成数、失败率、最近活跃等指标。
- 先复用现有接口与前端状态能力，后续再升级为真实鉴权和后端聚合接口。

## 二、第一阶段范围

### 1. 会话管理

会话管理页用于集中展示模拟面试会话列表。

建议字段：

- `sessionId`（会话标识）：用于定位一次模拟面试会话。
- `threadId`（对话线程标识）：用于定位一次连续问答链路。
- `scene`（面试场景）：如前端、后端、产品等。
- `status`（会话状态）：进行中、已结束、异常中断。
- `messageCount`（消息数量）：用于判断会话是否有效。
- `updatedAt`（更新时间）：用于排序和筛选。

核心能力：

- 按关键词搜索 `sessionId`（会话标识）或面试主题。
- 按状态、日期范围筛选。
- 查看会话详情，复用现有历史消息展示逻辑。
- 管理员可清理异常会话，普通用户不可执行高风险操作。

### 2. 报告管理

报告管理页用于展示已生成的面试报告。

建议字段：

- `reportId`（报告标识）：报告唯一标识。
- `sessionId`（会话标识）：关联面试会话。
- `score`（评分）：报告综合分。
- `summary`（摘要）：报告简短结论。
- `createdAt`（创建时间）：用于排序。
- `generateStatus`（生成状态）：已生成、生成中、生成失败。

核心能力：

- 按用户、会话、分数区间、生成状态筛选。
- 查看报告详情。
- 对失败报告提供重新生成入口。
- 支持导出为后续扩展项，第一阶段不强制实现。

### 3. 轻量数据看板

数据看板页用于给管理员展示运营和稳定性概览。

建议指标：

- 今日面试会话数。
- 累计面试会话数。
- 今日报告生成数。
- 报告生成成功率。
- 流式对话失败次数。
- 最近 7 天会话趋势。
- 最近 7 天报告生成趋势。
- 最近异常列表。

第一阶段可以先通过前端拉取现有会话列表和报告列表后计算派生指标；第二阶段再新增 `GET /api/admin/dashboard`（后台看板聚合接口），由后端统一聚合，减少前端多接口拼装。

### 4. 登录态与角色区分

第一阶段建议先做最小可用登录态：

- `AuthUser`（登录用户类型）：保存用户标识、昵称、角色。
- `role`（角色字段）：取值为 `user`（普通用户）或 `admin`（管理员）。
- `localStorage`（浏览器本地存储）：第一阶段保存演示登录态。
- `Pinia`（状态管理库）：维护当前登录用户和角色。
- `router.beforeEach`（路由前置守卫）：控制后台路由访问权限。

第二阶段再接入真实后端鉴权：

- `JWT`（JSON Web Token，令牌机制）：适合前后端分离快速落地。
- `httpOnly Cookie`（仅服务端可读 Cookie）：安全性更好，适合正式环境。
- `RBAC`（基于角色的访问控制）：支持普通用户、管理员和后续更多角色。

## 三、不做范围

第一阶段先不做以下内容，避免把小后台做成重后台系统：

- 不修改 `POST /api/interview/stream`（模拟面试流式接口）契约。
- 不新增复杂多租户体系。
- 不做细粒度按钮级权限配置中心。
- 不做完整用户注册、找回密码、短信验证码。
- 不先引入独立后台工程，仍放在当前前端项目内。
- 不把后台状态和模拟面试页面状态混在同一个 `store`（状态仓库）里。

## 四、路由设计

建议新增后台路由分组：

```ts
/admin/login
/admin
/admin/dashboard
/admin/sessions
/admin/reports
```

路由职责：

- `/admin/login`（后台登录页）：管理员登录入口，可与普通登录页分离。
- `/admin`（后台根路由）：默认重定向到 `/admin/dashboard`（后台数据看板页）。
- `/admin/dashboard`（后台数据看板页）：展示核心指标与趋势。
- `/admin/sessions`（会话管理页）：展示会话列表和详情入口。
- `/admin/reports`（报告管理页）：展示报告列表和详情入口。

建议路由元信息：

```ts
meta: {
  requiresAuth: true,
  requiredRole: 'admin',
}
```

其中 `requiresAuth`（是否需要登录）用于判断登录态，`requiredRole`（所需角色）用于判断是否有管理员权限。

## 五、页面与组件拆分

建议组件结构：

- `AdminLayout.vue`（后台布局组件）：负责侧边栏、顶部用户信息、退出登录。
- `AdminDashboard.vue`（后台数据看板组件）：负责看板容器和指标组装。
- `AdminMetricCard.vue`（后台指标卡片组件）：展示单个核心指标。
- `AdminTrendChart.vue`（后台趋势图组件）：展示最近 7 天趋势。
- `AdminSessionTable.vue`（后台会话表格组件）：展示会话列表、筛选和分页。
- `AdminReportTable.vue`（后台报告表格组件）：展示报告列表、筛选和分页。
- `AdminRoleGate.vue`（后台角色门禁组件）：用于局部权限控制。

状态边界：

- 页面组件负责筛选条件、分页状态、弹窗开关。
- `service`（服务层）负责请求后端接口。
- `Pinia`（状态管理库）只保存登录用户、角色和必要的全局后台状态。
- 统计指标优先用 `computed`（计算属性）从列表数据派生，避免重复存储。

## 六、接口复用与后端协作

第一阶段优先复用现有接口：

- `GET /api/interview/sessions`（面试会话列表接口）。
- `GET /api/interview/sessions/:sessionId/:threadId`（面试会话详情接口）。
- `GET /api/interview/reports`（面试报告列表接口）。
- `GET /api/interview/reports/:sessionId`（面试报告详情接口）。
- `POST /api/interview/reports/generate`（面试报告生成接口）。

前端服务层建议复用：

- `interview-session-api.ts`（面试会话接口服务）。
- `interview-report-api.ts`（面试报告接口服务）。
- `request.ts`（统一请求工具）。

第二阶段建议后端补充聚合接口：

```ts
GET /api/admin/dashboard
GET /api/admin/sessions
GET /api/admin/reports
```

聚合接口收益：

- 前端不用同时请求多个列表再计算指标。
- 管理员视角和普通用户视角可以在后端统一隔离。
- 后续接入真实数据库后，分页、筛选、统计都由后端完成。

## 七、权限模型

角色定义：

- `user`（普通用户）：只能查看自己的面试会话和报告。
- `admin`（管理员）：可以查看全部会话、全部报告和数据看板。

访问规则：

- 普通用户访问 `/admin/*`（后台路由）时，跳转到无权限页或普通首页。
- 未登录用户访问后台路由时，跳转到 `/admin/login`（后台登录页）。
- 管理员进入后台后，可以访问会话管理、报告管理和数据看板。
- 高风险操作需要二次确认，例如删除异常会话、重新生成报告。

第一阶段可前端控制展示和跳转；正式环境必须由后端再次校验角色，不能只依赖前端路由守卫。

## 八、数据看板指标口径

建议第一阶段先使用简单口径：

- 会话总数：会话列表总条数。
- 今日会话数：`updatedAt`（更新时间）落在当天的会话数。
- 报告总数：报告列表总条数。
- 今日报告数：`createdAt`（创建时间）落在当天的报告数。
- 报告成功率：已生成报告数 / 报告总数。
- 异常会话数：状态为异常中断或消息数量为 0 的会话数。

注意事项：

- 如果现有接口缺少 `status`（状态字段），第一阶段不要猜字段，可先用前端可确定字段做展示。
- 如果现有接口缺少用户标识，普通用户和管理员的数据隔离应等后端补齐。
- 数据看板只做辅助观察，不作为计费、审计等强一致数据来源。

## 九、实施步骤

### Phase 1：前端原型

目标：快速做出可演示后台。

- 新增后台路由和 `AdminLayout.vue`（后台布局组件）。
- 新增 `auth-store.ts`（登录态状态仓库），保存当前用户和角色。
- 新增后台登录页，支持普通用户和管理员演示账号。
- 新增会话管理页，复用会话列表接口。
- 新增报告管理页，复用报告列表接口。
- 新增数据看板页，先从列表数据派生统计指标。

### Phase 2：真实鉴权与聚合接口

目标：让后台具备真实业务边界。

- 后端新增登录接口和用户信息接口。
- 后端增加角色校验中间件。
- 新增 `GET /api/admin/dashboard`（后台看板聚合接口）。
- 会话和报告接口按角色返回不同数据范围。
- 前端从本地演示登录切换为真实登录态。

### Phase 3：运维增强

目标：让后台更适合长期维护。

- 增加操作日志。
- 增加异常会话标记。
- 增加报告重新生成记录。
- 增加导出能力。
- 扩展模型管理页，用于配置默认模型、启用状态和接口健康状态。

## 十、验证清单

功能验证：

- 未登录访问 `/admin/dashboard`（后台数据看板页）会跳转登录页。
- 普通用户登录后不能进入后台管理页。
- 管理员登录后可以进入后台管理页。
- 会话管理页可以加载、筛选、分页。
- 报告管理页可以加载、筛选、查看详情。
- 数据看板指标与列表数据一致。

异常验证：

- 会话列表接口失败时，页面显示错误提示和重试按钮。
- 报告列表接口失败时，不影响后台整体布局。
- 登录态过期时，路由能回到登录页。
- 刷新页面后，登录态能按第一阶段策略恢复。

边界验证：

- 不影响模拟面试流式对话。
- 不影响报告生成主流程。
- 不修改现有 `SSE`（服务端发送事件）协议。
- 不修改冻结的前后端接口契约。

## 十一、推荐优先级

建议优先顺序：

1. 后台路由、布局和角色守卫。
2. 会话管理页。
3. 报告管理页。
4. 轻量数据看板。
5. 真实鉴权和后端聚合接口。
6. 模型管理、操作日志和导出能力。

这样做的好处是先把后台的访问边界搭起来，再逐步挂业务数据，避免一开始就把看板、权限、接口聚合全部耦合在一起。
