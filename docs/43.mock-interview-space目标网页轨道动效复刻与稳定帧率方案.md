# 43.mock-interview-space 目标网页轨道动效复刻与稳定帧率方案

## 一、目标

本文档用于规划 `mock-interview-space`（宇宙模拟面试页）后续轨道动效升级，目标参考 `Thales Alenia Space`（泰雷兹阿莱尼亚宇航官网）首页的星球轨道切换效果。

本方案的核心目标有四个：

1. 轨道切换时，当前星球沿明确轨迹慢慢靠近、逐步变大、逐步变清晰。
2. 非当前星球向后退、变小、轻微模糊，形成稳定的景深关系。
3. 背景颜色切换使用渐变过渡，不再出现直接硬切。
4. 性能兜底优先级高于视觉复杂度，必须保证页面稳定丝滑，不因复刻目标网页而重新掉帧。

本方案只处理展示层动效，不修改 `POST /api/interview/stream`（模拟面试流式接口）、`SSE`（服务端发送事件）、路由协议、业务状态和后端代码。

## 二、目标网页实现结论

目标网页不是用简单 `CSS`（层叠样式表）写死星球位置，而是使用一套数据驱动的轨道系统。

目标网页的关键实现可以拆成三层：

| 层级 | 目标网页做法 | 对本项目的意义 |
| --- | --- | --- |
| 轨道层 | 使用 `SVG path`（矢量路径）定义椭圆轨道 | 本项目可增加一条轨道路径作为位置基准 |
| 轨道点层 | 使用 `GSAP`（动画库）和 `MotionPathPlugin`（路径动画插件）把按钮放到路径进度点 | 本项目可让轨道按钮沿真实路径移动 |
| 星球视觉层 | 星球本体用绝对定位，按槽位切换 `x`（横坐标）、`y`（纵坐标）、`scale`（缩放）、`blur`（模糊）和 `zIndex`（层级） | 本项目应保留星球独立视觉槽位，不强行让星球完全贴着路径跑 |

目标网页的关键点是：轨道按钮走 `SVG path`（矢量路径），星球本体走视觉槽位。两者通过同一套当前项索引同步，所以视觉上像星球沿轨道旋转，但实现上更可控。

## 三、目标网页轨迹机制

目标网页使用固定路径进度点控制轨道位置：

| 位置 | 路径进度 | 视觉含义 |
| --- | --- | --- |
| 左侧远点 | `0.2`（路径进度 20%） | 远处候选轨道 |
| 左中点 | `0.34`（路径进度 34%） | 即将进入主体区 |
| 中心点 | `0.5`（路径进度 50%） | 当前主体轨道 |
| 右中点 | `0.66`（路径进度 66%） | 刚离开主体区 |
| 右侧远点 | `0.8`（路径进度 80%） | 远处候选轨道 |

目标网页的移动不是一次性从起点跳到终点，而是按相邻段逐段推进。例如从第一个轨道切到第三个轨道时，不是直接 `0.2 -> 0.5`，而是先走 `0.2 -> 0.34`，再走 `0.34 -> 0.5`。中间段使用近似线性的节奏，最后落点使用 `Power3.easeOut`（三次方缓出曲线），所以停靠时更顺。

本项目如果要复刻，应避免只写一个目标位置并直接 `GSAP.to`（动画补间）到终点。更稳的方式是建立 `segmentMap`（分段映射表），让每次点击都被拆成 1 到 4 个相邻段过渡。

## 四、目标网页星球放置机制

目标网页的星球位置不是直接取 `SVG path`（矢量路径）上的坐标，而是用函数计算每个槽位的视觉参数。

每个槽位至少包含以下字段：

```ts
interface VisualSlotProfile {
  x: number
  y: number
  scale: number
  blur: number
  opacity: number
  zIndex: number
  brightness: number
}
```

字段含义：

| 字段 | 作用 |
| --- | --- |
| `x`（横坐标） | 控制星球在舞台内的水平位置 |
| `y`（纵坐标） | 控制星球在舞台内的垂直位置 |
| `scale`（缩放） | 控制远近和主体星球大小 |
| `blur`（模糊） | 控制景深虚化 |
| `opacity`（透明度） | 控制绕回或边缘星球显隐 |
| `zIndex`（层级） | 控制遮挡关系 |
| `brightness`（亮度） | 控制主体清晰度和远处暗化 |

本项目当前已经有 `SpaceVisualStage.vue`（星球舞台组件）负责星球展示和 `GSAP`（动画库）动效，因此不需要重写整页。推荐做法是把现有分散的位置逻辑收敛成 `visualSlotProfiles`（视觉槽位配置），让每个星球只是在槽位之间迁移。

## 五、本项目推荐架构

本项目应保持现有职责边界：

| 文件 | 保持或调整后的职责 |
| --- | --- |
| `mock-interview-space.vue`（宇宙模拟面试页主文件） | 只负责页面装配、当前场景、背景组件和内容面板，不下沉轨道动画细节 |
| `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件） | 继续负责 `WebGL`（网页图形渲染技术）背景渐变、星点和性能模式 |
| `SpaceVisualStage.vue`（星球舞台组件） | 负责星球层、景深槽位、轨道切换动画和大爆炸入场动效 |
| `SpaceOrbitNav.vue`（轨道导航组件） | 负责轨道按钮、点击入口和可访问交互 |
| `useMockInterviewSpaceOrbit.ts`（宇宙页轨道组合函数） | 负责当前轨道顺序、方向、自动播放和切换意图 |
| `mockInterviewSpaceScenes.ts`（宇宙页场景配置文件） | 负责每个轨道的主题色、文案、星球参数和背景映射 |

推荐落点是：视觉运动逻辑放在 `SpaceVisualStage.vue`（星球舞台组件），轨道状态仍由 `useMockInterviewSpaceOrbit.ts`（宇宙页轨道组合函数）提供，背景渐变仍由 `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件）承接。

## 六、实现步骤

### 6.1 第一步：建立轨道进度常量

新增或收敛一组轨道进度常量：

```ts
const orbitProgressSlots = [0.2, 0.34, 0.5, 0.66, 0.8] as const
```

其中 `0.5`（路径进度 50%）必须作为当前主体轨道。后续无论用户点击哪个轨道，最终都应该让目标轨道落到中心主体槽位。

### 6.2 第二步：建立视觉槽位配置

在 `SpaceVisualStage.vue`（星球舞台组件）中建立视觉槽位：

```ts
const visualSlotProfiles = [
  { x: -260, y: 120, scale: 0.62, blur: 8, opacity: 0.42, zIndex: 1, brightness: 0.7 },
  { x: -120, y: 80, scale: 0.78, blur: 4, opacity: 0.72, zIndex: 2, brightness: 0.86 },
  { x: 0, y: 30, scale: 1.18, blur: 0, opacity: 1, zIndex: 5, brightness: 1.08 },
  { x: 180, y: -10, scale: 0.72, blur: 5, opacity: 0.68, zIndex: 2, brightness: 0.82 },
  { x: 360, y: -80, scale: 0.56, blur: 10, opacity: 0.36, zIndex: 1, brightness: 0.66 },
] as const
```

实际数值需要按当前舞台宽度做响应式换算，不能直接照搬固定像素。建议封装 `resolveVisualSlotProfiles`（解析视觉槽位配置函数），根据 `stageWidth`（舞台宽度）和 `stageHeight`（舞台高度）返回最终槽位。

### 6.3 第三步：主体星球必须逐帧变大

当前用户反馈的核心问题是“到位置一瞬间才变大”。因此实现时必须保证 `scale`（缩放）进入同一个 `GSAP timeline`（动画时间线），和 `x`（横坐标）、`y`（纵坐标）、`blur`（模糊）同时开始、同时缓动。

错误做法：

```ts
// 错误：先移动，完成后才设置缩放，会出现瞬间变大。
gsap.to(layer, { x, y, onComplete: () => gsap.set(layer, { scale }) })
```

正确做法：

```ts
gsap.to(layer, {
  x,
  y,
  scale,
  filter: `blur(${blur}px) brightness(${brightness})`,
  opacity,
  duration: 0.84,
  ease: 'power3.out',
})
```

关键结论：主体星球的变大、变清晰、靠近，必须是同一段连续动画的一部分。

### 6.4 第四步：增加真实轨道路径

如果要更接近目标网页，可以在 `SpaceVisualStage.vue`（星球舞台组件）或 `SpaceOrbitNav.vue`（轨道导航组件）中增加一条 `SVG path`（矢量路径）：

```vue
<svg class="orbit-path-svg" viewBox="0 0 1440 800" aria-hidden="true">
  <path
    id="space-orbit-path"
    d="M0,516.5c0,0,250.1,155.5,719,155.5c470.9,0,721-155.4,721-155.4"
  />
</svg>
```

轨道按钮使用 `MotionPathPlugin`（路径动画插件）定位：

```ts
gsap.set(node, {
  motionPath: {
    path: '#space-orbit-path',
    align: '#space-orbit-path',
    alignOrigin: [0.5, 0.5],
    autoRotate: false,
    start: progress,
    end: progress,
  },
})
```

注意：这条路径主要服务轨道按钮和视觉参考，不建议强制让 `Three.js`（三维渲染库）星球或 `DOM`（文档对象模型）星球完全沿路径点运动。星球仍应由视觉槽位控制，这样更容易控制主体位置、景深和移动端适配。

### 6.5 第五步：背景渐变过渡

背景继续由 `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件）处理，不再回退到大面积 `CSS gradient`（CSS 渐变）逐帧插值。

每个轨道使用目标网页真实渐变方向：

| 本项目场景 | 参考目标网页模块 | 背景渐变 |
| --- | --- | --- |
| `overview`（总览场景） | `Explore`（探索） | `#B85800 -> #890019 -> #380000` |
| `library`（资料库场景） | `Observe & protect`（观测与保护） | `#1c96cb -> #184E8D -> #063974` |
| `mock`（模拟面试场景） | `Connect`（连接） | `#0d3485 -> #012457 -> #02305a` |
| `feedback`（专项刷题场景） | `Secure & defend`（安全防御） | `#015883 -> #016F7E -> #016F7E` |
| `report`（报告场景） | `Travel & navigate`（旅行与导航） | `#004F4F -> #007C7C -> #054F54` |

过渡策略：

1. 使用 `paletteTransitionMs`（调色板过渡毫秒数）控制 800ms 到 1000ms 的渐变。
2. 使用 `smoothstep`（平滑步进插值）做颜色混合，避免线性过渡机械感。
3. 背景过渡期间临时提高 `requestAnimationFrame`（浏览器逐帧回调）绘制频率。
4. 过渡结束后回到低频渲染或静态渲染，避免空闲时持续耗电和掉帧。

## 七、性能兜底原则

本方案的底线是：视觉效果可以降级，帧率不能失控。

### 7.1 不允许回退到高成本背景

禁止重新引入以下实现：

1. 大面积多层 `CSS gradient`（CSS 渐变）动画。
2. 全屏 `filter: blur()`（滤镜模糊）。
3. 全屏 `backdrop-filter`（背景滤镜）。
4. 滚动时持续变化的高频星点闪烁。
5. 每次轨道切换都触发布局测量和复杂重排。

背景必须优先走 `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件）的 `WebGL`（网页图形渲染技术）路径。

### 7.2 星球移动只动画合成属性

轨道切换期间，星球层只允许优先动画这些属性：

1. `transform`（变换）
2. `opacity`（透明度）
3. 小范围 `filter`（滤镜）

其中 `filter: blur()`（滤镜模糊）只能用在少数星球层，且数值必须克制。推荐远处星球 `blur`（模糊）控制在 `4px` 到 `10px`，主体星球必须为 `0px`。

不建议在切换过程中动画：

1. `width`（宽度）
2. `height`（高度）
3. `top`（顶部定位）
4. `left`（左侧定位）
5. `box-shadow`（盒阴影）大范围半径
6. 复杂渐变背景

星球变大必须用 `scale`（缩放），不能用 `width/height`（宽高）变化。

### 7.3 帧率监控兜底

建议新增轻量 `FrameBudgetGuard`（帧预算守卫）逻辑，用于在视觉层自我降级。

推荐规则：

| 监测结果 | 处理方式 |
| --- | --- |
| 连续 8 帧超过 24ms | 进入 `performanceMode`（性能模式） |
| 连续 20 帧恢复到 18ms 以下 | 退出 `performanceMode`（性能模式） |
| 页面不可见 | 暂停背景绘制和星球自转 |
| 用户开启 `prefers-reduced-motion`（减少动态效果偏好） | 默认进入低动态模式 |

`performanceMode`（性能模式）下应执行：

1. 背景星点数量减少。
2. 背景绘制频率降低。
3. 非主体星球暂停细节动画。
4. 非主体星球 `blur`（模糊）降低或取消。
5. `Three.js`（三维渲染库）星球 `pixelRatio`（像素比）降到 `1`。
6. 禁止多段拖拽实时高频重绘，只保留点击切换。

### 7.4 低端设备兜底

对于低端设备或 `WebGL`（网页图形渲染技术）不可用场景，需要提供降级路径。

降级顺序：

1. 保留 `DOM`（文档对象模型）星球外壳和静态渐变。
2. 关闭非主体星球自转。
3. 关闭背景星云动态，只保留静态底色。
4. 关闭 `blur`（模糊），改用 `opacity`（透明度）和 `scale`（缩放）表达远近。
5. 切换动画时长保留，但使用更少属性，避免突然硬切。

降级后的体验要求是：可以少一点高级质感，但不能卡顿、不能闪屏、不能出现星球瞬移。

## 八、动画时长与节奏建议

结合用户反馈“现在太快”，建议动效节奏如下：

| 动画 | 建议时长 | 缓动 |
| --- | --- | --- |
| 相邻轨道切换 | `840ms`（840 毫秒） | `power3.out`（三次方缓出） |
| 跨两个轨道切换 | `1260ms`（1260 毫秒） | 中间段 `none`（线性），最后段 `power3.out`（三次方缓出） |
| 背景渐变 | `900ms`（900 毫秒） | `cubic-bezier(.215,.61,.355,1)`（贝塞尔缓动）或 `smoothstep`（平滑步进） |
| 主体星球清晰化 | 与位移动画同长 | 同步缓动 |
| 非主体星球后退模糊 | 与位移动画同长 | 同步缓动 |

必须避免“移动结束后才变大”和“背景先切完星球后动”的割裂感。背景、星球、轨道按钮应从同一帧开始进入过渡。

## 九、实现细节建议

### 9.1 统一切换时间线

在 `SpaceVisualStage.vue`（星球舞台组件）中建议使用统一 `timeline`（动画时间线）：

```ts
const timeline = gsap.timeline({
  defaults: {
    duration: transitionDuration,
    ease: 'power3.out',
    overwrite: 'auto',
  },
})
```

每个星球层加入同一条 `timeline`（动画时间线）：

```ts
timeline.to(
  layer,
  {
    x: slot.x,
    y: slot.y,
    scale: slot.scale,
    opacity: slot.opacity,
    filter: `blur(${slot.blur}px) brightness(${slot.brightness})`,
  },
  0,
)
```

这样可以保证所有星球同时进入景深变化，不会出现某颗星球先变、某颗星球后跳。

### 9.2 主体星球识别规则

主体星球必须由 `activeSceneId`（当前场景标识）决定，不能由视觉层级最高或右上角位置决定。

规则：

1. 当前用户点击的轨道对应 `activeSceneId`（当前场景标识）。
2. `activeSceneId`（当前场景标识）对应的星球被分配到中心主体槽位。
3. 中心主体槽位拥有最大 `scale`（缩放）、最小 `blur`（模糊）、最高 `zIndex`（层级）和最高 `brightness`（亮度）。
4. 其它星球按环形顺序分配到左右远近槽位。

这可以避免再次出现“右上角星球成了主体”的问题。

### 9.3 图片星球立体感

如果当前主体星球仍使用图片贴图，需要补充局部光照和阴影层，而不是只把图片平铺到圆形容器里。

建议叠加三层：

1. `radial-gradient`（径向渐变）高光层：放在左上或右上，模拟主光源。
2. 内侧暗部层：放在背光侧，模拟球体明暗面。
3. 外侧辉光层：用较小范围 `box-shadow`（盒阴影）或伪元素模拟大气边缘光。

这些效果应保持静态或只跟随主体状态轻微变化，不能做高频动画。

## 十、验收标准

### 10.1 视觉验收

必须满足：

1. 点击任意轨道后，目标星球从原位置连续移动到主体槽位。
2. 目标星球在移动过程中逐步变大，不允许到终点瞬间变大。
3. 目标星球在移动过程中逐步变清晰，不允许先糊后突然清晰。
4. 非主体星球同步后退、变小、轻微模糊。
5. 当前主体必须是用户点击的那颗星球，不能被右上角或其它层级误抢。
6. 背景切换必须有渐变过渡，不能硬切。
7. 星点不能过大，不能像方块，滚动时不能闪烁。

### 10.2 性能验收

必须满足：

1. 桌面端连续点击轨道 10 次，不出现明显卡顿。
2. 移动端连续点击轨道 5 次，不出现明显掉帧和黑屏。
3. 滚动页面时，背景不闪、不重新大面积绘制。
4. 页面不可见后，`requestAnimationFrame`（浏览器逐帧回调）应暂停或降频。
5. 开发者工具中不应出现持续 `WebGL context lost`（网页图形上下文丢失）。
6. 构建命令 `npm run build`（执行生产构建验证）必须通过。

### 10.3 代码验收

必须满足：

1. 不修改后端接口。
2. 不修改 `SSE`（服务端发送事件）契约。
3. 不把轨道视觉状态写入持久化存储。
4. 不把展示层参数散落到多个组件互相同步。
5. `SpaceVisualStage.vue`（星球舞台组件）负责视觉槽位，`SpaceGpuBackdrop.vue`（GPU 宇宙背景组件）负责背景，职责不能混乱。

## 十一、风险与兜底

| 风险 | 影响 | 兜底 |
| --- | --- | --- |
| `filter: blur()`（滤镜模糊）过多 | 切换时掉帧 | 性能模式下降低或取消非主体模糊 |
| 多个 `Three.js`（三维渲染库）星球同时高频渲染 | 移动端发热、卡顿 | 非主体星球降频，必要时只保留主体星球真实渲染 |
| 背景过渡过重 | 点击轨道掉帧 | 过渡期高频，结束后静态；低端设备退回静态渐变 |
| 轨道点击过快 | 动画队列堆积 | 使用 `overwrite: 'auto'`（自动覆盖旧动画）并取消未完成 `timeline`（动画时间线） |
| 跨多轨道切换过长 | 用户感觉拖沓 | 最多压缩中间段时长，最后段保留缓出 |
| 响应式槽位不准 | 主体星球偏离目标位置 | 所有槽位通过舞台尺寸计算，不写死单一桌面尺寸 |

## 十二、推荐落地顺序

推荐按以下顺序执行：

1. 先把 `SpaceVisualStage.vue`（星球舞台组件）的星球槽位收敛成 `visualSlotProfiles`（视觉槽位配置）。
2. 修正主体星球识别，确保点击哪颗，哪颗进入中心主体槽位。
3. 把 `x/y/scale/filter/opacity`（横坐标 / 纵坐标 / 缩放 / 滤镜 / 透明度）放进同一条 `GSAP timeline`（动画时间线）。
4. 保留当前 `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件），继续优化背景过渡，不回退到 `CSS`（层叠样式表）大面积动画。
5. 再补 `SVG path`（矢量路径）和 `MotionPathPlugin`（路径动画插件）控制轨道按钮。
6. 最后加入 `FrameBudgetGuard`（帧预算守卫）和低端设备降级。

这个顺序可以先解决用户最敏感的主体错位、瞬间变大和掉帧问题，再逐步提高目标网页还原度。

## 十三、最终结论

本项目复刻目标网页的轨道动效是可行的，而且适合渐进式实现。

最稳的技术路线是：

1. 背景继续用 `SpaceGpuBackdrop.vue`（GPU 宇宙背景组件）承接，保证不掉帧。
2. 星球移动使用 `GSAP`（动画库）统一驱动 `transform`（变换）和少量 `filter`（滤镜）。
3. 轨迹参考目标网页的 `SVG path`（矢量路径）和固定进度点。
4. 星球位置使用独立 `visualSlotProfiles`（视觉槽位配置），不要完全依赖路径坐标。
5. 主体星球的靠近、变大、变清晰必须同步缓动。
6. 一旦监测到帧预算超限，立即降级背景频率、非主体模糊和三维细节。

因此，本方案的核心不是单纯“做得更炫”，而是先建立可控的轨道槽位系统，再把目标网页的顺滑切换和景深关系迁移过来。这样既能接近目标网页观感，也能守住稳定丝滑、不掉帧的底线。
