# 29. mock-interview-space 专项刷题真实题组第三阶段规划

## 文档目的

`27` 阶段完成真实流式 + 最小会话闭环；`28` 阶段完成报告后端化（宇宙页 **E1**）。  
`29` 阶段不再做联调收口，也不再扩后端平台能力，而是补齐 **文档 22 / 23 第二轮余量**：

> **让 `practicePlan`（专项训练计划）不再只驱动展示卡片，而是驱动一套可解释、可恢复、按序推进的真实题组，并在 `mock`（模拟面试场景）里真正影响下一题。**

本文用于明确方向、边界与推荐开发顺序，避免与 `28` 报告后端化或旧 `workbench` 专项页混做。

---

## 一、`29` 阶段的一句话目标

**报告弱项 → 生成专项题组 → 专项刷题确认 → 模拟面试按题组逐题训练。**

对用户可见的变化：

1. 从报告点「按弱项继续补练」，不只是卡片文案变了，**第一题真的来自报告推导的题组**。
2. 专项刷题场景能预览「本轮将练哪几题 / 练什么维度」，而不是只有题型、题数表单。
3. 模拟面试里「下一题」在专项模式下 **按题组顺序推进**，而不是从主题题库随机轮换。
4. 刷新或切场景回来后，**题组进度可恢复**（练到第几题、还剩几题）。

---

## 二、为什么现在该做专项真实题组

### 2.1 文档 22 / 23 已承诺但未闭环的能力

`docs/23.mock-interview-space功能层第一阶段接入开发文档.md` 第二轮要求：

1. 结构化弱项维度（`structure` / `case_detail` / `result_metric` / `principle_depth`）。
2. 报告续练计划与模拟面试展示一致。
3. 续练能解释「为什么练这个」。

当前 **类型字段与卡片展示已具备**，但 **题目选择仍偏粗**：

| 能力 | 当前状态 | 缺口 |
|------|----------|------|
| `practicePlan` 写入上下文 | ✅ | — |
| 专项卡片展示 plan | ✅ | — |
| 按 `questionType` / `difficulty` 筛题 | ⚠️ 正则匹配标题 | 无稳定题组，易重复、易漏 |
| 按 `focusArea` 选题 | ❌ | 只参与报告文案，不参与选题 |
| 按 `weaknessTag` 选题 | ❌ | 只拼进 prompt 上下文 |
| `questionCount` 题量 | ⚠️ 对 `topicQuestions` 取模循环 | 同一题可能重复出现 |
| 题组进度持久化 | ❌ | 刷新后不知练到第几题 |
| 专项刷题场景预览题组 | ❌ | 只有配置表单 |

### 2.2 与 `28` 的衔接关系

`28` 已在报告对象中沉淀 `practicePlan`（含 `focusArea`、`weaknessTag`、`zone`、`questionType`、`difficulty`、`questionCount`）。  
`29` **不重复做报告生成**，而是 **消费已有 `practicePlan` / `PersistedReportSummary`，把它编译成题组并驱动 mock 选题**。

## 三、`29` 阶段的边界

### 3.1 应该做的事

1. 建立 **题组（Question Group）** 最小数据模型与生成规则。
2. 从 `practicePlan` + 报告弱项信号 **编译出有序题目列表**（题组）。
3. 专项刷题场景 **预览题组**（题目标题、维度、顺序），用户确认后开始。
4. `mock` 场景在 `mockEntryMode === 'practice'` 时 **严格按题组顺序** 生成 `questionThreads`。
5. 题组与当前进度写入持久化（`PersistedWorkbenchContext` 或并列字段），支持刷新恢复。
6. 扩展 `questionBank`（题库）元数据，使选题可解释（见第五节）。

### 3.2 不应该急着做的事

1. 独立题库管理后台、大规模导入。
2. 后端题组 API / 数据库存储（本阶段 **前端题组编译 + 本地持久化** 即可）。
3. 旧 `workbench/practice.vue` 改造（宇宙页边界不变，见 `docs/18.mock-interview-space后续开发边界规范.md`）。
4. LLM 实时出题（题组来自现有 `questionBank` + 规则，不调模型生成新题）。
5. 多维打分、RAG、账号体系。

## 四、当前实现基线（开发前必读）

### 4.1 已有链路

```text
report.practicePlan
  → handleReportContinuePractice / handlePracticeStart
  → startMockRound({ mockEntryMode: 'practice', practicePlan, mockSessionConfig })
  → useMockInterviewSpaceMockState.topicQuestions（按 type/difficulty 过滤）
  → buildConfiguredQuestionThreads（取模循环 sourceQuestions）
  → mock 场景展示 + 流式问答
```

### 4.2 关键文件

| 文件 | 当前职责 |
|------|----------|
| `src/types/workbench.ts` | `PersistedPracticePlan` 类型 |
| `src/components/showcase/mock-interview-space/scenes/SpacePracticeScene.vue` | 专项配置 UI，可读 report 预填 |
| `src/views/showcase/mock-interview-space.vue` | `handlePracticeStart` 启动专项 |
| `src/composables/showcase/useMockInterviewSpaceMockState.ts` | `topicQuestions` 筛选 + `buildConfiguredQuestionThreads` |
| `src/views/workbench/mock-interview.data.ts` | `questionBank` 静态题库 |

### 4.3 已知问题（必须在 29 解决）

1. **`focusArea` 不参与选题**，与 23 第二轮「定向补练」目标不一致。
2. **题组=取模循环**，`questionCount=10` 时题库不足会重复同一题。
3. **无题组预览**，用户不知道「开始模拟面试」后会遇到什么题。
4. **无题组进度**，`generatedThreadCount` 与专项题组语义混在一起。
5. **`questionBank` 缺结构化维度字段**，只能靠标题正则猜题型。

## 五、推荐数据模型（最小冻结）

### 5.1 扩展题库条目 `InterviewQuestion`

在 `mock-interview.data.ts` 的 `InterviewQuestion` 上增加可选字段（不破坏现有题）：

```ts
interface InterviewQuestion {
  // ...现有字段
  /** 与 PersistedPracticeQuestionType 对齐 */
  questionType?: 'concept' | 'code' | 'scenario'
  /** 可命中多个弱项维度 */
  focusAreas?: Array<'structure' | 'case_detail' | 'result_metric' | 'principle_depth'>
  /** 与 PersistedPracticeZone 对齐，便于 zone 筛选 */
  practiceZones?: Array<'vue' | 'javascript' | 'typescript' | 'engineering' | 'performance'>
  /** 用于 weaknessTag 模糊匹配的 keywords */
  weaknessKeywords?: string[]
}
```

旧题逐步补元数据；未标注题走现有 regex 兜底，避免一次性改完。

### 5.2 新增题组对象 `PersistedPracticeQuestionGroup`

```ts
interface PersistedPracticeQuestionGroupItem {
  questionId: string
  order: number
  /** 展示用，来自题库 snapshot */
  title: string
  prompt: string
  focusArea?: PersistedPracticeFocusArea
  matchReason: 'focus_area' | 'weakness_tag' | 'type_zone' | 'fallback'
}

interface PersistedPracticeQuestionGroup {
  id: string
  /** 来源报告 sessionId，便于追溯 */
  sourceSessionId?: string
  /** 生成时的 plan 快照 */
  planSnapshot: PersistedPracticePlan
  items: PersistedPracticeQuestionGroupItem[]
  /** 当前做到第几题（0-based index，下一题 = currentIndex） */
  currentIndex: number
  status: 'pending' | 'in_progress' | 'completed'
  createdAt: string
  updatedAt: string
}
```

### 5.3 上下文挂载方式

推荐写入 `PersistedWorkbenchContext`：

```ts
interface PersistedWorkbenchContext {
  // ...现有字段
  practicePlan?: PersistedPracticePlan | null
  /** 29 新增：当前激活的专项题组 */
  practiceQuestionGroup?: PersistedPracticeQuestionGroup | null
}
```

**单一事实来源：**

- `practicePlan` = 用户意图 / 报告建议（配置层）
- `practiceQuestionGroup` = 已编译、可执行的题目列表（执行层）

开始训练时：**plan → 编译 group → 持久化 group → mock 只读 group**。

## 六、题组编译规则（B1：规则选题，不调 LLM）

新增 `src/services/practice/practice-question-group-builder.ts`（服务层，命名可微调）：

### 6.1 输入

- `PersistedPracticePlan`
- 可选 `PersistedReportSummary`（弱项标签、`weaknessFocusAreas`）
- `questionBank` + 当前 `activeTopic`

### 6.2 输出

- `PersistedPracticeQuestionGroup`，`items.length === plan.questionCount`（不足时允许 `< count`，但 UI 必须提示「题库不足，已用 N 题」）

### 6.3 推荐优先级（从高到低）

1. **topic 匹配**：`question.topic === plan.zone 映射后的 topicKey`
2. **focusArea 匹配**：题目的 `focusAreas` 含 `plan.focusArea`
3. **questionType 匹配**：`question.questionType === plan.questionType`
4. **difficulty 匹配**：`question.difficulty === plan.difficulty`
5. **weaknessTag / keywords 匹配**：`weaknessKeywords` 或标题 / `weaknessSignal` 与 `plan.weaknessTag` 有交集
6. **fallback**：同 topic 任意题，按 difficulty 接近度排序

### 6.4 去重与顺序

1. 同一 `questionId` 在一个题组内 **只出现一次**。
2. 不足 `questionCount` 时 **不取模循环**；返回实际条数并在专项场景提示。
3. 排序：先 focusArea 命中 → 再 weakness 命中 → 再 difficulty → 稳定 id 排序（保证可复现）。

### 6.4 与报告 `practicePlan` 的关系

- 报告生成时（前端 `buildReportSummary` / 后端 `report-service`）**继续只写 `practicePlan`**，不在 29 改报告契约。
- 题组 **在用户点击「开始训练 / 继续补练」时编译**，避免报告页加载时就绑死题目。

## 七、前端改造方向

### 7.1 专项刷题场景 `SpacePracticeScene.vue`

**从「配置表单」升级为「配置 + 题组预览」：**

1. 读取 `reportSummary.practicePlan` 预填（保持现有 watch hydrate）。
2. 用户改配置后，**实时调用 builder 预览** 将练题目（标题 + 维度标签 + 顺序）。
3. 「开始模拟面试」emit 时带上 **已编译的 `practiceQuestionGroup`**（或 emit plan 由父层编译，二选一，推荐父层统一编译）。

展示示例：

```text
本轮专项题组 · 共 5 题
1. [结构表达] 说说你会怎么设计一个可复用的组合式函数
2. [案例细节] 结合项目讲讲一次性能优化经历
...
```

### 7.2 主文件 `mock-interview-space.vue`

1. `handlePracticeStart` / `handleReportContinuePractice`：
   - `buildPracticeQuestionGroup(plan, reportSummary?)`
   - 写入 `saveWorkbenchContext({ practicePlan, practiceQuestionGroup })`
   - 再 `startMockRound(...)`
2. 报告直达补练与专项场景手动开始 **共用同一编译入口**。

### 7.3 模拟面试状态 `useMockInterviewSpaceMockState.ts`

**专项模式 (`mockEntryMode === 'practice'`) 下：**

1. **禁止** `buildConfiguredQuestionThreads` 的取模循环逻辑作为主编排。
2. 改为：`practiceQuestionGroup.items` → 顺序生成 `questionThreads`（一题一线程或沿用现有 thread 模型，但 **prompt / title 来自 group item**）。
3. `currentIndex` / `generatedThreadCount` 与题组进度对齐：
   - 提交一题 → `currentIndex += 1`
   - 「下一题」→ 打开 `items[currentIndex]`
4. 题组 `completed` 时：可选跳转 report 或展示「本轮专项完成」轻提示。

**非专项模式：** 保持现有 `buildQuestionThreads` + followUp 逻辑不变。

### 7.4 模拟面试场景 `SpaceMockScene.vue`

1. 专项卡片增加 **题组进度**：`第 2 / 5 题 · 结构表达`。
2. 展示 **当前题命中原因**（`matchReason` 文案映射），让用户感知「这题为什么出现」。

## 八、推荐开发顺序

### 第 1 步：类型与题库元数据

1. 扩展 `InterviewQuestion` 与 `PersistedPracticeQuestionGroup` 类型。
2. 给 `questionBank` 中 **至少每个 topic 2～3 题** 补 `questionType` / `focusAreas` / `practiceZones`。
3. 编写 builder 单元逻辑（纯函数，便于单测或 dev 脚本验证）。

### 第 2 步：题组编译服务

1. 实现 `practice-question-group-builder.ts`。
2. 在 dev 下用固定 `practicePlan` 打印题组，确认无重复、顺序稳定。

### 第 3 步：持久化与启动链路

1. `PersistedWorkbenchContext` 增加 `practiceQuestionGroup`。
2. `handlePracticeStart` / `handleReportContinuePractice` 编译并保存题组。
3. 刷新恢复：读取 context 中的 group + `currentIndex`。

### 第 4 步：mock 选题改造

1. 专项模式 thread 生成改读题组。
2. 下一题 / 进度与 `currentIndex` 同步。
3. 结束本轮时题组状态置 `completed`（或清空 group，见待拍板项）。

### 第 5 步：专项场景 UI

1. 题组预览列表。
2. 题库不足提示。
3. 与报告预填联动。

### 第 6 步：验收

1. 按第十节清单手测。
2. 在本方案文档或对应任务说明中补充必要验收结论。

## 九、待拍板项（实施前确认）

### A. 题组不足时的策略

| 选项 | 说明 |
|------|------|
| **A1（推荐）** | 不足则练满现有题为止，UI 明示「仅找到 N 题」 |
| A2 | 允许重复题凑满 `questionCount`（与现行为接近，不推荐） |

### B. 题组完成后

| 选项 | 说明 |
|------|------|
| **B1（推荐）** | 标记 `completed`，保留 group 供报告/回顾，新开专项再编译新 group |
| B2 | 完成后清空 `practiceQuestionGroup` |

### C. 用户改专项配置后

| 选项 | 说明 |
|------|------|
| **C1（推荐）** | 重新编译题组，重置 `currentIndex = 0` |
| C2 | 仅改 plan 不自动改 group，需用户点「重新生成题组」 |

### D. followUp 追问在专项模式

| 选项 | 说明 |
|------|------|
| **D1（推荐）** | 专项模式 **关闭多 followUp 链**，一题一线程，题组顺序即主链路 |
| D2 | 保留每题 followUp，题组只定 primary（复杂度高，29 不做） |

## 十、阶段成果与验收标准

### 10.1 对用户可见

1. 报告 → 继续补练 → 模拟面试 **第一题与报告弱项一致且可解释**。
2. 专项刷题场景能看到 **本轮题组预览**。
3. 「下一题」按题组顺序推进，**不重复同一题凑数**（A1）。
4. 刷新页面后 **题组进度仍在**。

### 10.2 对工程侧可见

1. `practicePlan`（配置）与 `practiceQuestionGroup`（执行）职责分离。
2. 选题逻辑集中在 builder + mock state，**不堆进 `SpacePracticeScene` 展示组件**。
3. `mock-interview-space.vue` 仍只做装配与场景切换。
4. `pnpm run build` 通过；中文文件无乱码。

### 10.3 最小手测清单

1. 完成一轮 mock → 报告有 `practicePlan` → 点「按弱项继续补练」→ mock 第一题标题与专项预览一致。
2. 从专项场景改难度/题数 → 预览题组变化 → 开始训练 → mock 题量一致。
3. 答 1 题后刷新 → 仍在第 2 题（或正确恢复进度）。
4. 非专项模式（资料库直达 mock）行为与改前一致，无回归。
5. 题组练完 → 状态为 `completed`（B1）或符合所选策略。

## 十一、风险与约束

1. **题库体量小**：`questionBank` 目前题量有限，29 必须做「不足提示」，不能假装总有 15 题。
2. **元数据迁移**：旧题无 `focusAreas` 时靠 regex 兜底，预览里应标注「规则兜底」。
3. **不要破坏 28 报告契约**：题组是 29 前端消费层，不要求后端新接口。
4. **不要改 workbench 专项页**：宇宙页闭环优先。

---

## 十二、最终结论

`28` 解决「报告进后端」；`29` 解决 **「报告建议如何变成真实可练的题目序列」**。

下一阶段最应该做的是：

**把 `practicePlan` 编译成 `practiceQuestionGroup`，让专项刷题与模拟面试共享同一套可恢复、可预览、按序推进的题组执行层。**

第九节 **A～D 已冻结（2026-05-24，全按推荐）**：A1 不足则练满并提示；B1 完成后保留 group；C1 改配置即重编译；D1 专项一题一线程。
