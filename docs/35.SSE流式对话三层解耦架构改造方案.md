# SSE流式对话三层解耦架构改造方案

## 一、结论

当前 `SSE`（服务端发送事件）流式对话已经完成了“底层传输”和“页面状态”的初步分离，但还不是完整的三层解耦架构。更准确地说，当前是“两层半”：

```text
fetch-stream-transport.ts（fetch 流传输适配文件）
  -> interview-stream.ts（面试流门面文件，职责偏重）
    -> useInterviewStream.ts（面试流状态组合函数）
```

目标不是重写主链路，而是在不修改 `POST /api/interview/stream`（面试流式接口）契约的前提下，把中间层收敛为高内聚的 `StreamController`（流控制器）/ `ConnectionManager`（连接管理器），让传输、连接控制、页面状态各自只处理自己的职责。

## 二、当前现状

### 1. 已完成的部分

| 能力 | 当前文件 | 判断 |
|---|---|---|
| 底层传输适配 | `frontend/src/services/sse/fetch-stream-transport.ts`（fetch 流传输适配文件） | 已拆出 |
| `SSE Frame`（SSE 数据帧）解析 | `frontend/src/services/sse/sse-client.ts`（SSE 客户端工具文件） | 已拆出 |
| 多事件路由 | `frontend/src/services/sse/interview-stream.ts`（面试流门面文件） | 基本具备 |
| 页面消息状态 | `frontend/src/composables/interview/useInterviewStream.ts`（面试流状态组合函数） | 已存在 |
| 本地消息持久化 | `frontend/src/services/storage/interview-message-store.ts`（面试消息存储文件） | 已接入 |

### 2. 主要缺口

`interview-stream.ts`（面试流门面文件）当前同时承担了：

- `prompt`（提示词）归一与压缩。
- `mock`（本地模拟）回复生成。
- `remote/mock`（远端/本地模拟）模式分流。
- `AbortController`（中止控制器）创建与中止。
- `chunk/done/error/activity`（分片/完成/错误/活动）事件映射。
- 传输错误归一。

这说明“传输控制处理层”还没有独立成稳定中间层。`useInterviewStream.ts`（面试流状态组合函数）里也仍然保存了连接生命周期细节，例如 `activeAbort`（当前中止句柄）、`retryableState`（可重试状态）、`lastStreamActivityAt`（最后活动时间）、`startStreamHealthCheck`（启动流健康检查函数）。

## 三、目标架构

目标三层如下：

```text
第一层：Transport Layer（传输层）
  fetch-stream-transport.ts（fetch 流传输适配文件）
  sse-client.ts（SSE 客户端工具文件）

第二层：Stream Control Layer（流控制层）
  interview-stream-controller.ts（面试流控制器文件）
  stream-connection-manager.ts（流连接管理器文件）
  stream-event-router.ts（流事件路由文件）
  stream-heartbeat.ts（流心跳工具文件，可选）

第三层：Application State Layer（应用状态层）
  useInterviewStream.ts（面试流状态组合函数）
  interview-message-store.ts（面试消息存储文件）
  页面 / 场景组件
```

### 第一层：Transport Layer（传输层）

职责只保留“把请求发出去，把字节流读回来，把基础错误分类”。

建议保留：

- `fetch`（网络请求接口）调用。
- `ReadableStream`（可读流）读取。
- `AbortSignal`（中止信号）透传。
- `HTTP`（超文本传输协议）错误、网络错误、空响应体、主动取消分类。
- `SSE Frame`（SSE 数据帧）文本解析。

禁止放入：

- 面试业务字段判断。
- `prompt`（提示词）组装。
- 页面消息状态更新。
- 重试策略。
- 业务级心跳提示文案。

### 第二层：Stream Control Layer（流控制层）

职责是连接生命周期、事件路由、心跳感知和重试策略，不直接管理页面消息数组。

建议拆分：

```text
frontend/src/services/sse/
  interview-stream.ts                 # 兼容旧入口，只做门面导出
  interview-stream-controller.ts      # 统一启动、停止、完成、失败
  stream-connection-manager.ts        # 管理 AbortController、连接状态、活动时间
  stream-event-router.ts              # 映射 chunk/done/error/activity 事件
  stream-heartbeat.ts                 # 心跳与超时判定工具，先做被动检测
  interview-stream-payload.ts         # prompt 归一与请求 payload 组装
  interview-stream-mock.ts            # 本地 mock 回复生成
```

其中：

- `interview-stream-controller.ts`（面试流控制器文件）负责统一对外暴露 `startInterviewStream`（启动面试流函数）和 `abort`（中止函数）。
- `stream-connection-manager.ts`（流连接管理器文件）负责 `AbortController`（中止控制器）、`lastActivityAt`（最后活动时间）、`status`（连接状态）。
- `stream-event-router.ts`（流事件路由文件）负责把 `SSE Frame`（SSE 数据帧）映射成 `InterviewStreamEvent`（面试流事件）。
- `interview-stream-payload.ts`（面试流请求组装文件）负责 `prompt`（提示词）归一。
- `interview-stream-mock.ts`（面试流本地模拟文件）负责本地降级回复，不污染连接控制逻辑。

### 第三层：Application State Layer（应用状态层）

职责是页面状态、消息落盘、滚动联动、错误展示和用户操作入口。

`useInterviewStream.ts`（面试流状态组合函数）应保留：

- `messages`（消息列表）。
- `isStreaming`（是否流式生成中）。
- `streamError`（流错误提示）。
- `streamConnectionHint`（连接提示）。
- `retryStream`（重试流函数）的用户入口。
- `stopStream`（停止流函数）的用户入口。
- `InterviewMessageQueue`（面试消息队列）的渲染节流。
- `interview-message-store.ts`（面试消息存储文件）读写。

应下沉到第二层：

- `activeAbort`（当前中止句柄）。
- 原始连接活动时间记录。
- 健康检查计时器。
- 可重试元信息的底层判定。

## 四、心跳管理边界

当前只有 `activity`（连接活动事件）与 15 秒无活动提示，这属于“被动超时感知”，还不是完整心跳。

### 第一阶段可做

不改后端契约，只做前端被动检测：

- 收到 `start/chunk/done/error`（开始/分片/完成/错误）时刷新 `lastActivityAt`（最后活动时间）。
- 超过阈值后产生 `inactive`（不活跃）状态。
- 页面只提示“不活跃”，不自动重连。

### 第二阶段需三方对齐

如果要做真正的应用层 `heartbeat`（心跳），需要先与 `Node`（后端运行时）和 `C++`（后端语言服务）确认：

- 是否新增 `heartbeat`（心跳）事件类型。
- 心跳间隔和超时阈值。
- 断线后是否支持 `checkpoint`（断点快照接口）。
- 是否支持 `idempotentKey`（幂等键），避免自动重连重复生成。

未完成这些能力前，不建议默认自动重连 `POST /api/interview/stream`（面试流式接口），否则可能触发重复回答。

## 五、建议实施顺序

### Phase 1：纯拆分，不改行为

目标：让代码职责清晰，但保持现有功能表现。

1. 新增 `interview-stream-payload.ts`（面试流请求组装文件），迁移 `normalizeInterviewStreamRequest`（归一化面试流请求函数）和提示词常量。
2. 新增 `interview-stream-mock.ts`（面试流本地模拟文件），迁移 `buildMockInterviewReply`（构建本地模拟回复函数）和 `splitTextToReader`（文本转流读取器函数）。
3. 新增 `stream-event-router.ts`（流事件路由文件），迁移 `chunk/done/error`（分片/完成/错误）映射逻辑。
4. 保留 `interview-stream.ts`（面试流门面文件）对外导出的 `startInterviewStream`（启动面试流函数），避免上层调用点大改。

### Phase 2：连接管理器下沉

目标：把连接生命周期从 `useInterviewStream.ts`（面试流状态组合函数）抽到中间层。

1. 新增 `stream-connection-manager.ts`（流连接管理器文件）。
2. 由连接管理器持有 `AbortController`（中止控制器）和活动时间。
3. 对外暴露 `abort`（中止函数）、`getSnapshot`（获取连接快照函数）等最小接口。
4. `useInterviewStream.ts`（面试流状态组合函数）只消费连接事件，不直接维护底层连接对象。

### Phase 3：被动心跳标准化

目标：把现在散落在上层的健康检查收敛为统一策略。

1. 新增 `stream-heartbeat.ts`（流心跳工具文件）。
2. 定义 `inactiveTimeoutMs`（不活跃超时时间）默认 15000。
3. 输出 `active/inactive/closed/error`（活跃/不活跃/关闭/错误）连接状态。
4. 页面只根据状态展示提示，不在这一阶段自动重连。

### Phase 4：真实心跳与自动恢复

目标：在后端支持后，升级完整可靠性。

前置条件：

- 后端支持 `heartbeat`（心跳）事件。
- 后端支持 `idempotentKey`（幂等键）或等价去重能力。
- 后端支持 `checkpoint`（断点快照接口）或可恢复消息快照。

完成后才能考虑：

- 心跳超时自动中止。
- 自动重连。
- 断点恢复。
- 自动补齐缺失消息。

## 六、不做范围

本次三层解耦改造方案不包含：

- 不修改 `POST /api/interview/stream`（面试流式接口）路径。
- 不修改 `chunk/done/error`（分片/完成/错误）事件契约。
- 不新增前端直连 `C++`（后端语言服务）的路径。
- 不把主链路切到 `EventSource`（浏览器原生 SSE 对象）。
- 不把 `mock`（本地模拟）行为删除。
- 不在没有后端幂等能力前默认自动重连。

## 七、验收标准

| 验收项 | 通过标准 |
|---|---|
| 行为兼容 | 原有模拟面试流式回复、停止、错误提示、重试入口表现不变 |
| 文件职责 | `interview-stream.ts`（面试流门面文件）不再直接承载大段提示词、模拟回复和事件路由细节 |
| 传输纯度 | `fetch-stream-transport.ts`（fetch 流传输适配文件）不出现面试业务判断 |
| 状态边界 | `useInterviewStream.ts`（面试流状态组合函数）不直接持有原始 `AbortController`（中止控制器） |
| 心跳边界 | 第一阶段只提示连接不活跃，不自动重连 |
| 契约安全 | 不改 `SSE`（服务端发送事件）事件名和请求/响应核心字段 |

## 八、推荐最终文件关系

```text
useInterviewStream.ts（面试流状态组合函数）
  -> startInterviewStream（启动面试流函数）
    -> InterviewStreamController（面试流控制器）
      -> StreamConnectionManager（流连接管理器）
      -> StreamEventRouter（流事件路由器）
      -> createFetchStreamReader（创建 fetch 流读取器函数）
        -> fetch-stream-transport.ts（fetch 流传输适配文件）
      -> consumeSSEStream（消费 SSE 流函数）
        -> sse-client.ts（SSE 客户端工具文件）

interview-stream-payload.ts（面试流请求组装文件）
interview-stream-mock.ts（面试流本地模拟文件）
```

最终目标是：页面层只关心“我要开始/停止/重试”和“收到什么事件”，中间层只关心“连接如何活着、如何结束、如何路由事件”，传输层只关心“网络流如何读取”。这样才算完整的 `SSE`（服务端发送事件）流式对话三层解耦架构。
