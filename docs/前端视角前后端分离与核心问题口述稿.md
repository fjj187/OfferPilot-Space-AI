# 前端视角前后端分离与核心问题口述稿

## 一、项目如何利用 Vibe Coding（AI 辅助开发方式）做好前后端分离

这个项目里的 Vibe Coding（AI 辅助开发方式）不是让 AI（人工智能）随意堆功能，而是在明确边界、明确契约、明确文档的基础上持续迭代。前端视角最核心的一点是：浏览器只认 `/api/*`（统一接口入口），不直接访问 `C++`（后端引擎服务），所有真实能力都先经过 `Node`（后端网关）收口。

整体数据流可以这样理解：

```text
Vue 3（前端框架） + Vite（构建工具）
  -> /api/*（统一接口入口）
  -> Node（后端网关）
  -> mock（本地模拟） / remote LLM（远程大模型） / C++（后端引擎服务）
```

前端主要负责 UI（用户界面）、交互体验、场景状态、流式内容渲染和用户侧模型选择；`Node`（后端网关）负责接口、鉴权、会话报告、模型配置、数据落盘和能力转发；`C++`（后端引擎服务）负责未来更高性能的生成、解析、检索等能力。这样即使底层从 `mock`（本地模拟）切到 `remote LLM`（远程大模型）或 `C++`（后端引擎服务），前端仍然只消费稳定的接口契约。

这个分离方式的价值是：前端可以持续优化体验，后端可以替换实现，双方不会因为字段、路径、事件格式随意变化而互相阻塞。

## 二、业务理解如何体现

这个项目不是简单的聊天页面，而是一个“AI（人工智能）模拟面试训练系统”。它的业务链路包括资料导入、题目生成、模拟面试、流式反馈、历史恢复、报告生成、弱项补练、后台管理和模型配置。

从前端视角看，业务复杂度主要体现在这些点：

| 业务点 | 体现方式 |
| --- | --- |
| 面试训练不是单轮问答 | 使用 `sessionId`（会话标识）、`threadId`（题线程标识）管理多轮训练 |
| 用户可能刷新或从外部直达 | 使用 `scene`（场景参数）、`reportSessionId`（报告会话标识）、`mockSessionId`（模拟会话标识）恢复上下文 |
| 训练过程有学习策略 | 使用 `unknownAnswerStreak`（连续不知道次数）、`forceRevealReferenceAnswer`（强制揭示参考答案标记）处理答不上来的场景 |
| 报告不是静态展示 | 使用 `weaknessTags`（弱项标签）、`practicePlan`（专项计划）驱动后续补练 |
| 系统需要运营能力 | 后台管理支持会话、报告、用户角色和模型配置 |

所以这个项目真正的业务亮点不是“能聊天”，而是把一次 AI（人工智能）问答变成了可恢复、可分析、可补练、可管理的训练闭环。

## 三、技术复杂度如何体现

技术复杂度主要体现在五个方向：

1. `SSE`（服务端发送事件）流式对话被拆成传输层、控制层、处理层，避免页面组件直接处理底层流。
2. `request.ts`（统一请求工具）在 `Axios`（HTTP 请求库）基础上加入 `retry`（重试）、`TokenBucket`（令牌桶）和 `circuit breaker`（熔断器）。
3. `route guard`（路由守卫）处理登录态、权限、动态路由、跳转参数恢复和 401（未授权）风暴。
4. `mock-interview-space.vue`（宇宙模拟面试页）把总览、资料库、模拟面试、专项练习、报告拆成场景组件。
5. `modelId`（模型标识）以可选字段方式接入主链路，不改路径、不改 `SSE`（服务端发送事件）事件结构。

这些点说明项目已经从“页面功能实现”进入“前端业务架构设计”阶段。

## 四、前端后续发展建议

### 1. 继续强化流式交互能力

重点理解 `SSE`（服务端发送事件）为什么需要分层：底层负责读流，中层负责控制，中上层负责把事件转成 UI（用户界面）状态。面试时要能讲清楚取消、错误、恢复、断点和消息收尾。

### 2. 继续强化状态建模能力

不要只靠 `if`（条件分支）写页面。像模拟面试这种场景，应该先区分未开始、进行中、历史预览、报告生成、恢复中、已结束等业务状态，再决定状态放在组件、`composable`（组合函数）、`Pinia`（状态管理库）还是后端。

### 3. 继续强化接口契约意识

前端不要随意猜字段，也不要为了页面方便单方修改 `DTO`（数据传输对象）。涉及 `POST /api/interview/stream`（模拟面试流式接口）、`SSE`（服务端发送事件）事件、报告字段、题池字段时，要先看类型定义和协作文档。

### 4. 继续强化可解释能力

Vibe Coding（AI 辅助开发方式）项目最怕“代码很多但讲不清”。建议把每个亮点都沉淀成“问题背景 -> 技术方案 -> 为什么这样做 -> 带来的收益 -> 风险边界”的表达结构。

## 五、五个高频问题回答思路与口述版

### 问题 1：`SSE`（服务端发送事件）三层解耦如何降低组件复杂度？

回答思路：

- 先说问题：流式对话如果写在组件里，会把请求、解析、错误、取消、消息更新混在一起。
- 再说分层：传输层负责读流，控制层负责启动和中止，事件层负责把后端事件转成前端可消费事件。
- 最后说收益：组件只关心“展示什么”和“用户点了什么”，不关心底层流协议。

标准口述版：

> 我们没有把 `SSE`（服务端发送事件）逻辑直接写进页面组件，而是拆成三层。底层 `sse-client.ts`（流式客户端文件）只负责读取和解析流数据；中间的 `interview-stream-controller.ts`（面试流控制器文件）负责启动、取消、错误兜底和模式切换；上层的 `stream-event-router.ts`（流事件路由文件）负责把 `chunk/done/error`（分片 / 完成 / 错误）、`heartbeat`（心跳事件）和 `checkpoint`（断点快照事件）转成前端统一事件。项目里已经有 `stream-heartbeat.ts`（流心跳工具文件）、`interview-stream-error.ts`（流错误文案文件）和 `interview-stream-checkpoint-recovery.ts`（断点恢复服务文件），说明心跳检测、错误归一化和断点快照恢复已经被收口到独立模块。这样 `useInterviewStream.ts`（面试流组合函数）和页面组件就不用关心底层协议，只需要处理消息状态和 UI（用户界面）更新。后续如果继续增强自动续流或更细的恢复策略，也可以沿着现有模块扩展，不会污染页面逻辑。

### 问题 2：`request.ts`（统一请求工具）如何提升请求稳定性？

回答思路：

- 先说普通请求层的问题：多个页面并发、网络抖动、接口短时失败、连续重试可能造成雪崩。
- 再说方案：统一 `Axios`（HTTP 请求库）实例，加 `retry`（重试）、`TokenBucket`（令牌桶）、`circuit breaker`（熔断器）。
- 最后说边界：只覆盖非流式 `REST API`（表述性状态转移接口），不混进 `SSE`（服务端发送事件）通道。

标准口述版：

> `request.ts`（统一请求工具）主要解决非流式接口的稳定性问题。项目里很多后台页、报告页、会话页会同时发请求，如果全部直接 `fetch`（浏览器网络请求 API）或散落写 `Axios`（HTTP 请求库），就很难统一处理超时、重试和错误提示。所以我把请求统一收口到 `request.ts`（统一请求工具），里面加了三个能力：第一是 `retry`（重试），只对适合重试的读取类请求做有限重试；第二是 `TokenBucket`（令牌桶），限制瞬时并发，避免页面同时加载时请求洪峰；第三是 `circuit breaker`（熔断器），当同一路径持续失败时短时间拦截后续请求，避免无意义重试。这样能提升用户体验，也能保护后端。

### 问题 3：`route guard`（路由守卫）如何解决登录态和跳转循环？

回答思路：

- 先说问题：本地有旧登录态但 `token`（令牌）失效时，容易反复请求 `/api/auth/me`（当前登录态接口）并重定向。
- 再说方案：公共路由、游客路由、鉴权路由分开；401（未授权）后清理本地状态；增加短冷却和单次导航结果。
- 最后说收益：避免白屏、重复跳转、参数丢失。

标准口述版：

> 这个项目的 `route guard`（路由守卫）不是只判断有没有 `token`（令牌），而是把路由分成公共路由、游客路由和需要鉴权的路由。以前容易出现的问题是：本地残留登录态，但后端 `/api/auth/me`（当前登录态接口）返回 401（未授权），前端又反复探测和重定向，最后形成跳转循环。现在的处理是：没有有效 `token`（令牌）就不发无意义探测；`/api/auth/me`（当前登录态接口）失败后立即清理本地状态，并进入短冷却，避免 401（未授权）风暴；同时通过 `route-guard-utils.ts`（路由守卫工具文件）、`dynamic-route-manager.ts`（动态路由管理文件）和 `route-param-recovery.ts`（路由参数恢复文件）把动态路由、跳转参数恢复、重定向结果收口。这样登录态异常时不会白屏，也不会无限跳转。

### 问题 4：`mock-interview-space.vue`（宇宙模拟面试页）如何把复杂业务拆成场景组件？

回答思路：

- 先说问题：宇宙页包含总览、资料库、模拟面试、专项练习、报告，全部写一个文件会失控。
- 再说拆法：主页面只做场景编排，具体内容交给 `SpaceOverviewScene.vue`（宇宙总览场景组件）、`SpaceMockScene.vue`（模拟面试场景组件）等。
- 最后说状态归属：业务状态放 `composable`（组合函数），展示组件通过 `props`（组件入参）和 `emit`（事件抛出）交互。

标准口述版：

> `mock-interview-space.vue`（宇宙模拟面试页）本身承载很多业务：总览、资料库、模拟面试、专项练习和报告。如果都写在一个页面里，组件会同时负责数据、状态、交互、动画和接口调用，后期很难维护。所以项目把它拆成场景组件，比如 `SpaceOverviewScene.vue`（宇宙总览场景组件）负责总览入口，`SpaceLibraryScene.vue`（资料库场景组件）负责资料场景，`SpaceMockScene.vue`（模拟面试场景组件）负责问题、回答和流式反馈，`SpacePracticeScene.vue`（专项练习场景组件）负责弱项补练，`SpaceReportScene.vue`（报告场景组件）负责报告展示。主页面只负责当前 `scene`（场景参数）的选择、路由参数同步和场景编排；真正的业务状态放在 `useMockInterviewSpaceMockState.ts`（宇宙模拟面试状态组合函数）等 `composable`（组合函数）里。这样场景之间边界清楚，后续加功能不会把主页面越写越大。

### 问题 5：`modelId`（模型标识）如何在不破坏原接口的前提下接入模型选择？

回答思路：

- 先说约束：不能改 `POST /api/interview/stream`（模拟面试流式接口）路径，也不能改 `chunk/done/error`（分片 / 完成 / 错误）事件结构。
- 再说方案：前端只拿启用模型列表，只保存 `modelId`（模型标识），不暴露 `apiKey`（接口密钥）和 `baseUrl`（接口基础地址）。
- 最后说兼容：`modelId`（模型标识）是可选字段，后端按“指定模型 -> 默认模型 -> 环境变量”解析。

标准口述版：

> 接入 `modelId`（模型标识）时，项目没有改 `POST /api/interview/stream`（模拟面试流式接口）的路径，也没有改 `SSE`（服务端发送事件）的 `chunk/done/error`（分片 / 完成 / 错误）结构，而是采用最小兼容方案。前端通过 `GET /api/models/enabled`（启用模型列表接口）只拿到 `modelId`（模型标识）、`displayName`（展示名称）、`provider`（模型供应商）和 `isDefault`（是否默认），不会拿到 `apiKey`（接口密钥）或 `baseUrl`（接口基础地址）。用户选择模型后，前端只把可选的 `modelId`（模型标识）透传到流式请求、报告生成和题池生成链路。后端通过 `resolveActiveModelConfig`（解析当前可用模型配置函数）按“请求指定模型 -> 后台默认模型 -> 任一启用模型 -> 环境变量配置”的优先级解析。这样既支持模型选择，又保持旧链路兼容。

## 六、面试表达模板

如果面试官问“你这个项目里前后端分离做得怎么样”，可以这样回答：

> 我这个项目不是简单分了 `frontend`（前端目录）和 `backend`（后端目录），而是用接口契约做边界。前端只请求 `/api/*`（统一接口入口），通过 `Vite proxy`（开发代理）转到 `Node`（后端网关），不会直接连接 `C++`（后端引擎服务）或模型服务。像 `POST /api/interview/stream`（模拟面试流式接口）和 `SSE`（服务端发送事件）的 `chunk/done/error`（分片 / 完成 / 错误）结构都是冻结契约，前端只消费稳定格式。这样后端可以在 `mock`（本地模拟）、`remote LLM`（远程大模型）和 `C++`（后端引擎服务）之间切换，前端不需要大改。我的前端重点是做好流式渲染、状态恢复、场景拆分、错误处理和用户体验。

如果面试官问“这个项目最能体现你业务理解的地方是什么”，可以这样回答：

> 我觉得最能体现业务理解的是它不是一次性的 AI（人工智能）问答，而是完整的面试训练闭环。用户可以从资料库进入面试，面试中有多题线程和流式反馈，结束后生成报告，报告再驱动弱项补练，后台还能管理会话、报告和模型配置。前端这里要处理的不只是页面展示，还包括 `sessionId`（会话标识）、`threadId`（题线程标识）、`scene`（场景参数）、历史恢复、报告直达、模型选择这些业务状态，所以我会先做状态建模，再拆组件和接口调用。

如果面试官问“你后面还会怎么优化”，可以这样回答：

> 后续我会继续往三个方向优化。第一是把流式恢复做得更完整，比如真正支持断点续流和更清晰的恢复提示；第二是继续收敛复杂状态，把场景状态、业务状态和临时 UI（用户界面）状态分得更清楚；第三是补测试和可观测性，比如给 `SSE`（服务端发送事件）事件解析、`request.ts`（统一请求工具）重试熔断、`route guard`（路由守卫）跳转分支加更细的验证。这样项目就不只是功能能跑，而是更接近真实工程质量。
