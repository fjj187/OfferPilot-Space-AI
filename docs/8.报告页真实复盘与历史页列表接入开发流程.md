# 8.报告页真实复盘与历史页列表接入开发流程

## 一、为什么这一阶段先做报告页和历史页

在完成了：

- 工作台多页面结构
- 资料页上下文承接
- 模拟面试页第一版交互
- 本地持久化第一阶段

之后，当前项目已经具备了一个关键前提：

**训练数据已经开始能够沉淀下来。**

这时候最自然的下一步，不是继续补更多输入页，而是把这些已经沉淀的数据真正展示出来。

也就是说，项目要从“能训练”进入“能复盘、能回看”的阶段。

这一阶段的目标就是：

1. 把 `report.vue` 从占位报告页升级成真实复盘页
2. 把 `history.vue` 从空壳页面升级成可查看 session 和 report summary 的历史列表页

---

## 二、这一阶段完成后的产品效果

### 1. 报告页完成后的用户体验

用户在模拟面试页结束一轮训练后：

- 自动跳转到报告页
- 报告页能够显示该轮训练的真实摘要
- 用户能看到：
  - 本轮主题
  - 来源资料
  - 模式
  - 已回答题数
  - 薄弱点标签
  - 本轮训练概述
  - 后续建议方向

这一版先不做复杂图表，也不做完整大报告导出，而是先把“真实复盘页”做出来。

### 2. 历史页完成后的用户体验

用户进入历史页后：

- 能看到历史训练 session 列表
- 能看到每轮训练的基础摘要
- 能看到是否已完成
- 能看到对应主题与来源
- 能看到关联的 report summary
- 能点击进入对应报告页继续查看

这一步完成后，项目才真正具备“训练闭环 + 回看闭环”。

---

## 三、这一阶段的开发目标

这一阶段先做成：

**前端真实数据承接版报告页 + 前端历史记录浏览版历史页**

也就是说，先解决：

- report 能读 session summary
- history 能列 session 和 report summary
- report 与 history 之间可以跳转

先不解决：

- 图表分析
- AI 自动总结长文报告
- PDF/Word 导出
- 高级筛选搜索
- 多维统计面板

---

## 四、报告页应该承接哪些真实数据

当前持久化层已经有两类关键数据：

- `PersistedInterviewSession`
- `PersistedReportSummary`

所以 `report.vue` 下一阶段应该基于它们渲染真实页面。

### 建议优先展示的字段

#### 1. 本轮基础信息

- 训练主题
- 训练模式
- 来源资料
- 当前文档类型
- 开始时间
- 结束时间
- 状态

#### 2. 本轮结果摘要

- 已回答题数
- 总题数
- 当前累计薄弱点数
- 是否存在跳题

#### 3. 薄弱点标签区

- 标签列表
- 标签数量
- 哪些标签最值得优先补练

#### 4. 复盘建议区

第一版可以先用规则生成的前端文案，例如：

- 如果薄弱点为空：说明本轮回答记录较少，建议继续完成更多题目
- 如果有多个薄弱点：展示“优先补练顺序”
- 如果 answeredCount < totalCount：提示用户本轮尚未完整完成

也就是说，这一版先做：

**基于 session / summary 的规则型复盘页**

后续再升级成：

**AI 生成型复盘页**

---

## 五、历史页应该展示哪些列表

历史页建议先拆成两个主区块。

### 1. 训练 session 列表

每一项建议展示：

- 主题
- 模式
- 来源
- 已回答题数 / 总题数
- 当前状态
- 开始时间
- 是否已完成

### 2. 报告 summary 列表

每一项建议展示：

- 主题
- 来源
- 薄弱点个数
- 已答题数
- 创建时间
- 跳转按钮

### 为什么要分两块

因为 session 和 report 不是完全同一种东西：

- session 更偏“训练过程记录”
- report 更偏“结果摘要记录”

后面如果你想继续扩展，也更好做：

- 历史训练列表
- 历史报告列表
- 按主题筛选
- 按状态筛选

## 九、这一阶段推荐开发顺序

### 第 1 步：先把 `report.vue` 升级成真实报告页

优先级更高，因为：

- `mock-interview.vue` 结束后第一跳就是报告页
- 用户感知最直接
- 现在 report 已经能读 `sessionId`

本步先做：

- 读取 session
- 读取 report summary
- 输出真实摘要卡片
- 渲染薄弱点与建议

### 第 2 步：再做 `history.vue` 的 session 列表

本步先做：

- 读取全部 session
- 按时间倒序展示
- 每项可跳转到 report

### 第 3 步：补 `history.vue` 的 report summary 列表

本步再做：

- 读取全部 report summary
- 按时间倒序展示
- 每项可跳转到具体报告页

### 第 4 步：补 report / history 之间跳转闭环

确保：

- 从 report 可以进入 history
- 从 history 可以回 report
- 从 history 进入 report 时能正确带 `sessionId`

## 十、这一阶段的数据读取关系

### 报告页读取关系

`report.vue` 建议按这个顺序读取：

1. 优先读 `sessionId`
2. 根据 `sessionId` 找对应 `report summary`
3. 再根据 `sessionId` 找对应 `session`
4. 合并成一份页面展示数据

也就是说，后面报告页最好不是“只看 summary”，而是：

**session + report summary 组合视图**

### 历史页读取关系

`history.vue` 建议：

1. 读取全部 sessions
2. 读取全部 report summaries
3. 各自独立展示
4. 通过 `sessionId` 建立跳转关联

---

## 十一、第一版规则型建议怎么做

报告页先不要急着接 AI 生成全文复盘。  
先做一版规则型建议，足够你演示，也更稳。

例如：

### 规则 1：已答题数不足

如果：

- `answeredCount < totalCount`

则展示：

- 建议先补完当前轮次，再看完整复盘

### 规则 2：薄弱点标签大于等于 3

则展示：

- 建议先去 `practice.vue` 做专项刷题

### 规则 3：来源是资料页导入

则展示：

- 建议回资料页继续补充相关主题资料

### 规则 4：模式是 guided

则展示：

- 说明当前为引导模式，后续可切换标准模式再练一轮

这样第一版报告页就已经是“真实复盘页”了，不只是静态介绍页。

## 十三、这一阶段特别注意的事项

### 1. 先不要把报告页做成超复杂分析大盘

第一版重点是：

- 真实承接
- 可读
- 可跳转
- 可解释

不是复杂图表。

### 2. 不要让 history 页承担太多筛选逻辑

第一版先做：

- 倒序列表
- 详情跳转

筛选、搜索、分组这些后面再说。

### 3. session 和 report summary 的职责不要混

- session：训练过程记录
- report summary：结果摘要

页面展示时可以组合，但存储上不要硬混成一种结构。

### 4. 页面文案和建议先规则化

这样更稳定，也更适合当前阶段。  
等你后面接好 SSE、消息流、完整训练数据，再升级 AI 生成复盘文案。

---

## 十四、这一阶段完成后，下一步最合理的方向

做完真实报告页和历史页之后，最自然的下一步就是：

### 方向一：开始做 SSE 流式主对话

因为到这一步：

- 资料承接有了
- 训练记录有了
- 报告和历史承接有了

这时候接 SSE，整条链路就闭环了。

### 方向二：增强报告页内容

比如：

- 更细的薄弱点分析
- 题目级复盘
- 知识点解析
- 导出能力

但更推荐的还是先上 SSE 主链路。

---

## 十五、结论

这一阶段的核心任务不是再补一个空页面，而是：

**把已经持久化下来的训练数据真正展示成“可复盘、可回看”的产品页面。**

所以接下来最合理的开发重点就是：

1. 把 `report.vue` 做成真实复盘页
2. 把 `history.vue` 接入 session 与 report summary 列表
3. 打通查看历史 -> 回到报告页的闭环

这一步完成后，项目会比现在更像一个完整产品，而不只是一个前端训练工作台。
